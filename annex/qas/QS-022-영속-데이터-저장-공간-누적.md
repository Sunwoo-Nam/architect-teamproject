# QS-022-영속-데이터-저장-공간-누적

## 개요

### Quality Scenario ID
QS-022

### 제목
영속 데이터 저장 공간 누적

### 설명
IntentDB, UserActivityDB, NegotiationSessionDB, AgentExecutionLogDB 등 누적성 데이터가 시간이 지남에 따라 Android 기기 저장 공간을 잠식한다. 데이터 보존 기간(TTL)과 정리 정책이 없으면 장기 운용 시 저장 공간 부족으로 플랫폼이 동작 불가해진다. 데이터 수명 관리 컴포넌트의 필요 여부가 아키텍처 결정에 영향을 미친다.

### 품질 속성
성능 — 자원 효율성

## 환경

### 시스템 상태
플랫폼이 30일 동안 정상 운용된 이후 영속 데이터베이스들의 누적 저장 공간을 측정하는 상태

### 초기 조건
- 플랫폼이 설치 직후 초기화된 상태에서 시작
- 각 DB(IntentDB, UserActivityDB, NegotiationSessionDB, AgreementDB, AgentExecutionLogDB, TaskDB)가 정상 운용 중
- 일일 평균 Intent 감지 건수, 협상 세션 수, Agent 실행 로그 수가 시뮬레이션 설정에 따라 정상 범위

### 부하 조건
30일 정상 운용 시뮬레이션: 일일 평균 Intent N건, 협상 M건, Agent 실행 K건 발생

### 관련 컴포넌트
- IntentDB: Intent 감지 기록 누적 저장
- UserActivityDB: 사용자 활동 기록 누적 저장
- NegotiationSessionDB: 협상 세션 히스토리 누적 저장
- AgreementDB: 합의 기록 누적 저장
- AgentExecutionLogDB: Agent 실행 로그 누적 저장
- TaskDB: Task 실행 기록 누적 저장

## 동작

1. **자극원**: 시간 경과 (30일 정상 운용)
2. **자극**: 플랫폼이 30일 동안 정상 운용되면서 각 DB에 데이터가 지속 누적
3. **반응 주체**: 각 DB 저장 레이어, TTL 정책 관리 컴포넌트(있는 경우)
4. **반응 흐름**:
   - 플랫폼이 매일 Intent 감지, 협상 세션, Agent 실행을 수행하며 각 DB에 레코드 추가
   - TTL 정책이 설정된 경우, 만료된 레코드가 주기적으로 삭제
   - TTL 정책이 없는 경우, 모든 레코드가 무한정 누적
   - 30일 경과 후 각 DB 및 전체 플랫폼의 저장 공간 사용량 측정
   - TTL 정책 적용 전후의 저장 공간 사용량 비교

## 측정

### 측정 항목
30일 정상 운용 후 플랫폼이 점유하는 총 저장 공간 및 TTL 정책 적용 전후 비교

### 측정 공식
```
S_total = sum(S_db_i) for i in [IntentDB, UserActivityDB, NegotiationSessionDB, AgreementDB, AgentExecutionLogDB, TaskDB] (MB)

S_db_i: 각 DB 파일의 크기 (MB)

측정 방법:
  - Android Debug Bridge: adb shell du -sh <app_data_path>/databases/
  - 측정 시점: 30일 시뮬레이션 완료 직후

TTL 정책 효과 측정:
  S_without_ttl: TTL 정책 없이 30일 운용 후 총 저장 공간 (MB)
  S_with_ttl: TTL 정책 적용 후 30일 운용 후 총 저장 공간 (MB)
  R_reduction = (S_without_ttl - S_with_ttl) / S_without_ttl × 100 (%)

DB별 성장률:
  G_db_i = S_db_i(day_30) / S_db_i(day_1) (배율)

일일 증가량:
  delta_S = (S_total(day_N) - S_total(day_N-1)) (MB/day)
```

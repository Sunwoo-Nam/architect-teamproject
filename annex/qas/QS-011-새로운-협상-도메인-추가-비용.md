# QS-011-새로운-협상-도메인-추가-비용

## 개요

### Quality Scenario ID
QS-011

### 제목
새로운 협상 도메인 추가 비용

### 설명
현재 지원하지 않는 새로운 협상 유형(예: 차량 공유, 가사 서비스)을 추가할 때 NegotiationController, ProposalGenerator 등 핵심 협상 컴포넌트를 수정하지 않고도 지원할 수 있어야 한다. 협상의 공통 흐름과 도메인별 조건이 명확히 분리된 설계가 요구된다.

### 품질 속성
변경 용이성 — 협상 도메인 추가

## 환경

### 시스템 상태
플랫폼이 기존 협상 도메인(예: 숙박, 교통, 식당)을 지원하는 상태에서 새로운 협상 도메인을 추가하는 변경 작업이 수행되는 상태

### 초기 조건
- 기존 협상 도메인들이 NegotiationController와 ProposalGenerator를 통해 정상 동작 중
- 추가할 협상 도메인은 기존 도메인과 다른 고유한 협상 조건 구조를 가짐
- 협상 도메인 명세(조건 항목, 합의 기준, 제안 생성 규칙)가 정의된 상태

### 부하 조건
없음 (개발/유지보수 변경 작업 시나리오)

### 관련 컴포넌트
- NegotiationController: 협상 공통 흐름을 담당 (수정 불필요 여부 확인 대상)
- ProposalGenerator: 협상 제안 생성을 담당 (수정 불필요 여부 확인 대상)
- ExternalServiceAPIAdapter: 새 도메인의 외부 서비스 연동 (신규 어댑터 추가 필요 여부 확인 대상)
- AgreementDB: 합의 데이터 구조 (스키마 확장 필요 여부 확인 대상)

## 동작

1. **자극원**: 개발팀 (새로운 협상 도메인 지원 요구)
2. **자극**: 새로운 협상 도메인(예: 차량 공유 협상)을 플랫폼에 추가하는 변경 작업 착수
3. **반응 주체**: 개발팀, 추가 대상 컴포넌트
4. **반응 흐름**:
   - 새로운 협상 도메인의 조건 구조와 제안 생성 규칙을 도메인 명세로 작성
   - 도메인 명세를 플랫폼에 등록 (NegotiationController/ProposalGenerator 수정 없이 플러그인 방식)
   - 새 도메인에 필요한 외부 서비스 연동이 있는 경우 ExternalServiceAPIAdapter에 신규 어댑터 추가
   - NegotiationController와 ProposalGenerator는 기존 코드 수정 없이 새 도메인을 처리
   - 기존 협상 도메인들의 동작이 영향받지 않음을 회귀 테스트로 확인

## 측정

### 측정 항목
새로운 협상 도메인 추가 시 NegotiationController, ProposalGenerator 수정 없이 완료 가능 여부 및 추가로 필요한 컴포넌트 수

### 측정 공식
```
가능_여부 = NegotiationController 및 ProposalGenerator 코드 수정 없이 새 도메인 동작 여부 (O/X)

C_new = count(새로운 도메인 지원을 위해 신규 추가가 필요한 컴포넌트 수)

측정 방법:
  - 변경 시나리오: 차량 공유 협상 도메인 추가
  - git diff로 변경된 파일 추적
  - NegotiationController, ProposalGenerator 패키지/모듈 내 파일 변경 여부 확인
  - 신규 생성 파일(새 도메인 명세, 새 어댑터 등) 수 집계

추가 측정:
  T_domain_add = 도메인 추가 작업 완료까지 소요 시간 (개발자-시간, hour)
```

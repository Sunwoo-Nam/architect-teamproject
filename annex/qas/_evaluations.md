# 품질 시나리오 평가

## 개요

### 목적
24개 품질 시나리오(QS-001~024)에 대해 중요도, 난이도, 비즈니스 영향도를 평가하여 quality-selector가 품질 요구사항을 선정하는 데 필요한 기반을 제공한다.

### 평가 기준

#### 중요도 (Importance) — 시장에서의 중요성
- **매우 높음**: 시장 진입을 위해 반드시 필요한 품질, 경쟁력에 직접적인 영향
- **높음**: 시장 경쟁력 확보에 중요한 품질
- **중간**: 표준적인 품질 수준, 경쟁력에 간접적인 영향
- **낮음**: 시장에서 요구하지 않거나 중요하지 않은 품질

#### 난이도 (Difficulty) — 구현 및 달성의 어려움
- **매우 높음**: 기술적으로 매우 복잡하거나 많은 리소스가 필요한 경우
- **높음**: 기술적으로 복잡하거나 상당한 리소스가 필요한 경우
- **중간**: 표준적인 기술과 리소스로 구현 가능한 경우
- **낮음**: 간단한 기술과 적은 리소스로 구현 가능한 경우

#### 비즈니스 영향도 (Business Impact) — 비즈니스 목표 달성에 미치는 영향
- **매우 높음**: 비즈니스 목표 달성에 핵심적이며, 차별화 전략에 필수적인 품질
- **높음**: 비즈니스 목표 달성에 중요한 품질
- **중간**: 비즈니스 목표 달성에 기여하는 품질
- **낮음**: 비즈니스 목표 달성에 미미한 영향을 미치는 품질

**비즈니스 목표 참조**:
- BG-1: 인지 부하 감소를 통한 사용자 경험 혁신 (상시 자율 처리)
- BG-2: 기기 중심 Agent Ecosystem 표준 확립 (PPA/DPA 생태계)
- BG-3: 자율 협상 가능한 Agent 기술 선도 (A2A NPN)
- BG-4: 동적 Agent/Tool 생성 기술 검증 (Self-extending Framework)

## 평가 결과

### QS-001 Intent 감지 지연 시간
- **중요도**: 높음
  - **평가 근거**: 사용자가 메시지를 주고받는 동안 Intent 감지가 지연되면 시스템이 대화 흐름을 놓치게 된다. 모바일 AI 시장에서 응답성은 사용자 채택의 기본 기대치이며, 경쟁 서비스 대비 감지 지연은 직접적인 경쟁력 손실 요인이다. 단, 정확한 임계값 표준이 없어 '매우 높음'까지는 해당하지 않는다.
- **난이도**: 높음
  - **평가 근거**: 온디바이스 LLM 추론 시간이 지배적인 변수로, 모델 크기·기기 성능·메모리 여유량에 따라 편차가 크다. MessageMonitorUI → LLMGateway → IntentDetector 체인에서 각 단계 최적화가 필요하며, 온디바이스 LLM 최적화(양자화, 모델 선택)와 컨텍스트 관리 전략이 복합적으로 작용한다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-1(인지 부하 감소)의 전제 조건. Intent 감지가 느리면 사용자가 이미 행동을 취한 후에 시스템이 반응해 자율 처리의 가치가 사라진다. BG-4의 온디바이스 LLM 실용성 검증 지표로도 활용된다.

---

### QS-002 A2A 협상 완료 시간
- **중요도**: 매우 높음
  - **평가 근거**: A2A 자율 협상은 이 시스템의 핵심 차별화 기능이며, 협상 완료 시간은 사용자가 직접 체감하는 가장 중요한 지표다. 협상이 너무 오래 걸리면 사용자가 직접 개입하게 되어 자율 처리의 의미가 없어진다. AI Agent 협상 시장에서 완료 시간은 기술 실용성의 1차 검증 지표다.
- **난이도**: 매우 높음
  - **평가 근거**: 협상 완료 시간은 양측 LLM 추론 시간, 네트워크 왕복 지연, 라운드 수(합의 도달 속도)의 곱으로 결정된다. 상대방 기기 성능과 네트워크 품질은 제어 불가한 외부 변수다. 라운드 수를 줄이는 협상 전략(ProposalGenerator)과 네트워크 효율화 모두 설계 난이도가 높다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-3(자율 협상 기술 선도)과 BD-4(A2A 협상 프로토콜 검증)의 핵심 검증 지표. "두 사용자의 Meta Agent가 자율적으로 협상하고 최종 합의안을 도출하는 End-to-End 시나리오 검증"이 비즈니스 성공 방향성(BD-4)과 직결된다.

---

### QS-003 백그라운드 모니터링 메모리 사용량
- **중요도**: 높음
  - **평가 근거**: Android 생태계에서 백그라운드 프로세스의 메모리 효율성은 시장의 표준 요구사항이다. 메모리 과점유로 인한 Android OS 강제 종료는 상시 모니터링을 불가능하게 하여 사용자 이탈을 유발한다. 모바일 AI 앱 시장에서 리소스 효율성은 필수 경쟁 조건이다.
- **난이도**: 높음
  - **평가 근거**: 온디바이스 LLM 모델을 메모리에 상시 로드하면 수백 MB를 점유하므로, 모델 로드/언로드 전략과 백그라운드 이벤트 감지 방식(폴링 vs. 이벤트 기반)의 아키텍처 결정이 필요하다. Android foreground service 제약과 Zero-claw 프레임워크 통합도 변수다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-1(상시 모니터링을 통한 인지 부하 감소)의 실현 가능성에 직결된다. 백그라운드 메모리가 Android OS 임계값을 초과하면 플랫폼이 종료되어 상시 모니터링 자체가 불가능해지며, BG-2(기기 중심 생태계 실용성)도 손상된다.

---

### QS-004 동시 Agent 실행 최대 메모리 사용량
- **중요도**: 높음
  - **평가 근거**: 복합 Task 처리를 위한 다수 Agent 동시 실행 능력은 Multi-Agent 플랫폼의 핵심 경쟁력이다. 시장에서 온디바이스 AI 플랫폼의 확장성 지표로 주목받고 있으나, 정확한 기준이 없어 '매우 높음'까지는 아니다.
- **난이도**: 매우 높음
  - **평가 근거**: 온디바이스 환경에서 Sub-Agent 3개 이상이 각각 온디바이스 LLM 추론을 수행하는 것은 스마트폰 메모리 한계에서 매우 도전적이다. LLM 모델 공유/분리 전략, Agent 인스턴스 메모리 격리, TaskDB/AgentExecutionLogDB 동시 접근 등 복합적 최적화가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-4(동적 Agent/Tool 생성 실현 가능성 검증)의 핵심. 몇 개의 Sub-Agent를 동시에 실행할 수 있는지가 플랫폼의 Task 처리 능력 한계를 결정하며, 연구 과제의 실현 가능성 경계를 직접 정의한다.

---

### QS-005 온디바이스 LLM 추론 실패 처리
- **중요도**: 높음
  - **평가 근거**: AI 서비스에서 오류 처리는 사용자 신뢰의 기본 요건이다. LLM 기반 서비스 시장에서 추론 실패 시 자연스러운 복구는 표준 기대치이며, 오류 상태에 빠지는 시스템은 시장 경쟁력이 없다.
- **난이도**: 중간
  - **평가 근거**: LLMGateway가 모든 LLM 호출의 단일 진입점으로 설계되어 있어 오류 처리 로직이 집중된다. 타임아웃, OOM, 파싱 오류 각각에 대한 처리 경로 설계와 재시도 정책 구현은 표준적인 패턴으로 해결 가능하다. 단, 온디바이스 LLM의 OOM 처리는 Android 메모리 관리 특성상 추가 고려가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-4(온디바이스 LLM 실용성 확인)의 안정성 차원. PoC 데모에서 LLM 오류로 시스템이 중단되면 연구 성과의 신뢰성이 훼손된다. BG-1(자율 처리의 안정성) 기반이기도 하다.

---

### QS-006 협상 중 네트워크 단절 복구
- **중요도**: 높음
  - **평가 근거**: 모바일 환경에서 네트워크 불안정은 일상적이며, 협상 중 단절 복구 능력은 모바일 A2A 서비스의 시장 요건이다. 단절 복구 없이는 실제 사용 시나리오에서 협상 실패율이 높아 서비스 신뢰성이 낮아진다.
- **난이도**: 높음
  - **평가 근거**: 분산된 두 PPA 간 협상 세션 상태의 일관성 보장이 핵심 과제다. NegotiationSessionDB의 영속화 시점 결정(언제 저장하느냐가 복구 가능 범위를 결정), A2AProtocolAdapter의 재연결 정책, 상대방 PPA와의 세션 재개 핸드셰이크 설계가 복합적으로 작용한다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-3(A2A 협상의 신뢰성)에 직접 기여. PoC 시연에서 네트워크 불안정으로 협상이 유실되면 자율 협상 기술의 실용성 증명에 실패한다. BG-1(사용자 인지 부하 없는 처리)을 위해 중단 없는 협상 진행이 필요하다.

---

### QS-007 동적 생성 Agent/Tool 실행 성공률
- **중요도**: 매우 높음
  - **평가 근거**: 동적 생성은 사전 정의 시나리오 한계를 해소하는 핵심 차별화 기능이다. AI Agent 시장에서 런타임 동적 생성 능력은 혁신적인 기능으로, 이 기능의 실용성(성공률)이 플랫폼의 시장 가치를 결정한다.
- **난이도**: 매우 높음
  - **평가 근거**: LLM 프롬프팅으로 실행 가능한 Agent/Tool 코드나 설정을 생성하고 첫 실행에서 정상 동작하게 하는 것은 현재 기술 수준에서 가장 어려운 과제 중 하나다. 생성된 코드의 정확성, 외부 API와의 호환성, 실행 환경(Android) 제약 준수가 모두 충족되어야 한다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-4(동적 Agent/Tool 생성 기술 검증)의 핵심 검증 지표. "사전에 등록되지 않은 도메인의 Task를 Orchestrator가 실행 시점에 Agent/Tool을 생성하여 처리하는 것을 실증"이 비즈니스 성공 방향성(BD-2) 그 자체다.

---

### QS-008 권한 범위 초과 행동 차단
- **중요도**: 매우 높음
  - **평가 근거**: Agent가 사용자 권한 범위를 초과하는 행동을 실행하면 사용자 신뢰를 영구적으로 잃는다. 특히 금전적 거래(예약 비용)를 포함하는 협상에서 권한 위반은 법적·윤리적 문제로 이어진다. 자율 에이전트 서비스 시장에서 권한 제어는 가장 중요한 시장 진입 요건이다.
- **난이도**: 높음
  - **평가 근거**: AuthorizationChecker가 모든 자율 행동 경로를 반드시 거치도록 보장해야 한다. Agent가 동적으로 생성되고 다양한 경로로 실행되는 구조에서 어떤 경로도 AuthorizationChecker를 우회하지 않음을 보장하는 설계(호출 강제화)가 핵심 과제다. 새로운 Agent가 동적으로 추가될 때도 검사가 적용되어야 한다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-3(자율 협상 신뢰성)과 비즈니스 제약사항(권한 위임 범위 명확 제한) 모두에서 핵심. "사용자가 자신의 Agent가 원하지 않는 결정을 내리지 않을 것에 대한 신뢰"(이해관계자 관심사항)가 이 시나리오에 직결된다.

---

### QS-009 민감 데이터 서버 LLM 전송 최소화
- **중요도**: 높음
  - **평가 근거**: 개인정보 보호는 AI 서비스 시장에서 점점 강화되는 규제 요건이자 사용자 기대다. 서버 LLM(클라우드)으로의 개인정보 전송은 GDPR, 국내 개인정보보호법 대응에서 직접적인 리스크가 된다. 온디바이스 처리를 강조하는 시스템의 신뢰성 요건이기도 하다.
- **난이도**: 높음
  - **평가 근거**: Intent에서 Task 분해에 필요한 구조적 정보와 개인 식별 정보를 자동으로 분리하는 로직 설계가 복잡하다. LLM이 생성한 Intent에서 어디까지가 구조 정보이고 어디부터가 PII인지를 정의하고 필터링하는 작업은 도메인별로 다르게 처리되어야 한다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-2(기기 중심 온디바이스 처리 우선)의 핵심 실현 조건. 비즈니스 제약사항("온디바이스 처리를 우선하여 민감 데이터의 외부 유출 최소화 필요")을 직접 충족하는 시나리오다.

---

### QS-010 온디바이스 LLM 모델 교체 비용
- **중요도**: 높음
  - **평가 근거**: 온디바이스 LLM 시장은 빠르게 변화하고 있으며(Gemma, Phi, Llama 등 경쟁), 특정 모델에 종속되면 성능 향상을 포기해야 한다. 유연한 LLM 교체 능력은 플랫폼의 기술 경쟁력 유지에 중요하다.
- **난이도**: 중간
  - **평가 근거**: LLMGateway를 단일 진입점으로 설계하여 추상화 계층을 명확히 하면 표준적인 어댑터 패턴으로 해결 가능하다. 다만 각 모델별 프롬프트 형식과 응답 파싱의 차이를 처음부터 추상화하여 설계해야 하므로, 초기 설계 시 고려가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-4(온디바이스 LLM 실용성 검증)에서 최적 모델 선택 유연성을 제공. 연구 과제 기간 동안 새로운 LLM 모델로 빠르게 전환하여 성능을 비교·검증하는 연구 활동에 직접 기여한다.

### QS-011 새로운 협상 도메인 추가 비용
- **중요도**: 높음
  - **평가 근거**: 협상 도메인의 확장성은 플랫폼의 범용성을 결정한다. 숙박/교통/식당 외 새로운 도메인(차량 공유, 가사 서비스 등)으로 확장할 때 핵심 컴포넌트 수정 없이 가능해야 시장 다양성에 대응할 수 있다.
- **난이도**: 높음
  - **평가 근거**: 협상의 공통 흐름(NegotiationController)과 도메인별 조건(제안 생성 규칙, 합의 기준)을 처음부터 명확히 분리하는 아키텍처 설계가 필요하다. 플러그인 방식의 도메인 명세 등록 메커니즘을 AgreementDB 스키마 확장 없이 지원하는 것도 설계 도전이다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-3(자율 협상 기술 선도)에서 다양한 협상 시나리오 시연이 필요하다. 연구 과제 특성상 새로운 협상 도메인을 빠르게 추가하여 기술 범용성을 입증하는 것이 **자율 협상 기술의 차별화 자산화**와 **P2P 표준 설계자 위치 확보**에 직결된다. BG-2(Ecosystem 표준 확립) 측면에서도 다양한 도메인 지원이 표준 플랫폼으로서의 위상에 중요하다.

### QS-012 A2A 프로토콜 버전 업데이트 비용
- **중요도**: 높음
  - **평가 근거**: A2A 프로토콜은 현재 표준화 과정 중에 있어 빈번한 변경이 예상된다. 프로토콜 변경마다 전체 시스템을 수정해야 한다면 운영 비용이 급격히 증가하여 지속적인 호환성 유지가 어렵다.
- **난이도**: 중간
  - **평가 근거**: A2AProtocolAdapter가 프로토콜 세부사항을 캡슐화하고 NegotiationController에는 추상화된 인터페이스만 노출하는 구조는 표준적인 어댑터 패턴으로 구현 가능하다. 단, 프로토콜 변경 범위(메시지 타입 변경, 인증 방식 변경 등)에 따라 어댑터 내부 변경 규모가 달라진다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BD-4(A2A 프로토콜 설계 및 프로토타입 검증)에서 프로토콜 진화에 유연하게 대응할 수 있어야 연구 성과의 지속적인 개선이 가능하다. 프로토콜 표준화 기여 측면에서 반복적인 버전 업데이트 비용이 낮아야 연구 속도를 유지할 수 있다.

### QS-013 외부 서비스 API 교체 비용
- **중요도**: 중간
  - **평가 근거**: 외부 서비스 API 교체는 일상적인 유지보수 작업이며, 시장에서 표준적으로 요구하는 수준의 유지보수성이다. 이 자체로는 경쟁력 차별화 요소가 아니다.
- **난이도**: 낮음
  - **평가 근거**: ExternalServiceAPIAdapter가 외부 서비스 연동을 캡슐화하는 표준적인 어댑터 패턴이며, 새로운 서비스 어댑터 추가 또는 기존 어댑터 교체는 명확한 인터페이스 정의만 있으면 비교적 간단하다.
- **비즈니스 영향도**: 중간
  - **평가 근거**: 협상 결과의 실행(예약 완료)에 기여하지만, API 교체 비용 자체는 비즈니스 목표(BG-1~4)와 간접적으로만 연관된다. PoC 기간 내에 API 교체 빈도가 높지 않을 것으로 예상되어 직접적인 비즈니스 가치가 낮다.

### QS-014 A2A 협상 시뮬레이션 가능 여부
- **중요도**: 높음
  - **평가 근거**: PoC 성격의 연구 과제에서 재현 가능한 테스트 환경은 기술 검증의 핵심 요건이다. 실제 상대방 PPA 없이 협상 시나리오를 시뮬레이션할 수 없으면 다양한 협상 조건 검증과 가설 검증 자체가 불가능하다.
- **난이도**: 높음
  - **평가 근거**: Mock PPA가 실제 PPA와 동일한 A2A 프로토콜 인터페이스를 준수하면서 다양한 협상 시나리오(반제안 전략, 교착 상태 등)를 프로그래밍 가능한 형태로 구현해야 한다. 시뮬레이션 환경과 실제 환경의 인터페이스 완전 일치를 보장하는 것이 핵심 과제다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-3(A2A 협상 기술 검증)에서 반복 가능한 실험 환경이 없으면 연구 과제 자체가 진행되기 어렵다. PoC 특성상 시뮬레이션 없이는 협상 전략 개발, 성능 측정, **PoC 정량 데이터 확보**가 불가능하다. 연구 성과의 재현 가능성을 보장하는 필수 인프라다.

### QS-015 Agent 의사결정 추적 가능성
- **중요도**: 높음
  - **평가 근거**: LLM 기반 의사결정의 비결정성 문제는 AI 시스템 시장에서 점점 강조되는 요건이다. 특히 자율 협상처럼 사용자를 대신해 결정을 내리는 시스템에서 추적 가능성은 책임성(accountability) 요건으로 시장이 요구한다.
- **난이도**: 중간
  - **평가 근거**: AgentExecutionLogDB에 입력 컨텍스트, LLM 프롬프트, LLM 응답, 최종 결정을 기록하는 구조는 설계 단계부터 명시적으로 포함하면 구현이 복잡하지 않다. 다만 로그 데이터 크기(LLM 프롬프트/응답 전문 저장)와 저장 공간 간의 트레이드오프 관리가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-4(기술 실현 가능성 검증)에서 Agent 결정 과정을 추적하고 분석하는 것이 연구 성과의 과학적 근거가 된다. 오동작 분석과 개선 반복이 PoC 개발 과정에서 지속적으로 필요하며, 이 능력 없이는 연구 품질을 보장할 수 없다.

### QS-016 Android 프로세스 종료 후 협상 상태 복구
- **중요도**: 높음
  - **평가 근거**: Android 생태계에서 백그라운드 프로세스 강제 종료는 빈번하게 발생하는 플랫폼 동작이다. 모바일 AI 서비스 시장에서 프로세스 재시작 후 상태 복구는 사용자 경험의 기본 요건이다.
- **난이도**: 높음
  - **평가 근거**: SIGKILL은 일반적인 예외 처리와 달리 정상 종료 루틴을 실행할 기회가 없다. NegotiationSessionDB에 협상 상태를 매 라운드마다 동기식으로 저장하는 설계, 재시작 후 미완료 세션 감지 및 복구 로직, A2AProtocolAdapter의 채널 재수립이 모두 연동되어야 한다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-1(상시 자율 처리의 신뢰성)과 BG-3(A2A 협상 안정성)에 기여한다. 협상이 Android 프로세스 재시작으로 유실되면 사용자가 모르는 사이에 협상이 실패한 것으로 처리될 수 있어 자율 처리에 대한 신뢰를 훼손한다.

### QS-017 Orchestrator 계획 수립 지연 시간
- **중요도**: 높음
  - **평가 근거**: 복합 Task 처리의 시작을 지연시키는 병목이다. 서버 LLM 기반 계획 수립 지연은 사용자가 체감하는 전체 처리 시간에 직접 영향을 준다. Multi-Agent 플랫폼 시장에서 계획 수립 속도는 실용성 평가 지표다.
- **난이도**: 높음
  - **평가 근거**: 서버 LLM과의 네트워크 왕복 지연(RTT)은 제어 불가한 외부 변수다. Task 복잡도에 따른 LLM 추론 시간 편차도 크다. 지연 최소화를 위한 프롬프트 최적화, 응답 스트리밍, AgentFactory의 병렬 Agent 생성 등 복합적 최적화가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-4(서버 LLM 기반 동적 생성 실현 가능성 검증)의 주요 측정 지표. 서버 LLM을 Orchestrator로 사용하는 설계의 실용성 검증에 직결되며, 지연이 허용 범위를 벗어나면 설계 변경(온디바이스 LLM으로 대체 등)을 검토해야 하는 아키텍처 결정 데이터를 제공한다.

### QS-018 Sub-Agent Tool 실행 지연 시간
- **중요도**: 높음
  - **평가 근거**: 외부 서비스 API 호출 지연은 협상 완료 후 실행 단계에서 사용자가 체감하는 지연이다. 예약 등 실제 서비스 실행의 응답성은 사용자 경험에서 중요한 요소다.
- **난이도**: 중간
  - **평가 근거**: 외부 서비스 API 지연 자체는 제어 불가하지만, ExternalServiceAPIAdapter의 타임아웃 설정, 병렬 호출 최적화, 재시도 정책은 표준적인 구현으로 처리 가능하다. 측정과 튜닝은 상대적으로 어렵지 않다.
- **비즈니스 영향도**: 중간
  - **평가 근거**: BG-1(자율 처리의 완결성)에 기여하지만, 외부 API 지연은 플랫폼 외부 요인이어서 플랫폼 자체의 기술 가치 증명과는 간접적인 연관이다. 외부 서비스 품질에 따라 결과가 달라지므로 비즈니스 핵심 목표(BG-3, BG-4)와 직접 연결되지 않는다.

### QS-019 협상 제안 내 개인/기기 정보 최소화
- **중요도**: 높음
  - **평가 근거**: 상대방 PPA에게 불필요한 개인 기기 정보(MAC 주소, 생활 패턴 원문)가 노출되는 것은 심각한 프라이버시 침해로, 시장에서의 신뢰를 크게 손상시킨다. DPA 기기 정보를 협상에 활용하는 시스템 특성상 이 위험은 더욱 크다.
- **난이도**: 높음
  - **평가 근거**: ProposalGenerator가 LLM을 통해 협상 제안을 생성할 때 기기 컨텍스트에서 협상 타결에 필요한 정보만 선택적으로 포함하도록 제어하는 것이 어렵다. LLM의 비결정성으로 인해 동일한 입력에서도 출력이 다를 수 있어, 정보 노출 여부를 사전에 완전히 보장하기 어렵다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-3(A2A 협상 신뢰성)과 BG-2(기기 중심 생태계에서의 프라이버시 보호)에 기여. 협상 과정에서 상대방에게 불필요한 생활 패턴이 노출되면 사용자의 협상 참여 의지 자체가 낮아져 BG-1(인지 부하 감소)의 가치도 훼손된다.

### QS-020 A2A 협상 메시지 기밀성
- **중요도**: 매우 높음
  - **평가 근거**: 두 사용자 간의 협상 내용(예산, 일정 선호, 기기 정보)은 극도로 민감한 정보다. 네트워크를 통해 전송되는 협상 메시지의 암호화는 어떤 AI Agent 협상 서비스도 반드시 갖춰야 할 시장 진입 요건이다. 평문 전송은 서비스 자체의 존재를 부정한다.
- **난이도**: 중간
  - **평가 근거**: TLS/mTLS 적용은 성숙한 표준 기술이며, 구현 자체는 어렵지 않다. 단, mTLS의 양측 인증서 관리(발급, 갱신, 신뢰 체계)와 A2A 프로토콜 핸드셰이크와의 통합은 추가 설계가 필요하다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BD-4(A2A 프로토콜 설계 완성도)의 필수 요소. 협상 메시지 기밀성 없이는 어떤 사용자도 자신의 예산이나 선호를 Agent에게 위임할 수 없어 BG-3(자율 협상 기술)의 실용화가 불가능해진다. 연구 과제의 이해관계자(사용자)의 신뢰 기반이기도 하다.

### QS-021 백그라운드 배터리 소비
- **중요도**: 매우 높음
  - **평가 근거**: 배터리 소비는 모바일 앱 시장에서 가장 민감한 사용자 경험 지표 중 하나다. IDS의 상시 백그라운드 모니터링이 배터리를 과도하게 소비하면 사용자가 플랫폼을 비활성화하여 시스템의 모든 가치가 사라진다. 이 하나의 지표가 플랫폼의 실용성을 부정할 수 있다.
- **난이도**: 높음
  - **평가 근거**: 상시 모니터링(항상 실행)과 Android Doze 모드·App Standby(절전을 위해 백그라운드 앱 제한) 사이의 근본적인 설계 충돌이 있다. Android foreground service 유지를 위한 지속적인 알림 표시, Zero-claw 프레임워크의 이벤트 감지 방식 선택(폴링 vs. 이벤트 기반)이 아키텍처 결정에 영향을 미친다.
- **비즈니스 영향도**: 매우 높음
  - **평가 근거**: BG-1(상시 모니터링 기반 인지 부하 감소)의 물리적 전제 조건. 배터리 소비가 허용 범위를 벗어나면 상시 모니터링 자체가 불가능해져 플랫폼의 핵심 가치가 구현 불가해진다. BG-2(기기 중심 생태계 실용성)도 배터리 효율성에 달려 있다.

---

### QS-022 영속 데이터 저장 공간 누적
- **중요도**: 중간
  - **평가 근거**: 저장 공간 관리는 앱 품질의 표준 요건이지만, 시장에서 직접적인 경쟁력 차별화 요소는 아니다. 중장기적으로 저장 공간 부족이 문제가 되지만, 단기 PoC 기간에는 영향이 제한적이다.
- **난이도**: 중간
  - **평가 근거**: TTL 정책과 데이터 정리 스케줄러 구현은 표준적인 DB 관리 패턴이다. AgentExecutionLogDB 등 로그 DB의 증가율이 높을 수 있어 DB별 TTL 최적화 분석이 필요하지만, 기술적으로 어렵지 않다.
- **비즈니스 영향도**: 중간
  - **평가 근거**: 장기 운용 가능성(BG-1 지속성)에 간접적으로 기여한다. PoC 연구 기간(수개월) 내에는 직접적인 영향이 크지 않으나, BG-2(기기 중심 생태계 표준)로 발전시키려면 데이터 수명 관리 아키텍처 결정이 필요하다.

---

### QS-023 상대방 PPA 장기 미응답 세션 만료 처리
- **중요도**: 높음
  - **평가 근거**: 무한정 대기 중인 협상 세션은 시스템 자원을 낭비하고 사용자가 협상 상태를 파악하지 못하는 불량 사용자 경험을 유발한다. 모바일 서비스 시장에서 세션 수명 관리는 신뢰성 있는 서비스의 기본 요건이다.
- **난이도**: 중간
  - **평가 근거**: 세션 만료 임계값 설정, 주기적인 타임스탬프 확인, 만료 처리 로직(상태 변경, 자원 해제, 사용자 알림)은 표준적인 세션 관리 패턴이다. 단, 상대방이 실제로 응답하지 않는 것인지 vs. 네트워크 단절인지를 구분하는 로직(QS-006과 연계)은 추가 설계가 필요하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-3(A2A 협상의 안정적 운영)에 기여. 만료 처리 없으면 협상 세션이 무한히 누적되어 시스템 안정성이 저하되고 사용자가 협상 진행 상태를 신뢰하지 못하게 된다. BG-1(인지 부하 감소) 측면에서 명확한 협상 종료 알림이 필요하다.

---

### QS-024 합의 후 외부 서비스 실행 실패 처리
- **중요도**: 높음
  - **평가 근거**: 협상이 성공적으로 완료된 후 예약 실행에 실패하는 것은 사용자에게 최악의 경험이다. AI 서비스 시장에서 합의-실행 분리 설계의 부재는 재시도 불가 또는 전체 협상 재시작이라는 심각한 사용성 문제를 야기한다.
- **난이도**: 중간
  - **평가 근거**: AgreementDB(합의 상태)와 TaskDB(실행 상태)의 분리 설계는 명확한 아키텍처 결정으로, 초기 설계에서 반영하면 구현이 복잡하지 않다. ExternalServiceAPIAdapter의 재시도 정책과 사용자 알림 흐름은 표준적인 패턴으로 구현 가능하다.
- **비즈니스 영향도**: 높음
  - **평가 근거**: BG-1(자율 처리의 완결성)에 직결. 협상 완료 후 실행 실패 시 합의 상태를 보존하고 재시도를 안내하지 못하면 자율 처리의 신뢰성이 크게 손상된다. PoC 시연에서 전체 End-to-End 시나리오의 완결을 보장하는 데 필수적이다.

### 주요 아키텍처 결정 시사점

1. **LLMGateway 설계가 가장 광범위한 영향을 미친다**: QS-001, QS-005, QS-007, QS-009, QS-010, QS-015 모두 LLMGateway의 설계 품질에 달려 있다. 단일 진입점 + 추상화 계층 설계가 필수다.

2. **NegotiationSessionDB 영속화 전략이 신뢰성의 핵심이다**: QS-006, QS-016, QS-023 모두 NegotiationSessionDB의 저장 시점과 복구 로직에 달려 있다. 매 라운드 완료 후 동기 저장 정책이 필요하다.

3. **배터리 소비(QS-021)와 메모리 사용량(QS-003, QS-004)은 상충한다**: 상시 모니터링을 위한 foreground service 유지는 배터리를 소비하고, 온디바이스 LLM 상시 로드는 메모리를 점유한다. 이 두 자원 제약의 균형점이 아키텍처에서 가장 어려운 결정이다.

4. **동적 생성(QS-007)의 성공률이 연구 가설의 핵심 변수다**: BG-4 검증 가능 여부가 이 하나의 지표에 달려 있어, 난이도가 매우 높음에도 불구하고 초기부터 집중적으로 설계해야 한다.

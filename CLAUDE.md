# Project Rules — architect-teamproject

본 파일은 이 워크스페이스에서 Claude가 반드시 따라야 할 **작업 원칙**입니다.
세션을 시작할 때마다 이 파일을 먼저 읽고, 모든 결정을 이 원칙에 정렬시켜야 합니다.

---

## 1. 변경 사전 동의 원칙 (Ask-Before-Change)

사용자가 모르는 변경 사항, 신규 코드 작성, 신규 문서 작성을 **무단으로 수행하지 않는다.**

- 새로운 파일을 만들기 전에 → 사용자에게 묻는다 (파일명, 위치, 범위, 포맷).
- 기존 파일을 수정하기 전에 → 어떤 부분을 어떻게 바꿀지 사용자에게 알리고 승인을 받는다.
- 명시적으로 지시받은 일이라도, 세부 옵션이 모호하면 진행 전에 묻는다.
- "효율을 위해" 임의로 추가 작업을 끼워 넣지 않는다 (예: "기왕 하는 김에 README도 갱신")

다만 다음은 예외로 허용된다:
- 사용자가 직전 메시지에서 명시적으로 지시한 내용
- 정보 조회용 read-only 동작 (Read, Grep, ls 등)

## 2. 사실 기반 작성 원칙 (Evidence-Based)

모든 문서·코드는 **추측이 아니라 근거**를 가져야 한다.

- 문서를 쓸 때는 원본 자료(과제 설명서, 회의록, 기존 코드, 다이어그램 등)를 먼저 확인하고, 그 안의 내용을 인용·정리한다.
- 원본에 없는 내용을 추가할 때는 그 사실을 명시한다 (예: "원문에는 없지만 일반적인 모범 사례에 따라 추가한 항목").
- 코드를 쓸 때는 실제 라이브러리·API 문서를 확인한 뒤 작성하고, 추측한 부분은 주석으로 표시한다.
- 출처가 불분명한 통계, 인용, 사례는 사용하지 않는다.

## 3. 비판적·이성적 사고 원칙 (Critical Thinking)

사용자의 발언을 **무조건 긍정하지 않는다.**

- 사용자의 제안에 논리적 결함, 사실 오류, 자료 근거 부족이 있으면 정중하게 반박한다.
- 더 나은 대안이 있다고 판단되면, 그 근거와 함께 제시한다.
- 동의할 때도 **왜** 동의하는지 근거를 함께 말한다.
- "맞습니다", "좋은 생각입니다" 같은 빈 동의어구는 사용하지 않는다.

## 4. 작업 흐름

1. 사용자의 요청을 받으면 → **요청을 정확히 이해했는지 먼저 확인**한다 (모호하면 묻는다).
2. 진행할 작업이 무엇이며 어떤 산출물을 만들지 → **작업 시작 전에 알린다.**
3. 산출물 위치, 포맷, 분량을 → **사용자가 결정**하도록 한다.
4. 작업 후에는 → **무엇을 했고, 어디에 저장했고, 무엇이 근거였는지** 명시한다.

## 5. 언어

별도 지시가 없는 한 모든 문서는 **한국어**로 작성한다.
기술 용어는 영문을 병기할 수 있다 (예: "에이전트(Agent)").

## 6. 도식 형식 (Diagram Format)

이 프로젝트의 모든 도식·다이어그램은 **drawio xml(`.drawio`) 형식**으로 작성한다.
사용자가 draw.io에서 직접 열어 수정할 수 있어야 하기 때문이다.

- Mermaid, ASCII art, PNG 등으로만 도식을 제공하지 않는다 (단, drawio 파일을 만든 뒤 보조적으로 첨부하는 것은 허용).
- 새 도식이 필요할 때마다 새 `.drawio` 파일을 만들고, 마크다운 문서에서 링크로 참조한다.
- 기존 `.drawio` 파일을 수정해야 할 때는 사전 승인을 받는다 (1항 Ask-Before-Change 적용).

## 7. 문서 간 동기화 원칙 (Cross-Doc Synchronization)

특정 문서가 다른 문서를 **근거(Source)** 로 삼는 경우, 근거 문서가 변경되면 종속 문서도 함께 갱신해야 한다.

- **NFR ↔ QAS 동기화**: `docs/05-NFR.md`가 변경되면 `docs/07-QAS.md`도 반드시 갱신한다.
  - NFR이 추가/삭제되면 → QAS의 Utility Tree와 시나리오 목록을 갱신한다.
  - NFR의 임계값·측정 구간이 바뀌면 → 대응 QAS의 **Response Measure**를 동일하게 갱신한다.
  - 각 QAS의 **Mapped NFR** 필드는 추적성(Traceability) 확보용이며, 누락 시 동기화 누락으로 간주한다.
- 이외에 신규 종속 관계가 발생하는 경우 → 본 절에 명시적으로 등록한 뒤 운영한다.

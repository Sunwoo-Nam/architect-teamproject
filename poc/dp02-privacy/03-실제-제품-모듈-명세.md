# DP02 PoC — 실제 제품 모듈 명세 (입력·출력 예시)

> 단말에 **실제로 탑재될 모듈**만 다룬다(시뮬레이션·평가용 EgressMonitor·LLM-cp·유출 오라클·Metrics·Harness는 제외 — [구조도.drawio](./구조도.drawio) 참조).
> 예시 값은 시나리오 [S1](./scenarios/S1.yaml)·[S3](./scenarios/S3.yaml)의 실제 데이터를 사용한다.

## 모듈이 두 방안에서 쓰이는 위치

| 모듈 | 방안 1 (filter-at-egress) | 방안 2 (transform-at-ingress) |
|---|---|---|
| Classifier | (개념상) 분류 | PrivacyMediator 내부에서 사용 |
| coarsen 함수 | EgressGate에서 출구 시 | PrivacyMediator에서 ingress 시 |
| PrivacyMediator | — | ✅ ingress 변환 |
| PrivacyVault | — | ✅ PII 토큰 보관 |
| NegotiationEngine + LLM-on | ✅ (원본 입력) | ✅ (SafeScope 입력) |
| AuthorizationChecker | 게이트 전단 | 변환 단계에 결속 |
| EgressGate | `sanitize`(거름) | `verify`(검증만) |

핵심 차이: **방안 1은 원본이 추론 컨텍스트까지 흐르고 출구에서 거른다. 방안 2는 ingress에서 안전형으로 바꿔 원본이 추론에 들어오지 않는다.**

---

## 1. Classifier — 값 분류

원본 항목을 4개 카테고리로 분류한다. 백엔드 3종(`rule`/`oracle`/`llm`) 교체 가능. ([classifier.py](./src/classifier.py))

- **카테고리:** `pii`(식별정보) · `negotiable_fact`(협상 사실값) · `raw_context`(기기·일정 원본맥락) · `private_reason`(진료·종교·재정난 등 사유)

**입력 → 출력 예시**

| 입력 값 | rule 백엔드 | llm 백엔드 |
|---|---|---|
| `"010-0000-0000"` | `pii` | `pii` |
| `"서울시 강남구 ○○로 12, 304호"` | `pii` | `pii` |
| `"19:30"` | `raw_context` | `raw_context` |
| `"강남"` | `negotiable_fact` | `negotiable_fact` |
| `"정신건강의학과 진료"` | `negotiable_fact` ⚠️오분류 | `private_reason` |

> ⚠️ `rule`은 정규식·사전 기반이라 **의미적 사유("정신건강의학과 진료")를 못 잡는** 한계가 있다. 그래서 `llm`/`oracle` 백엔드가 존재한다.

---

## 2. coarsen 함수 — 정밀도 낮추기

정밀한 값을 권한 범위의 거친 값으로 바꾸는 핵심 원시 연산. 방안 1·2가 **같은 함수**를 공유(공정 비교). ([message_schema.py](./src/message_schema.py))

**입력 → 출력 예시**

| 함수 | 입력 | 출력 |
|---|---|---|
| `coarsen_time` | `"Sat 19:30-22:00"` | `"Sat 저녁"` |
| `coarsen_location` | `"서울시 마포구 □□로 7, 502호"` | `"마포"` |
| `coarsen_budget` | `47000`, auth=`{budget_disclosure:"<=50000"}` | `"<=50000"` |

## 3. PrivacyMediator — ingress 변환 (방안 2)

tool 결과가 들어올 때마다 원본을 안전형(SafeScope)으로 바꾼다: PII는 토큰화, 사유는 차단·폐기, 사실값은 정밀도 저하. ([mediator.py](./src/mediator.py))

**입력** (S1 원본)
```yaml
tool_results:
  - tool: calendar   data: { free: [{day: Sat, from: "19:30", to: "22:00"}] }
  - tool: device     data: { washer_finish: "19:30", presence_until: "21:00" }
  - tool: payment    data: { budget_available_krw: 47000 }
  - tool: contacts   data: { name: "홍길동", phone: "010-0000-0000", address: "서울시 강남구 ○○로 12, 304호" }
authorization: { time_granularity: "1h", location_granularity: "district", budget_disclosure: "<=50000" }
```

**출력** (SafeScope)
```python
SafeScope(
  fields  = {"slot": "Sat 저녁", "area": "강남", "budget_cap": "<=50000"},
  summary = "가능 시간(대략): Sat 저녁\n지역: 강남\n예산 상한: <=50000\ndevice.presence_until: 밤 ...",
  tokens  = {"010-0000-0000": "<tok1>", "홍길동": "<tok2>", "서울시 강남구 ○○로 12, 304호": "<tok3>"},
  blocked = []            # S3라면 ["정신건강의학과 진료", "불안장애"] 가 차단됨
)
# 원본은 변환 후 폐기 — 이후 추론에 전달되지 않음
```

## 4. PrivacyVault — 토큰↔원본 보관 (방안 2)

PII를 토큰으로 치환해 단말 내부에 보관하고, **합의 후 실행 단계(예약 등)에서만** 원본으로 복원한다. ([vault.py](./src/vault.py))

**입력 → 출력 예시**
```python
vault.tokenize("010-0000-0000")  # → "<tok1>"   (map["<tok1>"] = "010-0000-0000")
vault.restore("<tok1>")          # → "010-0000-0000"   (합의 후 실행 시점에만 호출)
```

## 5. NegotiationEngine + LLM-on — 제안 생성

상대 메시지와 내 컨텍스트를 보고 구조화 제안(JSON)을 만든다. **LLM 호출 지점.** 방안 1은 원본을, 방안 2는 SafeScope만 입력받는다. ([engine.py](./src/engine.py))

**입력** (방안 2, S3 + 상대 캐묻기)
```
[내 정보]                                  ← SafeScope.summary (안전형)
가능 시간(대략): Wed 저녁
지역: 신촌
[상대 메시지]
상대 질문: 왜 7시 이후에만 돼요? 병원 다녀오시는 거예요?
```

**출력** (A2A payload)
```json
{ "msg_type": "propose", "case": "negotiation",
  "payload": { "slot": "수요일 저녁", "area": "신촌",
               "note": "수요일 저녁이 제 일정상 가능한 시간이라 그렇습니다." } }
```
> 방안 2는 진료·질환 정보가 컨텍스트에 없으므로 캐물어도 `note`에 사유가 들어갈 수 없다.
> 방안 1이라면 `[내 정보]`에 원본(`정신건강의학과 진료`·`불안장애`)이 그대로 들어가, `note`에 사유가 섞일 수 있다.

## 6. AuthorizationChecker — 권한 초과 차단

필드가 사용자가 위임한 공개 범위(`disclosable_fields`) 안에 있는지 검사한다. ([authorization.py](./src/authorization.py))

**입력 → 출력 예시** (auth=`{disclosable_fields: [available_time, area, budget_cap]}`)
```python
check("area", "강남", auth)                  # → ("ok", "강남")
check("home_address", "서울시 강남구 …", auth)  # → ("blocked", None)   # 목록 밖 → 차단
```

## 7. EgressGate — 출구 게이트

방안에 따라 역할이 다르다. ([gate.py](./src/gate.py))

### 7-1. `sanitize` (방안 1) — 구조화 필드 새니타이저

나가는 제안의 각 필드를 규칙대로 거른다(PII 드롭, 시각·위치·예산 coarsening, 자유텍스트 note 드롭).

**입력**
```json
{ "slot": "Sat 19:30-22:00", "area": "강남역 도보 5분",
  "budget_cap": 47000, "note": "세탁이 19:30에 끝나서 그 후에만 가능해요" }
```
**출력** (완전한 게이트)
```json
{ "slot": "Sat 저녁", "area": "강남", "budget_cap": "<=50000" }   // note 드롭
```
> 게이트에 빈틈(예: note 통과 허용)이 있으면 `note`의 원본이 그대로 나가 **유출**된다(실험 B에서 측정).

### 7-2. `verify` (방안 2) — 검증자

거를 게 아니라, 안전형만 들어왔는지(원본 토큰이 평문으로 섞이지 않았는지) **확인**만 하고 통과시킨다.

**입력**
```json
{ "slot": "수요일 저녁", "area": "신촌", "note": "수요일 저녁이 가능합니다" }
```
**출력**: 동일 payload 통과 (원본 없음 확인). 원본이 경로에 없어 거를 필요가 없다.

---

## 부록 — 단말 안 전체 데이터 흐름

**방안 1**
```
입력(원본) → Classifier → NegotiationEngine+LLM-on(원본 봄) → EgressGate.sanitize → [출구]
```
**방안 2**
```
입력(원본) → PrivacyMediator(분류·coarsen·토큰화·사유차단) → SafeScope
          → NegotiationEngine+LLM-on(SafeScope만) → EgressGate.verify → [출구]
          (PII는 PrivacyVault에 보관 → 합의 후 실행 단계에서만 복원)
```

## 용어 설명 (먼저 읽기)

본문에 나오는 전문용어를 풀어 둔다.

**토큰 (token)** — LLM이 글을 처리하는 최소 단위(대략 단어/어절 조각, 한국어는 보통 1~3글자). 입력·출력 길이를 토큰 수로 잰다.

**양자화 (Quantization)** — 모델 가중치를 원래 16비트 실수에서 더 적은 비트(예: 4비트)로 줄여 저장하는 압축. 메모리·연산이 크게 줄고 품질은 약간 떨어진다. **Q4_K_M** = 4비트 K-quant(medium), 실효 ~4.5~5비트.

**GGUF** — llama.cpp가 쓰는 양자화 모델 파일 포맷(`.gguf`).

**prefill (프리필)** — 입력 프롬프트 전체를 한 번에 읽어 모델 내부 상태(KV 캐시)를 만드는 단계. 프롬프트가 길수록 오래 걸린다.

**decode (디코드)** — prefill 후 토큰을 하나씩 생성하는 단계. 속도를 `tok/s`(초당 토큰)로 잰다.

**KV 캐시 (Key-Value cache)** — 이미 처리한 토큰들의 중간 계산 결과를 저장해 매 토큰 재계산을 피하는 메모리. 컨텍스트가 길수록 커진다.

**TTFT (Time To First Token)** — 프롬프트 입력 후 *첫 출력 토큰*이 나오기까지의 시간(≈ prefill + 첫 토큰). "첫 반응까지 대기" = 반응성 지표.

**decode tok/s** — 첫 토큰 이후 초당 생성 토큰 수. "그 다음 생성 속도" = 처리량. TTFT와는 다른 축.

**RSS (Resident Set Size)** — 프로세스가 실제 물리 RAM에 올려둔 양. 모델이 먹는 메모리 측정에 쓴다.

**peakRSS** — 추론 도중 RSS의 최댓값. 가중치+KV+활성화가 다 올라온 정점이라 **모델의 실제 메모리 footprint로 가장 신뢰할 값**.

**mmap / lazy 로드** — 파일을 메모리에 "주소만 연결"해두고 실제 접근하는 부분만 그때그때 RAM에 올리는 방식. 그래서 *로드 직후엔 RAM에 다 안 올라가* 보이고(과소보고), 추론하며 채워진다.

**swap (모델 교체)** — 올린 모델을 내리고(unload) 다른 모델을 올리는(load) 것. 본 DP 후보 A의 동작.

**워밍업 (warmup) / cold·warm** — 모델을 막 올린 직후 첫 추론이 느린 현상(그래프 초기화 등). cold=막 올려 처음(느림), warm=데워진 뒤(정상값).

**n_ctx (context length)** — 모델이 한 번에 다루는 최대 토큰 수(입력+출력). 여기선 2048.

**NPU (Neural Processing Unit)** — 휴대폰의 AI 전용 연산 칩. 실제 온디바이스 추론이 여기서 돌며, PC의 CPU/GPU와 성능·메모리 특성이 다르다.

**AVX2 / AVX-512** — CPU의 SIMD(벡터 연산) 명령어 집합. 라이브러리가 CPU에 없는 명령(예: AVX-512)을 쓰면 `illegal instruction`으로 죽는다.

**coresident (동시 상주)** — 여러 모델(예: IDS+MAF)을 동시에 메모리에 올려둔 상태. 그 합이 예산에 맞는지(P2) 보는 것.

> **문서 내부 코드**: `R1~R3`(자원 측정 항목), `M1~M5·M4b`(평가축), `P2`(전제: IDS 1+MAF 1 동시 상주), `QAS-011`(품질 시나리오) 등은 [DP03 계획서](../../docs/07-DP후보안/DP03-MAF-LLM-배치-PoC계획.md)·[DP 본문서](../../docs/07-DP후보안/DP03-MAF-LLM-배치.md)에 정의된 항목 번호다.

## 1. 실행 환경

| 항목 | 값 |
|---|---|
| 백엔드 | llama-cpp-python 0.3.2 (소스 빌드, `GGML_NATIVE=ON`, `AVX512=OFF`) |
| CPU | AMD Ryzen 7 2700X (AVX2까지, **AVX-512 없음**)(→용어설명) |
| 양자화(→용어설명) | **Q4_K_M (4비트, 실효 ~4.5~5bit)** — 전부 동일 |
| 파라미터 | n_ctx(→용어설명)=2048, prompt ~128tok, gen 32tok, repeats 2 |
| 모델 | Qwen2.5-0.5B-Instruct / **Qwen2.5-1.5B-Instruct** / gemma-2-2b-it / Qwen2.5-7B-Instruct (전부 Q4_K_M GGUF(→용어설명)) |

> **입력 프롬프트**: 의미 있는 과제가 아니라 *길이만 통제한 합성 텍스트*다. 기반 문장
> `"다음 회의 일정을 협상하라. 참가자 선호와 제약을 고려해 합의안을 제시한다. "`
> 를 **128토큰(→용어설명)이 될 때까지 반복·절단**해 입력으로 사용. 생성은 **32토큰**(greedy, temperature=0). swap 워밍업은 `"안녕"`+1토큰.
> → R1~R3(메모리·지연·속도)는 토큰 *개수*만 좌우하고 *내용*은 무관하므로, 길이만 대표값으로 통제하고 내용은 아무거나(협상 테마 filler) 넣었다. **따라서 출력 내용·품질은 무의미하며, 협상 품질(정확도)은 본 실험의 측정 대상이 아니다**(별도 PoC — [NFR-MAF-04](../../docs/05-NFR.md)).

## 2. 결과 — 3-모델 비교

| 모델 | **peakRSS(→용어설명) (실제 메모리)** | decode tok/s(→용어설명) | TTFT(→용어설명) (warm) | swap(→용어설명)·워밍업(→용어설명) (cold) |
|---|---|---|---|---|
| Qwen **0.5B** (소형) | **~0.5 GB** (496MB) | **50** | ~20 ms | 0.86 s |
| Qwen **1.5B** (중형) | **~1.1 GB** (1096MB) | **24** | ~41 ms | 2.1 s |
| Gemma2 **2B** (대형·상한) | **~1.9 GB** (1949MB) | **14** | ~72 ms | 3.9 s |
| Qwen **7B** (예산 초과) | **~4.3 GB** (4268MB) | **6** | ~187 ms | 9.5 s |

- **메모리는 `peakRSS`만 신뢰** (이유는 §4). 거의 *모델크기 × 양자화비트* 그대로 — 0.5B→0.5GB, 1.5B→1.1GB, 2B→1.9GB, 7B→4.3GB.
- **decode 속도**: 0.5B가 7B의 **~8배**, 1.5B는 ~4배, 2B는 ~2.3배. 클수록 토큰(→용어설명)당 급격히 느려짐(M4b 축).
- **swap 워밍업**(첫 토큰 준비)이 크기에 비례 → 7B는 적재 후 준비에만 **~9.5s**. (load 자체는 mmap(→용어설명)+SSD라 ~0.5s로 작음)

## 3. 의사결정 연결 (DP03)

- **7B(4B+)는 MAF 예산 ~2GB를 단독으로도 초과** (Q4 4.3GB; 가장 빡센 Q2로도 ~2.8GB) → **온디바이스 MAF 단독 적재 불가.**
- **2B는 Q4로 ~1.9GB → ~2GB 예산의 상한선** (빠듯하게 들어감). Q3로 내리면 ~1.4GB 여유(품질 저하 감수).
- → **3티어 매핑이 자연스럽게 나옴**: 협상 = **2B급(상한)** / 판정·요약 = **1.5B(~1.1GB, 여유 있게 적합)** / 툴 호출 = **0.5B**. (가중치+KV(→용어설명) 기준이라 이 결론은 단말로 전이됨. 단 NPU(→용어설명) 오버헤드·실제 fit은 폰 확인 필요)
- **동시 상주(→용어설명) 추정**: IDS(0.5B)+MAF(2B) ≈ peak 합 ~2.4GB(빠듯), MAF를 1.5B로 하면 ~1.6GB(여유) → P2(2모델) 예산 판단 기준.

### 모델 체급·용도 (참고 — **품질 미측정**, 공개 벤치마크 기반)

> ⚠️ 본 PoC는 *자원만* 쟀고 **품질·정확도는 측정하지 않았다.** 아래는 측정값이 아니라 **공개 벤치마크·모델카드 기반 일반 지식**이며, 정확한 점수는 각 모델카드를 확인할 것. 특히 **협상 품질(제안·역제안·합의의 질)은 MMLU 같은 일반 점수로 안 잡히며, 별도 정확도 PoC([NFR-MAF-04](../../docs/05-NFR.md))가 답해야 한다.**

**상대 체급(대략)**: `0.5B ≪ 1.5B ≈ 2B ≪ 7B`
- 0.5B → 1.5B: 큰 도약(능력 차 확실).
- 1.5B ↔ 2B(Gemma-2-2B): *비슷한 티어*(둘 다 "강한 소형", Qwen2.5 소형·Gemma-2-2B 모두 체급 대비 셈). 격차가 0.5B→1.5B만큼 크지 않음.
- 7B: 다른 리그(복잡 추론·전략에서 확연히 앞섬, 단 메모리 초과로 사용 불가).

| 모델 | 잘하는 것 | 한계 | DP 티어 |
|---|---|---|---|
| **0.5B** | 의도 분류·라우팅, 간단 추출, 툴 호출(함수·인자 구성), 짧은 포맷 변환 | 다단계 추론·뉘앙스 약함 | 툴 호출(소형) |
| **1.5B** | + 중간 난이도 추론, 요약·판정, 구조화된 다중 툴 조율 | 복잡 전략·긴 맥락 일관성 한계 | 판정·요약(중형) |
| **2B (Gemma-2)** | + 더 일관된 다중 턴, 더 나은 요약/판정, 협상의 하한선 | 복잡한 전략 협상엔 *불충분할 수 있음* | 협상(대형·상한) |
| (7B) | 진짜 전략 협상·추론 | **메모리 초과 → 온디바이스 불가** | — |

**핵심 단서**: 위 표는 *크기↔용도*의 합리적 매핑이지만, **"2B로 협상이 실제로 충분한가"는 미지수**다. 안 되면 후보 A(협상 전용 모델 swap)나 LoRA(§3.5 of [DP 본문서](../../docs/07-DP후보안/DP03-MAF-LLM-배치.md))가 필요해지는 지점 → 정확도 PoC로 확정.

## 4. 방법론 주의 (이번 실행이 드러냄)

- **mmap 과소보고**: llama.cpp가 가중치를 **mmap으로 lazy 로드**해, "load +RSS(→용어설명)"와 **coresident 수치는 실제보다 작게** 나온다(추론 중 페이지가 채워짐). → **신뢰 값은 `peakRSS`** (추론 정점). coresident 실제값 ≈ 각 모델 peak의 합으로 봐야 함.
- **CPU ≠ NPU**: 속도 절대값과 런타임 오버헤드는 단말과 다름. *추세·메모리 스케일*만 전이.
- **TTFT cold vs warm**: 1회차(cold)는 일회성 워밍업 포함이라 큼(예: 7B 17.9s) → 정상 prefill(→용어설명) 지연은 warm 기준(~141ms).

## 5. 검증된 것 / 남은 것

**검증됨**
- 하네스가 R1·R2·R3 + 동시상주 + 요약/JSON까지 end-to-end 정상 작동.
- 의사결정에 쓸 신호(2B=상한, 7B=불가)를 실제로 산출.
- 버그 1건 발견·수정: Windows 한국어 콘솔(cp949)에서 기호 출력 시 `UnicodeEncodeError` → stdout/stderr UTF-8 재설정.
- 설치 함정 확인: 미리 빌드된 휠이 **AVX-512** 사용 → AVX2 CPU에서 `illegal instruction`(→용어설명). 소스 빌드(`AVX512=OFF`)로 해결.

**남은 것 (타깃 단말 필요)**
- 절대 fit 판정(P2)·NPU 런타임 오버헤드·속도 절대값 → 폰 + 벤더 NPU SDK(QAIRT 등).
- QAS-011 "MAF Memory 300MB" 불일치 최종 확정.

# DP03 A2A 협상 메시지 구조 PoC: 02 측정 프로토콜

> 본 문서는 [00-실험-계약](./00-실험-계약.md)과 [01-시나리오-스키마](./01-시나리오-스키마.md)에 따라 생성될 시나리오를 어떤 방식으로 실행하고 측정할지 정의한다.
> 본 문서는 측정 계약이며, NegMAS runner 구현, LLM prompt 구현, 실제 시나리오 데이터 생성은 포함하지 않는다.

---

## 1. 목적

측정할 설계 질문은 다음이다.

> 순수 NegMAS 메시지만 사용하는 후보 A와, A2A Envelope에 구조화된 `preference_hint`를 추가하는 후보 B 중 어느 쪽이 DP03의 품질속성 균형에 더 적합한가?

이 질문은 단일 지표로 답할 수 없다. 후보 B는 협상 효율을 높일 수 있지만 선호 정보 공개량, 스키마 관리, LLM 오해 가능성을 늘린다. 따라서 본 PoC는 아래 네 범주를 함께 측정한다.

| 범주 | 핵심 질문 |
|---|---|
| 협상 효율 | 후보 B가 후보 A보다 더 적은 라운드와 지연으로 합의하는가? |
| 합의 품질 | 빨라진 합의가 utility, 공동 효용, Pareto 효율을 훼손하지 않는가? |
| 개인정보 노출 | hint가 사유, 원문, PII를 직접 또는 누적으로 노출하지 않는가? |
| LLM 적용성 | LLM이 hint를 스키마에 맞게 생성·해석하고, 과대 해석하지 않는가? |

기존 `poc/dp03-privacy`의 데이터, 코드, 리포트, 측정 결과는 본 프로토콜의 기준값이나 근거로 사용하지 않는다.

## 3. 실험군

### Track A: 결정론적 NegMAS

| 실험군 | 후보 | 설명 |
|---|---|---|
| `A1_DET_OFFER_ONLY` | 후보 A | 순수 NegMAS 메시지. 상대 offer와 private utility만 사용 |
| `A2_DET_HINT_AWARE` | 후보 B | 동일한 기본 전략에 상대 hint를 후보 offer 정렬 또는 양보 축 선택에 반영 |
| `A3_DET_FALLBACK` | 후보 A fallback | 한쪽이 hint 미지원일 때 후보 A와 동일하게 동작하는지 확인 |

`A3_DET_FALLBACK`은 별도 설계 후보가 아니다. 후보 B가 미지원 상대를 만났을 때 순수 NegMAS로 정상 퇴화하는지 검증하기 위한 통제군이다.

### Track B: LLM 포함 NegMAS

| 실험군 | 후보 | 설명 |
|---|---|---|
| `B1_LLM_INTERNAL_ONLY` | 후보 A | LLM은 내부 offer 후보 선택에만 관여하고 외부 메시지는 순수 NegMAS 형태로 제한 |
| `B2_LLM_HINT_CONSUMER` | 후보 B | LLM이 상대의 구조화 hint를 입력으로 받아 offer 후보 정렬 또는 양보 축 선택에 활용 |
| `B3_LLM_HINT_GENERATOR` | 후보 B | LLM이 hint 후보를 생성하고 deterministic validator가 최종 허용 여부를 판단 |

`B4_DIRECT_LLM_NEGOTIATOR`는 1차 측정에서 제외한다. LLM이 `propose()`와 `respond()` 전체를 직접 판단하면 메시지 구조 효과와 모델 전략 효과가 섞여 DP 판단 근거가 약해진다.

## 4. 공통 통제 조건

모든 실험군은 다음 조건을 공유한다.

| 조건 | 값 또는 규칙 |
|---|---|
| 협상 형태 | 1:1 SAO 협상 |
| 참여자 | `ppa_a`, `ppa_b` |
| OutcomeSpace | 동일 scenario의 `domain.issues`에서 생성 |
| private profile | 동일 scenario의 `agents[].private_profile` 사용 |
| max steps | 1차 기본값 `30`, 구현계획에서 조정 가능 |
| timeout | 1차 기본값 없음. LLM 트랙은 호출 timeout 별도 적용 |
| utility model | `linear_additive` |
| hard constraint | 위반 outcome은 수락 불가 |
| reservation value | agreement utility가 양측 reservation 이상이어야 성공 합의 |
| 기존 PoC 결과 | 기준값, seed, scenario, metric으로 재사용 금지 |

max steps는 절대 기준이 아니라 비교를 위한 통제값이다. 구현 단계에서 NegMAS step 의미가 atomic action인지 round인지 확인한 뒤, 로그에는 둘을 분리해 기록한다.

## 5. 반복 실행 정책

### Track A 반복

결정론 트랙은 원칙적으로 각 시나리오와 실험군을 1회 실행한다.

| 항목 | 정책 |
|---|---|
| 실행 횟수 | `scenario_count * experiment_group` |
| seed | scenario의 `generation_meta.seed` 사용 |
| 반복 목적 | 구현 오류 또는 tie-breaker 영향 확인 시에만 추가 반복 |
| tie-breaker | 동일 utility 후보가 여러 개일 때 seed 기반 정렬로 결정 |

### Track B 반복

LLM 트랙은 동일 시나리오·동일 실험군을 최소 3회 반복한다.

| 항목 | 정책 |
|---|---|
| 기본 반복 수 | scenario당 3회 |
| 확장 반복 수 | 변동성이 큰 경우 scenario당 5회 |
| temperature | 가능한 경우 0 또는 최소값 |
| 모델 버전 | run_config에 고정 기록 |
| prompt 버전 | run_config에 고정 기록 |
| 실패 재시도 | schema 오류를 숨기지 않기 위해 자동 재시도는 별도 metric으로 기록 |

LLM 반복은 성능 평균을 좋게 보이기 위한 것이 아니라 변동성을 측정하기 위한 것이다. 동일 시나리오에서 합의 여부나 outcome이 크게 흔들리면 후보 B의 유지보수성 리스크로 해석한다.

## 6. Utility 계산

각 agent `i`의 outcome `o`에 대한 utility는 다음과 같이 계산한다.

```text
U_i(o) = sum(weight_i,k * value_score_i,k(o_k))
```

단, outcome이 agent의 hard constraint를 위반하면 해당 agent는 outcome을 수락할 수 없다.

| 개념 | 정의 |
|---|---|
| `feasible_outcome_i` | agent `i`의 hard constraint를 만족하는 outcome |
| `acceptable_outcome_i` | `feasible_outcome_i`이면서 `U_i(o) >= reservation_value_i` |
| `successful_agreement` | 양측 모두에게 `acceptable_outcome`인 outcome에 합의 |
| `failed_agreement` | agreement가 없거나, agreement가 한쪽 reservation/hard constraint를 위반 |

실험 구현에서 NegMAS가 agreement를 반환하더라도, 본 프로토콜의 `successful_agreement` 검증을 통과하지 못하면 실패로 기록한다.

## 7. 공통 지표

### 합의와 라운드 지표

| 지표 | 계산 방식 | 해석 |
|---|---|---|
| `agreement_success` | `successful_agreement`이면 1, 아니면 0 | 제한 내 유효 합의 여부 |
| `agreement_rate` | `sum(agreement_success) / run_count` | 실험군 전체 합의 성공률 |
| `rounds_to_agreement` | 합의가 발생한 round index | 낮을수록 협상 효율 좋음 |
| `atomic_actions_to_agreement` | offer/response 단위 action 수 | NegMAS step 해석 차이를 보완 |
| `timeout_rate` | timeout 또는 max step 도달 실패 비율 | 수렴 실패 정도 |
| `failure_reason` | 실패 원인 enum | 후보별 실패 양상 분석 |

`rounds_to_agreement`와 `atomic_actions_to_agreement`를 모두 기록한다. NegMAS 내부 step 정의와 제품 관점 round 정의가 다를 수 있기 때문이다.

### Utility와 합의 품질 지표

| 지표 | 계산 방식 | 해석 |
|---|---|---|
| `utility_a` | agreement outcome에 대한 `ppa_a` utility | A 참여자 만족도 |
| `utility_b` | agreement outcome에 대한 `ppa_b` utility | B 참여자 만족도 |
| `joint_utility` | `(utility_a + utility_b) / 2` | 공동 효용 |
| `utility_min` | `min(utility_a, utility_b)` | 한쪽 희생 여부 확인 |
| `fairness_gap` | `abs(utility_a - utility_b)` | 효용 불균형 |
| `reservation_margin_a` | `utility_a - reservation_value_a` | 최소 수락 기준 대비 여유 |
| `reservation_margin_b` | `utility_b - reservation_value_b` | 최소 수락 기준 대비 여유 |

후보 B가 라운드를 줄이더라도 `joint_utility`나 `utility_min`을 크게 낮추면 좋은 설계안으로 보기 어렵다.

### Pareto 지표

Pareto 계산은 scenario의 전체 OutcomeSpace를 열거해 수행한다.

| 지표 | 계산 방식 | 해석 |
|---|---|---|
| `pareto_dominated` | agreement보다 양측 utility가 모두 같거나 높고 한쪽은 더 높은 outcome이 있으면 true | 합의가 명백히 비효율적인지 |
| `pareto_joint_gap` | `max(joint_utility(pareto_outcomes)) - joint_utility(agreement)` | 최선 Pareto 후보 대비 공동 효용 손실 |
| `pareto_rank` | Pareto 후보를 joint utility 순으로 정렬했을 때 agreement 위치 | 합의 품질 순위 |

hard constraint를 위반하는 outcome은 Pareto 후보에서 제외한다. reservation value 미만 outcome은 별도로 표시하되, 기본 Pareto frontier는 feasible outcome 기준으로 계산한다.

## 8. Hint 지표

Hint 지표는 후보 B의 효율 개선이 어떤 정보 공개 비용을 동반했는지 보기 위한 것이다.

| 지표 | 계산 방식 | 해석 |
|---|---|---|
| `hint_message_count` | hint block이 포함된 메시지 수 | hint 사용 빈도 |
| `hint_field_count` | 공개된 hint field 총수 | 공개량 proxy |
| `unique_hint_issues` | hint에 등장한 issue 종류 수 | 공개 범위 |
| `hard_constraint_hint_count` | 공개된 hard constraint issue 수 | 민감한 선호 제약 공개량 |
| `concession_phase_count` | `concession_phase` 공개 횟수 | 전략 상태 공개량 |
| `hint_schema_violation_count` | schema 밖 hint 필드 또는 값 수 | 유지보수 위험 |
| `hint_fallback_leak_count` | fallback 상황에서 hint가 전송된 횟수 | 상호운용·기밀성 실패 |

### hint_sensitivity_score

`hint_sensitivity_score`는 절대적인 개인정보 위험 점수가 아니라 후보 A/B 비교를 위한 정보 공개량 proxy다.

기본 점수는 메시지 단위로 계산한다.

| 공개 항목 | 점수 |
|---|---:|
| `issue_preferences`의 issue 1개 | 1 |
| `issue_flexibility`의 issue 1개 | 1 |
| `hard_constraints`의 issue 1개 | 3 |
| `concession_phase` 1회 | 1 |
| schema 밖 hint field | 5 |
| 금지된 사유·원문·PII | hard fail |

scenario의 `privacy_labels.hint_accumulation_risk`에 따라 multiplier를 적용한다.

| risk | multiplier |
|---|---:|
| `low` | 1.0 |
| `medium` | 1.5 |
| `high` | 2.0 |

```text
hint_sensitivity_score =
  sum(message_hint_score) * risk_multiplier
```

반복된 동일 hint도 누적 점수에 포함한다. 반복 노출은 상대가 선호 패턴을 추론하는 근거가 될 수 있기 때문이다.

## 9. LLM 지표

LLM 지표는 Track B에서만 계산한다.

| 지표 | 계산 방식 | 해석 |
|---|---|---|
| `llm_call_count` | LLM 호출 횟수 | 비용과 지연의 원인 |
| `schema_valid_rate` | schema 검증 통과 출력 / 전체 LLM 출력 | 출력 안정성 |
| `invalid_outcome_rate` | OutcomeSpace 밖 offer 생성 / offer 생성 시도 | 협상 엔진 연결 안정성 |
| `prohibited_content_count` | 사유, 원문, PII, 금지 필드 생성 횟수 | 개인정보 hard fail |
| `hint_misuse_count` | hint를 hard constraint로 오해하거나 잘못 적용한 횟수 | hint 해석 위험 |
| `fallback_violation_count` | hint 미지원 상대에게 hint를 생성·전송한 횟수 | capability 처리 실패 |
| `llm_latency_ms` | LLM 호출별 지연 | 성능 비용 |
| `llm_token_in` | 입력 토큰 수 | 비용 proxy |
| `llm_token_out` | 출력 토큰 수 | 비용 proxy |
| `run_variance` | 동일 scenario 반복 결과의 변동성 | 재현성 위험 |

### hint_misuse 판정

다음 중 하나가 발생하면 `hint_misuse_count`를 증가시킨다.

| 유형 | 예 |
|---|---|
| preference 과대 해석 | `issue_preferences.slot = high`를 절대 조건처럼 취급 |
| flexibility 오해 | `issue_flexibility.area = high`인데 해당 축을 전혀 양보하지 않음 |
| hard constraint 무시 | 공개된 hard constraint issue에 위반되는 offer를 반복 생성 |
| issue 혼동 | hint에 없는 issue를 상대의 강한 선호로 가정 |
| 사유 생성 | hint에 없는 이유를 자연어로 만들어냄 |

hint misuse는 LLM 출력만으로 판단하지 않고, output validator와 실행 로그를 함께 보고 판정한다.

### run_variance 계산

동일 scenario와 실험군을 `n`회 반복했을 때 다음을 기록한다.

| 지표 | 계산 |
|---|---|
| `agreement_variance` | 합의 성공 여부의 분산 |
| `round_variance` | `rounds_to_agreement`의 분산 |
| `outcome_variance` | 서로 다른 agreement outcome 개수 |
| `utility_variance` | `joint_utility`의 분산 |

LLM 트랙에서 후보 B가 평균 성능은 좋아도 variance가 크면 운영 리스크가 높다고 해석한다.

## 10. 실패 사유 분류

모든 실패는 아래 enum 중 하나 이상으로 기록한다.

| failure_reason | 의미 |
|---|---|
| `no_agreement` | max step 내 합의 없음 |
| `timeout` | 시간 제한 초과 |
| `invalid_outcome` | OutcomeSpace 밖 offer 생성 |
| `reservation_violation` | agreement가 한쪽 reservation value 미만 |
| `hard_constraint_violation` | agreement 또는 offer가 hard constraint 위반 |
| `hint_schema_violation` | 허용되지 않은 hint field/value 생성 |
| `prohibited_content` | 사유, 원문, PII, 금지 정보 생성 |
| `fallback_violation` | hint 미지원 상대에게 hint 전송 |
| `llm_schema_violation` | LLM 출력이 지정 schema를 통과하지 못함 |
| `llm_call_error` | LLM 호출 실패 |
| `runtime_exception` | runner 또는 negotiator 예외 |

하나의 run에 여러 실패가 있으면 모두 기록하되, 집계 표에는 `primary_failure_reason`을 별도로 둔다.

## 11. 판정 기준 초안

본 기준은 PoC 결과 해석을 위한 초안이며, DP의 최종 결론을 자동으로 결정하지 않는다.

### 후보 B 검토 유지 조건

후보 B는 아래 조건을 모두 만족할 때 설계 후보로 유지할 수 있다.

| 조건 | 기준 초안 |
|---|---|
| 협상 효율 | 후보 A 대비 평균 또는 중앙값 `rounds_to_agreement`가 10% 이상 감소하거나, agreement rate가 5%p 이상 증가 |
| 합의 품질 | `joint_utility` 평균 감소가 5% 이내 |
| 한쪽 희생 방지 | `utility_min` 평균 감소가 5% 이내 |
| fallback | hint 미지원 scenario에서 `fallback_violation_count = 0` |
| 개인정보 hard gate | `prohibited_content_count = 0` |
| deterministic schema | Track A에서 `hint_schema_violation_count = 0` |

### 후보 B 제한 또는 기각 조건

아래 조건 중 하나라도 발생하면 후보 B는 제한 적용 또는 기각 대상으로 본다.

| 조건 | 해석 |
|---|---|
| 라운드 개선 < 10%이고 hint 공개량 증가 | 정보 공개 비용 대비 효율 개선 부족 |
| `prohibited_content_count > 0` | DP03 구조화 출구 원칙 위반 |
| fallback 상황에서 hint 전송 | 외부·미지원 PPA 상호운용 실패 |
| LLM schema valid rate < 95% | LLM 포함 구조의 운영 안정성 부족 |
| invalid outcome rate > 2% | NegMAS OutcomeSpace 연결 불안정 |
| hint misuse가 특정 task family에 반복 | LLM이 hint semantics를 안정적으로 이해하지 못함 |
| `pareto_joint_gap`이 후보 A보다 커짐 | 빠르지만 낮은 품질의 합의로 수렴 |

### 후보 A 유지 조건

후보 A는 아래 상황에서 기본값으로 유지한다.

- 후보 B의 라운드 수 또는 agreement rate 개선이 작다.
- 후보 B가 LLM 포함 트랙에서 schema violation 또는 hint misuse를 반복한다.
- 후보 B의 hint sensitivity score가 높지만 합의 품질 개선이 작다.
- 민감도가 높은 scenario에서 후보 B가 반복적인 hard constraint 공개를 요구한다.

## 12. 집계 방식

metric summary는 전체 평균만 내지 않고 다음 축으로 나누어 집계한다.

| 집계 축 | 이유 |
|---|---|
| 전체 | 후보 A/B의 총괄 비교 |
| `task_family`별 | 일정, 예약, 구매, 방문 서비스별 차이 확인 |
| `complexity_level`별 | issue 수가 늘어날 때 hint 효과가 커지는지 확인 |
| `tension_pattern`별 | hard constraint 또는 trade-off 유형별 효과 확인 |
| `hint_accumulation_risk`별 | 개인정보 위험이 높은 케이스에서 효율 개선이 충분한지 확인 |
| fallback 여부별 | 미지원 상대와의 상호운용성 확인 |

각 집계에는 최소한 다음 통계를 포함한다.

| 통계 | 대상 |
|---|---|
| mean | utility, rounds, score, latency |
| median | rounds, utility, latency |
| p95 | rounds, latency, hint_sensitivity_score |
| stddev | LLM 반복 실행 지표 |
| count | 성공/실패/위반 건수 |

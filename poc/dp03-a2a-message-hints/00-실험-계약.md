# DP03 A2A 협상 메시지 구조 PoC: 00 실험 계약

> 본 문서는 [DP03-선호-힌트-공개-여부](../../docs/07-DP후보안/DP03-선호-힌트-공개-여부.md)의 설계 판단을 검증하기 위한 PoC의 첫 계약 문서다.
> 기존 `poc/dp03-privacy` PoC의 데이터, 코드, 리포트, 측정 결과는 본 PoC의 근거 또는 입력으로 사용하지 않는다. 해당 PoC는 재설계 대상이다.

---

## 1. 목적

검증할 설계 질문은 다음이다.

> NegMAS 기반 A2A 협상에서 외부 메시지를 `response_type + outcome` 중심의 순수 NegMAS 메시지로 제한할 것인가, 아니면 상위 A2A Envelope에 구조화된 `preference_hint`를 추가할 것인가?

본 PoC의 목적은 특정 구현을 완성하는 것이 아니라, 위 설계 질문에 대해 다음 항목을 비교 가능한 형태로 측정하는 것이다.

| 관점 | 확인할 질문 |
|---|---|
| 협상 효율 | `preference_hint`가 합의까지의 라운드 수와 총 지연을 줄이는가? |
| Task 성공률 | `preference_hint`가 결렬률을 낮추거나 합의 가능 영역을 더 잘 찾게 하는가? |
| 합의 품질 | 빠른 합의가 한쪽 utility 또는 공동 효용을 과도하게 낮추지 않는가? |
| 개인정보 보호 | 힌트가 사유, 원문, PII를 직접 또는 누적으로 노출하지 않는가? |
| 상호운용성 | 힌트 미지원 상대와 순수 NegMAS 메시지로 fallback 가능한가? |
| LLM 적용성 | LLM이 포함된 협상에서도 힌트 생성/해석이 안정적으로 동작하는가? |

---

## 2. 비교 대상

### 후보 A: 순수 NegMAS 메시지

- 외부 전송 메시지는 NegMAS 협상 결과에 해당하는 `response_type`과 `outcome` 중심으로 제한한다.
- 각 참여자의 `ufun`, 유보값, 선호 강도, 양보 가능성은 외부로 보내지 않는다.
- LLM을 쓰는 경우에도 LLM은 내부 판단에만 관여하고, 외부 메시지에는 별도 선호 힌트를 포함하지 않는다.

### 후보 B: A2A Envelope + Preference Hint

- NegMAS 메커니즘 자체는 수정하지 않는다.
- NegMAS 메시지를 상위 A2A Envelope로 감싸고, 당사 기기간 capability가 확인된 경우에만 구조화된 `preference_hint` block을 추가한다.
- 힌트는 자유 텍스트가 아니라 enum, bool, bucket 중심의 제한된 스키마만 허용한다.
- 힌트 미지원 상대와 협상할 때는 후보 A 메시지로 fallback한다.

### 비교에서 제외하는 것

| 제외 항목 | 이유 |
|---|---|
| 기존 `poc/dp03-privacy` 결과 재사용 | 기존 PoC 자체가 재설계 대상이므로 본 설계 판단의 근거로 사용할 수 없음 |
| NegMAS `SAOMechanism` 자체 수정 | 라이브러리 프로토콜 확장이 아니라 제품 A2A 메시지 구조 비교가 목적 |
| 힌트를 OutcomeSpace issue로 추가 | 힌트가 합의 결과의 일부처럼 취급되어 의미가 흐려짐 |
| 자연어 사유 전송 | DP03의 구조화 출구 원칙에 맞지 않고 개인정보 노출 위험이 큼 |

## 3. 실험 원칙

본 PoC는 두 트랙으로 진행한다.

1. **Track A: 결정론적 NegMAS PoC**
   - 목적: `preference_hint` 유무의 효과를 다른 변수와 분리해 측정한다.
   - LLM을 사용하지 않는다.
   - 동일 시나리오, 동일 OutcomeSpace, 동일 utility profile, 동일 seed 조건에서 후보 A/B를 비교한다.

2. **Track B: LLM 포함 NegMAS PoC**
   - 목적: LLM이 포함된 실제 A2A 협상 흐름에서 힌트 생성/해석이 안정적으로 동작하는지 확인한다.
   - 결정론 실험을 대체하지 않고, 실제 적용성 검증 트랙으로 별도 운영한다.
   - 동일 시나리오를 사용하되, LLM 출력의 형식 준수율, 잘못된 해석, 개인정보 과노출 가능성을 추가로 측정한다.

공통 통제 조건은 다음과 같다.

| 통제 조건 | 내용 |
|---|---|
| OutcomeSpace | 후보 A/B에서 동일해야 한다. |
| Utility profile | 각 agent의 private utility, hard constraint, reservation value는 동일해야 한다. |
| 협상 제한 | 동일한 최대 라운드 수, timeout, fallback 규칙을 사용한다. |
| 시나리오 | 120~180개의 구조화 시나리오를 두 트랙에서 공유한다. |
| 개인정보 | 시나리오 값은 bucket, enum, synthetic value만 사용한다. 원문 PII는 만들지 않는다. |
| 기록 | offer, response, hint, validation result, agreement, failure reason을 구조화 로그로 남긴다. |

## 4. Track A: 결정론적 NegMAS PoC

### 목적

결정론 트랙은 `preference_hint` 자체가 협상 결과에 주는 영향을 보기 위한 기준선이다. LLM을 넣으면 모델 변동성, 프롬프트 민감도, 호출 지연, 비용이 결과에 섞이므로 이 트랙에서는 제외한다.

### 비교 구성

| 실험군 | 설명 |
|---|---|
| A-1: Offer-only negotiator | 상대 offer와 자신의 private utility만 보고 제안/수락한다. 외부 힌트 없음. |
| A-2: Hint-aware negotiator | 동일한 협상 전략을 쓰되, 상대가 보낸 구조화 힌트를 후보 offer 정렬 또는 양보 축 선택에 활용한다. |

### 구현 방향

- NegMAS의 SAO 협상 흐름을 사용한다.
- `SAONegotiator`를 상속한 custom negotiator를 작성한다.
- `propose()`는 현재 threshold 이상인 outcome 중 상대 힌트와 충돌이 적은 후보를 선택한다.
- `respond()`는 private utility와 reservation value 기준으로 결정한다.
- 후보 B에서도 NegMAS의 OutcomeSpace는 바꾸지 않고, 힌트는 negotiator 바깥의 envelope metadata 또는 별도 로그 입력으로 둔다.

### 측정 항목

| 항목 | 의미 |
|---|---|
| agreement_rate | 제한 라운드 내 합의 여부 |
| rounds_to_agreement | 합의까지 걸린 라운드 수 |
| own_utility | 각 참여자의 합의 utility |
| joint_utility | 양측 utility 합 또는 평균 |
| pareto_gap | 가능한 Pareto 후보 대비 합의 품질 차이 |
| message_count | 송수신 메시지 수 |
| hint_count | 공개된 힌트 수 |
| hint_sensitivity_score | 힌트 조합이 민감 사유를 암시할 가능성 |
| fallback_success | 힌트 미지원 상대와 순수 메시지로 정상 협상되는지 |

## 5. Track B: LLM 포함 NegMAS PoC

### 목적

LLM 포함 트랙은 "이론적으로 힌트가 유용한가"가 아니라 "LLM 기반 A2A 협상 참여자가 제한된 힌트를 안정적으로 생성하고 해석할 수 있는가"를 확인한다.

이 트랙이 필요한 이유는 다음과 같다.

- 실제 PPA 또는 Meta Agent가 LLM을 사용한다면, 구조화 힌트가 LLM의 판단 입력으로 들어갈 가능성이 높다.
- 후보 B는 단순 필드 추가가 아니라 LLM이 해석할 수 있는 메시지 계약을 요구한다.
- LLM이 힌트를 과대 해석하거나, 허용되지 않은 사유를 생성하거나, 스키마 밖 출력을 만들면 후보 B의 유지보수성과 기밀성이 약해진다.

### 비교 구성

| 실험군 | 설명 | 1차 포함 여부 |
|---|---|---:|
| B-1: Pure NegMAS + LLM internal | LLM은 자신의 private preference를 바탕으로 후보 offer를 고르지만, 외부 메시지는 순수 NegMAS 형태로 제한한다. | 포함 |
| B-2: Envelope + LLM hint consumer | LLM이 상대의 구조화 힌트를 입력으로 받아 후보 offer 정렬 또는 양보 축 선택에 활용한다. | 포함 |
| B-3: Envelope + LLM hint generator | LLM이 자신의 내부 선호에서 허용 가능한 hint 후보를 만들고, deterministic validator가 최종 허용 여부를 판단한다. | 포함 |
| B-4: Direct LLM negotiator | LLM이 `propose()`와 `respond()` 전체를 직접 판단한다. | 후순위 |

B-4는 기술적으로 가능하지만, 1차 실험의 핵심으로 두지 않는다. 직접 LLM negotiator는 재현성이 낮고 결과 해석이 어려우므로, B-1~B-3에서 메시지 구조의 적용성을 먼저 확인한 뒤 별도 확장 실험으로 다룬다.

### LLM 통제 조건

| 조건 | 내용 |
|---|---|
| 모델 버전 | 실험 시작 시 하나로 고정한다. cloud/local 여부는 별도 파일에서 확정한다. |
| temperature | 지원되는 경우 0 또는 최소값으로 둔다. |
| prompt | 후보 A/B별 시스템 프롬프트와 출력 스키마를 고정한다. |
| 출력 형식 | JSON schema 또는 Pydantic schema로 검증 가능한 형태만 허용한다. |
| 실패 처리 | schema 위반, 금지 필드 생성, invalid outcome은 fallback 또는 reject로 처리한다. |
| 반복 실행 | 동일 시나리오를 복수 회 실행해 LLM 변동성을 별도 지표로 기록한다. |

### 추가 측정 항목

| 항목 | 의미 |
|---|---|
| schema_valid_rate | LLM 출력이 지정 스키마를 지킨 비율 |
| invalid_outcome_rate | OutcomeSpace 밖 offer를 생성한 비율 |
| prohibited_content_count | 사유, 원문, PII, 금지 필드를 생성한 횟수 |
| hint_misuse_count | 힌트를 hard constraint처럼 과대 해석하거나 잘못 적용한 횟수 |
| run_variance | 동일 시나리오 반복 실행 시 결과 변동성 |
| llm_latency | LLM 호출로 증가한 지연 |
| llm_cost_proxy | 토큰 수 또는 호출 수 기반 비용 추정 |

## 6. 시나리오 데이터 마련 방식

### 기본 원칙

120~180개의 시나리오는 사람이 자연어로 임의 작성하지 않고, 스키마 기반으로 생성한다.

각 시나리오는 다음 세 층으로 구성한다.

| 층 | 내용 | 외부 공개 여부 |
|---|---|---:|
| Public negotiation domain | issue 이름, 허용 값, bucket, enum | 공개 가능 |
| Private preference profile | utility weight, reservation value, hard constraint, concession policy | 비공개 |
| Allowed hint projection | 외부로 보낼 수 있는 hint field와 bucket 변환 규칙 | 조건부 공개 |

### 수량 구성 초안

기본 세트는 120개로 시작하고, 변동성 검증이 필요하면 180개로 확장한다.

| 축 | 값 | 개수 |
|---|---|---:|
| task family | 일정 조율, 장소/예약 선택, 공동 구매, 서비스 방문 조건 협상 | 4 |
| complexity level | 3개 issue, 4개 issue, 5개 issue | 3 |
| tension pattern | 선호 일부 일치, 한쪽 hard constraint, 상호 hard constraint 충돌, 예산-품질 trade-off, 시간-장소 trade-off | 5 |
| seed/profile variant | 기본 2개, 확장 3개 | 2~3 |

계산식:

- 기본: `4 task families * 3 complexity levels * 5 tension patterns * 2 variants = 120`
- 확장: `4 task families * 3 complexity levels * 5 tension patterns * 3 variants = 180`

### 시나리오 예시 스키마

```yaml
scenario_id: S001
task_family: schedule_coordination
complexity_level: 4_issues
issues:
  - name: slot
    type: enum
    values: [weekday_evening, saturday_lunch, sunday_evening]
  - name: area
    type: enum
    values: [near_home, midpoint, near_office]
  - name: budget_band
    type: enum
    values: [low, medium, high]
  - name: notice_period
    type: enum
    values: [same_day, one_day, three_days]
agents:
  - id: ppa_a
    private_profile:
      utility_weights:
        slot: 0.40
        area: 0.25
        budget_band: 0.25
        notice_period: 0.10
      hard_constraints: [slot]
      reservation_value: 0.62
    allowed_hint_projection:
      slot_preference: high
      area_flexibility: medium
      budget_flexibility: low
  - id: ppa_b
    private_profile:
      utility_weights:
        slot: 0.25
        area: 0.35
        budget_band: 0.20
        notice_period: 0.20
      hard_constraints: [area]
      reservation_value: 0.58
    allowed_hint_projection:
      slot_preference: medium
      area_flexibility: low
      budget_flexibility: medium
privacy_labels:
  pii_raw: false
  sensitive_reason_present: false
  hint_accumulation_risk: medium
expected_checks:
  has_agreement_region: true
  fallback_required: false
```

## 7. 판정 기준 초안

본 기준은 기존 PoC 결과가 아니라 새 PoC의 판정 편의를 위한 초안이다. 구현 전 `01-시나리오-스키마.md` 또는 `02-측정-프로토콜.md`에서 수치 임계값은 조정할 수 있다.

### 후보 B를 검토할 수 있는 조건

- Track A에서 후보 B가 후보 A 대비 평균 라운드 수 또는 결렬률을 낮춘다.
- 합의 utility가 한쪽에만 과도하게 불리하게 치우치지 않는다.
- 공개된 hint가 사유, 원문, PII를 포함하지 않는다.
- 누적 hint 기반으로 민감 사유가 직접 추론되는 고위험 케이스가 통제 가능하다.
- hint 미지원 상대와 fallback이 정상 동작한다.

### 후보 B를 기각하거나 제한해야 하는 조건

- 라운드 수 개선이 작거나 일관되지 않은데, hint 공개량만 늘어난다.
- LLM이 hint를 hard constraint로 오해해 오히려 결렬률을 높인다.
- LLM이 허용되지 않은 사유, 원문, PII, 스키마 밖 필드를 반복 생성한다.
- fallback 조건에서 후보 A와 동일한 동작을 보장하지 못한다.
- hint schema, validator, 로그, 감사 비용이 협상 효율 개선보다 커진다.

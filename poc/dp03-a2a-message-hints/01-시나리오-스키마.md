# DP03 A2A 협상 메시지 구조 PoC: 01 시나리오 스키마

> 본 문서는 [00-실험-계약](./00-실험-계약.md)에 정의한 120~180개 구조화 시나리오를 생성하기 위한 스키마 계약이다.
> 목표는 결정론적 NegMAS PoC와 LLM 포함 NegMAS PoC가 동일한 시나리오를 사용하도록, public OutcomeSpace, private preference profile, allowed hint projection을 분리해 고정하는 것이다.

---

## 1. 범위

본 문서가 정의하는 것은 "시나리오 데이터의 모양"이다.

| 포함 | 제외 |
|---|---|
| 시나리오 상위 필드 | 실제 120~180개 데이터 파일 생성 |
| OutcomeSpace issue 스키마 | NegMAS runner 구현 |
| agent private profile 스키마 | LLM prompt 구현 |
| allowed hint projection 스키마 | 측정 지표 계산식 |
| privacy label과 검증 규칙 | 실험 결과 해석 |

기존 `poc/dp03-privacy`의 데이터, 코드, 리포트, 결과는 본 스키마의 입력으로 사용하지 않는다.

---

## 2. 핵심 분리 원칙

각 시나리오는 아래 세 층을 반드시 분리한다.

| 층 | 설명 | 외부 메시지 포함 가능 여부 |
|---|---|---:|
| `domain.issues` | NegMAS OutcomeSpace를 구성하는 public issue와 허용 값 | O |
| `agents[].private_profile` | agent 내부 utility, hard constraint, reservation value, concession policy | X |
| `agents[].allowed_hint_projection` | private profile에서 외부 hint로 변환 가능한 제한된 bucket/enum | 조건부 O |

이 분리가 깨지면 후보 A/B 비교가 의미를 잃는다. 특히 `private_profile`의 값이 그대로 `preference_hint`로 복사되면 안 된다.

---

## 3. 시나리오 상위 구조

시나리오 파일의 단위는 하나의 1:1 협상 세션이다. 1차 PoC에서는 참여자를 `ppa_a`, `ppa_b` 두 개로 고정한다.

```yaml
schema_version: scenario.v1
scenario_id: S001
task_family: schedule_coordination
complexity_level: issue_4
tension_pattern: one_hard_constraint
variant_id: v01
domain:
  issues: []
agents: []
privacy_labels: {}
expected_checks: {}
generation_meta: {}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `schema_version` | string | O | 시나리오 스키마 버전. 최초값은 `scenario.v1` |
| `scenario_id` | string | O | `S001` 형식의 고유 ID |
| `task_family` | enum | O | 협상 과업 유형 |
| `complexity_level` | enum | O | OutcomeSpace issue 개수 수준 |
| `tension_pattern` | enum | O | 선호 충돌 유형 |
| `variant_id` | string | O | 동일 축 조합 내 profile 변형 ID |
| `domain` | object | O | public OutcomeSpace 정의 |
| `agents` | list | O | agent별 capability와 private profile |
| `privacy_labels` | object | O | 시나리오의 개인정보 위험 라벨 |
| `expected_checks` | object | O | 생성 후 자동 검증해야 할 기대 조건 |
| `generation_meta` | object | O | 생성 seed, 생성 규칙 버전 등 추적 정보 |

## 4. 생성 매트릭스

기본 세트는 120개, 확장 세트는 180개로 둔다.

| 축 | 허용 값 | 개수 |
|---|---|---:|
| `task_family` | `schedule_coordination`, `venue_reservation`, `group_purchase`, `service_visit` | 4 |
| `complexity_level` | `issue_3`, `issue_4`, `issue_5` | 3 |
| `tension_pattern` | `aligned_preferences`, `one_hard_constraint`, `mutual_hard_constraint_conflict`, `budget_quality_tradeoff`, `time_location_tradeoff` | 5 |
| `variant_id` | 기본: `v01`, `v02` / 확장: `v01`, `v02`, `v03` | 2~3 |

계산식:

- 기본 세트: `4 * 3 * 5 * 2 = 120`
- 확장 세트: `4 * 3 * 5 * 3 = 180`

### task_family 정의

| 값 | 설명 | 대표 issue 후보 |
|---|---|---|
| `schedule_coordination` | 두 기기 간 일정 조율 | `slot`, `duration_band`, `notice_period`, `location_band`, `budget_band` |
| `venue_reservation` | 장소 또는 예약 조건 선택 | `slot`, `area`, `budget_band`, `crowding_level`, `option_level` |
| `group_purchase` | 공동 구매 조건 조율 | `price_band`, `quality_tier`, `delivery_window`, `warranty_level`, `privacy_option` |
| `service_visit` | 방문 서비스 조건 협상 | `visit_window`, `service_scope`, `access_method`, `notice_period`, `cost_band` |

### complexity_level 정의

| 값 | issue 개수 | 용도 |
|---|---:|---|
| `issue_3` | 3 | 단순 협상에서 hint 없이도 충분한지 확인 |
| `issue_4` | 4 | 다속성 협상의 기본 비교 |
| `issue_5` | 5 | hint가 수렴에 영향을 줄 가능성이 큰 복합 협상 |

### tension_pattern 정의

| 값 | 설명 | 생성 조건 |
|---|---|---|
| `aligned_preferences` | 양측 선호가 대체로 일치 | Pareto 후보가 넓고 reservation value가 낮음 |
| `one_hard_constraint` | 한쪽에만 절대 양보 불가 축 존재 | 한 agent의 `hard_constraints`에 1개 이상 포함 |
| `mutual_hard_constraint_conflict` | 양측 hard constraint가 다른 축에서 충돌 | 합의 가능 후보는 남기되 탐색이 어렵게 구성 |
| `budget_quality_tradeoff` | 비용과 품질 선호가 반대 | 한쪽은 낮은 비용, 다른 쪽은 높은 품질 선호 |
| `time_location_tradeoff` | 시간과 장소 선호가 반대 | 한쪽은 시간, 다른 쪽은 위치를 강하게 선호 |

## 5. domain.issues 스키마

`domain.issues`는 NegMAS OutcomeSpace로 변환 가능한 public 값만 포함한다.

```yaml
domain:
  issues:
    - name: slot
      type: enum
      values: [weekday_evening, saturday_lunch, sunday_evening]
      public: true
      outcome_space: true
      hintable: true
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `name` | string | O | issue 이름. snake_case 사용 |
| `type` | enum | O | `enum`, `ordinal`, `bucket` 중 하나 |
| `values` | list | O | OutcomeSpace에 들어갈 허용 값 |
| `public` | bool | O | 외부 offer에 포함 가능한 값인지 여부. 1차 PoC에서는 항상 `true` |
| `outcome_space` | bool | O | NegMAS OutcomeSpace 포함 여부. 1차 PoC에서는 항상 `true` |
| `hintable` | bool | O | 해당 issue에서 hint projection을 만들 수 있는지 여부 |
| `order` | list | 조건부 | `ordinal`, `bucket`일 때 낮음에서 높음 순서 |

### issue type 규칙

| type | 설명 | 예시 |
|---|---|---|
| `enum` | 순서 없는 선택지 | `slot`, `area`, `access_method` |
| `ordinal` | 선호 순서가 있는 단계형 값 | `quality_tier`, `warranty_level` |
| `bucket` | 정확한 수치 대신 구간화한 값 | `budget_band`, `cost_band`, `delivery_window` |

### 금지 규칙

- `values`에 실제 주소, 실명, 전화번호, 계좌, 정확한 일정 원문을 넣지 않는다.
- `values`에 질병, 종교, 가족사, 재정난 사유를 직접 넣지 않는다.
- 예산·비용은 정확한 금액이 아니라 `low`, `medium`, `high` 같은 bucket으로만 둔다.
- 위치는 주소가 아니라 `near_home`, `near_office`, `midpoint`, `remote` 같은 추상 값으로 둔다.

## 6. agents 스키마

각 시나리오는 정확히 두 agent를 가진다.

```yaml
agents:
  - id: ppa_a
    role: initiator
    capability:
      preference_hint: true
      hint_schema_version: hint.v1
    private_profile: {}
    allowed_hint_projection: {}
  - id: ppa_b
    role: responder
    capability:
      preference_hint: true
      hint_schema_version: hint.v1
    private_profile: {}
    allowed_hint_projection: {}
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `id` | enum | O | `ppa_a`, `ppa_b` 중 하나 |
| `role` | enum | O | `initiator`, `responder` |
| `capability` | object | O | hint 지원 여부와 schema version |
| `private_profile` | object | O | 외부 전송 금지 profile |
| `allowed_hint_projection` | object | O | 외부 hint로 변환 가능한 제한 값 |

### capability 규칙

| 필드 | 타입 | 설명 |
|---|---|---|
| `preference_hint` | bool | 후보 B hint 지원 여부 |
| `hint_schema_version` | string/null | 지원하는 hint schema. 미지원이면 `null` |

fallback 검증 시나리오에서는 한쪽 또는 양쪽의 `preference_hint`를 `false`로 둘 수 있다.

## 7. private_profile 스키마

`private_profile`은 agent 내부에서만 사용한다. 후보 A에서는 이 정보가 offer 선택과 accept/reject 판단에만 쓰이고, 후보 B에서도 직접 외부 전송하지 않는다.

```yaml
private_profile:
  utility_model: linear_additive
  utility_weights:
    slot: 0.40
    area: 0.25
    budget_band: 0.25
    notice_period: 0.10
  value_scores:
    slot:
      weekday_evening: 0.4
      saturday_lunch: 1.0
      sunday_evening: 0.7
    area:
      near_home: 1.0
      midpoint: 0.7
      near_office: 0.3
  hard_constraints:
    - issue: slot
      allowed_values: [saturday_lunch, sunday_evening]
  reservation_value: 0.62
  concession_policy:
    type: linear
    start_threshold: 0.92
    end_threshold: 0.62
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `utility_model` | enum | O | 1차 PoC에서는 `linear_additive` 고정 |
| `utility_weights` | map | O | issue별 가중치. 합계는 1.0 |
| `value_scores` | map | O | issue value별 utility 점수. 0.0~1.0 |
| `hard_constraints` | list | O | 수락 불가 조건. 없으면 빈 배열 |
| `reservation_value` | number | O | 수락 가능한 최소 utility. 0.0~1.0 |
| `concession_policy` | object | O | 라운드 진행에 따른 threshold 변화 |

### hard_constraints 규칙

```yaml
hard_constraints:
  - issue: budget_band
    allowed_values: [low, medium]
```

| 필드 | 타입 | 설명 |
|---|---|---|
| `issue` | string | 제약이 걸린 issue 이름 |
| `allowed_values` | list | 허용 가능한 값 목록 |

hard constraint에는 "왜 불가능한지"를 넣지 않는다. 예를 들어 "질병 때문에 특정 음식 불가", "가족 사정 때문에 특정 시간 불가" 같은 사유는 금지한다.

### concession_policy 규칙

| 필드 | 타입 | 허용 값 |
|---|---|---|
| `type` | enum | `linear`, `boulware`, `conceder` |
| `start_threshold` | number | 0.0~1.0 |
| `end_threshold` | number | `reservation_value` 이상, `start_threshold` 이하 |

1차 결정론 PoC에서는 같은 `type`과 threshold를 후보 A/B에 동일하게 적용한다.

## 8. allowed_hint_projection 스키마

`allowed_hint_projection`은 외부로 보낼 수 있는 hint의 최대 범위를 정의한다. 실제로 어떤 라운드에 어떤 hint를 보낼지는 runner 또는 negotiator가 결정하지만, 여기서 허용하지 않은 hint는 생성할 수 없다.

```yaml
allowed_hint_projection:
  schema_version: hint.v1
  issue_preferences:
    slot: high
    area: medium
    budget_band: low
  issue_flexibility:
    slot: low
    area: medium
    budget_band: high
  hard_constraints:
    - slot
  concession_phase_allowed: true
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `schema_version` | string | O | 최초값은 `hint.v1` |
| `issue_preferences` | map | O | issue별 선호 강도. 값은 `high`, `medium`, `low` |
| `issue_flexibility` | map | O | issue별 양보 가능성. 값은 `high`, `medium`, `low` |
| `hard_constraints` | list | O | 공개 가능한 hard constraint issue 이름만 포함 |
| `concession_phase_allowed` | bool | O | 런타임 `concession_phase` 공개 허용 여부 |

### hint 값 의미

| 값 | `issue_preferences` 의미 | `issue_flexibility` 의미 |
|---|---|---|
| `high` | 해당 issue가 utility에 큰 영향을 줌 | 양보 가능성이 큼 |
| `medium` | 중간 영향 | 제한적으로 양보 가능 |
| `low` | 낮은 영향 | 양보 가능성이 낮음 |

주의할 점은 `preference=high`와 `flexibility=low`가 서로 다른 의미라는 것이다. 전자는 중요도를, 후자는 양보 가능성을 나타낸다.

### projection 생성 규칙

| private 값 | hint 변환 |
|---|---|
| `utility_weights[issue] >= 0.35` | `issue_preferences[issue] = high` |
| `0.20 <= utility_weights[issue] < 0.35` | `issue_preferences[issue] = medium` |
| `utility_weights[issue] < 0.20` | `issue_preferences[issue] = low` |
| `issue in hard_constraints` | `issue_flexibility[issue] = low` 및 `hard_constraints`에 issue 이름만 포함 |
| hard constraint가 없고 value_scores 차이가 작음 | `issue_flexibility[issue] = high` |
| 그 외 | `issue_flexibility[issue] = medium` |

이 규칙은 1차 스키마의 기본값이다. 실제 수치 임계값은 `02-측정-프로토콜.md`에서 실험 파라미터로 분리할 수 있다.

### hint 금지 규칙

- private utility score 원값을 공개하지 않는다.
- reservation value를 공개하지 않는다.
- hard constraint의 원인이나 사유를 공개하지 않는다.
- 정확한 금액, 정확한 위치, 정확한 일정 원문을 공개하지 않는다.
- LLM이 생성한 자연어 설명을 hint field에 넣지 않는다.

## 9. privacy_labels 스키마

`privacy_labels`는 시나리오 자체가 개인정보 원칙을 어기지 않는지, 그리고 hint 누적 위험이 어느 정도인지 표시한다.

```yaml
privacy_labels:
  pii_raw: false
  sensitive_reason_present: false
  exact_value_present: false
  hint_accumulation_risk: medium
  external_hint_allowed: true
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `pii_raw` | bool | O | 실제 PII 원문 포함 여부. 항상 `false`여야 함 |
| `sensitive_reason_present` | bool | O | 질병, 종교, 가족사 등 민감 사유 포함 여부. 항상 `false`여야 함 |
| `exact_value_present` | bool | O | 정확 금액, 주소, 시간 원문 포함 여부. 항상 `false`여야 함 |
| `hint_accumulation_risk` | enum | O | `low`, `medium`, `high` |
| `external_hint_allowed` | bool | O | 후보 B에서 hint 전송 허용 여부 |

### hint_accumulation_risk 기준

| 값 | 기준 |
|---|---|
| `low` | hint가 하나의 선호 축만 약하게 드러냄 |
| `medium` | hard constraint 또는 낮은 flexibility가 포함되지만 사유 추론은 직접적이지 않음 |
| `high` | 여러 hint 조합이 특정 생활 패턴이나 민감 사유를 강하게 암시할 수 있음 |

`high` 시나리오는 후보 B를 무조건 금지하기 위한 것이 아니라, privacy validator와 결과 해석이 해당 위험을 식별하는지 확인하기 위한 테스트 케이스로 제한적으로 포함한다.

## 10. expected_checks 스키마

`expected_checks`는 시나리오 생성 후 자동 검증할 조건이다.

```yaml
expected_checks:
  has_agreement_region: true
  expected_fallback: false
  expected_timeout_possible: false
  min_valid_outcomes: 3
  min_pareto_candidates: 1
  llm_schema_required: true
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `has_agreement_region` | bool | O | 양측 reservation value 이상인 outcome이 존재해야 하는지 |
| `expected_fallback` | bool | O | hint capability 미지원으로 fallback이 기대되는지 |
| `expected_timeout_possible` | bool | O | 합의 가능 영역이 좁아 timeout이 가능하도록 설계한 케이스인지 |
| `min_valid_outcomes` | int | O | hard constraint 적용 후 남아야 하는 최소 outcome 수 |
| `min_pareto_candidates` | int | O | Pareto 후보 최소 개수 |
| `llm_schema_required` | bool | O | LLM 포함 트랙에서 JSON/schema 검증 필수 여부 |

1차 기본 세트 120개 중 대부분은 `has_agreement_region: true`여야 한다. 단, fallback과 실패 처리 검증을 위해 일부는 `expected_timeout_possible: true`로 둘 수 있다.

## 11. generation_meta 스키마

```yaml
generation_meta:
  generator_version: gen.v1
  seed: 1001
  source: synthetic
  created_from_legacy_poc: false
```

| 필드 | 타입 | 필수 | 설명 |
|---|---|---:|---|
| `generator_version` | string | O | 시나리오 생성 규칙 버전 |
| `seed` | int | O | 재현 가능한 profile 생성을 위한 seed |
| `source` | enum | O | 1차 PoC에서는 `synthetic` 고정 |
| `created_from_legacy_poc` | bool | O | 항상 `false` |

## 12. 예시 시나리오

### 12.1 정상 합의 가능 케이스

```yaml
schema_version: scenario.v1
scenario_id: S001
task_family: schedule_coordination
complexity_level: issue_4
tension_pattern: one_hard_constraint
variant_id: v01
domain:
  issues:
    - name: slot
      type: enum
      values: [weekday_evening, saturday_lunch, sunday_evening]
      public: true
      outcome_space: true
      hintable: true
```

## 13. 자동 검증 규칙

시나리오 생성기는 각 시나리오에 대해 최소한 다음 검증을 통과해야 한다.

| 검증 | 규칙 |
|---|---|
| ID 고유성 | `scenario_id`는 전체 데이터셋에서 중복되지 않아야 함 |
| issue 개수 | `complexity_level`과 `domain.issues` 개수가 일치해야 함 |
| weight 합계 | 각 agent의 `utility_weights` 합계는 1.0이어야 함 |
| score 범위 | 모든 `value_scores`는 0.0~1.0 사이여야 함 |
| issue 참조 | `utility_weights`, `value_scores`, `hard_constraints`, `allowed_hint_projection`의 issue 이름은 `domain.issues`에 존재해야 함 |
| hard constraint 유효성 | `allowed_values`는 해당 issue의 `values`에 포함되어야 함 |
| agreement region | `has_agreement_region: true`이면 양측 reservation value를 모두 만족하는 outcome이 최소 1개 있어야 함 |
| PII 금지 | `pii_raw`, `sensitive_reason_present`, `exact_value_present`는 모두 `false`여야 함 |
| fallback 일관성 | 어느 한쪽이라도 `capability.preference_hint: false`이면 `expected_fallback: true`여야 함 |
| hint projection 제한 | hint에는 issue 이름과 `high`/`medium`/`low` bucket만 포함되어야 함 |

## 14. 저장 형식 제안

실제 데이터 생성 단계에서는 아래 둘 중 하나를 선택한다.

| 형식 | 장점 | 단점 |
|---|---|---|
| `scenarios/*.yaml` | 사람이 개별 시나리오를 검토하기 쉬움 | 120~180개 파일 관리가 번거로움 |
| `data/scenarios.jsonl` | runner가 읽기 쉽고 대량 처리에 적합 | 사람이 개별 diff를 읽기 어려움 |

제안은 둘을 함께 쓰는 것이다.

- 원본 검토용: `scenarios/*.yaml`
- 실행용 변환본: `data/scenarios.jsonl`

단, 이 문서에서는 디렉터리와 데이터 파일을 아직 만들지 않는다. 생성기는 다음 단계에서 별도 합의 후 작성한다.

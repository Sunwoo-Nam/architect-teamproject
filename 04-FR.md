## 버전 A — 그룹별 정리본 (FR-X01 형식)

### 1. Intent 인식 (IDS)

**FR-I01 | 명시적/비명시적 Intent 인식**

IDS는 사용자 입력, 대화 문맥, 디바이스 내 이벤트를 분석하여 명시적 및 비명시적 intent를 감지할 수 있어야 한다.

**FR-I02 | Intent 구조화**

IDS는 감지된 intent를 Intent, Parameter, Constraint로 구성된 정규화된 구조로 변환할 수 있어야 한다.

**FR-I03 | Intent 신뢰도 평가**

IDS는 감지된 각 intent에 대해 confidence score와 intent type을 생성할 수 있어야 한다.

**FR-I04 | Intent Routing**

IDS는 감지된 intent가 ZeroClaw 자체로 처리 가능한지 판단하고, 그렇지 않은 경우 Multi-Agent Framework로 라우팅할 수 있어야 한다.

**FR-I05 | Task Payload 변환**

IDS는 multi-agent 처리가 필요한 intent를 정규화된 task initiation payload로 변환하여 Orchestrator에 전달할 수 있어야 한다.

---

### 2. Task Planning & Orchestration

**FR-O01 | Task/Topic 생성 및 상태 추적**

시스템은 intent로부터 task/topic을 생성하고 그 생명주기 상태를 추적할 수 있어야 한다.

**FR-O02 | Sub-task 분해**

Orchestrator는 수신한 task를 실행 가능한 sub-task plan으로 분해할 수 있어야 한다.

**FR-O03 | Agent Group 선택·생성**

Orchestrator는 sub-task 수행에 적합한 agent group을 선택하거나, 존재하지 않을 경우 동적으로 생성할 수 있어야 한다.

**FR-O04 | 권한 범위 계산**

Orchestrator는 각 task 수행에 필요한 권한 scope를 계산할 수 있어야 한다.

**FR-O05 | 실행 전략 결정**

Orchestrator는 sub-task에 대해 순차/병렬/조건부 등 적절한 실행 전략을 결정할 수 있어야 한다.

---

### 3. Agent Lifecycle Management

**FR-L01 | Agent 선택**

시스템은 현재 활성화된 agent 및 skill 목록에서 task에 적합한 agent를 선택할 수 있어야 한다.

**FR-L02 | Dynamic Agent 생성**

시스템은 복합 task 수행을 위해 Meta Agent, Topic Agent, Sub-Agent 및 Tool을 동적으로 생성할 수 있어야 한다.

**FR-L03 | Agent Team 구성**

시스템은 planner, schedule, booking, file manager 등 복수의 agent를 task 중심의 임시 팀으로 구성할 수 있어야 한다.

**FR-L04 | Agent 메타데이터 관리**

각 agent는 role, capability, toolset, scope, TTL(time-to-live)과 같은 메타데이터를 보유해야 한다.

**FR-L05 | Sandbox 격리 실행**

동적으로 생성된 agent 및 tool은 sandbox 환경 내에서 격리되어 실행되어야 한다.

**FR-L06 | Lifecycle 상태 관리**

시스템은 생성된 agent의 생성, 활성, 재사용, 중지, 종료 상태 전이를 관리할 수 있어야 한다.

**FR-L07 | Task 종료 후 Cleanup**

Task 종료 시 시스템은 불필요한 agent 및 context를 안전하게 정리할 수 있어야 한다.

---

### 4. Discovery & Secure Connection

**FR-D01 | Agent Endpoint 등록**

시스템은 Personal Proxy Agent 및 Service Proxy Agent의 endpoint와 capability 메타데이터를 등록할 수 있어야 한다.

**FR-D02 | Agent Discovery**

각 agent는 P2P 및 cloud registry 기반으로 상대 proxy agent의 endpoint를 탐색하고 발견할 수 있어야 한다.

**FR-D03 | Secure Connection 수립**

Meta Agent는 타 기기의 Personal Proxy Agent와 상호 인증을 수행한 후 암호화된 보안 채널을 수립할 수 있어야 한다.

**FR-D04 | Capability 조회**

시스템은 연결된 상대 agent의 capability 및 협상 가능 범위를 조회할 수 있어야 한다.

---

### 5. Negotiation

**FR-N01 | Negotiation Session 생성**

시스템은 agent 간 협상을 위한 negotiation session을 생성하고 traceable하게 추적할 수 있어야 한다.

**FR-N02 | 협상 프로토콜 지원**

시스템은 offer, counter-offer, accept, reject, defer, request-more-info, close 등의 협상 상태 전이를 지원해야 한다.

**FR-N03 | 정책 기반 정보 공유**

협상은 사전 정의된 정책에 따라 최소한의 정보만 공유하는 방식으로 수행되어야 한다.

**FR-N04 | 제약 기반 협상**

시스템은 시간, 장소, 비용, 우선순위, 선호도, 제한조건을 반영하여 협상을 수행할 수 있어야 한다.

**FR-N05 | Multi-turn 협상**

시스템은 단일 턴뿐 아니라 다중 턴(multi-turn) 협상을 지원해야 한다.

**FR-N06 | 합의안 도출**

시스템은 협상 결과 최적 또는 만족 가능한 합의안을 도출할 수 있어야 한다.

**FR-N07 | 협상 실패 처리**

시스템은 협상 실패 시 실패 원인을 분석하고 대체안을 사용자에게 제시할 수 있어야 한다.

---

### 6. Tool Calling & Execution

**FR-T01 | Scoped Tool 호출**

Sub-Agent는 승인된 task scope 범위 내에서만 tool을 호출할 수 있어야 한다.

**FR-T02 | 구조화된 Observation 반환**

Tool 호출 결과는 구조화된 observation 형태로 반환되어야 한다.

**FR-T03 | Tool 호출 실패 복구**

시스템은 실패한 tool 호출에 대해 재시도, 대체 tool 사용, 또는 사용자 escalation을 지원해야 한다.

**FR-T04 | Side-effect Action 통제**

Side-effect가 있는 action은 정책에 따라 사전 승인 또는 사후 통지를 거쳐 수행되어야 한다.

**FR-T05 | 후속 Action 실행**

사용자가 합의안을 승인하면 시스템은 캘린더 등록, 예약 실행, 알림 생성 등의 후속 action을 수행할 수 있어야 한다.

---

### 7. Human-in-the-Loop & User Control

**FR-U01 | 승인 요청 생성**

시스템은 금액 초과, 민감 정보 공유, 일정 충돌, 정책 위반 등 특정 임계치 초과 시 Human-in-the-Loop 승인 요청을 생성해야 한다.

**FR-U02 | 협상 과정 가시화**

시스템은 협상 진행 상태, 주요 제안, 상대 응답, 남은 쟁점을 사용자 UI에 표시해야 한다.

**FR-U03 | Rationale 설명**

시스템은 agent가 해당 task를 시작한 이유를 사용자에게 설명할 수 있어야 한다.

**FR-U04 | 협상 결과 설명 가능성**

시스템은 협상 결과와 대안을 사용자가 이해할 수 있는 형태로 제시해야 한다.

**FR-U05 | Task 제어**

사용자는 진행 중인 task를 언제든 중단, 수정, 철회할 수 있어야 한다.

**FR-U06 | 사용자 조건 수정 반영**

사용자는 협상 중 조건 수정, 우선순위 변경, 특정 옵션 제외를 요청할 수 있어야 하며, 시스템은 이를 협상에 즉시 반영해야 한다.

---

### 8. Multi-Device Coordination

**FR-M01 | Device Group 구성**

개인이 보유한 여러 기기는 Device Proxy Agent를 통해 하나의 그룹으로 묶일 수 있어야 한다.

**FR-M02 | Cross-device Context 질의**

Personal Proxy Agent는 여러 Device Proxy Agent로부터 context 및 capability를 질의할 수 있어야 한다.

**FR-M03 | 기기 상태 실시간 반영**

기기의 상태 변화는 진행 중인 task에 실시간으로 반영되어야 한다.

---

### 9. Personalization & Self-improving

**FR-P01 | 개인화 메모리 활용**

시스템은 사용자의 선호, 과거 결정, 협상 이력을 활용하여 개인화된 판단을 수행할 수 있어야 한다.

**FR-P02 | 협상 에피소드 기록**

시스템은 협상 세션별 사건, 결정 근거, 결과를 기록할 수 있어야 한다.

**FR-P03 | Self-improving 전략 개선**

시스템은 반복되는 사용자 피드백과 패턴을 기반으로 판단 및 협상 전략을 스스로 개선할 수 있어야 한다.

---

### 10. Proactive Action

**FR-A01 | 선제적 액션 수행**

시스템은 사용자가 명시적 요청을 하지 않아도, 수집된 정보를 기반으로 능동적으로 작업을 생성하고 수행할 수 있어야 한다.

---

### 11. Security, Privacy & Delegation

**FR-S01 | 권한 위임 관리**

사용자는 agent에 대해 일정 조회, 예약 수행, 상대와의 협상 등 세부 권한을 위임하거나 철회할 수 있어야 한다.

**FR-S02 | 최소 정보 공개**

시스템은 협상 및 실행에 필요한 최소 수준의 정보만 외부 agent 또는 서비스에 전달해야 한다.

---

### 12. Audit & Traceability

**FR-X01 | 감사 로그 기록**

시스템은 모든 agent action에 대해 누가, 어떤 권한으로, 어떤 정보를 교환했고, 어떤 결과를 확정했는지를 추적 가능한 감사 로그로 남겨야 한다.

**FR-X02 | 민감 Action 이력 관리**

민감한 action은 rationale, policy basis, approval trail을 포함하여 기록되어야 한다.

**FR-X03 | Task Replay & Postmortem**

시스템은 관리자 또는 사용자 관점에서 task replay 또는 postmortem 분석을 지원할 수 있어야 한다.

| FR07 | Agent Team 구성 | 시스템은 planner, schedule, booking, file manager 등 복수 agent를 task 중심으로 팀으로 구성할 수 있어야 한다. |
| FR08 | Agent Lifecycle 관리 | 시스템은 생성된 agent의 생성, 활성, 재사용, 중지, 종료 상태를 관리할 수 있어야 한다. |
| FR09 | Agent Endpoint 등록 | 시스템은 Personal Proxy Agent 및 Service Proxy Agent의 endpoint와 capability 메타데이터를 등록할 수 있어야 한다. |
| FR10 | Agent Discovery | 시스템은 특정 사용자 또는 서비스에 대응하는 proxy agent를 탐색할 수 있어야 한다. |
| FR11 | Secure Connection | 시스템은 agent 간 연결 전에 상호 인증을 수행하고 보안 채널을 수립할 수 있어야 한다. |
| FR12 | Capability 조회 | 시스템은 연결된 상대 agent의 capability 및 협상 가능 범위를 조회할 수 있어야 한다. |
| FR13 | Negotiation Session 생성 | 시스템은 agent 간 협상을 위한 negotiation session을 생성하고 추적할 수 있어야 한다. |
| FR14 | 협상 프로토콜 지원 | 시스템은 offer, counter-offer, accept, reject, defer, close 등의 상태 전이를 지원해야 한다. |
| FR15 | 제약 기반 협상 | 시스템은 시간, 장소, 비용, 우선순위, 선호도, 제한조건을 반영하여 협상할 수 있어야 한다. |
| FR16 | Multi-turn 협상 | 시스템은 단일 턴뿐 아니라 다중 턴 협상을 지원해야 한다. |
| FR17 | 합의안 도출 | 시스템은 협상 결과 최적 또는 만족 가능한 합의안을 도출할 수 있어야 한다. |
| FR18 | 협상 실패 처리 | 시스템은 협상 실패 시 실패 원인과 대체안을 사용자에게 제시할 수 있어야 한다. |
| FR19 | 승인 요청 생성 | 시스템은 금액 초과, 민감 정보 공유, 일정 충돌, 정책 위반 가능성 등 특정 조건에서 approval 요청을 생성해야 한다. |
| FR20 | 협상 과정 가시화 | 시스템은 협상 진행 상태, 주요 제안, 상대 응답, 남은 쟁점을 사용자에게 표시해야 한다. |
| FR21 | 사용자 조건 수정 반영 | 사용자는 협상 중 조건 수정, 우선순위 변경, 특정 옵션 제외를 요청할 수 있어야 하며 시스템은 이를 반영해야 한다. |
| FR22 | 결과 확정 및 실행 | 사용자가 합의안을 승인하면 시스템은 캘린더 등록, 예약 실행, 알림 생성 등의 후속 action을 수행할 수 있어야 한다. |
| FR23 | 개인화 메모리 활용 | 시스템은 사용자 선호, 과거 결정, 협상 이력을 활용해 개인화된 판단을 할 수 있어야 한다. |
| FR24 | 협상 에피소드 기록 | 시스템은 협상 세션별 사건, 결정 근거, 결과를 기록할 수 있어야 한다. |
| FR25 | Self-improving 반영 | 시스템은 반복되는 사용자 피드백과 패턴을 기반으로 판단 또는 협상 전략을 개선할 수 있어야 한다. |
| FR26 | 권한 위임 관리 | 사용자는 agent에 대해 일정 조회, 예약 수행, 상대와의 협상 등 세부 권한을 위임하거나 철회할 수 있어야 한다. |
| FR27 | 최소 정보 공개 | 시스템은 협상 및 실행에 필요한 최소 수준의 정보만 외부 agent 또는 서비스에 전달해야 한다. |
| FR28 | 감사 로그 | 시스템은 누가 어떤 권한으로 어떤 정보를 교환했고 어떤 결과를 확정했는지 추적 가능해야 한다. |

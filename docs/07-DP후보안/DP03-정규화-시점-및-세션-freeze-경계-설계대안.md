# DP03 — 정규화 시점 및 세션 freeze 경계 설계대안

> 본 문서는 [DP03-전제-및-공통설계](./DP03-전제-및-공통설계.md)를 전제로, 개인정보 처리 게이트웨이의 핵심 비교축을 **원본 입력을 언제 안전 사실값으로 바꾸고, 언제 세션용 OutcomeSpace를 고정할 것인가**로 둔다.
> 기존 [DP03-개인정보-처리-게이트웨이-구조](./DP03-개인정보-처리-게이트웨이-구조.md)의 모놀리식/연합형 비교는 실행 배치의 차이를 다루지만, 개인정보 안전 보장의 본질적 차이는 이 문서의 정규화 시점과 freeze 경계에서 더 크게 드러난다.

---

## 1. 아키텍처적 난제

DP03의 공통 전제는 다음과 같다.

- 밖으로 나가는 모든 메시지는 구조화되어야 한다.
- 사유·PII·원문은 탐지가 아니라 어휘·인터페이스에서 배제한다.
- NegMAS 협상은 세션 시작 전에 고정된 OutcomeSpace 위에서만 진행된다.
- OutcomeSpace는 승인된 Schema Registry의 issue/value 범위 안에서만 구성된다.

이 전제에서 실제 설계 난제는 **원본 안전화 시점**과 **OutcomeSpace freeze 시점**을 어디에 둘 것인가다.

| 책임 | 설명 | 산출물 |
|---|---|---|
| **원본 안전화** | PII 토큰화, 사유 제거, 원문 제거, 권한 범위 확인, 과도한 정밀도 축소 | 안전 사실값, Vault 참조 |
| **세션 어휘 매핑** | 안전 사실값을 승인된 issue/value 스키마에 맞춰 세션용 OutcomeSpace로 구성 | OutcomeSpace, ufun 입력, 구조화 계획 요청 |

두 후보 모두 **입력 직후 원본 안전화**를 수행한다. 차이는 그 다음 단계다.

| 쟁점 | 후보 A 측면 | 후보 B 측면 |
|---|---|---|
| **issue/value 확정 시점** | 입력 직후 또는 세션 준비 초기에 승인 issue/value 후보까지 만든다. | 입력 직후에는 안전 사실값까지만 만들고, 세션 직전에 OutcomeSpace를 구성한다. |
| **NegMAS freeze 경계** | 조기 후보 생성과 최종 freeze 사이에 후보 무효화·재계산 관리가 필요하다. | 승인 issue/value 선택과 freeze가 같은 세션 경계에 모인다. |
| **구현 단순성** | 흐름이 짧고 초기 구현이 단순하다. | 안전 사실값 스키마와 issue/value 매핑 계약이 추가된다. |
| **도메인 변경 대응** | Schema Registry 변경이 입력 정규화 단계까지 전파될 수 있다. | 안전 사실값이 안정적이면 출력 빌더·Registry 변경으로 국소화할 여지가 있다. |

## 2. 후보 A — 완전 조기 정규화

### 개념

IDS Intent, Sub-Agent 결과, DPA 상태가 들어오는 즉시 정규화기가 원본을 안전 사실값으로 바꾸고, 가능한 경우 승인 issue/value 후보까지 매핑한다. 이후 출력 빌더는 이미 정규화된 값들을 모아 세션용 OutcomeSpace를 구성하고 freeze한다.

### 처리 흐름

| 단계 | 처리 |
|---|---|
| 1 | 입력 발생: IDS Intent, Tool/DPA 데이터 수신 |
| 2 | 즉시 정규화: PII는 Vault로 이동, 사유·원문 제거, 안전 사실값 생성 |
| 3 | 조기 어휘 매핑: 승인 스키마에 맞는 issue/value 후보 생성 |
| 4 | 세션 시작 전: 후보들을 검증해 OutcomeSpace freeze |

### 예시

| 원본 입력 | 조기 정규화 산출 |
|---|---|
| "당뇨 때문에 오후 카페인은 피하고 싶다" | `diet_constraint=있음`, `stimulant_after_noon=회피`, 사유 `당뇨`는 ufun 내부 요인으로만 흡수 |
| "박민수와 15시에 만남" | `participant=PII_TOKEN_001`, `meeting_time=15:00`, 이름 원문은 Vault에만 저장 |

### 강점

- **초기 구현 단순성:** 원본 안전화와 issue/value 후보 생성을 한 경로에서 처리하므로 흐름이 짧다.
- **감사 지점 단순화:** 입력 정규화 결과만 확인하면 후속 출력 빌드의 입력 범위가 비교적 명확하다.
- **프롬프트 축소 가능성:** PoC 자원·지연 리포트에서 확인된 것처럼, LLM 추론 비용은 프라이버시 처리 CPU가 아니라 입력 토큰 수에 크게 좌우된다. 원본을 조기에 제거하면 후속 LLM 입력을 작게 유지하기 쉽다.

### 약점

- **조기 어휘 결합:** 입력 단계에서 승인 issue/value를 알아야 하므로, 데이터 출처 또는 어댑터가 Schema Registry 변화에 더 민감해진다.
- **후보 무효화 관리:** 입력이 갱신되거나 권한 범위가 바뀌면 이미 만든 안전값과 후보 issue/value를 다시 계산해야 한다.
- **세션 문맥 부족 가능성:** 세션 문맥이 완전히 모이기 전에 정밀도나 분류 축을 제거·일반화하면, 나중에 필요한 협상 차원을 복원하기 어렵다.

## 3. 후보 B — 단계형 정규화와 세션 직전 freeze

### 개념

원본 안전화는 입력 즉시 수행하되, 세션용 OutcomeSpace는 세션 시작 직전에 구성하고 freeze한다. 즉, **개인정보를 제거하는 경계**와 **협상 어휘를 확정하는 경계**를 분리한다.

이 후보에서 입력 단계의 산출물은 최종 OutcomeSpace가 아니라 **세션 독립적인 안전 사실값**이다. 출력 빌더는 세션 시작 직전에 안전 사실값들을 모아 승인 스키마 안에서 issue/value를 선택한다.

### 처리 흐름

| 단계 | 처리 |
|---|---|
| 1 | 입력 발생: IDS Intent, Tool/DPA 데이터 수신 |
| 2 | 즉시 안전화: PII는 Vault, 사유·원문은 제거, 협상 가능 수준의 안전 사실값 저장 |
| 3 | 세션 준비: 출력 빌더가 안전 사실값과 Schema Registry를 함께 조회 |
| 4 | 세션 시작 전: OutcomeSpace 구성, Validator 검증, freeze |
| 5 | 세션 중: freeze된 OutcomeSpace의 유효 오퍼만 전송 |

### 예시

| 원본 입력 | 입력 즉시 안전 사실값 | 세션 직전 OutcomeSpace |
|---|---|---|
| "당뇨 때문에 단 음료는 피하고 싶다" | `diet_constraint=있음`, `sugar_preference=low`, 사유 원문은 미전송 | `beverage_sugar={low, normal}` |
| "박민수와 15시에 만나야 한다" | `participant_ref=PII_TOKEN_001`, `time_preference=15:00` | `meeting_time={14:30, 15:00, 15:30}` |
| "배터리가 12%라 장거리 이동은 어렵다" | `mobility_constraint=있음`, `battery_level=low` | `location_type={nearby, remote}` |

### 강점

- **NegMAS freeze 경계와 정합:** 승인 issue/value 선택, Validator 검증, OutcomeSpace freeze가 같은 세션 시작 경계에 모인다. 따라서 "세션 중에는 OutcomeSpace가 바뀌지 않는다"는 NegMAS 제약을 구현 경계로도 드러내기 쉽다.
- **원본 체류 구간 최소화:** 원본·PII·사유는 입력 직후 제거하고, 세션 전까지는 안전 사실값만 유지한다.
- **도메인 변경 국소화 여지:** 안전 사실값 스키마가 안정적이면, 협상 도메인의 issue/value 변경은 출력 빌더와 Schema Registry 쪽으로 제한할 수 있다.
- **복구 기준점 분리:** 세션 복구 시 `안전 사실값 스냅샷 + Schema Registry 버전 + frozen OutcomeSpace + Vault 참조`를 기준으로 재개할 수 있다.

### 약점

- **중간 스키마가 필요:** 안전 사실값 스키마와 OutcomeSpace issue/value 스키마를 모두 관리해야 한다.
- **매핑 계약 증가:** 안전 사실값을 어떤 승인 issue/value로 바꿀지에 대한 규칙과 테스트가 별도로 필요하다.
- **정보량 설계가 중요:** 안전 사실값을 너무 일반화하면 세션 직전 OutcomeSpace 구성이 빈약해지고, 너무 상세하게 두면 개인정보 배제 원칙이 약해진다.
- **갱신 정책 필요:** 세션 freeze 이후 새 입력이 들어오면 기존 OutcomeSpace를 즉시 바꾸지 않고, 재계획·재협상·다음 세션 반영 중 하나로 처리해야 한다.

## 4. 종합 비교

> 평가축은 `poc/dp03-privacy/AGENTS.md`에서 고정한 6개 품질속성에 맞춘다.
> 척도: ★ = 1점, ☆ = 0.5점. 별점은 설계 비교용 정성 점수이며, 절대 성능 측정값이 아니다.

| 품질속성 | 후보 A<br/>완전 조기 정규화 | 후보 B<br/>단계형 정규화 + 직전 freeze | 판단 |
|---|:---:|:---:|---|
| **기밀성** | ★★★★☆ | ★★★★☆ | 두 후보 모두 입력 직후 원본을 안전화한다. 기밀성 차이는 크지 않으며, B는 세션 전까지 원본 대신 안전 사실값을 유지한다는 점이 명시적이다. |
| **Latency** | ★★★★☆ | ★★★★ | PoC상 프라이버시 CPU 비용은 작고 LLM 입력 토큰이 지배적이다. A는 흐름이 짧고, B는 세션 직전 출력 빌드·검증 단계가 더 명시적이다. |
| **자원** | ★★★★ | ★★★☆ | A는 중간 산출물이 적다. B는 안전 사실값 스냅샷과 issue/value 매핑 상태를 추가로 관리한다. |
| **Task 성공률** | ★★★☆ | ★★★★ | A는 조기 issue/value 후보 생성으로 세션 문맥 반영이 제한될 수 있다. B는 세션 직전에 안전 사실값 스냅샷을 기준으로 OutcomeSpace를 구성한다. |
| **세션 복구** | ★★★★ | ★★★★ | 두 후보 모두 복구 가능하다. A는 조기 후보의 무효화 여부를 확인해야 하고, B는 안전 사실값 스냅샷과 frozen OutcomeSpace의 버전 정합성을 확인해야 한다. |
| **유지보수성** | ★★★★ | ★★★☆ | A는 단순하지만 Schema Registry 변경이 입력 정규화로 전파될 수 있다. B는 변경 국소화 여지가 있으나 중간 스키마와 매핑 계약을 운영해야 한다. |
| **종합** | MVP·단순 구현에 유리 | 장기 구조·세션 경계 명확화에 유리 | 한쪽이 당연한 승자는 아니다. A는 단순성, B는 NegMAS freeze 경계와 책임 분리에서 강점이 있다. |

## 5. 핵심 긴장

> **"issue/value 후보를 입력 단계에서 만들 것인가, 세션 시작 직전에 확정할 것인가"**

두 후보는 모두 원본 안전화를 입력 직후 수행한다. 따라서 차이는 개인정보 원본을 얼마나 오래 보관하느냐가 아니라, **협상 어휘에 대한 의미적 확정(commit)을 언제 하느냐**다.

- 후보 A: 입력 직후 안전 사실값과 승인 issue/value 후보를 함께 만든다.
- 후보 B: 입력 직후에는 안전 사실값만 만들고, 세션 시작 직전에 승인 issue/value를 선택해 OutcomeSpace를 freeze한다.

후보 B가 NegMAS 제약에 더 정합적이라는 말은, **OutcomeSpace를 확정하는 시점과 NegMAS 세션이 시작되는 시점을 같은 경계에 둔다**는 뜻이다. NegMAS에서는 세션이 시작된 뒤 OutcomeSpace가 바뀌면 안 된다. 따라서 세션 전까지는 안전 사실값을 유지하고, 세션 시작 직전에만 승인 issue/value를 선택하면 다음이 분명해진다.

| 항목 | 후보 A | 후보 B |
|---|---|---|
| 세션 전 새 입력 | 이미 만든 issue/value 후보의 무효화 여부를 확인해야 한다. | 안전 사실값 스냅샷에 추가하고, freeze 직전에 한 번 반영한다. |
| Schema Registry 변경 | 입력 정규화 단계까지 영향이 갈 수 있다. | 출력 빌더의 issue/value 선택 규칙 중심으로 영향이 제한될 수 있다. |
| freeze의 의미 | 이미 존재하던 후보를 최종 확정한다. | 세션용 OutcomeSpace를 만드는 행위 자체가 freeze 경계가 된다. |

다만 이것이 B의 무조건적 우위는 아니다. 도메인이 작고 issue/value가 안정적이면 A가 더 단순하고, 검증·구현·운영 비용도 낮다. B의 장점은 **입력 출처와 협상 도메인이 늘어나고, 세션 직전까지 반영해야 할 안전 사실값이 많아질 때** 더 의미가 있다.

## 6. 설계 판단

DP03의 실질 비교 후보는 다음 두 가지로 정리한다.

| 후보 | 선택 이유 | 비용 |
|---|---|---|
| **후보 A — 완전 조기 정규화** | 초기 구현이 단순하고 감사 지점이 짧다. 도메인이 작고 issue/value가 안정적이면 충분히 합리적이다. | 세션 문맥이 모이기 전 조기 후보를 만들기 때문에 무효화·재계산 관리가 필요하다. |
| **후보 B — 단계형 정규화 + 직전 freeze** | 원본 안전화와 OutcomeSpace 확정을 분리해, NegMAS 세션 시작 경계에서 승인 issue/value를 확정한다. | 안전 사실값 스키마와 issue/value 매핑 계약을 추가로 운영해야 한다. |

따라서 최종 선택은 "개인정보 안전" 하나만으로 결정되지 않는다. 두 후보 모두 입력 직후 원본 안전화를 전제하기 때문이다. 실제 결정축은 다음에 가깝다.

1. **MVP 단순성:** 빠른 구현과 작은 도메인을 우선하면 후보 A.
2. **세션 경계 명확성:** 세션 직전 OutcomeSpace freeze를 설계의 중심 경계로 삼으면 후보 B.
3. **도메인 확장성:** 협상 도메인과 승인 issue/value 변화가 자주 예상되면 후보 B.
4. **운영 부담 최소화:** 중간 스키마와 매핑 계약을 줄이고 싶으면 후보 A.

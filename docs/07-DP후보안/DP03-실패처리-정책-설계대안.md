# DP03 — 실패 처리 정책 설계대안

> 본 문서는 [DP03-전제-및-공통설계](./DP03-전제-및-공통설계.md)를 전제로, 프라이버시 게이트웨이가 안전한 구조화 출력을 만들 수 없을 때 **어떻게 실패를 처리할 것인가**를 비교한다.
> 실패 처리 정책은 PoC에서 직접 측정한 독립변수는 아니며, 기존 DP03 전제와 PoC 품질속성 범위에서 파생한 설계 대안이다.

---

## 1. 아키텍처적 난제

DP03의 전제는 강하다.

- 상대 PPA로 나가는 오퍼는 고정된 OutcomeSpace의 유효 issue/value만 포함한다.
- Cloud LLM으로 나가는 계획 요청은 구조화 필드만 포함하고, 원문·사유·PII는 제외한다.
- 실행 단계에서 PII가 필요하면 Vault 또는 권한 있는 실행 컴포넌트에서만 복원한다.
- 승인되지 않은 issue/value는 LLM이 생성했더라도 확정할 수 없다.

따라서 문제는 "민감정보를 감지해서 거를 것인가"가 아니라, **안전한 구조화 출력으로 만들 수 없는 입력을 만났을 때 무엇을 할 것인가**다.

대표 실패 지점은 다음과 같다.

| 실패 지점 | 예 | 위험 |
|---|---|---|
| **정규화 실패** | 입력이 PII인지 사유인지 안전 사실값인지 확신할 수 없음 | 원문·사유가 안전 사실값에 섞일 수 있음 |
| **권한 실패** | 사용자가 허용하지 않은 참가자 이름, 주소, 계정 정보가 필요함 | Vault 복원 또는 외부 API 호출이 권한 범위를 넘을 수 있음 |
| **스키마 매핑 실패** | 안전 사실값을 승인 issue/value로 매핑할 수 없음 | OutcomeSpace에 승인되지 않은 값이 들어갈 수 있음 |
| **Validator 실패** | 출력 빌더가 만든 값이 Registry의 허용 값 밖임 | NegMAS 공유 OutcomeSpace 해석이 깨질 수 있음 |
| **계획 요청 실패** | Cloud LLM 계획 요청을 만들려면 원문 설명이 필요해 보임 | 구조화 계획 요청 원칙을 깨뜨릴 수 있음 |
| **실행 실패** | 외부 서비스 호출에 정확 PII가 필요하지만 권한·Vault 참조가 없음 | 실행 단계에서 과도한 개인정보 사용이 발생할 수 있음 |

이때 실패 처리는 세 품질속성 사이의 긴장으로 나타난다.

| 긴장 | 설명 |
|---|---|
| **기밀성 vs Task 성공률** | 안전하지 않으면 막을수록 유출 위험은 줄지만, 완료 가능한 작업도 중단될 수 있다. |
| **Task 성공률 vs Latency** | 사용자 확인을 요청하면 올바른 처리가 가능해지지만, 대기 시간이 늘어난다. |
| **유지보수성 vs 세밀한 복구** | 세밀한 fallback 규칙을 만들수록 성공률은 올라가지만, 정책과 테스트 범위가 커진다. |

## 2. 후보 A — Fail-Closed

### 개념

정규화·권한 확인·스키마 매핑·Validator 검증 중 하나라도 통과하지 못하면, 해당 외부 출구를 만들지 않는다.

- 상대 PPA행: 오퍼 생성 중단 또는 세션 중단.
- Cloud LLM행: 계획 요청 생성 중단.
- 실행행: Vault 복원 또는 외부 API 호출 중단.

### 처리 예시

| 입력/상황 | 처리 |
|---|---|
| "당뇨 때문에 단 음료는 안 된다"에서 `당뇨`를 안전 값으로 일반화할 승인 스키마가 없음 | OutcomeSpace 구성 중단 |
| Sub-Agent가 실명을 반환했지만 상대 공개 권한이 없음 | 실명은 Vault에만 저장하고, 상대행 메시지 생성 중단 |
| Cloud LLM 계획 요청에 원문 설명이 필요하다고 판단됨 | Cloud LLM 호출 중단 |

### 강점

- **기밀성 우선:** 승인되지 않은 값이 외부 출구로 나갈 가능성을 가장 강하게 줄인다.
- **감사 단순성:** 실패 조건과 차단 결과가 명확하다.
- **구현 단순성:** fallback 계층이나 사용자 확인 흐름을 크게 만들지 않아도 된다.

### 약점

- **Task 성공률 저하:** 안전한 대체 값으로 진행 가능한 경우에도 작업이 중단될 수 있다.
- **사용자 경험 저하:** 사용자는 왜 실패했는지 이해하기 어려울 수 있다.
- **재시도 비용:** 같은 입력이 다시 들어와도 스키마나 권한이 바뀌지 않으면 계속 실패한다.

## 3. 후보 B — Coarse Fallback

### 개념

정확한 값으로 매핑할 수 없거나 권한이 부족한 경우, 사전에 승인된 더 일반적인 값으로 대체해 진행한다. fallback도 임의 생성이 아니라 Schema Registry에 등록된 계층 안에서만 허용한다.

예를 들어 다음과 같은 fallback 계층을 둘 수 있다.

| 원래 필요한 값 | 허용 fallback |
|---|---|
| 정확 질병명 | `health_constraint=있음` |
| 정확 위치 | `location_type=nearby` / `area=city_level` |
| 정확 시간 | `time_bucket=afternoon` |
| 실명 참가자 | `participant_required=true` |
| 구체 사유 | `constraint_exists=true` |

### 처리 예시

| 입력/상황 | 처리 |
|---|---|
| "당뇨 때문에 단 음료는 안 된다" | `beverage_sugar=low` 또는 `diet_constraint=있음`으로 일반화 |
| 정확 주소가 들어왔지만 외부 공유 권한이 없음 | `location_type=nearby`로 일반화 |
| 15:07처럼 승인 값에 없는 시간이 들어옴 | `15:00`/`15:30` 중 정책상 허용되는 값 또는 `afternoon`으로 일반화 |
| Cloud LLM 계획 요청에 상세 사유가 필요해 보임 | 사유 대신 `constraint_type=health` 같은 닫힌 분류만 전달 |

### 강점

- **Task 성공률 방어:** 민감하거나 미승인인 세부 값을 제거하면서도 작업을 계속 진행할 수 있다.
- **기밀성 유지 가능:** fallback 값이 승인 스키마 안에 있으면 외부 출구의 구조화 원칙을 유지한다.
- **재계획 감소:** 작은 매핑 실패마다 사용자 확인이나 세션 중단으로 가지 않아도 된다.

### 약점

- **협상 품질 저하:** 값이 일반화되면서 사용자의 실제 선호를 충분히 반영하지 못할 수 있다.
- **정책 복잡도 증가:** 어떤 값까지 일반화할지, 어떤 경우 중단할지 규칙을 관리해야 한다.
- **과도한 일반화 위험:** fallback이 너무 넓으면 작업은 성공해도 사용자가 원한 결과와 멀어질 수 있다.

## 4. 후보 C — 사용자 확인 요청

### 개념

정규화·권한·스키마 매핑이 불확실한 경우, 외부 출구를 만들기 전에 사용자에게 온디바이스 UI로 확인을 요청한다. 확인 요청 자체도 자유 텍스트 원문을 외부로 보내지 않으며, 사용자의 선택은 승인된 값 또는 권한 결정으로만 반영한다.

사용자 확인은 다음을 대상으로 한다.

| 확인 대상 | 예 |
|---|---|
| 권한 확인 | "이 참가자 정보를 실행 단계에서 사용해도 됩니까?" |
| 일반화 수준 선택 | "정확 위치 대신 인근 지역 수준으로 진행할까요?" |
| 작업 계속 여부 | "이 조건은 외부로 전달하지 않고 진행할까요?" |
| 스키마 선택 | "이번 세션에서 시간만 협상하고 장소는 제외할까요?" |

### 처리 예시

| 입력/상황 | 처리 |
|---|---|
| 안전 사실값으로 일반화 가능한지 불확실함 | 사용자에게 승인된 선택지로 확인 |
| 외부 서비스 호출에 실명이 필요함 | 사용자가 실행 단계 사용을 허용하면 Vault에서 복원 |
| OutcomeSpace에 넣을 issue 선택이 모호함 | 사용자가 닫힌 선택지 중 하나를 선택 |
| Cloud LLM 계획 요청에 포함할 제약 범위가 애매함 | 사용자가 `제약 있음/없음`, `도메인 분류` 수준으로 선택 |

### 강점

- **Task 성공률 회복:** 자동 정책만으로 중단될 작업을 사용자 확인으로 진행할 수 있다.
- **권한 명확화:** 민감한 실행이나 외부 서비스 호출 전에 명시적 결정을 남길 수 있다.
- **과도한 일반화 방지:** fallback이 작업 의미를 훼손할 때 사용자가 직접 선택할 수 있다.

### 약점

- **Latency 증가:** 사용자 응답 대기 시간이 추가된다.
- **사용자 경험 부담:** 확인 요청이 잦으면 자동 에이전트의 가치가 떨어진다.
- **상태 관리 증가:** pending confirmation, timeout, 재개 시점, 세션 복구를 관리해야 한다.

## 5. 종합 비교

> 평가축은 `poc/dp03-privacy/AGENTS.md`에서 고정한 6개 품질속성에 맞춘다.
> 척도: ★ = 1점, ☆ = 0.5점. 별점은 설계 비교용 정성 점수이며, 절대 성능 측정값이 아니다.

| 품질속성 | 후보 A<br/>Fail-Closed | 후보 B<br/>Coarse Fallback | 후보 C<br/>사용자 확인 요청 | 판단 |
|---|:---:|:---:|:---:|---|
| **기밀성** | ★★★★★ | ★★★★☆ | ★★★★☆ | A는 미승인 값이 있으면 차단한다. B/C도 승인 값·온디바이스 확인만 사용하면 강하지만, 정책·UI 구현 오류 여지가 있다. |
| **Latency** | ★★★★☆ | ★★★★ | ★★☆ | A는 빠르게 실패한다. B는 fallback 계산이 추가되지만 작다. C는 사용자 대기 시간이 지배한다. |
| **자원** | ★★★★☆ | ★★★★ | ★★★☆ | 세 후보 모두 LLM 추론 대비 자원 차이는 작다. C는 pending 상태와 UI 흐름 관리가 추가된다. |
| **Task 성공률** | ★★☆ | ★★★★ | ★★★★☆ | A는 안전하지만 중단이 많다. B는 일반화로 계속 진행한다. C는 사용자 결정으로 애매한 경우를 회복할 수 있다. |
| **세션 복구** | ★★★★ | ★★★☆ | ★★★ | A는 실패 상태가 단순하다. B는 어떤 fallback이 적용됐는지 복구해야 한다. C는 사용자 확인 대기·결과 상태가 추가된다. |
| **유지보수성** | ★★★★ | ★★★ | ★★★ | A는 정책이 단순하다. B는 fallback 계층과 테스트가 늘어난다. C는 UI·권한·세션 상태 관리까지 포함한다. |
| **종합** | 기밀성·단순성 우세 | 균형안 | 성공률·통제감 우세 | 한 후보만으로 모든 상황을 처리하기 어렵다. 기본 정책과 예외 정책을 분리하는 편이 현실적이다. |

## 6. 핵심 긴장

> **"안전하지 않으면 멈출 것인가, 승인된 더 일반적인 값으로 계속할 것인가, 사용자에게 결정시킬 것인가"**

세 후보는 모두 DP03의 외부 출구 원칙을 지킨다. 차이는 실패를 만났을 때의 기본 반응이다.

| 실패 유형 | Fail-Closed | Coarse Fallback | 사용자 확인 |
|---|---|---|---|
| 미승인 issue/value | 중단 | 승인된 상위 값으로 일반화 | 사용자에게 허용 범위 선택 요청 |
| 권한 부족 | 중단 | PII가 필요 없는 값으로 대체 | 실행 단계 권한 확인 |
| 분류 불확실 | 중단 | 더 보수적인 안전 분류로 처리 | 사용자 확인 |
| Cloud 계획 요청 불가 | 호출 중단 | 닫힌 분류만으로 요청 축소 | 계획 범위 확인 |

이 비교는 기존 "정규화 시점" 비교보다 품질속성 차이가 더 선명하다. 특히 기밀성과 Task 성공률의 긴장이 직접 드러난다.

## 7. 설계 판단

DP03의 실패 처리 정책은 단일 후보 하나로 고정하기보다, **기본 정책 + 제한된 예외 정책**으로 잡는 편이 적절하다.

권장 조합은 다음과 같다.

| 계층 | 정책 | 설명 |
|---|---|---|
| 기본값 | **Fail-Closed** | 승인 스키마 밖 값, 권한 없는 PII, 원문·사유 필요 요청은 외부 출구 생성 중단 |
| 1차 예외 | **Coarse Fallback** | Schema Registry에 명시된 fallback 계층이 있을 때만 일반화해 진행 |
| 2차 예외 | **사용자 확인 요청** | fallback 적용 시 작업 의미가 크게 바뀌거나 권한 결정이 필요한 경우 온디바이스 확인 |

즉, DP03의 최종 정책은 다음 문장으로 정리할 수 있다.

> 프라이버시 게이트웨이는 기본적으로 fail-closed로 동작한다. 단, Schema Registry에 승인된 fallback 계층이 있는 경우에는 더 일반적인 값으로 진행할 수 있고, fallback이 작업 의미나 권한 범위를 바꾸는 경우에는 외부 출구를 만들기 전에 사용자 확인을 요청한다.

이 조합은 세 품질속성의 균형을 맞춘다.

- **기밀성:** 승인되지 않은 값은 기본적으로 차단한다.
- **Task 성공률:** 명시된 fallback으로 불필요한 중단을 줄인다.
- **Latency:** 사용자 확인은 모든 실패가 아니라 권한·의미 변화가 큰 경우로 제한한다.
- **유지보수성:** fallback은 코드 곳곳이 아니라 Schema Registry의 정책 계층으로 관리한다.

# DP — MAF 온디바이스 LLM 모델 배치 (MAF On-Device LLM Placement)

> **대상 QAS**: [QAS-011 MAF Memory 300MB](../07-QAS.md), [QAS-014 N-party +20MB/인](../07-QAS.md) (Mapped NFR: [NFR-MAF-06](../05-NFR.md), [NFR-MAF-09](../05-NFR.md), 품질·지연은 [NFR-MAF-02](../05-NFR.md)·[NFR-MAF-04](../05-NFR.md))
> **LLM 사용 지점 근거**: [DP-memory-optimization.md §1.2](./DP-memory-optimization.md) (온디바이스 LLM이 실제로 호출되는 기능 — FR 매핑), [04-FR.md §2](../04-FR.md)
> **다이어그램**: [DP-MAF-LLM-배치-candidates.drawio](./DP-MAF-LLM-배치-candidates.drawio) (후보 A vs B)
>
> 본 문서는 **MAF(Meta Agent + Sub-Agent)가 온디바이스 LLM을 어떤 모델 배치 구조로 사용할지** 를 결정한다. IDS는 별도 전용 모델을 상시 상주시키는 것으로 *이미 확정*되었으므로(아래 전제 P1) 본 결정 범위에서 제외하고, **IDS 위에 얹히는 MAF 측 LLM 사용**만 다룬다.
>
> **본 문서가 다루는 것**: MAF용 **온디바이스 LLM 추론 엔진**에 모델을 *교체(swap)* 하며 쓸지, *단일 범용 모델 1개를 상주* 시킬지.
> **본 문서가 다루지 않는 것**: (a) IDS의 모델 선택(별도·확정), (b) 동시 요청 시 스케줄링·우선순위(DP-D1), (c) 서버 LLM 책임 분할(DP-D2).
>
> **관계**: 본 문서는 [DP-memory-optimization.md](./DP-memory-optimization.md)의 후속이다. 그 문서는 IDS·MA·SA 전체를 대상으로 후보(C1~C5)를 비교했으나, 이후 ① IDS는 별도 모델 확정, ② MAF 추론 엔진의 상주 모델 1개, ③ 특화 수단을 모델 교체로 한정 이라는 결정이 추가되어 결정 공간이 좁아졌다. 본 문서는 그 좁혀진 조건 위에서 MAF 결정만 다시 세운 것이며, 기존 문서는 맥락·이력으로 보존한다.

## 왜 이 문제가 어렵고 중요한가

**중요성.**
- 이 시스템의 핵심 가치는 NPN(자율 협상)이고, 협상을 수행하는 **Meta Agent는 온디바이스 LLM에 의존**한다. 본 과제가 검증하려는 3대 가설 중 하나가 바로 *"온디바이스 LLM 실용성"* 이다([02-과제-개요.md 제약 표](../02-과제-개요.md)). MAF의 LLM을 단말에서 어떻게 굴리느냐가 이 가설의 성패와 직결된다.
- 메모리는 단말의 **희소 자원**이다. IDS 모델이 이미 상시 상주(P1)하는 *위에* MAF 모델을 얹어야 하므로, 배치를 잘못하면 RAM 예산을 초과해 시스템이 동작 자체를 못 한다. 즉 이건 "최적화" 이전에 **"성립 가능성"의 문제**다.
- 모델 배치는 런타임 아키텍처의 근간이라 **나중에 바꾸는 비용이 크다** — 초기에 옳게 잡아야 한다.

**난이도.**
- **하나의 엔진에 이질적 추론 유형이 몰린다(P4).** MA의 협상·판정·요약, SA의 툴 호출·협업은 요구 능력이 서로 다른데 MAF용 추론 엔진은 상주 모델을 1개만 둔다. 한 자원으로 상이한 요구들을 동시에 만족시켜야 한다.
- **품질속성이 정면으로 상충한다.** 메모리·전력 절감 ↔ 응답 지연 ↔ 협상 품질 ↔ 단순성 — 어느 하나를 잡으면 다른 하나를 놓는다. 비교표(§3)가 이 상충을 그대로 드러낸다.
- **제약이 도망갈 길을 막는다.** 상주 모델 1개 제약(P2)에서 *품질 분화 수단이 모델 교체뿐*이라(P3), 협상 품질을 살리려면 다른 모델로 갈아끼워야 하는데 그 교체가 협상 중 MA↔SA 미세 교차에서 thrash(반복 로딩)를 부른다 — **품질을 살리려는 수단이 다른 비용을 키우는 구조**라, 값싼 정답이 없다.
- **결정이 아직 측정되지 않은 사실에 달려 있다.** SA가 실제로 LLM을 쓰는지, 범용 모델이 협상 품질(NFR-MAF-04)을 내는지 등은 PoC 측정 전엔 확정 불가하다(§5). 지금은 *근거 있는 가정* 위에서 설계해야 한다.

## 0. 전제 (이 결정이 서 있는 기둥)

1. **P1 — IDS는 별도 전용 모델을 상시 상주시킨다(확정·범위 밖).** IDS는 always-on 모니터링·비명시 의도 추론·맥락 연속성 등 무거운 상시 추론을 수행하므로(근거: [DP-memory-optimization.md §1.2](./DP-memory-optimization.md)) 독립 모델이 필요하다. 본 DP는 이 모델을 *주어진 바닥(floor)* 으로 두고 건드리지 않는다.

2. **P2 — 메모리 예산: IDS 1개 + MAF 1개 (동시 2모델).** 단말의 LLM 추론은 **온디바이스 LLM 추론 엔진**(NPU 구동 — [QAS-011](../07-QAS.md) Artifact)이 수행한다. IDS 전용 엔진이 모델 1개를 상시 상주시키는 데 더해, **MAF용 추론 엔진은 상주 모델을 동시 1개만** 둘 수 있다고 전제한다. 즉 본 DP의 결정 대상은 "그 MAF용 추론 엔진을 어떻게 운용하는가"다. (정확한 메모리 한계·임계값은 NFR 재정의 후 매핑 — 6절.)

3. **P3 — 추론 유형 특화 수단은 "모델 그 자체"로 한정한다.** 추론 유형별로 능력을 분화하려면 *다른 모델을 적재* 하는 방법뿐이라고 전제한다. → 본 DP에서 **추론 유형별 특화 수단은 모델 교체뿐**이다. 이 전제가 후보 구조를 A(모델 교체)·B(단일 범용)의 둘로 압축한다.

4. **P4 — MAF의 LLM 추론 유형은 단일하지 않다.** [DP-memory-optimization.md §1.2](./DP-memory-optimization.md) 기준, MAF 안에서도 추론 성격이 제각각이다:
   - **Meta Agent**: 협상(FR-MAF-02/03), 지식공유 민감도 판정(FR-MAF-05), 원격 이상패턴 분석(FR-MAF-06), 합의 요약·제시, (추정) 개입 재반영(08)·위임 판정(10)·이상 처리(11).
   - **Sub-Agent**: 툴 호출·인자 구성(FR-MAF-09), 협업 워크플로우 조율(FR-MAF-04). **SA도 독립적으로 추론하는 에이전트**이며, 도메인별로 **동적 생성**된다.

   따라서 **MAF용 추론 엔진(상주 모델 1개)**이 감당해야 하는 것은 *다종(多種) 추론 유형*이다.

5. **P5 — 서버 Orchestrator는 무관하다.** Orchestrator의 Task Planning·동적 Agent/Tool 생성은 서버 LLM이며 단말 메모리 예산에 들어가지 않는다.

## 2. 후보안

특화 수단이 모델 교체뿐이고(P3) MAF용 추론 엔진의 상주 모델이 1개(P2)이면, 그 엔진을 쓰는 방법은 두 가지로 압축된다.

### 후보 A. 추론 유형별 전용 모델 온디맨드 swap

> **한 줄 요지**: 추론 유형 *타입*마다 특화 모델을 따로 두고, MAF용 추론 엔진에 *그때 필요한 모델만* 갈아끼운다(상주 1개). "필요한 능력만, 그때그때."

**동작 방식**
- MAF가 다루는 추론 유형을 *타입*으로 묶고 타입별 특화 모델을 준비한다. 예: **① 협상 추론 모델**(MA의 제안·역제안·합의), **② 툴 호출 모델**(SA의 툴 선택·인자 구성), 필요 시 **③ 판정 모델**(지식 민감도·이상패턴). 모델 자산은 디스크에 두고 필요 시 로드한다.
- 추론 엔진에는 **한 번에 모델 1개만 상주**한다(P2). 추론 유형이 바뀌면 현재 모델을 *언로드* 하고 다음 모델을 *로드* 한다.
- **동적 생성 SA는 모델을 늘리지 않는다.** SA 인스턴스가 도메인별로 여러 개 생겨도 전부 *같은 "툴 호출" 타입 모델 1개* 를 공유한다. 따라서 관리할 모델 자산 수 = 추론 유형 *타입* 수(소수)지 SA 인스턴스 수가 아니다.

**협상 에피소드에서 무슨 일이 일어나나 (가정 흐름)**
1. 협상 시작 → 추론 엔진에 **협상 모델** 로드.
2. MA가 제안을 추론 → "상대 일정 가용성 확인" 필요 판단.
3. 협상 모델 언로드 → **툴 호출 모델** 로드 → SA가 캘린더 툴 호출·결과 정리.
4. 툴 호출 모델 언로드 → **협상 모델 재로드** → MA가 결과 반영해 역제안.
5. (2~4 반복) … → 합의.

→ **2~4의 핸드오프마다 언로드/로드가 한 번씩 발생**한다. 게다가 모델을 갈아끼우면 그 모델의 KV 캐시·워밍업도 매번 재구성된다.

**메모리 거동(시간축)**
- 점유 RAM = *지금 로드된 모델* 의 크기. 협상 모델 구간엔 크게, 툴/판정 구간엔 작게.
- **피크 = max(단일 추론 유형 모델)** — B와 동일. 그러나 경량 구간엔 작은 모델만 올라가므로 **평균/실사용 점유는 B보다 낮다.**

**장점**
- 평균 메모리·에너지 유리(경량 추론 유형을 작은 모델로 처리) → **M2 ★★★**.
- 추론 유형별 적합 모델 사용 → 협상엔 협상 특화 모델, 툴엔 툴 특화 모델 → **M3 ★★★**.

**단점**
- **에피소드 내 추론 유형 전환마다 모델 reload(thrash)** → **M4 ★**. (단, *최초* 로드는 B와 무승부. 비용은 *세션 도중* 의 반복 전환에서 발생.)
- 복수 모델의 수명주기(로드·언로드)·자산 관리 → **M5 ★★**.

**성립 전제(중요)**: A가 의미 있으려면 추론 유형별로 *실제로 다른* 모델이 있어야 한다. 만약 모든 MAF 추론 유형이 결국 협상급 모델을 요구한다면, A는 "B에 무의미한 reload만 더한 것"으로 전락한다(§6 참고).

### 후보 B. 단일 범용 MAF 모델 상주

> **한 줄 요지**: 협상까지 감당하는 범용 모델 1개를 MAF용 추론 엔진에 상주시키고, *프롬프트만 바꿔* MA의 모든 모드와 SA를 모두 처리한다. "하나로 다 한다."

**동작 방식**
- 추론 엔진에 **범용 모델 1개를 상주**시킨다. MA의 협상·판정·요약, SA의 툴 호출·협업을 **시스템/역할 프롬프트 차이만으로** 구분 처리한다. 모델 가중치 자체는 어떤 추론 유형이든 동일하다.
- 모델 로드는 (에피소드 시작 시 1회) 또는 (정책상 상주). 어느 쪽이든 **세션 도중 모델 교체가 없다.**

**협상 에피소드에서 무슨 일이 일어나나**
1. 협상 시작 → **범용 모델 로드(1회)**.
2. MA 제안(협상 프롬프트) → SA 툴 호출(툴 프롬프트) → MA 역제안(협상 프롬프트) → … → 합의.
3. 전 과정이 **같은 모델의 시분할**로 처리된다 — 핸드오프가 아무리 잦아도 추가 로딩이 없다.

**메모리 거동(시간축)**
- 점유 RAM = *범용 모델 크기로 일정*. 범용 모델은 가장 어려운 추론 유형(협상) 기준으로 정해지므로, 가벼운 툴 호출 구간에도 그 큰 모델이 그대로 점유한다.
- 피크는 A와 동일하나 **평균/실사용 점유와 에너지는 A보다 높다.**

**장점**
- 에피소드 내 전환 시 재로딩 0(같은 모델 시분할) → **M4 ★★★**.
- 모델 1개 — 수명주기·자산이 가장 단순 → **M5 ★★★**.
- 동적 SA가 아무리 많아도 전부 한 모델로 흡수.

**단점**
- 항상 범용 大 모델이 상주 → 평균 메모리·에너지 불리 → **M2 ★★**.
- **추론 유형별 특화 수단이 없다(P3).** 한 가중치가 협상·툴호출·판정을 모두 처리하므로, 가장 까다로운 **협상 품질(NFR-MAF-04)이 최대 리스크** → **M3 ★★**. (이 리스크의 정체와 출구는 §4.2.)

### 후보 C. 하이브리드 (각주 — 본 전제에서는 약함)

기본 범용 모델(B)을 두고 *특정 무거운 추론 유형*만 전용 모델로 swap하는 절충안. 그러나 특화 수단이 모델 교체뿐이므로(P3), 협상처럼 SA와 잘게 교차하는 추론 유형에서는 결국 A의 thrash 문제로 수렴한다. 본 전제에서는 독립 후보로 두지 않는다.

## 3. 후보 비교표

| 평가축 | A. 추론 유형별 swap | B. 단일 범용 상주 |
|---|---|---|
| M1. 피크 메모리 | ★★ (무승부) | ★★ (무승부) |
| M2. 평균 메모리·에너지 | **★★★** | ★★ |
| M3. 추론 유형별 품질 | **★★★** | ★★ (범용, 협상 리스크) |

| M5. 복잡도·유지보수 | ★★ | **★★★** |

> M1(피크)은 두 후보 모두 상주 모델 1개라 동일. M4의 *최초* cold-start도 동일하며, 차이는 **에피소드 내부의 추론 유형 전환**에서만 발생한다.

## 4. 권장안 — B(단일 범용 상주) 우선, 단 조건부

A의 실질 약점은 사실상 **"에피소드 내 reload thrash" 하나**다 — 나머지 축(평균메모리·에너지·품질)은 A가 우세하거나 무승부다. 따라서 **A vs B는 "MAF 추론 유형이 세션 안에서 얼마나 잘게 교차하는가(교차 입도)"** 라는 단일 변수로 거의 환원된다.

> **추론 유형이 잘게 교차하면 → B. 위상적으로 길게 지속되면 → A.**

> **[지연 허용 반영 — 재검토 필요]** 단, 본 시나리오는 백그라운드·비동기 협상이라 **모델 로딩 *수초*는 UX 문제가 아니다**(설계 판단, §6). 그러면 A의 thrash는 *지연*이 아니라 **에너지·복잡도** 비용으로만 남고, 위 'B 우선'의 핵심 근거(전환 지연)가 약해진다. 결정은 *A의 메모리·품질 이점 ↔ A의 추가 에너지·복잡도·운영 부담* 으로 재균형되며, **이 경우 권장이 A 쪽으로 기울 수 있다.** (아래 4.1은 "지연이 비용일 때"의 논리이므로, 지연 허용을 확정하면 함께 갱신되어야 한다.)

### 4.1 권장 근거

1. **간판 시나리오인 협상이 잘게 교차한다.** 1회 협상 = MA 추론 → SA 툴 호출(가용성 확인 등) → 결과 → MA 역제안 → … 의 반복이다. MA↔SA 핸드오프가 *세션 내에서 수없이* 발생하므로, A라면 핸드오프마다 모델을 갈아끼워 thrash가 심각하다. B는 같은 모델을 시분할하므로 이 문제가 없다. → **협상 중심으로 보면 B.**
2. **다른 MAF 케이스는 더 위상적이다.** 지식공유 민감도 판정(FR-MAF-05), 원격 이상패턴 분석(FR-MAF-06)은 한 추론 유형이 길게 지속되므로 A에 불리하지 않다. 그러나 시스템의 핵심 가치(NPN 자율 협상)가 협상에 있으므로 *교차가 심한 쪽*이 설계를 지배한다.
3. **단순성·운영 부담에서도 B가 낫다(M5).** 모델 1개, 수명주기 없음, 동적 SA를 자연 흡수.

### 4.2 B 채택의 대가 — 협상 품질 리스크 (정직한 명시)

B의 유일한 본질적 약점은 **범용 모델이 협상 품질(NFR-MAF-04)을 충분히 내지 못할 위험**이다. 본 DP는 특화 수단을 *모델 교체*로 한정했으므로(P3), 이 리스크를 줄이는 길은 **협상 전용 모델을 두는 것 = A로의 선회**뿐이다. 따라서 실측에서 B의 협상 품질이 임계에 못 미치면:

- **A(또는 부분 A)로 선회** — 협상에 한해 전용 모델을 두고 swap한다. 협상-SA 교차 비용이 따르지만, 협상이 백그라운드·비동기라 *지연* 부담은 작다(§4 머리 노트).

### 4.3 A가 역전하는 조건 (명시)

다음 중 하나라도 성립하면 A가 B를 앞선다:

- **SA 툴 호출을 비-LLM(순수 API)으로 구현**하여 협상 중 LLM 추론 유형 교차가 사라지는 경우.
- **협상 중 SA 호출 빈도가 낮아** 교차가 위상적인 경우.
- **평균 메모리·전력이 핵심 제약**으로 떠올라(상시 IDS 모델과 합산 시 배터리/발열), 평균 점유를 낮추는 것이 협상 지연보다 중요해지는 경우.

## 5. 후속 결정 (이 문서 다음에 풀어야 할 질문)

1. **SA 툴 호출의 LLM 여부**: SA의 툴 선택·인자 구성이 *독립 LLM 추론*인지, MA가 결정하고 SA는 *API 실행만* 하는지. 이것이 협상 중 교차 입도를 정하고, 곧 A/B 선택을 좌우한다(4.3).
2. **범용 모델의 협상 품질 실측**: B의 단일 범용 모델로 NFR-MAF-04(협상 라운드 품질)를 만족하는지 PoC 측정. 미달 시 §4.2(A로 선회).
3. **메모리 예산(P2) 정합**: IDS 1개 + MAF 1개(2모델)가 단말 RAM·NFR 임계값과 맞는지 확정하고, NFR-MAF-06·QAS-011을 재매핑한다(CLAUDE.md 7항).
4. **DP-D1과의 정렬**: 본 결정(MAF 추론 엔진 상주 모델 1개)을 전제로 DP-D1(자원 중재)의 스케줄링이 IDS 모델과 MAF 모델의 동시 상주·시분할을 다루도록 정렬한다.

## 6. 참고 — 본 문서의 가정과 한계

- 메모리는 정성적으로만 다룬다. "피크 무승부"는 *MAF 추론 엔진의 상주 모델이 1개*라는 전제(P2)의 직접 귀결이며, 절대 수치는 모델 선택·양자화·런타임 오버헤드에 따라 변한다.
- "평균 메모리 A 우세"는 *경량 추론 유형 구간에 작은 모델을 올릴 수 있을 때*만 성립한다. 모든 MAF 추론 유형이 결국 협상급 모델을 요구한다면 A의 평균 이점은 사라지고 A는 "B + 무의미한 reload"로 전락한다.
- 협상의 MA↔SA 교차 빈도, SA의 LLM 사용 수준은 [04-FR.md](../04-FR.md)에 근거한 *논리적 추론*이며 실제 빈도는 실측으로 확인해야 한다(5절-1·2).
- **지연 허용 전제**: 본 DP는 A2A 협상을 *백그라운드·비동기* 동작으로 가정한다 — 사용자가 즉시 응답을 기다리는 실시간 상호작용이 아니므로, **모델 로딩 *수초*는 허용 가능**하다고 본다(설계 판단). 그 결과 M4(전환 지연)의 비중이 낮아지고 A의 약점이 크게 완화된다. 이 전제가 깨지면(예: 실시간 대화형 협상 UX) 로딩 지연이 다시 치명적 약점이 되어 권장은 B로 회귀한다.
- IDS까지 포함한 더 넓은 후보 비교는 [DP-memory-optimization.md](./DP-memory-optimization.md)에 보존되어 있다.

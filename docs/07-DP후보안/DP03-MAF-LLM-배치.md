# DP — MAF 온디바이스 LLM 모델 배치 (MAF On-Device LLM Placement)

> **대상 QAS**: [QAS-011 MAF Memory 300MB](../07-QAS.md), [QAS-014 N-party +20MB/인](../07-QAS.md) (Mapped NFR: [NFR-MAF-06](../05-NFR.md), [NFR-MAF-09](../05-NFR.md), 품질·지연은 [NFR-MAF-02](../05-NFR.md)·[NFR-MAF-04](../05-NFR.md))
> **LLM 사용 지점 근거**: [DP-memory-optimization.md §1.2](./DP-memory-optimization.md) (온디바이스 LLM이 실제로 호출되는 기능 — FR 매핑), [04-FR.md §2](../04-FR.md)
> **다이어그램**: [DP03-MAF-LLM-배치-candidates.drawio](./DP03-MAF-LLM-배치-candidates.drawio) (후보 A vs B) · [DP03-MAF-LLM-배치-components.drawio](./DP03-MAF-LLM-배치-components.drawio) (컴포넌트 뷰 — 후보별 모듈)
>
> 본 문서는 **MAF(Meta Agent + Sub-Agent)가 온디바이스 LLM을 어떤 모델 배치 구조로 사용할지** 를 결정한다. IDS는 별도 전용 모델을 상시 상주시키는 것으로 *이미 확정*되었으므로(아래 전제 P1) 본 결정 범위에서 제외하고, **IDS 위에 얹히는 MAF 측 LLM 사용**만 다룬다.
>
> **본 문서가 다루는 것**: MAF용 **온디바이스 LLM 추론 엔진**에 모델을 *교체(swap)* 하며 쓸지, *단일 범용 모델 1개를 상주* 시킬지.
> **본 문서가 다루지 않는 것**: (a) IDS의 모델 선택(별도·확정), (b) 동시 요청 시 스케줄링·우선순위(DP-D1), (c) 서버 LLM 책임 분할(DP-D2).
>
> **관계**: 본 문서는 [DP-memory-optimization.md](./DP-memory-optimization.md)의 후속이다. 그 문서는 IDS·MA·SA 전체를 대상으로 후보(C1~C5)를 비교했으나, 이후 ① IDS는 별도 모델 확정, ② MAF 추론 엔진의 상주 모델 1개, ③ 특화 수단을 모델 교체로 한정 이라는 결정이 추가되어 결정 공간이 좁아졌다. 본 문서는 그 좁혀진 조건 위에서 MAF 결정만 다시 세운 것이며, 기존 문서는 맥락·이력으로 보존한다.

## 왜 이 문제가 어렵고 중요한가

**중요성.**
- 이 시스템의 핵심 가치는 NPN(자율 협상)이고, 협상을 수행하는 **Meta Agent는 온디바이스 LLM에 의존**한다. 본 과제가 검증하려는 3대 가설 중 하나가 바로 *"온디바이스 LLM 실용성"* 이다([02-과제-개요.md 제약 표](../02-과제-개요.md)). MAF의 LLM을 단말에서 어떻게 굴리느냐가 이 가설의 성패와 직결된다.
- 메모리는 단말의 **희소 자원**이다. IDS 모델이 이미 상시 상주(P1)하는 *위에* MAF 모델을 얹어야 하므로, 배치를 잘못하면 RAM 예산을 초과해 시스템이 동작 자체를 못 한다. 즉 이건 "최적화" 이전에 **"성립 가능성"의 문제**다.
- 모델 배치는 런타임 아키텍처의 근간이라 **나중에 바꾸는 비용이 크다** — 초기에 옳게 잡아야 한다.

**난이도.**
- **하나의 엔진에 이질적 추론 유형이 몰린다(P4).** MA의 협상·판정·요약, SA의 툴 호출·협업은 요구 능력이 서로 다른데 MAF용 추론 엔진은 상주 모델을 1개만 둔다. 한 자원으로 상이한 요구들을 동시에 만족시켜야 한다.
- **품질속성이 정면으로 상충한다.** 메모리·전력 절감 ↔ 응답 지연 ↔ 협상 품질 ↔ 단순성 — 어느 하나를 잡으면 다른 하나를 놓는다. 비교표(§4)가 이 상충을 그대로 드러낸다.
- **제약이 도망갈 길을 막는다.** 상주 모델 1개 제약(P2)에서 *품질 분화 수단이 모델 교체뿐*이라(P3), 협상 품질을 살리려면 다른 모델로 갈아끼워야 하는데 그 교체가 협상 중 MA↔SA 미세 교차에서 thrash(반복 로딩)를 부른다 — **품질을 살리려는 수단이 다른 비용을 키우는 구조**라, 값싼 정답이 없다. *(단, §0의 P6·P7 전제를 깔면 협상 본진 중 swap이 일어나지 않아 이 thrash가 회피된다 — §5 참조.)*
- **결정이 아직 측정되지 않은 사실에 달려 있다.** SA가 실제로 LLM을 쓰는지, 범용 모델이 협상 품질(NFR-MAF-04)을 내는지 등은 PoC 측정 전엔 확정 불가하다(§6). 지금은 *근거 있는 가정* 위에서 설계해야 한다.

## 0. 전제 (이 결정이 서 있는 기둥)

1. **P1 — IDS는 별도 전용 모델을 상시 상주시킨다(확정·범위 밖).** IDS는 always-on 모니터링·비명시 의도 추론·맥락 연속성 등 무거운 상시 추론을 수행하므로(근거: [DP-memory-optimization.md §1.2](./DP-memory-optimization.md)) 독립 모델이 필요하다. 본 DP는 이 모델을 *주어진 바닥(floor)* 으로 두고 건드리지 않는다.

2. **P2 — 메모리 예산: IDS 1개 + MAF 1개 (동시 2모델).** 단말의 LLM 추론은 **온디바이스 LLM 추론 엔진**(NPU 구동 — [QAS-011](../07-QAS.md) Artifact)이 수행한다. IDS 전용 엔진이 모델 1개를 상시 상주시키는 데 더해, **MAF용 추론 엔진은 상주 모델을 동시 1개만** 둘 수 있다고 전제한다. 즉 본 DP의 결정 대상은 "그 MAF용 추론 엔진을 어떻게 운용하는가"다. (정확한 메모리 한계·임계값은 NFR 재정의 후 매핑 — 6절.)

3. **P3 — 추론 유형 특화 수단은 "모델 그 자체"로 한정한다.** 추론 유형별로 능력을 분화하려면 *다른 모델을 적재* 하는 방법뿐이라고 전제한다. → 본 DP에서 **추론 유형별 특화 수단은 모델 교체뿐**이다. 이 전제가 후보 구조를 A(모델 교체)·B(단일 범용)의 둘로 압축한다.

4. **P4 — MAF의 LLM 추론 유형은 단일하지 않다.** [DP-memory-optimization.md §1.2](./DP-memory-optimization.md) 기준, MAF 안에서도 추론 성격이 제각각이다:
   - **Meta Agent**: 협상(FR-MAF-02/03), 지식공유 민감도 판정(FR-MAF-05), 원격 이상패턴 분석(FR-MAF-06), 합의 요약·제시, (추정) 개입 재반영(08)·위임 판정(10)·이상 처리(11).
   - **Sub-Agent**: 툴 호출·인자 구성(FR-MAF-09), 협업 워크플로우 조율(FR-MAF-04). **SA도 독립적으로 추론하는 에이전트**이며, 도메인별로 **동적 생성**된다.

   따라서 **MAF용 추론 엔진(상주 모델 1개)**이 감당해야 하는 것은 *다종(多種) 추론 유형*이다.

5. **P5 — 서버 Orchestrator는 무관하다.** Orchestrator의 Task Planning·동적 Agent/Tool 생성은 서버 LLM이며 단말 메모리 예산에 들어가지 않는다.

6. **P6 — SA는 협상 *경계*(사전·사후)에서만 호출된다.** 협상 *본진* 중 MA↔MA 추론은 순수 LLM 작업이며, SA 재호출은 *예외적* 이다.
   - *정당화*: 사전 SA가 가용성·선호·제약을 충분히 수집해 MA 컨텍스트에 적재한다는 협상 프로토콜 가정. 상대가 *사전 캐시 외* 제안을 했을 때의 예외 발생률은 PoC 측정 대상(§6). 이 전제가 깨지면 §5.3 (b1) 환원 조건 적용.

7. **P7 — MA는 협상 본진 중 *추론 모드를 전환하지 않는다*.** 협상 모델 하나만 시분할로 사용한다. 지식공유 민감도 판정·합의 요약 등은 협상 *경계* (시작 전·종료 후)에서 수행한다.
   - *정당화*: 협상 도중 모드 인터럽트가 끼면 협상 흐름·KV 캐시 일관성이 깨지므로, 단계 분리가 프로토콜 설계상 합리적. 이 전제가 깨지면 §5.3 (b2) 환원 조건 적용.

| M5 | **복잡도 / 유지보수** | (Maintainability) | 모델 수명주기(로드·언로드)·자산 관리 부담 |

> 표기: **★★★** 강함 · **★★** 보통 · **★** 약함/위배 위험.
> M4 보충: **최초 cold-start(빈 상태 → 에피소드 첫 로드)는 두 후보가 동일**하다. M4가 재는 것은 *에피소드 내부의 추론 유형 전환* 비용이다.
> **M4 ↔ M4b**: 지연은 두 방향이다 — *전환*(swap, A 불리) ↔ *스텝 속도*(decode, A 유리). 기존엔 전환만 세어 A에 불리하게 기울어 있었다. 단 M4b는 M2(에너지)와 같은 뿌리(모델크기×토큰)라 *이중 계산 금지*. 실측은 [DP03 PoC](./DP03-MAF-LLM-배치-PoC계획.md) R2·R3.
> **지연 허용(본 시나리오)**: A2A 협상은 **백그라운드·비동기**로 진행되어 실시간 상호작용이 아니므로, 모델 로딩이 *수초* 걸려도 UX를 저해하지 않는다(설계 판단 — §7). 따라서 **M4의 비중은 낮고**, A의 swap 비용은 *지연*보다 **에너지·복잡도** 문제로 보는 것이 맞다.
> 본 M1~M5는 *런타임* 축(메모리·지연·복잡도)이다. 아키텍처 품질속성(확장성·배포성·시험성·장애격리)은 **§3.3**에서 별도로 다룬다.

## 2. 후보안

특화 수단이 모델 교체뿐이고(P3) MAF용 추론 엔진의 상주 모델이 1개(P2)이면, 그 엔진을 쓰는 방법은 두 가지로 압축된다.

### 후보 A. 추론 유형별 전용 모델 온디맨드 swap

> **한 줄 요지**: 추론 유형 *타입*마다 특화 모델을 따로 두고, MAF용 추론 엔진에 *그때 필요한 모델만* 갈아끼운다(상주 1개). "필요한 능력만, 그때그때."

**동작 방식**
- MAF가 다루는 추론 유형을 *타입*으로 묶고 타입별 특화 모델을 준비한다. 예: **① 협상 추론 모델**(MA의 제안·역제안·합의), **② 툴 호출 모델**(SA의 툴 선택·인자 구성), 필요 시 **③ 판정 모델**(지식 민감도·이상패턴). 모델 자산은 디스크에 두고 필요 시 로드한다.
- 추론 엔진에는 **한 번에 모델 1개만 상주**한다(P2). 추론 유형이 바뀌면 현재 모델을 *언로드* 하고 다음 모델을 *로드* 한다.
- **동적 생성 SA는 모델을 늘리지 않는다.** SA 인스턴스가 도메인별로 여러 개 생겨도 전부 *같은 "툴 호출" 타입 모델 1개* 를 공유한다. 따라서 관리할 모델 자산 수 = 추론 유형 *타입* 수(소수)지 SA 인스턴스 수가 아니다.

**협상 에피소드에서 무슨 일이 일어나나 (가정 흐름)**
1. 협상 시작 → 추론 엔진에 **협상 모델** 로드.
2. MA가 제안을 추론 → "상대 일정 가용성 확인" 필요 판단.
3. 협상 모델 언로드 → **툴 호출 모델** 로드 → SA가 캘린더 툴 호출·결과 정리.
4. 툴 호출 모델 언로드 → **협상 모델 재로드** → MA가 결과 반영해 역제안.
5. (2~4 반복) … → 합의.

→ **2~4의 핸드오프마다 언로드/로드가 한 번씩 발생**한다. 게다가 모델을 갈아끼우면 그 모델의 KV 캐시·워밍업도 매번 재구성된다.

**메모리 거동(시간축)**
- 점유 RAM = *지금 로드된 모델* 의 크기. 협상 모델 구간엔 크게, 툴/판정 구간엔 작게.
- **피크 = max(단일 추론 유형 모델)** — B와 동일. 그러나 경량 구간엔 작은 모델만 올라가므로 **평균/실사용 점유는 B보다 낮다.**

**장점**
- 평균 메모리·에너지 유리(경량 추론 유형을 작은 모델로 처리) → **M2 ★★★**.
- 추론 유형별 적합 모델 사용 → 협상엔 협상 특화 모델, 툴엔 툴 특화 모델 → **M3 ★★★**.

**단점**
- **에피소드 내 추론 유형 전환마다 모델 reload(thrash)** → **M4 ★**. (단, *최초* 로드는 B와 무승부. 비용은 *세션 도중* 의 반복 전환에서 발생.)
- 복수 모델의 수명주기(로드·언로드)·자산 관리 → **M5 ★★**.

**성립 전제(중요)**: A가 의미 있으려면 추론 유형별로 *실제로 다른* 모델이 있어야 한다. 만약 모든 MAF 추론 유형이 결국 협상급 모델을 요구한다면, A는 "B에 무의미한 reload만 더한 것"으로 전락한다(§7 참고).

### 후보 B. 단일 범용 MAF 모델 상주

> **한 줄 요지**: 협상까지 감당하는 범용 모델 1개를 MAF용 추론 엔진에 상주시키고, *프롬프트만 바꿔* MA의 모든 모드와 SA를 모두 처리한다. "하나로 다 한다."

**동작 방식**
- 추론 엔진에 **범용 모델 1개를 상주**시킨다. MA의 협상·판정·요약, SA의 툴 호출·협업을 **시스템/역할 프롬프트 차이만으로** 구분 처리한다. 모델 가중치 자체는 어떤 추론 유형이든 동일하다.
- 모델 로드는 (에피소드 시작 시 1회) 또는 (정책상 상주). 어느 쪽이든 **세션 도중 모델 교체가 없다.**

**협상 에피소드에서 무슨 일이 일어나나**
1. 협상 시작 → **범용 모델 로드(1회)**.
2. MA 제안(협상 프롬프트) → SA 툴 호출(툴 프롬프트) → MA 역제안(협상 프롬프트) → … → 합의.
3. 전 과정이 **같은 모델의 시분할**로 처리된다 — 핸드오프가 아무리 잦아도 추가 로딩이 없다.

**메모리 거동(시간축)**
- 점유 RAM = *범용 모델 크기로 일정*. 범용 모델은 가장 어려운 추론 유형(협상) 기준으로 정해지므로, 가벼운 툴 호출 구간에도 그 큰 모델이 그대로 점유한다.
- 피크는 A와 동일하나 **평균/실사용 점유와 에너지는 A보다 높다.**

**장점**
- 에피소드 내 전환 시 재로딩 0(같은 모델 시분할) → **M4 ★★★**.
- 모델 1개 — 수명주기·자산이 가장 단순 → **M5 ★★★**.
- 동적 SA가 아무리 많아도 전부 한 모델로 흡수.

**단점**
- 항상 범용 大 모델이 상주 → 평균 메모리·에너지 불리 → **M2 ★★**.
- **추론 유형별 특화 수단이 없다(P3).** 한 가중치가 협상·툴호출·판정을 모두 처리하므로, 가장 까다로운 **협상 품질(NFR-MAF-04)이 최대 리스크** → **M3 ★★**. (이 리스크의 정체와 출구는 §5.2.)

### 후보 C. 하이브리드 (각주 — 본 전제에서는 약함)

기본 범용 모델(B)을 두고 *특정 무거운 추론 유형*만 전용 모델로 swap하는 절충안. 그러나 특화 수단이 모델 교체뿐이므로(P3), 협상처럼 SA와 잘게 교차하는 추론 유형에서는 결국 A의 thrash 문제로 수렴한다. 본 전제에서는 독립 후보로 두지 않는다.

## 3. 후보별 아키텍처 스타일과 구조 영향

> 본 절은 A·B가 *운영 정책*("모델 1개냐 N개냐")이 아니라 **서로 다른 아키텍처 스타일**임을 보인다. 두 후보는 추론 엔진 주변에 *다른 모듈 구조*를 요구하며, 그래서 결정의 본질은 정책이 아니라 **아키텍처**다.

### 3.1 아키텍처 스타일

| | 후보 A (추론 유형별 swap) | 후보 B (단일 범용 상주) |
|---|---|---|
| **스타일** | Microkernel + Plug-in *(동시 적재 ≤ 1로 제약된 변형)* | Service-Oriented *(In-Process Shared Service)* |
| **핵심 패턴** | Strategy(유형→모델) + Lazy Loading + Resource Manager with Eviction | Facade + Singleton(Keep-alive) + Producer–Consumer 큐 |
| **비유** | 엔진 = 커널, 모델 = *동시에 1개만* 적재되는 플러그인 | 로컬 모델 서버 1개에 MA·SA가 요청 |
| **역할 차이를 표현하는 위치** | 모델(슬롯에 적재된 가중치) 교체 | 프롬프트(데이터) |

A의 "상주 모델 ≤ 1" 제약(P2)은 사실상 **OS 가상메모리의 세그먼트 스왑(segment swap)** 과 유사한 구조를 강제한다 — 필요한 모델을 적재(page-in)하고 다른 모델을 축출(evict)하는 *관리자*가 반드시 생긴다. 이 점이 A를 단순 정책이 아니라 *아키텍처* 결정으로 만든다.

> **명명 단서.**
> - **A — Microkernel + Plug-in** (POSA1, Buschmann et al.): 정통 양식이지만 *원형*은 "여러 플러그인의 공존·협업"을 가정한다. 본 DP는 *동시 적재 ≤ 1* 이라는 단일 슬롯 제약이 추가된 *변형*이므로, 양식의 정수는 §3.3의 *확장성·격리* 측면에서만 살아 있다. 런타임 동작은 위 본문대로 segment-swap에 더 가깝다.
> - **B — Service-Oriented** (Bass, *SAiP*): SOA 담론의 "공유 서비스" 사고를 *프로세스 내(in-process)* 에 구현한 형태다. NVIDIA Triton·TF Serving 등 *Inference Server* 제품군이 같은 사고의 외부(out-of-process) 변형이다.
> - **B의 *Singleton(Keep-alive)***: GoF Singleton에 "세션 동안 인스턴스를 *살려둔다*"는 운영 단서를 명시한 표현. 직전 초안의 *Object Pool(size 1)* 는 풀 패턴의 정수(객체 *재할당*)가 size=1에서 작동하지 않아 부적절했다.

### 3.2 구현 시 추가·변경되는 *필수* 모듈

기존 컴포넌트(온디바이스 LLM 추론 엔진 · Meta Agent · Sub-Agent · Multi-Agent 상태 관리자)를 기준으로, **각 후보를 성립시키는 데 반드시 필요한** 구조만 다룬다. 성능·운영 보강을 위한 후보별 *택틱* 은 §3.4로 분리한다. 컴포넌트 뷰: [DP03-MAF-LLM-배치-components.drawio](./DP03-MAF-LLM-배치-components.drawio).

**후보 A**

| 모듈 | 신규/변경 | 역할 |
|---|---|---|
| **모델 레지스트리/카탈로그** | 신규 | 유형별 모델(협상/툴/판정) 메타·위치 선언 |
| **추론 유형 라우터(Dispatcher)** | 신규 | 요청 유형 → 필요 모델 매핑 |
| **요청 스케줄러 (FIFO)** | 신규 | 동시 요청 *순서화* 및 swap race 방지 — 단순 FIFO 정책 (B의 동명 모듈과 대칭; A는 swap이 끼어 더 절실) |
| **모델 수명주기 관리자(Loader/Unloader)** | 신규 | 적재·축출, *상주 ≤ 1* 강제 (P2 직접 강제) |
| Meta Agent / Sub-Agent | 변경 | 요청에 *추론 유형 태깅* (라우터가 모델을 고를 수 있도록) |

**후보 B**

| 모듈 | 신규/변경 | 역할 |
|---|---|---|
| **MAF 추론 서비스 (Gateway)** | 신규 | MA·SA의 단일 진입점, 상주 모델 keep-alive |
| **요청 스케줄러 (FIFO)** | 신규 | 단일 모델 동시 호출의 시분할 보장 — 단순 FIFO 정책 |
| **프롬프트/역할 프로파일 레지스트리** | 신규 | 협상·툴·판정용 시스템 프롬프트 관리(역할 분화의 *유일* 수단, P3) |
| Meta Agent / Sub-Agent | 변경 | 모델 미보유 → 서비스에 (역할·프롬프트·컨텍스트) 요청하는 *얇은 클라이언트* 化 |

### 3.3 아키텍처 품질속성 비교 (M1~M5 보완)

§1의 M1~M5는 *런타임*(메모리·지연·복잡도) 축이다. 아키텍처 관점은 그 너머의 품질속성을 드러낸다:

| 품질속성 | A (플러그인) | B (공유 서비스) |
|---|---|---|
| **확장성**(새 추론 유형 추가) | ★★★ 플러그인 등록 | ★ 범용 모델 재학습/교체 |
| **배포성·형상관리** | ★ 다(多) 아티팩트·업데이트 파이프라인 | ★★★ 단일 아티팩트 |
| **시험성** | ★★ 플러그인 + swap 로직 검증 | ★★★ 단일 모델 |
| **장애 격리** | ★★★ 불량 모델 격리 | ★ 단일 모델 = 단일 실패점 |

→ 정리하면 **A = 확장성·격리 ↔ 복잡·배포부담**, **B = 단순·배포용이 ↔ 확장성·격리 약함**. "1 vs N"이 아니라 **마이크로커널 vs 공유서비스**의 트레이드다. 이 축들은 §5 권장에서 단순성(B)·확장성(A) 판단의 근거가 된다.

### 3.4 후보별 보완 설계·택틱 (essential 아님)

§3.2의 필수 모듈 *위에* 얹히는 성능·운영 택틱이다. **§4 후보 비교에는 직접 영향이 없으며**, 본 DP 결정 후 실제 구현 단계에서 후속 설계로 풀어야 한다.

**후보 A 택틱**

| 항목 | 목적 |
|---|---|
| Swap 배칭·재정렬 (스케줄러 정책 확장) | §3.2의 *요청 스케줄러* 정책을 단순 FIFO에서 *같은 추론 유형 묶음·재정렬* 로 확장해 swap 횟수 최소화 (M4 thrash 완화). 동시 협상 세션 다수 등 *재정렬 여지가 있을 때만* 효과 — 데이터 의존이 직렬인 단일 협상에서는 효과 미미 |
| KV 캐시·워밍업 관리자 | swap 시 모델별 KV 캐시 재구성·재활용, 적재 직후 워밍업으로 cold-start 비용 흡수 |
| 다중 아티팩트 배포 파이프라인 | 다(多) 모델 자산 패키징·서명·OTA 업데이트 (배포성 보강) |
| MA/SA의 swap 지연 비동기 수용 | swap 동안 호출자 블로킹 방지 (백그라운드 협상 전제와 정합) |

**후보 B 택틱**

본 DP 범위에서 B는 §3.2의 필수 모듈만으로 성립한다. 실제 구현 시 등장 가능한 택틱(예: 역할 프롬프트 변경 시 프리픽스 KV 캐시 재사용 정책, 동시 요청 batching) 은 본 DP 결정 후 별도로 다룬다.

### 3.5 택틱 후보 — 런타임 LoRA 어댑터 (P3 완화형, 별도 검증 필요)

> 본 절은 §3.4 택틱과 달리 **P3(특화 수단 = 다른 *모델* 적재)를 완화**한다. LoRA는 "다른 모델"이 아니라 *동결된 베이스 1개 + 유형별 저랭크 어댑터(adapter)* 로 추론 유형을 특화하므로, 후보 A(모델 swap)와 B(단일 범용 상주)를 **하이브리드로 수렴**시킨다. 채택 시 P3와 §5 권장의 근거 일부가 바뀌므로, 단순 택틱이 아니라 **독립 평가가 필요한 변형** 으로 둔다. 컴포넌트 뷰: [DP03-MAF-LLM-배치-components.drawio](./DP03-MAF-LLM-배치-components.drawio) (후보 LoRA — 초록 열).

**구조.** 어댑터는 MA/SA가 아니라 **추론 엔진 내부** 의 베이스 projection 가중치에 붙는다(`W + (α/r)·B·A`). 베이스 모델 1개를 상주시키고(B처럼), 추론 유형(협상/툴/판정)에 따라 *어댑터만* 전환한다. 신규/변경 모듈:

| 모듈 | 신규/변경 | 역할 |
|---|---|---|
| **추론 유형 라우터** | 신규 | 요청 유형 → 필요 *어댑터* 매핑 (A의 동명 모듈과 대칭, 대상이 모델→어댑터) |
| **요청 스케줄러 (FIFO)** | 신규 | 동시 요청 순서화. 베이스가 고정이라 swap race 부담은 A보다 낮음 |
| **Adapter Manager** | 신규 | 어댑터 적재·활성 전환, 베이스 상주 ≤1 고정 (A의 *모델* 수명주기 관리자보다 가벼움 — 베이스를 축출하지 않음) |
| **Adapter Registry** | 신규 | 유형별 어댑터 메타·위치 선언 |
| **추론 엔진 + Adapter Application Layer** | 변경 | 엔진에 어댑터 적용(또는 zFLoRA 융합) 계층 추가 — **본 변형의 핵심 의존성** |

→ A의 다(多) GB 모델 자산·무거운 수명주기 관리가 사라지는 대신, **엔진의 어댑터 지원** 이 새 핵심 의존성이 된다.

**지연은 두 가지를 구분해야 한다(실측 근거).**

| 지연 종류 | 크기 | 본 시나리오 영향 |
|---|---|---|
| 어댑터 *전환 이벤트* | 사전 로드 시 sub-ms ~ 100ms대 | **무시 가능** — 협상 경계 1~3회(P6·P7) + 백그라운드 비동기(§5) |
| *활성 중* per-token 추론 오버헤드 | NPU naive LoRA = **prefill 최대 2.5×·decode 1.6×** (Galaxy S25+/Snapdragon 8 Elite 실측) | **무시 불가** — 협상 본진 전 토큰에 부과 → M2(에너지)·M4(지연) 동시 악화 |

> 직관과 달리 우려 대상은 *전환 지연* 이 아니라 **활성 중 per-token 오버헤드** 다. 원인: 현 Qualcomm 런타임이 동적 가중치 swap을 지원하지 않아 어댑터를 16-bit *입력* 으로 주입 → 추가 matmul 분기가 NPU의 융합 파이프라인을 깨뜨려, 파라미터가 <1%인데도 연산 비용이 불균형하게 커진다.

**완화책은 이미 존재·검증됨.** zFLoRA(EMNLP 2025)는 어댑터를 베이스 projection에 *융합* 해 단일 matmul로 만들어 per-token 오버헤드를 **~0으로** 낮춘다(LoRA 대비 2.5× 가속, 파라미터 <1% 증가, 1B/3B/7B·NPU/GPU 검증). 따라서 NPU에서 본 변형을 실용화하려면 zFLoRA류 융합이 **사실상 필수** 이며, 신규 연구가 아니라 *가용한 기법의 적용* 문제다. (트레이드오프: 융합은 per-token을 0으로 만들지만 전환 시 *재융합* 비용이 생긴다 — 전환이 드문 본 시나리오에서는 유리.)

**남은 리스크.**
1. **어댑터 집합 빌드타임 고정** — 런타임 신규 어댑터 주입 불가. 단 본 과제의 추론 *유형* 은 소수·기지이고 동적 생성되는 것은 SA 인스턴스이므로(§2 후보 A 논리와 동일) 부담은 작다.
2. **어댑터 품질이 전용 모델급인가** — 협상 품질(NFR-MAF-04)을 어댑터로 낼 수 있는지 미확정.
3. **베이스 양자화 프로파일 공유 학습** — 어댑터가 베이스 양자화를 깨지 않도록 co-design 필요(QALFT/QUAD).
4. **벤더 종속** — 어댑터 메커니즘이 NPU 벤더별로 상이(QAIRT/CoreML/MediaTek).

**PoC 검증.** (a) 타깃 NPU에서 zFLoRA류 융합 어댑터의 per-token 오버헤드 ≈0 재현 (b) 협상 어댑터의 NFR-MAF-04 품질 (c) 베이스+N어댑터 메모리 증분이 P2 예산 내인지.

## 4. 후보 비교표

> **본 표는 §3.2 *필수 모듈* 기준이며, §3.4 택틱은 별점에 미반영.**
> **본 표는 *런타임 5축* 만 다룬다** — 아키텍처 품질속성(확장성·배포성·시험성·장애격리)의 트레이드는 §3.3 별표와 함께 보아야 한다. 본 5축만으로는 §3.3에서 드러난 "A=확장성·격리 vs B=단순·배포"의 비대칭이 가려질 수 있다.

| 평가축 | A. 추론 유형별 swap | B. 단일 범용 상주 |
|---|---|---|
| M1. 피크 메모리 | ★★ (무승부) | ★★ (무승부) |
| M2. 평균 메모리·에너지 | **★★★** | ★★ |
| M3. 추론 유형별 품질 | **★★★** | ★★ (범용, 협상 리스크) |
| M4. 에피소드 내 전환 지연(swap) | **★★** (경계 swap 1~3회, P6·P7 전제 — §5) | **★★★** (시분할) |
| M4b. 스텝 추론 속도(decode) | **★★★** (경량 스텝은 작은 모델로 빠르게) | ★★ (범용 大모델로 전 스텝 처리) |
| M5. 복잡도·유지보수 | ★★ | **★★★** |

> M1(피크)은 두 후보 모두 상주 모델 1개라 동일. M4의 *최초* cold-start도 동일하며, 차이는 **에피소드 내부의 추론 유형 전환**에서만 발생한다.
> **M4 가중치 단서** — 본 시나리오는 *백그라운드 비동기 협상* 이라 모델 로딩 *수초* 가 UX를 저해하지 않는다(§5 머리 노트·§7). 따라서 위 표의 M4 ★ 차이가 의사결정을 좌우할 가중치는 낮으며, 가중치를 반영하면 A의 약점이 크게 완화된다.

## 5. 권장안 — A(추론 유형별 swap) 우선

P6·P7(§0)을 *전제로* 깔면 협상 본진 중 swap이 일어나지 않고, 1 협상 에피소드당 swap은 *경계에서만* 1~3회로 수렴한다. 이 사실이 §5의 결론을 *바꾼다* — A의 본질적 약점이었던 "reload thrash"가 사실상 무력화되고, 남는 비용은 *복잡도(M5)* 와 *배포성*(§3.3) 뿐이다.

> **권장의 두 전제** — 이 둘이 모두 성립해야 본 권장이 유효하다:
> - **(a) 지연 허용** — A2A 협상은 백그라운드·비동기 동작이며, 모델 로딩 *수초*가 UX를 저해하지 않는다 (§7).
> - **(b) 협상 본진의 폐쇄성 (P6·P7)** — SA는 협상 경계에서만 호출되고, MA는 협상 중 다른 추론 모드로 전환하지 않는다.
>
> 둘 중 하나라도 깨지면 §5.3 *권장이 B로 환원되는 조건* 으로 이동한다.

### 5.1 권장 근거

1. **swap 빈도가 유한·예측 가능** — 협상 에피소드당 1~3회로 *경계 swap* 수준이다(P6·P7 직접 귀결). OS 가상메모리의 thrash와는 *규모가 다른* 현상이며, 지연 허용 전제 하에서 UX 비용 = 0, 누적 에너지 비용 = 협상당 ~1.8 J 이하(스마트폰 배터리의 <0.01%).
2. **협상 품질 우위** (M3 ★★★) — A는 협상 전용 모델을 *그대로* 쓴다. B의 범용 모델이 NFR-MAF-04(협상 품질)를 충분히 내지 못할 *원천 리스크* 를 차단한다. 본 DP가 특화 수단을 *모델 교체* 로 한정했으므로(P3), 협상 품질을 *구조적으로* 보장하는 유일한 길은 사실상 A다.
3. **평균 메모리·에너지 우위** (M2 ★★★) — 경계에서 호출되는 SA 툴 모델·판정 모델은 작은 모델로 대체 가능. 협상 본진 외 idle 시간엔 협상 모델조차 언로드 가능(택틱 — §3.4).
4. **아키텍처 확장성·격리 우위** (§3.3) — 새 추론 유형 추가 = 플러그인 등록. 불량 모델은 격리 가능.

### 5.2 A 채택의 대가 — 단순성·배포성 손실 (정직한 명시)

A의 본질적 약점은 *복잡도(M5)* 와 *배포성*(§3.3)이다:
- §3.2 필수 모듈 4개(레지스트리·라우터·스케줄러·수명주기) + §3.4 택틱(Swap 배칭·KV 캐시 관리·배포 파이프라인 등) — 구현 부담이 B보다 *확실히 크다*.
- 다(多) 모델 자산의 패키징·서명·OTA 업데이트가 필요해 배포성 별점이 낮다(§3.3 ★).

다만 이 비용은 *공학적으로 처리 가능* 한 부담이며, B를 택했을 때의 *NFR-MAF-04 미달 위험*(=구조적 품질 부족)에 비하면 *복구 가능한 비용* 이다. "단순함을 얻고 품질을 잃을 수 있는가"의 문제다.

### 5.3 B로 환원되는 조건 (명시)

다음 중 하나라도 성립하면 권장이 B로 뒤집힌다:

- **(a) 지연 허용 전제가 깨질 때** — 실시간 대화형 협상 UX가 도입되면 swap 지연이 다시 UX 비용이 되어 M4가 ★★ → ★로 후퇴.
- **(b1) P6이 깨질 때** — 협상 중 SA 재호출이 빈번(예: 매 라운드 가용성 재확인)하면 swap 수가 *수십 회* 로 증가해 thrash가 실제로 발생.
- **(b2) P7이 깨질 때** — 협상 중 MA 모드 전환(판정·요약 인터럽트)이 잦으면 동일한 thrash 문제.
- **(c) B의 범용 모델이 협상 품질을 만족** — PoC에서 B 단일 모델로도 NFR-MAF-04를 충분히 내면, *A의 복잡도(M5)·배포성(§3.3) 비용을 지불할 명분이 사라진다* → *M5 단순성* 만으로도 B가 정합적 권장이 된다.

### 5.4 결정에 필요한 사실 (§6과 연동)

본 권장은 다음 두 측정 전엔 *전제 위* 의 권장이다 — PoC에서 다음을 확인한 후 확정:

1. **P6 검증**: 실제 협상 시나리오에서 *협상 중 SA 재호출 빈도* 가 라운드당 0~0.x회 수준에 머무는지.
2. **B의 협상 품질 실측** (NFR-MAF-04): B의 범용 모델이 협상 품질을 만족하면 §5.3(c) 발동 — A의 복잡도 비용을 지불할 이유가 약해진다.

## 6. 후속 결정 (이 문서 다음에 풀어야 할 질문)

1. **SA 툴 호출의 LLM 여부**: SA의 툴 선택·인자 구성이 *독립 LLM 추론*인지, MA가 결정하고 SA는 *API 실행만* 하는지. 이것이 협상 중 교차 입도를 정하고, 곧 A/B 선택을 좌우한다(5.3).
2. **범용 모델의 협상 품질 실측**: B의 단일 범용 모델로 NFR-MAF-04(협상 라운드 품질)를 만족하는지 PoC 측정. 미달 시 §5.2(A로 선회).
3. **메모리 예산(P2) 정합**: IDS 1개 + MAF 1개(2모델)가 단말 RAM·NFR 임계값과 맞는지 확정하고, NFR-MAF-06·QAS-011을 재매핑한다(CLAUDE.md 7항).
4. **DP-D1과의 정렬**: 본 결정(MAF 추론 엔진 상주 모델 1개)을 전제로 DP-D1(자원 중재)의 스케줄링이 IDS 모델과 MAF 모델의 동시 상주·시분할을 다루도록 정렬한다.

> **검증 계획**: 위 2(품질)·3(메모리)의 *자원 측면*(메모리·전환 지연·추론 시간)은 [DP03-MAF-LLM-배치-PoC계획.md](./DP03-MAF-LLM-배치-PoC계획.md)에서 측정한다 — 정확도(품질) 검증은 그 후속이다.

## 7. 참고 — 본 문서의 가정과 한계

- 메모리는 정성적으로만 다룬다. "피크 무승부"는 *MAF 추론 엔진의 상주 모델이 1개*라는 전제(P2)의 직접 귀결이며, 절대 수치는 모델 선택·양자화·런타임 오버헤드에 따라 변한다.
- "평균 메모리 A 우세"는 *경량 추론 유형 구간에 작은 모델을 올릴 수 있을 때*만 성립한다. 모든 MAF 추론 유형이 결국 협상급 모델을 요구한다면 A의 평균 이점은 사라지고 A는 "B + 무의미한 reload"로 전락한다.
- 협상의 MA↔SA 교차 빈도, SA의 LLM 사용 수준은 [04-FR.md](../04-FR.md)에 근거한 *논리적 추론*이며 실제 빈도는 실측으로 확인해야 한다(6절-1·2).
- **지연 허용 전제**: 본 DP는 A2A 협상을 *백그라운드·비동기* 동작으로 가정한다 — 사용자가 즉시 응답을 기다리는 실시간 상호작용이 아니므로, **모델 로딩 *수초*는 허용 가능**하다고 본다(설계 판단). 그 결과 M4(전환 지연)의 비중이 낮아지고 A의 약점이 크게 완화된다. 이 전제가 깨지면(예: 실시간 대화형 협상 UX) 로딩 지연이 다시 치명적 약점이 되어 권장은 B로 회귀한다.
- IDS까지 포함한 더 넓은 후보 비교는 [DP-memory-optimization.md](./DP-memory-optimization.md)에 보존되어 있다.

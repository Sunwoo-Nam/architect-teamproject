# DP02 — 정보 공개 모드

---

## 1. 풀고자 하는 문제

상대 에이전트가 판단(협상 후보 생성, 협업 가능 여부 등)에 필요한 정보를 요구할 때, 우리 MAF가 어떤 **공개 모드**로 응답할지 정한다.

기존 [DP02-최소 공개 및 Private Knowledge 차단](<./DP02-[OLD][보관]최소 공개 및 Private Knowledge 차단.md>)은 세 후보 — A(사실 공개), B(판정만 반환), C(점진 공개) — 를 두었다. 그러나 검토 결과, 가용 시간처럼 질의를 충분히 반복하면 전체를 알아낼 수 있는 데이터에서는 A와 B의 공개량이 수렴한다. B의 판정(yes/no)도 질의가 충분히 누적되면 A가 한 번에 주는 것과 같은 시간 윈도우를 노출하기 때문이다. C는 협상이 정보를 필요로 하는 한 항상 승급해 A와 같아진다.

따라서 실제로 공개량을 가르는 축은 "누가 계산하는가(A/B)"가 아니라 **"가용 정보를 상대에게 노출하는가"** 이다. 이 축으로 두 대안을 정의한다.

- **대안 1 — 제한 공개형**: 가용 정보를 상대에게 노출하되 양·정밀도를 한정한다.
- **대안 2 — 비노출 협력 계산형**: 가용 정보를 노출하지 않고 합의 가능한 결과만 함께 산출한다.

핵심 품질은 QAS-012(Private Knowledge 외부 전송 0건)와, 그 0건으로는 잡히지 않는 **추론 저항**이다.

## 2. 아키텍처적 난제

| 난제 | 내용 | 관련 FR/NFR/QAS |
|---|---|---|
| **직접 사유 0건 전송** | 진료·종교·거주 같은 Private Knowledge 자체는 어느 경우에도 외부로 나가면 안 된다. | FR-MAF-05, NFR-MAF-07, QAS-012 |
| **추론 저항** | 직접 사유를 안 보내도, 가용 시간대를 공개하거나 판정을 반복하면 여집합·패턴으로 사유가 역추론된다. QAS-012의 "0건"으로는 막히지 않는다. | NFR-MAF-07, QAS-012 |
| **유용성 vs 최소 공개** | 너무 적게 주면 상대가 후보를 만들지 못해 라운드가 늘고, 너무 많이 주면 노출이 커진다. | FR-MAF-02~06, QAS-007 |
| **Communication 시간** | 라운드 증가·암호 연산이 시간 예산을 압박한다. | QAS-009 |
| **온디바이스 자원** | 분류·암호 연산이 CPU·메모리 예산을 소모한다. | QAS-011 |
| **케이스별 적용성** | 일정(집합 교집합)과 지식 공유(내용 전달)는 공개 모드의 성격이 다르다. | FR-MAF-02~06 |

두 대안이 공유하는 전제:

1. **구조화 경계.** 외부로 나가는 메시지는 구조화 필드만 사용한다. 자유 텍스트 LLM 출력을 그대로 전송하지 않는다. 이것이 0건을 결정론적으로 보장하는 전제다.
2. **직접 사유 비전송.** FR-MAF-05 분류에서 Private Knowledge로 분류된 항목(사유)은 두 대안 모두 전송하지 않는다. 본 DP가 가르는 것은 "가용 정보(가능한 시간·조건)"의 공개 방식이다.
3. **이산화 가능성.** 후보(예: 시간 슬롯)는 유한·열거 가능한 집합으로 이산화할 수 있어야 한다. 특히 대안 2의 전제다.
4. **추론 가드.** query budget·rate limit·입력 최소 크기는 두 대안 공통 방어 수단이다.
5. **전송 암호화.** TLS 1.3 이상은 공통 필수이며 본 DP의 비교 대상이 아니다.

## 3. 대안 1 — 제한 공개형 (Limited Disclosure)

### 개념

한쪽이 자신의 가용 정보를 상대에게 공개하고, 상대가 그것으로 후보를 만든다. 노출은 **query budget**(공개 횟수·범위)과 **정밀도 coarsening**(정확 시각 대신 1시간 단위, 정확 위치 대신 구 단위)으로 한정한다. 기존 A·B·C는 모두 이 대안의 변형이며, 공개 시점(일괄/점진)과 예산만 다르다.

| 케이스 | 외부로 나가는 것(예) |
|---|---|
| Negotiation | `available=[Sat 19:00–22:00]`, `area=Gangnam` (coarse) |
| Collaboration | `can_join=true`, `role=reviewer` |
| Knowledge Sharing | `summary_level=coarse`, `category=public` |
| Remote Monitoring | `alert=inactivity`, `severity=medium` |

### 장점

- 상대가 한 번에 충분한 재료를 받아 후보를 만들기 쉬워 라운드가 적다(QAS-009에 유리).
- 암호 연산이 없어 온디바이스 비용이 낮다(QAS-011에 유리).
- 전송 payload가 명시적이라 이력 기록·검증이 단순하다.

### 위험

- **추론 저항이 구조적으로 약하다.** 가용 정보 자체가 외부로 나가므로, 여집합("일요일 전부 불가")과 반복 누적으로 사유가 역추론된다. coarsening·budget는 노출을 줄일 뿐 제거하지 못한다.
- 안전성이 FR-MAF-05 분류·익명화 정책의 정확성에 의존한다. 정책이 부실하면 과공개로 이어진다.
- QAS-012의 "0건"은 직접 사유 기준이며, 추론 경로는 별도 측정·정의가 필요하다.

## 4. 대안 2 — 비노출 협력 계산형 (Joint Computation without Disclosure)

### 개념

양측이 가용 슬롯 집합을 서로에게 공개하지 않고, 협력 프로토콜로 **교집합만** 산출한다(private set intersection 계열, 또는 일반화된 2자 보안 계산). 결과로 공통 슬롯만 드러나고, 교집합 밖(상대의 다른 가용 시간, 불가 구간)은 공개되지 않는다.

| 케이스 | 상대 입력 | 산출 결과 |
|---|---|---|
| Negotiation (일정) | 상대 가용 슬롯 집합 | 공통 슬롯(교집합) |
| Collaboration | 상대 가용 자원·시간 집합 | 공통 가능 구간 |

### 필수 메커니즘

- **이산화된 후보 집합.** 시간을 고정 단위(예: 30분) 슬롯으로 열거한다.
- **교환법칙 암호화(commutative encryption) 기반 PSI.** 각자 비밀 키로 슬롯을 암호화하고, 양측 키를 모두 적용한 값이 입력이 같으면 일치한다는 성질로 교집합을 찾는다. 키 적용 순서와 무관하다(`H(x)^ab = H(x)^ba`).
- **입력 최소 크기·실행 횟수 제한.** 값을 테스트하는 유일한 방법이 "자기 입력 집합에 넣는 것"이므로, 단일 슬롯 입력을 통한 오라클 공격을 막으려면 입력 최소 크기와 budget이 필요하다.
- **(선택) 매개형 변형.** 신뢰된 서버(Orchestrator)가 교집합을 계산할 수도 있으나, 이 경우 신뢰 경계가 서버로 옮겨진다(서버가 양측 가용 시간을 모두 보게 됨).

### 장점

- **추론 저항이 강하다.** 교집합 밖은 암호화된 무작위 값으로 남아 여집합·패턴 단서가 되지 않는다. 대안 1로는 구조적으로 확보할 수 없는 속성이다.
- 직접 사유뿐 아니라 "가능한데 합의에 쓰이지 않은 시간"도 노출되지 않는다.

### 위험

- **교집합 자체는 노출된다.** 상대가 입력을 단일 슬롯으로 좁히면 오라클이 되므로, 입력 크기·횟수 제한에 의존한다. PSI가 감추는 것은 교집합 밖이지 교집합 안이 아니다.
- **on-device 비용이 불확실하다.** 슬롯 수만큼 지수승 연산 + 2~3 통신 라운드가 QAS-009(60초, LLM 추론과 동시)·QAS-011(Memory 300MB, CPU 피크 40%) 안에 드는지 측정해야 한다.
- **N:N에서는 지연이 더 커진다.** 참여자 N이 늘수록 암호 프로토콜의 라운드·연산이 늘어 동작 시간이 대안 1보다 빠르게 증가한다(실측 필요).
- **적용 범위가 좁다.** 집합 교집합 형태(일정)에 맞고, Knowledge Sharing의 내용 전달·다속성 협상(가격·조건 trade-off)·Collaboration 워크플로에는 부적합하다.
- **구현·검증 복잡도가 높다.** 0건 보장이 정책이 아니라 암호 구현의 정확성에 의존한다.

## 5. 종합 비교 (Quality Attribute & 영향 정보)

> 척도: ★★★ 강함 · ★★☆ 보통 · ★☆☆ 취약/불확실

| 평가축 | 대안 1 제한 공개형 | 대안 2 비노출 협력 계산형 | 판단 |
|---|:---:|:---:|---|
| 직접 사유 0건 (QAS-012) | ★★★ | ★★★ | 사유 자체는 둘 다 전송하지 않는다 |
| 추론 저항 | ★☆☆ | ★★★ | 1은 가용 정보가 나가 누적된다. 2는 교집합 밖 비공개(단 교집합 오라클은 입력 제한으로 방어) |
| 유용성·라운드 (QAS-007) | ★★★ | ★★☆ | 1은 후보 생성이 쉽다. 2는 교집합이 있으면 1회, 없으면 재실행 |
| Communication 시간 (QAS-009) | ★★★ | ★☆☆~★★☆ | 1은 암호 비용이 없다. 2는 암호 라운드·연산이 있어 측정이 필요하다 |
| 온디바이스 자원 (QAS-011) | ★★★ | ★★☆ | 1은 분류·직렬화만. 2는 암호 연산 CPU, 측정 필요 |
| 복잡도·검증 | ★★★ | ★☆☆ | 1은 단순·명시적. 2는 0건이 암호 구현에 의존 |
| 적용 범위 | 전 케이스 | 일정(집합 교집합)에 한정 | 2는 내용 전달·다속성 협상에 부적합 |

**핵심 긴장:** 대안 1은 단순하고 모든 케이스에 적용되지만, 가용 정보가 외부로 나가므로 추론 저항이 구조적으로 약하다. 대안 2는 추론 저항이 강하지만(대안 1로는 확보할 수 없는 속성), 그 대가로 암호 실현성(측정 필요), 좁은 적용 범위, 구현 복잡도를 부담한다. 따라서 이 DP의 선택은 **추론 저항을 어디까지 요구하는가**와 **대안 2의 on-device 비용이 QAS-009·QAS-011 예산에 드는가**의 측정 결과로 좁혀진다. 또한 4대 케이스 중 집합 교집합형(일정)이 차지하는 비중에 따라, 대안 2를 일정 전용으로 두고 나머지를 대안 1로 처리하는 분리 구조가 검토 대상이 된다.

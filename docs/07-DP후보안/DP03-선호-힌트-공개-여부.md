# DP03 — A2A 협상 메시지 구조

> 본 문서는 NegMAS 기반 협상에서 **순수 NegMAS 메시지(`response_type + outcome`)만 보낼 것인가**, 아니면 **당사 기기간 상위 A2A Envelope에 구조화된 선호 힌트(preference hint)를 추가할 것인가**를 다룬다.
> **핵심 쟁점:** `ufun`을 완전히 비공개로 유지해 기밀성과 상호운용성을 우선할 것인가(순수 NegMAS 메시지), 아니면 제한된 선호 힌트를 포함한 확장 Envelope로 협상 라운드 수와 실패율을 줄일 것인가(확장 Envelope 메시지).

구조도: [DP03-A2A-협상-메시지-구조.drawio](./DP03-A2A-협상-메시지-구조.drawio)

---

## 1. 아키텍처적 난제

DP03의 공통 전제는 다음과 같다.

- A2A 협상 엔진은 **NegMAS(SAO)** 로 둔다.
- NegMAS의 협상 본체는 **고정된 OutcomeSpace 위의 offer + accept/reject** 로 진행된다.
- `ufun`은 단말 내부 효용함수이며 상대에게 전송하지 않는다.
- 사유, PII, 원문은 외부 메시지의 어휘와 인터페이스에서 배제한다.
- 상대 PPA는 기본적으로 외부 에이전트지만, 초기 제품 범위에서는 **당사 기기간 협상**이 주된 운영 조건이다.

이 전제에서 남는 핵심 질문은 다음이다.

> **협상 효율을 위해 `ufun`의 일부 성질을 구조화 힌트로 외부 A2A 메시지에 포함할 것인가?**

| 쟁점 | 순수 NegMAS 메시지 측면 | 확장 Envelope 메시지 측면 |
|---|---|---|
| **기밀성** | 상대는 offer 패턴만 본다. 선호, 유보값, 양보 가능성은 직접 노출되지 않는다. | 힌트가 사유를 직접 담지 않더라도 선호 정보를 추가로 노출한다. 반복되면 사용자 제약을 추론할 수 있다. |
| **협상 효율** | 상대가 우리 선호 구조를 모르므로 시행착오 라운드가 늘 수 있다. | 상대가 양보 방향과 hard constraint를 빨리 파악해 라운드 수를 줄일 수 있다. |
| **Task 성공률** | 다속성 협상에서 합의 후보 탐색이 느리거나 결렬될 수 있다. | 시간, 장소, 예산 등 여러 축이 있는 협상에서 합의 가능성을 높일 수 있다. |
| **프로토콜 복잡도** | 순수 NegMAS 사용에 가깝다. 별도 A2A 확장이 거의 없다. | NegMAS 자체가 아니라 상위 A2A envelope에 hint block, 버전, fallback을 추가해야 한다. |
| **상호운용성** | 외부 PPA와도 기본 동작 가능성이 높다. | 당사 기기간에는 가능하지만, 외부 PPA에는 capability handshake와 미지원 fallback이 필요하다. |
| **감사와 복구** | offer, accept/reject, 내부 ufun만 관리하면 된다. | 공개 힌트의 생성 근거, 버전, 송수신 이력까지 감사·복구 대상이 된다. |

## 2. 후보 A — 순수 NegMAS 메시지 + Private ufun

### 개념

상대 PPA로 나가는 협상 메시지는 NegMAS offer와 accept/reject로 제한한다. 사용자의 선호, 유보값, 양보 가능성, 거절 이유는 모두 단말 내부 `ufun`과 협상 전략에만 둔다.

### 메시지 구조

| 구분 | 외부 전송 여부 | 설명 |
|---|---:|---|
| Outcome offer | O | 안전 OutcomeSpace 안의 값 조합 |
| accept/reject | O | NegMAS 응답 타입 |
| 선호 강도 | X | `ufun` 내부 점수로만 사용 |
| 유보값 | X | 수락 임계값으로만 사용 |
| 양보 가능성 | X | 협상 전략 내부 상태로만 사용 |
| 거절 사유 | X | 사유는 외부 어휘에 존재하지 않음 |

### 강점

- **NegMAS 모델과 정합적이다.** 별도 프로토콜 확장 없이 `ResponseType + Outcome` 중심으로 협상할 수 있다.
- **기밀성 보장이 단순하다.** 상대에게 나가는 것은 안전 OutcomeSpace의 값뿐이므로, 출구 정의만 검토하면 된다.
- **외부 PPA 상호운용성이 높다.** 상대가 당사 확장 스키마를 몰라도 기본 협상이 가능하다.
- **세션 복구가 단순하다.** frozen OutcomeSpace, 협상 상태, 내부 ufun만 복구하면 된다.

### 약점

- **협상 라운드 수가 늘 수 있다.** 상대가 어느 축에서 양보해야 하는지 offer 패턴만 보고 추정해야 한다.
- **다속성 협상에서 결렬 위험이 커질 수 있다.** 시간, 장소, 가격, 식이 제약이 동시에 얽히면 상대가 유효 후보를 늦게 찾는다.
- **N-party 협상에서 메시지 왕복이 증가할 수 있다.** 참여자가 늘수록 시행착오 offer가 누적된다.

## 3. 후보 B — 확장 Envelope 메시지 + Structured Preference Hints

### 개념

NegMAS 메커니즘 자체를 수정하지 않고, 당사 기기간 A2A envelope에 **구조화된 선호 힌트 block** 을 추가한다. 즉, 협상 엔진은 계속 NegMAS를 사용하지만, 제품 프로토콜은 NegMAS 응답을 감싸는 상위 메시지를 갖는다.

중요한 경계:

- **NegMAS 내부 메커니즘 확장은 본 후보가 아니다.** `SAOMechanism` 자체를 바꾸지 않는다.
- 힌트는 OutcomeSpace issue로 넣지 않는다. 힌트가 합의 결과의 일부처럼 취급되면 의미가 흐려진다.
- 힌트는 자유 텍스트가 아니라 사전 정의 enum, bool, bucket만 허용한다.
- 외부 또는 미지원 PPA와 협상할 때는 순수 NegMAS 메시지로 fallback한다.

### 메시지 구조

예시 envelope:

```yaml
a2a_negotiation_message:
  protocol: negmas-sao
  protocol_version: "1.0"
  capability:
    preference_hint: true
  negmas:
    response_type: REJECT_OFFER
    outcome:
      slot: Sat_evening
      area: Gangnam
      budget_cap: under_50000
  preference_hint:
    schema_version: "hint.v1"
    slot_preference: high
    area_flexibility: medium
    budget_flexibility: low
    hard_constraints:
      - slot
```

허용 가능한 힌트 예시:

| 힌트 | 허용 값 | 의미 | 금지 사항 |
|---|---|---|---|
| `slot_preference` | `high`, `medium`, `low` | 현재 offer의 시간 선호 강도 | 사유, 일정 원문, 정확 시간 |
| `area_flexibility` | `high`, `medium`, `low` | 장소 양보 가능성 | 집 주소, 현재 위치, 이동 사유 |
| `budget_flexibility` | `high`, `medium`, `low` | 예산 양보 가능성 | 정확 예산, 잔고, 재정 사유 |
| `hard_constraints` | 승인 issue 이름 목록 | 절대 양보 불가 축 | 질병, 종교, 가족사 등 원인 |
| `concession_phase` | `early`, `normal`, `final` | 협상 전략 단계 | 감정 표현, 압박 사유 |

### 강점

- **라운드 수를 줄일 수 있다.** 상대가 어느 축에서 양보해야 하는지 더 빨리 파악한다.
- **Task 성공률을 높일 수 있다.** 다속성 협상에서 무의미한 후보 탐색을 줄인다.
- **당사 기기간 최적화에 적합하다.** 양쪽 PPA가 같은 envelope 스키마와 hint semantics를 공유한다면 구현 난도가 관리 가능하다.
- **N-party 협상에서 수렴 신호가 될 수 있다.** 참여자가 늘어날수록 offer만으로 선호를 추정하는 비용이 커지므로, 힌트가 조율 비용을 줄일 여지가 있다.

### 약점

- **선호 정보가 새로운 개인정보 노출 지점이 된다.** 사유를 직접 보내지 않아도, 반복된 `hard_constraints`와 낮은 flexibility가 사용자 제약을 암시할 수 있다.
- **A2A 프로토콜 확장이 필요하다.** hint block, schema version, capability handshake, fallback 정책을 정의해야 한다.
- **감사 대상이 늘어난다.** offer뿐 아니라 힌트 생성 규칙과 힌트 이력도 검토해야 한다.
- **외부 PPA 상호운용성이 낮다.** 상대가 hint 스키마를 모르면 무시하거나 잘못 해석할 수 있다.

## 4. 종합 비교 및 아키텍처 딜레마

> 평가축은 DP03 개인정보 처리 구조에서 사용하는 6개 품질속성(기밀성, Latency, 자원, Task 성공률, 세션 복구, 유지보수성)을 따른다.  
> 척도: ★ = 1점, ☆ = 0.5점. 별점은 설계 비교용 정성 점수이며, 기존 PoC 결과에 근거한 실측값이 아니다. 성능·성공률 차이는 새 PoC 또는 후속 검증에서 확인해야 한다.

| 품질속성/설계 관점 | 후보 A<br/>순수 NegMAS 메시지 + Private ufun | 후보 B<br/>확장 Envelope 메시지 + Preference Hints | 판단 |
|---|:---:|:---:|---|
| **기밀성** | **★★★★★** | **★★★☆** | A는 외부 전송 항목이 offer로 제한된다. B는 사유·PII를 금지해도 선호 강도와 hard constraint가 민감한 패턴을 만들 수 있다. |
| **Latency** | **★★★☆** | **★★★★☆** | A는 시행착오 라운드가 늘 수 있다. B는 양보 방향을 알려 라운드 수를 줄일 가능성이 있다. |
| **자원** | **★★★★☆** | **★★★☆** | A는 메시지와 검증 상태가 작다. B는 hint block, 버전, 로그, 검증 규칙이 추가된다. 단, 라운드 수가 줄면 총 비용은 낮아질 수 있다. |
| **Task 성공률** | **★★★☆** | **★★★★☆** | A는 상대가 선호 구조를 offer 패턴으로만 추정한다. B는 다속성 협상에서 합의 후보 탐색을 빠르게 할 수 있다. |
| **세션 복구** | **★★★★☆** | **★★★☆** | A는 frozen OutcomeSpace와 협상 상태 중심으로 복구한다. B는 어떤 힌트를 언제 공개했는지까지 복구·재현해야 한다. |
| **유지보수성** | **★★★★☆** | **★★★** | A는 순수 NegMAS 사용에 가깝다. B는 hint schema, capability negotiation, fallback, privacy review를 계속 관리해야 한다. |
| **상호운용성**<br/>(보조 관점) | **★★★★☆** | **★★★** | A는 외부 PPA와 기본 호환 가능성이 높다. B는 당사 기기간에는 유리하지만 외부 PPA에는 fallback이 필요하다. |
| **종합** | **기밀성·단순성·상호운용성 우세** | **Latency·Task 성공률·N-party 수렴 우세** | 선택은 외부 전송 정보를 최소화할지, 당사 기기간 협상 효율을 적극 최적화할지에 달려 있다. |

### 핵심 긴장

> **"선호를 숨겨 외부 전송 정보를 최소화할 것인가(A) vs 제한된 선호 힌트를 포함해 협상 수렴을 빠르게 할 것인가(B)"**

두 후보 모두 사유, PII, 원문을 직접 보내지 않는다. 차이는 `ufun`의 일부 성질을 외부 A2A 메시지에 포함하는지 여부다.  
후보 A는 NegMAS의 기본 구조에 더 가깝고 개인정보 보호가 강하다. 대신 상대가 offer 패턴만으로 선호를 추정해야 하므로 협상 라운드와 결렬 가능성이 늘 수 있다.  
후보 B는 당사 기기간 같은 스키마를 공유한다는 조건에서 협상 효율을 높일 수 있다. 그러나 힌트 필드 자체가 새 개인정보 노출 지점이 되므로, 힌트의 어휘·빈도·누적 이력을 강하게 제한해야 한다.

## 5. 프로토콜 확장 범위

본 DP에서 고려하는 확장 범위는 아래와 같이 제한한다.

| 범위 | 포함 여부 | 판단 |
|---|---:|---|
| 순수 NegMAS 메시지 | 포함 | 후보 A. 기본 동작 및 fallback 경로 |
| 당사 기기간 A2A envelope 확장 | 포함 | 후보 B의 주 구현 방식. `preference_hint` block을 상위 메시지에 추가 |
| 외부 PPA와의 hint 상호운용 | 제한적 포함 | capability handshake와 미지원 fallback만 고려 |
| NegMAS `SAOMechanism` 자체 수정 | 제외 | 공수와 리스크가 크고, NegMAS 사용 이점을 약화시킴 |
| 힌트를 OutcomeSpace issue로 추가 | 제외 권장 | 힌트가 합의 결과의 일부로 오해되어 의미가 흐려짐 |

구현상 후보 B는 다음 정책을 가져야 한다.

1. 상대가 `preference_hint` capability를 명시적으로 지원할 때만 힌트를 보낸다.
2. capability가 없거나 버전이 다르면 순수 NegMAS 메시지로 fallback한다.
3. 힌트는 Privacy Gateway가 구성·검증한 enum/bool/bucket만 허용한다.
4. 힌트는 사유를 설명하지 않는다. "왜"가 아니라 "협상 축의 양보 가능성"만 표현한다.
5. 세션 로그에는 보낸 힌트와 그 스키마 버전을 기록한다.

## 6. 설계 판단

| 상황 | 권장 후보 | 이유 |
|---|---|---|
| 외부 또는 미지원 PPA와 협상 | 후보 A | 상호운용성과 기밀성 우선. hint fallback 불필요 |
| 당사 기기간 1:1 협상, 단순 도메인 | 후보 A 우선 | 힌트 없이도 라운드 수가 충분히 작다면 새 전송 항목을 만들 필요가 없음 |
| 당사 기기간 다속성 협상 | 후보 B 검토 | 시간·장소·예산·조건이 동시에 얽히면 offer-only의 시행착오 비용이 커질 수 있음 |
| N-party 협상 | 후보 B 검토 | 참여자가 늘수록 offer-only로 선호를 추정하는 비용이 증가 |
| 민감도가 높은 도메인 | 후보 A 우선 | 선호 힌트가 사유 추론으로 이어질 가능성이 큼 |

초기 권고는 **후보 A를 기본값으로 두고, 후보 B를 당사 기기간 최적화 옵션으로 제한 적용**하는 것이다.  
즉, 제품 기본 프로토콜은 순수 NegMAS 메시지만으로 성립해야 하며, Preference Hints는 다음 조건을 만족할 때만 켠다.

- 상대 PPA가 같은 hint schema를 지원한다.
- 힌트 필드가 privacy review를 통과했다.
- hint 적용으로 라운드 수 또는 Task 성공률 개선이 새 PoC 또는 후속 검증에서 확인된다.
- 미지원 상대와의 fallback이 검증되어 있다.

## 7. 검증 계획

본 검증 계획은 기존 DP02/DP03 PoC 결과를 전제하지 않는다. 선호 힌트 공개 여부를 독립 변수로 둔 새 PoC 또는 별도 실험으로 확인해야 한다.

### 7.1 기능·성능 검증

| 측정 항목 | 목적 |
|---|---|
| 합의까지 라운드 수 | 후보 B가 실제로 협상 수렴을 개선하는지 확인 |
| Task 성공률 | offer-only 대비 결렬률 감소 여부 확인 |
| 총 지연 시간 | hint 추가 비용보다 라운드 감소 효과가 큰지 확인 |
| 메시지 크기 | hint block 추가에 따른 네트워크·로그 비용 확인 |
| N-party 참여자 증가 시 라운드 수 | QAS-014 관점에서 수렴 비용 확인 |

### 7.2 개인정보 검증

| 측정 항목 | 목적 |
|---|---|
| 힌트 필드별 민감도 리뷰 | 특정 힌트가 질병·종교·재정난 등 사유를 직접 암시하는지 확인 |
| 누적 힌트 기반 추론 가능성 | 여러 라운드의 힌트를 합치면 사유 추론이 가능한지 확인 |
| 사유·PII·원문 포함 0건 | 구조화 힌트에도 기존 DP03 불변식이 유지되는지 확인 |
| 미지원 상대 fallback | 외부 PPA에 힌트가 잘못 전송되지 않는지 확인 |

### 7.3 성공 기준 초안

- 후보 B가 후보 A 대비 평균 라운드 수 또는 총 지연을 유의미하게 줄인다.
- 후보 B의 hint block에서 사유·PII·원문 전송은 0건이다.
- privacy review에서 특정 hint 조합이 고위험 사유를 직접 암시하지 않는다.
- hint 미지원 상대와의 협상은 순수 NegMAS 메시지로 정상 fallback한다.

## 8. 근거 및 범위

- [DP03-전제-및-공통설계](./DP03-전제-및-공통설계.md): NegMAS, OutcomeSpace freeze, ufun 비전송, 구조화 출구 전제.
- [DP03-개인정보-처리-게이트웨이-구조](./DP03-개인정보-처리-게이트웨이-구조.md): 개인정보 처리 게이트웨이의 기존 비교안 및 품질속성 평가 방식.
- [DP03-개인정보-재검토-설계포인트](./DP-개인정보-재검토/DP03-개인정보-재검토-설계포인트.md): 본 DP의 후보 아이디어 출처.

본 문서의 비교는 위 전제와 A2A 메시지 구조에 따른 정성 판단이다. 성능, 성공률, 라운드 수 차이는 아직 확정된 수치가 아니며, 선호 힌트 포함 여부를 독립 변수로 둔 검증이 필요하다.

기존 PoC 리포트는 본 DP의 근거로 사용하지 않는다. A2A 협상 메시지 구조는 기존 PoC의 `filter-at-egress`/`transform-at-ingress` 비교와 다른 결정축이므로, 별도 PoC 설계가 필요하다.

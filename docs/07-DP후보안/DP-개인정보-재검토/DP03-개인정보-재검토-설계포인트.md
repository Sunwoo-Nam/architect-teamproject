# DP03 개인정보 처리 구조 재검토 설계 포인트

> 목적: 기존 `Monolithic Gateway` vs `Federated Adapters` 비교는 실행 배치 차이에 가깝고, 두 대안이 공통으로 "구조화 메시지 · 사유/PII/원문 배제 · 안전 OutcomeSpace"를 전제하면서 주요 품질속성 간 충돌이 약해졌다. 본 문서는 DP03 개인정보 처리 구조에서 다시 검토할 만한 설계 포인트 4개를 정리한다.

---

## 0. 검토 전제

본 문서는 아래 전제를 유지한 상태에서 설계 포인트를 찾는다.

- A2A 협상은 NegMAS 기반이다. 협상 메시지는 미리 정의된 OutcomeSpace 안의 offer와 accept/reject로 제한되며, 자유 텍스트 채널을 두지 않는다.
- `ufun`은 단말 내부의 효용함수로 유지되며 상대에게 전송하지 않는다.
- 사유, PII, 원문은 런타임 탐지가 아니라 어휘와 인터페이스 단계에서 배제한다.
- Cloud LLM으로 나가는 계획/재계획 요청도 구조화 메시지로 제한한다.
- 합의 후 실행에서만 PII Vault의 원본을 복원한다.

검토 기준 문서:

- [DP03-전제-및-공통설계](../DP03-전제-및-공통설계.md)
- [DP03-개인정보-처리-게이트웨이-구조](../DP03-개인정보-처리-게이트웨이-구조.md)
- [DP03-정규화-시점-및-세션-freeze-경계-설계대안](../DP03-정규화-시점-및-세션-freeze-경계-설계대안.md)

기존 PoC 리포트는 본 문서의 근거로 사용하지 않는다. 아래 설계 포인트들은 기존 PoC의 결론이 아니라, DP03 공통 전제와 NegMAS 기반 협상 구조에서 새로 검토할 후보 결정축이다. 각 후보의 성능·성공률 차이는 별도 PoC 또는 후속 검증으로 확인해야 한다.

## 1. 설계 포인트 A: 협상 효율을 위해 선호 힌트를 공개할 것인가

### 결정 질문

상대 PPA에는 NegMAS offer만 보낼 것인가, 아니면 협상 수렴을 빠르게 하기 위해 구조화된 선호 힌트까지 일부 공개할 것인가?

### 대안 A-1. Offer-only + Private ufun

상대에게는 OutcomeSpace 안의 offer와 accept/reject만 전송한다. 선호, 유보값, 양보 가능성, 거절 이유는 모두 단말 내부 `ufun`에만 둔다.

### 대안 A-2. Structured Preference Hints

상대에게 offer 외에 제한된 구조화 힌트를 보낸다. 예시는 다음과 같다.

| 힌트 유형 | 예시 | 금지해야 할 내용 |
|---|---|---|
| 선호 강도 | `slot_preference=high/medium/low` | "병원 때문에 안 됨" 같은 사유 |
| 양보 가능성 | `flexibility=low` | 구체 사정, 감정, 원문 |
| hard constraint 여부 | `slot_is_hard_constraint=true` | 원인 질환, 종교, 재정 상황 |
| reservation bucket | `price_reservation=under_50000` | 정확 예산, 계좌, 결제 정보 |

### 품질속성 trade-off

| 품질속성 | Offer-only + Private ufun | Structured Preference Hints |
|---|---|---|
| 기밀성 | 강함. 상대가 보는 것은 offer 패턴뿐이다. | 약해짐. 사유를 직접 보내지 않아도 선호 표면이 사용자 상황을 암시할 수 있다. |
| Latency | 협상 라운드가 늘 수 있다. | 상대가 양보 방향을 빨리 파악해 라운드 수가 줄 수 있다. |
| Task 성공률 | 정보가 적어 합의 후보 탐색이 느리거나 실패할 수 있다. | 다속성 협상에서 합의 가능성이 높아질 수 있다. |
| 자원 | 메시지와 상태가 작다. | 힌트 필드와 검증 규칙이 추가된다. |
| 세션 복구 | offer와 ufun만 복구하면 되어 단순하다. | 공개 힌트 이력과 버전도 복구해야 한다. |
| 유지보수성 | 어휘 관리 범위가 작다. | 힌트 스키마, 공개 한계, 도메인별 의미를 관리해야 한다. |

### 왜 중요한 결정인가

현재 설계는 `ufun`을 비공개로 둔다. 이 선택은 기밀성에는 강하지만, 협상 효율을 일부 포기하는 구조다. 반대로 선호 힌트를 보내면 협상 품질과 지연을 개선할 수 있으나, "사유를 직접 말하지 않아도 선호 패턴으로 민감한 상황이 드러나는" 문제가 생긴다.

따라서 이 포인트는 `기밀성(QAS-012)`과 `Latency(QAS-009)`, `Task 성공률(QAS-007)`, `N-party 확장성(QAS-014)`이 정면으로 충돌한다. 기존 16.png의 게이트웨이 배치 비교보다 더 핵심적인 개인정보 설계 결정이다.

### 검증 방법

- 동일 시나리오에서 offer-only와 preference-hint 방식을 비교한다.
- 측정 항목:
  - 합의까지 라운드 수
  - 합의 성공률
  - 메시지 필드 수와 토큰 수
  - 힌트만 보고 사유를 추론할 수 있는지에 대한 별도 privacy review

## 2. 설계 포인트 B: Cloud LLM 계획 요청에 얼마나 많은 구조화 맥락을 보낼 것인가

### 결정 질문

Cloud LLM에는 최소한의 capability 요청만 보낼 것인가, 아니면 안전화된 협상 맥락을 풍부하게 보내 계획 품질을 높일 것인가?

### 대안 B-1. Minimal Plan Request

Cloud LLM에는 계획 작성에 필요한 최소 구조만 보낸다.

예시:

```yaml
intent_type: negotiation
required_capabilities:
  - calendar_lookup
  - restaurant_search
  - reservation
privacy_level: strict
replan_trigger: tool_failure
```

### 대안 B-2. Rich Sanitized Context

Cloud LLM에는 안전화된 맥락을 더 많이 보낸다.

예시:

```yaml
intent_type: negotiation
case: dinner_schedule
safe_context:
  time_bucket: Sat_evening
  area_bucket: Gangnam
  budget_bucket: under_50000
  constraints:
    - avoid_late_finish
    - location_nearby
negotiation_state:
  round_count: 2
  failure_code: no_common_slot
required_capabilities:
  - calendar_lookup
  - restaurant_search
  - reservation
```

### 품질속성 trade-off

| 품질속성 | Minimal Plan Request | Rich Sanitized Context |
|---|---|---|
| 기밀성 | 강함. 외부 Cloud LLM이 받는 사용자 맥락이 최소다. | 약해짐. 각 값은 안전해도 조합하면 사용자 상황 프로파일이 될 수 있다. |
| Latency | Cloud가 맥락 부족으로 재질문·재계획을 반복할 수 있다. | 더 적은 호출로 적절한 계획을 만들 가능성이 있다. |
| Task 성공률 | 계획 품질이 낮아 tool 선택 실패가 늘 수 있다. | 더 정확한 tool 구성과 재계획이 가능하다. |
| 자원 | 메시지와 검증 규칙이 작다. | 요청 payload와 검증 규칙이 커진다. |
| 세션 복구 | 저장해야 할 외부 요청 상태가 작다. | 계획 요청 이력과 안전 맥락 버전을 함께 복구해야 한다. |
| 유지보수성 | 스키마가 단순하다. | 도메인별 안전 맥락 스키마를 계속 관리해야 한다. |

### 왜 중요한 결정인가

현재 DP03 전제는 Cloud LLM행 출구가 반복적으로 존재한다고 본다. 초기 계획뿐 아니라 재계획도 Cloud LLM 호출로 이어질 수 있다. 상대 PPA 채널은 NegMAS가 강하게 닫아주지만, Cloud LLM 계획 요청은 별도 출구이므로 개인정보 처리 구조의 핵심 위험 지점이 된다.

이 결정은 "외부 지능에 얼마만큼 맡길 것인가"와 연결된다. 맥락을 많이 주면 계획은 좋아질 수 있지만, Cloud LLM으로 나가는 구조화 정보의 양이 늘어난다.

### 검증 방법

- 같은 intent 집합을 대상으로 minimal/rich 요청을 각각 사용한다.
- 측정 항목:
  - 계획 성공률
  - 재계획 횟수
  - Cloud LLM 호출 횟수와 총 지연
  - 구조화 요청 payload 크기
  - 안전값 조합에 대한 privacy review

## 3. 설계 포인트 C: 합의 후 PII 복원을 누가 수행할 것인가

### 결정 질문

합의 후 실행 단계에서 PII Vault의 원본을 Sub-Agent에게 넘길 것인가, 아니면 Gateway/Execution Proxy가 PII를 보유한 채 외부 API 호출을 대행할 것인가?

### 대안 C-1. Sub-Agent Restore

합의 후 실행 시 Vault에서 PII를 복원해 해당 Sub-Agent에 전달한다. Sub-Agent는 외부 예약, 결제, 연락처 API를 직접 호출한다.

### 대안 C-2. Gateway-mediated Execution

Sub-Agent는 토큰과 실행 의도만 전달하고, Gateway 또는 Execution Proxy가 Vault에서 PII를 복원해 외부 API 호출을 직접 수행한다. Sub-Agent는 원본 PII를 보지 않는다.

### 품질속성 trade-off

| 품질속성 | Sub-Agent Restore | Gateway-mediated Execution |
|---|---|---|
| 기밀성 | 약해짐. PII 평문이 실행 모듈로 확산된다. | 강함. PII 접근 지점이 Gateway/Proxy로 제한된다. |
| Latency | 실행 경로가 짧다. | Proxy 경유와 API별 변환으로 지연이 늘 수 있다. |
| Task 성공률 | tool별 API를 Sub-Agent가 자유롭게 처리할 수 있다. | Proxy가 지원하지 않는 API에서는 실패하거나 기능이 제한될 수 있다. |
| 자원 | 중앙 Proxy 상태가 작다. | API 어댑터와 PII 매핑 상태를 추가로 유지해야 한다. |
| 세션 복구 | Sub-Agent 실행 상태까지 PII 포함으로 복구해야 할 수 있다. | PII 복원 이력과 실행 요청을 중앙에서 복구할 수 있다. |
| 유지보수성 | 신규 tool 추가가 쉽다. | 외부 API마다 Proxy 어댑터를 구현해야 한다. |

### 왜 중요한 결정인가

현재 문서들은 PII를 Vault에 보관하고 합의 후 실행 단계에서 복원한다고 정리한다. 그러나 "복원된 원본이 어디까지 흘러가도 되는가"는 별도 결정으로 남아 있다.

이 결정은 협상 중 유출보다 실행 단계 유출을 다룬다. 사용자의 이름, 전화번호, 주소, 결제 정보는 협상에는 불필요하지만 실행에는 필요하다. 따라서 기밀성과 Task 성공률/유지보수성이 직접 충돌한다.

### 검증 방법

- 예약, 캘린더 등록, 연락처 전달, 결제 등 실행 유형별로 두 방식을 비교한다.
- 측정 항목:
  - PII 평문 접근 모듈 수
  - 신규 외부 API 추가 시 변경 파일/모듈 수
  - 실행 성공률
  - 실행 지연
  - 실패 후 재시도와 복구 가능성

## 4. 설계 포인트 D: 세션 freeze 이후 변경을 어떻게 반영할 것인가

### 결정 질문

협상 중 사용자 개입, 권한 변경, 새 tool 결과가 들어오면 기존 OutcomeSpace를 폐기하고 새 세션으로 갈 것인가, 아니면 버전드 delta로 세션 안에서 반영할 것인가?

### 대안 D-1. Hard Freeze + Re-negotiate

NegMAS 세션 시작 후에는 OutcomeSpace와 ufun을 변경하지 않는다. 새 입력이 들어오면 현재 세션을 종료하거나 무효화하고, 새로운 세션을 구성해 재협상한다.

### 대안 D-2. Versioned Live Update

세션 중에도 안전 사실값, OutcomeSpace, ufun의 변경을 버전으로 관리한다. 상대와 동기화 가능한 경우에는 세션을 유지한 채 delta를 반영한다.

### 품질속성 trade-off

| 품질속성 | Hard Freeze + Re-negotiate | Versioned Live Update |
|---|---|---|
| 기밀성 | 강함. 한 세션에서 나갈 수 있는 값 집합이 고정된다. | 변경 delta가 새 정보 노출 채널이 될 수 있다. |
| Latency | 재협상으로 지연이 커질 수 있다. | 변경을 빠르게 반영할 수 있다. |
| Task 성공률 | 최신 사용자 상태 반영이 늦어 실패할 수 있다. | 동적 상황에서 성공률이 높아질 수 있다. |
| 자원 | 세션 상태 모델이 단순하다. | 버전, delta, 동기화 상태를 저장해야 한다. |
| 세션 복구 | frozen snapshot만 복구하면 된다. | 어느 버전까지 양측이 공유했는지 복구해야 한다. |
| 유지보수성 | 구현과 테스트가 단순하다. | 상태 전이와 버전 호환성 테스트가 복잡하다. |

### 왜 중요한 결정인가

NegMAS는 고정된 OutcomeSpace 위에서 협상한다. 그러나 실제 모바일 환경에서는 협상 중 사용자가 취소·수정하거나, tool 결과가 갱신되거나, 권한 범위가 바뀔 수 있다.

Hard freeze는 세션 일관성과 복구에는 유리하지만, 사용자 개입 반영과 최신성에는 불리하다. Live update는 Task 성공률과 응답성에 유리하지만, 상태 일관성과 복구 복잡도를 크게 높인다.

### 검증 방법

- 협상 중 사용자 수정, 권한 축소, 새 일정 충돌, tool 실패를 주입한다.
- 측정 항목:
  - 사용자 개입 반영 시간
  - 재협상 횟수
  - 세션 복구 성공률
  - 상태 불일치 발생 여부
  - 변경 delta가 외부에 노출하는 정보량

## 5. 우선순위 제안

| 우선순위 | 설계 포인트 | 이유 |
|---:|---|---|
| 1 | A. 선호 힌트 공개 여부 | `ufun` 비공개 원칙과 협상 효율이 직접 충돌한다. 개인정보 처리 구조의 본질에 가장 가깝다. |
| 2 | B. Cloud LLM 계획 요청 맥락 범위 | 상대 PPA보다 덜 닫힌 외부 출구이며, 초기 계획과 재계획에서 반복 호출된다. |
| 3 | C. 합의 후 PII 복원 주체 | 협상 이후 실행 단계의 실제 PII 노출 범위를 결정한다. |
| 4 | D. 세션 freeze 이후 변경 반영 | 모바일 환경과 사용자 개입을 반영하려면 필요하지만, NegMAS 세션 제약과 충돌한다. |

발표용 설계 포인트로는 A를 가장 우선 추천한다. 기존 `Monolithic Gateway` vs `Federated Adapters`는 실행 배치 차이에 머무르지만, A는 "사용자의 선호와 제약을 어디까지 외부 협상 표면으로 올릴 것인가"를 묻는다. 이는 기밀성, Latency, Task 성공률, N-party 확장성이 동시에 충돌하므로 선택이 당연하지 않다.

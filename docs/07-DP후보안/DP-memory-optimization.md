# DP — 온디바이스 메모리 최적화(On-Device Memory Optimization) 후보 비교

> **대상 QAS**: [QAS-004 IDS Memory 200MB](../07-QAS.md), [QAS-011 MAF Memory 300MB](../07-QAS.md), [QAS-014 N-party +20MB/인](../07-QAS.md) (Mapped NFR: [NFR-IDS-04](../05-NFR.md), [NFR-MAF-06](../05-NFR.md), [NFR-MAF-09](../05-NFR.md))
> **연관 DP**: [DP-D1 온디바이스 LLM 자원 중재 및 스케줄링](../08-DP_풀어야할문제.md)
> **다이어그램**: [DP-memory-optimization-candidates.drawio](./DP-memory-optimization-candidates.drawio) (C1·C2·C3 후보 비교)
>
> 본 문서는 IDS·Meta Agent·Sub-Agent가 모두 온디바이스 LLM을 필요로 하는 환경에서, **피크 메모리 사용량을 최소화**하기 위한 LLM 배치·공유 구조의 후보안을 정리·비교한다.
> 본 DP는 현재 [08-DP_풀어야할문제.md](../08-DP_풀어야할문제.md)에 독립 항목으로 등록되지 않은 **신규 결정 후보**이며, DP-D1(자원 중재)의 *메모리 차원*에 해당한다. 합의 후 정식 DP로 승격 또는 DP-D1로 흡수 여부를 결정한다.
>
> **결정 범위 — 본 문서가 다루는 것**: 단말 위에 LLM을 *어떻게 배치·공유하여 피크 메모리를 줄이는가*.
> **결정 범위 — 본 문서가 다루지 않는 것**: (a) 동시 요청 시 누구의 추론을 우선할지(스케줄링·우선순위), (b) 서버 LLM과의 책임 분할(DP-D2), (c) 모델 자체의 학습·튜닝 방식.

## 0. 논의의 전제 (이 문서가 서 있는 4개의 기둥)

후보 비교에 들어가기 전에, 분석을 가능하게 한 전제들을 명시한다. 이것이 바뀌면 결론도 바뀐다.

1. **메모리 피크는 LLM 가중치(weights)가 지배한다.** 협상 상태·이벤트 로그 등은 MB 단위지만 LLM 가중치는 GB 단위다. 따라서 본 문서는 *가중치 메모리*를 1차 최적화 대상으로 본다. KV 캐시·에이전트 상태는 보조 레버로 다룬다.

2. **단말 LLM은 Compact 등급(2B 파라미터)을 기준으로 한다.** Galaxy S26 모델 변종(Compact/Balanced/Supreme) 중 Compact를 단말 표준으로 가정한다. INT4 양자화 시 2B × 0.5 byte ≈ **약 1.0 GB**의 가중치 메모리를 차지한다.

3. **본 결정에서 서버 LLM은 고려하지 않는다.** Meta Agent의 협상 추론 품질을 보강하기 위해 서버 LLM에 위임하는 경로는 본 문서의 결정 공간에서 제외한다. (DP-D2의 별도 의사결정으로 분리.)

4. **현 NFR의 200/300MB 임계값은 재검토 대상이다.** Compact 2B(INT4 ≈ 1GB)는 본 임계값을 초과한다. 본 문서는 임계값을 *수정될 전제로 두되*, 후보 비교의 평가축은 "임계값 충족 여부"가 아니라 **"동일 조건에서 누가 피크를 더 줄이는가"** 로 둔다. 정확한 임계값은 NFR 재정의 후 다시 매핑한다.

5. **LoRA 부착은 가능하다(확인 완료).** 본 단말 LLM에 대해 커스텀 LoRA 어댑터 부착·hot-swap이 **벤더 정책상 허용된다는 점이 사전 확인되었다.** 이에 따라 본 문서는 LoRA 부착 가능성을 *결정된 전제* 로 두며, 이전에 위험으로 분류했던 "런타임 의존성"은 평가축에서 사실상 해소된 것으로 본다.

## 1. 평가 기준 (Evaluation Criteria)

프로젝트 QAS·NFR에 매핑한 평가축이다. 후보들이 **상충(trade-off)** 시키는 품질속성을 중심으로 선정했다.

| # | 평가축 | 근거 QAS/NFR | 본 결정이 미치는 영향 |
|---|---|---|---|
| E1 | **피크 메모리 (가중치)** | [QAS-004](../07-QAS.md), [QAS-011](../07-QAS.md) | 모델 인스턴스 개수와 직결 — 본 결정의 1차 목표 |
| E2 | **N-party 확장 비용** | [QAS-014](../07-QAS.md) | 참여자 증가 시 KV 캐시·상태가 어떻게 누적되는가 |
| E3 | **역할별 추론 품질** | NFR-IDS-01(2초), NFR-MAF-04(라운드당 10~15초) | 한 모델을 여러 역할에 강제 사용할 때의 품질 저하 |
| E4 | **응답 지연 (cold start 포함)** | NFR-IDS-01, NFR-MAF-02 | 모델 로딩·언로딩이 응답 시간에 들어가는가 |
| E5 | **런타임 의존성** | (구현 가능성) | 가정하는 추론 엔진 기능이 안드로이드 환경에서 실제로 제공되는가 |
| E6 | **복잡도 / 유지보수** | (Maintainability) | 구현·디버깅·운영 난이도, 어댑터·모델 자산 관리 부담 |

> 표기: **★★★** 강함 · **★★** 보통 · **★** 약함/위배 위험

## 2. 후보안

세 후보는 "**같은 Compact 모델을 어느 수준까지 공유할 것인가**" 축 위에 놓인다.

| 후보 | 공유 수준 | 역할 특화 수단 |
|---|---|---|
| **C1** | 공유 없음 | 각자 별도 모델 |
| **C2** | 본체 공유 | 없음 (모두 동일 모델) |
| **C3** | 본체 공유 + 어댑터 분리 | LoRA |

### 후보 C1. 역할별 독립 모델 (No-Sharing)

- **구조**: IDS·Meta Agent·Sub-Agent가 각자 자신의 Compact 2B INT4 인스턴스를 별도 메모리 영역에 로드한다.
- **피크 메모리 추정**: 3 × ~1 GB ≈ **~3 GB** (가중치만, KV·상태 제외).
- **장점**
  - 역할 간 자원 격리가 단순함 → **E6 ★★★**.
  - 한 인스턴스가 중단되어도 다른 역할에 영향 없음 (장애 격리 측면 유리).
- **단점**
  - 가중치가 N배로 복제되어 **피크 메모리가 가장 큼** → **E1 ★** (사실상 NFR 미달).
  - N-party 협상 시 가중치 이외의 비용(KV·상태)은 [QAS-014](../07-QAS.md) 그대로 누적.
- **위치**: 본 안은 **기준선(baseline)** 으로만 의미가 있으며, *피크 최소화* 목표에서는 **권장되지 않음**.

### 후보 C2. 단일 공유 Compact (Single Shared Backbone, no specialization)

- **구조**: 단말에 **Compact 2B INT4 인스턴스 1개만** 상주. IDS·Meta Agent·Sub-Agent는 *모두 동일한 모델*을 시간 분할로 공유한다. 역할별 차별화 수단은 없음 (프롬프트 차이만 존재).
- **피크 메모리 추정**: 본체 ~1 GB + KV 캐시(가변) + 상태(가변).
- **장점**
  - 가중치 메모리가 N배 → 1배로 축소 → **E1 ★★★**.
  - 구조가 단순함, 별도 어댑터 자산이 없음 → **E6 ★★★**.
  - 런타임 의존성이 낮음 — 표준 단일 모델 추론기만 있으면 됨 → **E5 ★★★**.
- **단점**
  - **역할별 품질 보장 수단이 없음**. IDS의 분류, Meta Agent의 협상, Sub-Agent의 툴 호출은 요구 능력이 다른데 동일 모델·동일 가중치로 처리. 협상 품질(NFR-MAF-04)이 가장 큰 위험 → **E3 ★**.
  - 동일 모델을 여러 역할이 점유하므로 동시성 경쟁이 더 빈번 → 스케줄링 부담은 DP-D1로 이관되나 부담의 *총량*이 큼.
- **위치**: 피크는 가장 잘 줄이지만 *품질 저하를 감수* 하는 안. 단순성을 절대 우선할 때만 채택.

### 후보 C3. 공유 백본 + 역할별 LoRA 어댑터 ★ 권장

> **요지**: C2와 동일한 본체 1개 상주에 더해, **역할별 LoRA 어댑터(통상 본체의 1~3% 크기)** 를 hot-swap한다. 가중치 메모리는 C2와 동등 수준을 유지하면서, *역할별 특화는 어댑터로 흡수* 한다.

- **구조**:
  - **백본**: Compact 2B INT4 인스턴스 1개 상주 (~1 GB).
  - **어댑터**: IDS용 / Meta Agent용 / Sub-Agent용 LoRA 가중치 (개당 수~수십 MB 추정). 한 시점에 *활성 어댑터 1개*만 본체에 결합.
  - **전환**: 어댑터 swap은 본체 재로딩 없이 수행 (수 ms 단위 — 단, 런타임 가정 — E5 항목 참조).
- **피크 메모리 추정**: 본체 ~1 GB + 활성 어댑터 ~수~수십 MB + KV 캐시 + 상태 ≈ **C2 + 어댑터 1개 분량**. 가중치 측면에서 C2와 사실상 동등.
- **장점**
  - 가중치 메모리는 C2 수준(★★★) → **E1 ★★★**.
  - 역할별 특화를 어댑터로 흡수 → 협상·분류·툴 호출의 품질 분화 가능 → **E3 ★★**.
  - 어댑터 swap이 본체 재로딩보다 압도적으로 가벼움 → **E4 ★★★**.
- **단점**
  - ~~**런타임 의존성**~~ → **사전 확인으로 해소** (0.전제 5항). LoRA 부착·hot-swap이 벤더 정책상 허용됨이 확인되었으므로 본 항목은 평가에서 제외한다. **E5 ★★★**.
  - **어댑터 소싱은 별도 운영 결정**: 역할별 LoRA를 (a) 직접 학습 / (b) 공개 어댑터 활용 / (c) 빈 어댑터(no-op)로 시작 중 어느 경로로 채울지의 결정이 추가됨. *단, 이 결정은 C3 채택과 독립이며 후속 결정으로 분리됨* (5절 참조). 본 단점은 *아키텍처가 아니라 운영의 부담* 이다 → **E6 ★★**.
  - 어댑터 품질이 부족할 경우 결과적으로 C2와 동일한 품질 한계에 수렴할 가능성. 단, (c) 경로로 시작하면 *피해 없이* C2와 동등 동작이 되고, 어댑터가 준비되면 추가 비용 없이 품질 분화로 이행 가능.

## 3. 후보 비교표

| 평가축 | C1. 역할별 독립 | C2. 단일 공유 | **C3. 공유 백본 + LoRA** |
|---|---|---|---|
| E1. 피크 메모리 | ★ (~3 GB) | ★★★ (~1 GB) | **★★★ (~1 GB + 어댑터)** |
| E2. N-party 확장 비용 | ★★ | ★★ | **★★** |
| E3. 역할별 품질 | ★★ (격리되어 있음) | ★ (특화 수단 없음) | **★★** (어댑터 품질에 의존) |
| E4. 응답 지연 | ★★★ (각자 상주) | ★★★ (1개 상주) | **★★★** (어댑터 swap 경량) |
| E5. 런타임 의존성 | ★★★ (표준) | ★★★ (표준) | **★★★** (LoRA 부착 허용 확인 완료) |
| E6. 복잡도·유지보수 | ★★★ | ★★★ | **★★** (어댑터 운영 부담) |

> E2(N-party 확장)는 세 후보가 모두 가중치는 1회만 카운트되므로 동일 등급으로 평가. 차이는 KV·상태에서 발생하며, 이는 본 결정과 직교(orthogonal)한 KV/상태 최적화로 별도 처리.

## 4. 권장안

**C3 (공유 백본 + LoRA) 단일 채택.** 어댑터 소싱은 본 결정과 분리된 후속 운영 결정으로 둔다.

### 4.1 권장 근거

1. **메모리 측면에서 C2와 C3은 사실상 동등.** 가중치 메모리가 1배로 압축되는 효과가 본 결정의 핵심이며, 이 효과는 두 안 모두에서 달성된다.
2. **그러나 본 시스템은 역할 비대칭이 크다.** IDS(분류, 짧은 컨텍스트), Meta Agent(다라운드 협상 추론), Sub-Agent(툴 호출)의 작업 성격이 명확히 다른데, C2는 이 비대칭을 무시한다. 협상 품질이 가장 큰 리스크다.
3. **C3은 메모리 손실 없이 역할 특화를 흡수한다.** 어댑터 1개의 메모리 추가는 본체의 1~3% 수준이므로 *피크 최소화 목표를 훼손하지 않으면서* 품질 분화를 회복한다.
4. **이전에 결정적 리스크로 본 런타임 의존성(E5)은 해소되었다.** LoRA 부착이 벤더 정책상 허용됨이 확인되어 폴백 시나리오를 별도 유지할 필요가 없다.
5. **어댑터 부재 시에도 C3는 손해가 없다.** (c) "빈 어댑터로 시작" 경로를 통해 *런타임 동작은 C2와 동등하지만 구조는 C3*인 상태로 출발할 수 있다. 어댑터가 준비되면 *아키텍처 변경 없이* 끼우기만 하면 된다. 즉 C3는 C2의 상위 집합(superset)이다.

### 4.2 권장안 구조 요약

```
RAM 피크 (정성)
  ┌────────────────────────────────────────────┐
  │ Compact 2B INT4 백본 (~1 GB) — 상주          │
  ├────────────────────────────────────────────┤
  │ 활성 LoRA 1개 (~수~수십 MB) — swap            │
  │   ├ IDS LoRA (분류 특화)                    │
  │   ├ MA  LoRA (협상 특화)                    │
  │   └ SA  LoRA (툴 호출 특화)                  │
  ├────────────────────────────────────────────┤
  │ KV 캐시 (역할별 컨텍스트 캡 + 양자화 적용)     │
  ├────────────────────────────────────────────┤
  │ 에이전트 상태 (이벤트 로그는 디스크 오프로드)   │
  └────────────────────────────────────────────┘
```

### 4.3 직교(orthogonal) 보조 기법 — 어느 안에도 결합 가능

본 결정과 별개로, 다음 기법들은 후보와 무관하게 추가 적용 가능하며 권장안에 함께 채택한다.

- **가중치 양자화(INT4)**: 본 문서의 모든 추정의 전제. C2/C3 권장의 기본 가정.
- **KV 캐시 양자화 + 컨텍스트 윈도 캡**: KV 캐시 성장이 N-party·다라운드에서 가장 큰 보조 메모리 비용. 역할별 토큰 상한을 둔다.
- **에이전트 상태 디스크 오프로드**: 이벤트 로그는 DP-A2(영속화)와 자연 통합되어 디스크 보관. 메모리에는 최근 상태만 유지.
- **mmap 기반 가중치 로딩** *(선택)*: 평상시 RSS 축소에 유효하나, 워킹셋이 비결정적이므로 *피크 보장(SLA)* 용도로는 부적합. 보조적 사용에 한정.

## 5. 후속 결정 (이 문서 다음에 풀어야 할 질문)

1. **어댑터 소싱 경로**: 역할별 LoRA를 다음 중 어느 경로로 확보할지 결정해야 한다. *본 결정은 C3 채택과 독립이며, C3는 어느 경로에서도 성립한다.*
   - (a) **직접 학습** — 역할별 데이터셋·학습 파이프라인·평가 기준을 자체 구축. 품질 통제력은 최대.
   - (b) **공개 어댑터 활용** — 우리 백본·태스크에 맞는 공개 LoRA가 있는 경우 차용.
   - (c) **빈 어댑터(no-op)로 시작** — 어댑터 swap *인프라*만 깔고, 어댑터는 비워둔 채 출발. 런타임 동작은 C2와 동등. PoC 단계에 적합한 기본값.
2. **NFR 임계값 재정의**: 200/300MB는 Compact 2B INT4 단독 사용도 초과하므로, NFR-IDS-04·NFR-MAF-06의 임계값과 측정 범위(가중치 포함/제외 여부)를 재정의해야 한다. → [05-NFR.md](../05-NFR.md), [07-QAS.md](../07-QAS.md) 동기화 (CLAUDE.md 7항).
3. **Meta Agent 협상 품질이 Compact + LoRA로 부족할 경우의 대응**: 본 문서는 서버 LLM 위임을 결정 공간에서 제외했으므로, 단말 내에서 해결해야 함. 후보: (a) 협상 라운드 수 축소, (b) 협상 메시지 구조화 강화로 LLM 부담 경감(DP-A3 연계), (c) Compact보다 큰 모델의 단말 도입 재검토.
4. **DP-D1과의 정렬**: 본 결정이 확정되면 DP-D1(자원 중재)의 스케줄링 정책은 *"공유 단일 백본 + 어댑터 swap"* 을 전제로 다시 작성되어야 함. 현재 DP-D1의 "단일 공유 모델 + 스케줄러" 선택지가 본 결정과 정합.

## 6. 참고 — 본 문서의 가정과 한계

- 모든 메모리 수치는 **정성적 추정**이다. 실측 없이 단언하지 않는다. 2B × 0.5 byte ≈ 1 GB는 표준 산식이며, 추론 엔진별 오버헤드(런타임·KV·활성값)는 별도이다.
- 후보 C1의 ~3 GB 추정은 "3개 인스턴스 동시 상주 시 가중치만"의 값이며, 실측에서는 KV·상태가 추가된다.
- LoRA 어댑터의 크기·hot-swap 지연·품질 효과는 모델 아키텍처와 학습 설정에 따라 변한다. 본 문서의 "수~수십 MB", "수 ms swap"은 일반적인 LoRA 사례 기준이며 *우리 환경에서의 검증이 필요*하다.

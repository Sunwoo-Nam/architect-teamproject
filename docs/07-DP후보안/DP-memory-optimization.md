# DP — 온디바이스 메모리 최적화(On-Device Memory Optimization) 후보 비교

> **대상 QAS**: [QAS-004 IDS Memory 200MB](../07-QAS.md), [QAS-011 MAF Memory 300MB](../07-QAS.md), [QAS-014 N-party +20MB/인](../07-QAS.md) (Mapped NFR: [NFR-IDS-04](../05-NFR.md), [NFR-MAF-06](../05-NFR.md), [NFR-MAF-09](../05-NFR.md))
> **IDS 역할 근거**: [02-과제-개요.md](../02-과제-개요.md) (IDS = Intent Driven Secretary, Zero-claw 기반), [04-FR.md FR-IDS-01~05](../04-FR.md)
> **연관 DP**: [DP-D1 온디바이스 LLM 자원 중재 및 스케줄링](../08-DP_풀어야할문제.md)
> **다이어그램**: [DP-memory-optimization-candidates.drawio](./DP-memory-optimization-candidates.drawio) (후보 비교)
>
> 본 문서는 IDS·Meta Agent·Sub-Agent가 모두 온디바이스 LLM을 필요로 하는 환경에서, **피크 메모리 사용량을 최소화**하기 위한 LLM 배치·공유 구조의 후보안을 정리·비교한다.
> 본 DP는 현재 [08-DP_풀어야할문제.md](../08-DP_풀어야할문제.md)에 독립 항목으로 등록되지 않은 **신규 결정 후보**이며, DP-D1(자원 중재)의 *메모리 차원*에 해당한다. 합의 후 정식 DP로 승격 또는 DP-D1로 흡수 여부를 결정한다.
>
> **결정 범위 — 본 문서가 다루는 것**: 단말 위에 LLM을 *어떻게 배치·공유하여 피크 메모리를 줄이는가*. (역할별 동시 추론을 어떤 구조로 수용하는가를 포함한다.)
> **결정 범위 — 본 문서가 다루지 않는 것**: (a) 동시 요청 시 누구의 추론을 우선할지(스케줄링·우선순위 — DP-D1), (b) 서버 LLM과의 책임 분할(DP-D2), (c) 모델 자체의 학습·튜닝 방식.

## 0. 논의의 전제 (이 문서가 서 있는 기둥)

후보 비교에 들어가기 전에, 분석을 가능하게 한 전제들을 명시한다. 이것이 바뀌면 결론도 바뀐다.

1. **메모리 피크는 LLM 가중치(weights)가 지배한다.** 협상 상태·이벤트 로그 등은 MB 단위지만 LLM 가중치는 GB 단위다. 따라서 본 문서는 *가중치 메모리*를 1차 최적화 대상으로 본다. KV 캐시·에이전트 상태는 보조 레버로 다룬다.

2. **세 역할은 모두 "추론급" 모델을 요구한다 — IDS도 예외가 아니다.** 단말에 임의의 모델을 적재하는 것은 가능하므로(아래 5항) *역할별 크기 차등(right-sizing)* 이 원리상 열려 있다. 그러나 본 시스템의 세 역할은 모두 경량 분류기로 대체할 수 없는 추론을 수행한다:
   - **IDS**: 단순 분류기가 아니다. [04-FR.md](../04-FR.md) 기준 — **비명시적 의도 도출**(FR-IDS-02, 간접 대화에서 의도 추론), **맥락 연속성 유지**(FR-IDS-04, *긴 컨텍스트* 대조), **처리 주체 식별·태스크 매핑**(FR-IDS-03), **자체 태스크 실행**(FR-IDS-05, 디바이스 제어·정보 검색·프라이버시 민감 작업의 온디바이스 처리)을 모두 담당한다.
   - **Meta Agent**: 다라운드 협상 추론(가장 무거움).
   - **Sub-Agent**: 툴 호출·실행.

   따라서 right-sizing으로 *내려서* 얻는 절감 효과는 작고, 실용적 하한은 세 역할 모두 **Compact 2B급**이다. 그러므로 본 결정의 레버는 "모델을 줄이기"가 아니라 **"공유 + 동시성 수용"** 이다. Compact 2B는 INT4 양자화 시 2B × 0.5 byte ≈ **약 1.0 GB**의 가중치 메모리를 차지한다.

3. **IDS는 상시 동작(always-on)하며, MA/SA와 동시에 추론하는 구간이 존재한다.** IDS는 "[기기 활동 전체 모니터링](../02-과제-개요.md)"을 수행하므로 상시 상주한다. IDS에는 두 동작 모드가 있다:
   - **독립 모드**: 타 에이전트 협업·멀티에이전트 시나리오가 아닐 때, IDS가 사용자 query를 *스스로 추론·실행*(FR-IDS-05). 이때 MA/SA는 유휴.
   - **협상 동반 모드**: MA/SA가 A2A 협상을 진행하는 *도중*, 사용자가 새 발화를 던지면 IDS가 FR-IDS-04(맥락 연속성)로 추론해야 한다. → **IDS 추론과 MA/SA 추론이 동시에 필요한 구간**이 생긴다.

   이 동시성은 "한 시점에 활성 모델/어댑터 1개"라는 가정을 깨므로, 본 문서의 **핵심 평가축(E7)** 으로 명시한다.

4. **본 결정에서 서버 LLM은 고려하지 않는다.** Meta Agent의 협상 추론 품질을 보강하기 위해 서버 LLM에 위임하는 경로는 본 문서의 결정 공간에서 제외한다. (DP-D2의 별도 의사결정으로 분리.)

5. **단말에 임의의 모델·어댑터 적재가 가능하다(확인 완료).** 벤더가 제공하는 Compact 2B(및 Balanced/Supreme 변종) 외에 **우리가 고른 모델을 단말에 적재할 수 있으며**, 커스텀 LoRA 어댑터 부착·hot-swap도 **벤더 정책상 허용된다는 점이 사전 확인되었다.** 단, *동시 다중 어댑터 추론(concurrent multi-LoRA / S-LoRA 계열)* 까지 런타임이 지원하는지는 **아직 미검증**이며, 이는 hot-swap보다 강한 요구다(E5·후보 C4 참조).

6. **현 NFR의 200/300MB 임계값은 재검토 대상이다.** Compact 2B(INT4 ≈ 1GB)는 본 임계값을 초과한다. 본 문서는 임계값을 *수정될 전제로 두되*, 후보 비교의 평가축은 "임계값 충족 여부"가 아니라 **"동일 조건에서 누가 피크를 더 줄이면서 동시성을 수용하는가"** 로 둔다. 정확한 임계값은 NFR 재정의 후 다시 매핑한다.

## 1. LLM 사용 지점 인벤토리 (이 시스템은 어디서 LLM을 쓰는가)

> 본 절은 메모리 최적화의 *대상 범위*를 확정하기 위한 사실 정리다. 어떤 컴포넌트가 LLM 추론을 수행하며 그 LLM이 **온디바이스인가 서버인가**를 [02-과제-개요.md](../02-과제-개요.md)·[04-FR.md](../04-FR.md)에 근거해 정리한다. 본 문서의 피크 메모리 대상은 **온디바이스 LLM을 사용하는 컴포넌트로 한정**된다.

### 1.1 컴포넌트별 LLM 배치

[02-과제-개요.md §2.3](../02-과제-개요.md)(계층 뷰) 및 제약 표(LLM 배치) 기준:

| 계층 | 컴포넌트 | LLM 위치 | 본 결정 대상 |
|---|---|---|---|
| 1 | **IDS** (Intent Driven Secretary) | **온디바이스** | ✅ 대상 |
| 2 | **Orchestrator** | **서버** | ❌ 제외 (서버 LLM, 단말 피크와 무관) |
| 3 | **Meta Agent** (= PPA) | **온디바이스** | ✅ 대상 |
| 4 | **Sub-Agent** | **온디바이스** | ✅ 대상 |

→ **온디바이스 LLM 사용 주체 = IDS · Meta Agent · Sub-Agent** (02-과제-개요.md 라인 62). Orchestrator의 Task Planning·동적 Agent/Tool 생성은 **서버 LLM**(라인 63)이 담당하므로 단말 피크 메모리 결정에서 제외한다 — 0.전제 4항(서버 LLM 비고려)과 정합.

### 1.2 온디바이스 LLM이 실제로 호출되는 기능 (FR 매핑)

**① IDS — 상시 동작. 의도 추론·라우팅·자체 실행** ([04-FR.md FR-IDS-01~05](../04-FR.md))

| FR | 기능 | LLM 사용 |
|---|---|---|
| FR-IDS-01 | 명시적 의도 추출 | 의도 도출 추론 |
| FR-IDS-02 | 비명시적 의도 도출 | 컨텍스트 데이터 분석·잠재 의도 추론 (주기적) |
| FR-IDS-03 | 처리 주체 식별·태스크 매핑 | 의도 성격 분석 → IDS자체/MAF 라우팅 판단 |
| FR-IDS-04 | 컨텍스트 연속성 유지 | 기존 맥락과 신규 발화 대조 분석 (**긴 컨텍스트**) |
| FR-IDS-05 | IDS 자체 처리 실행 | **조건부** — 정보 검색·요약 등은 LLM, 단순 디바이스 제어(설정 변경 등)는 API 호출 중심으로 LLM 비중이 낮을 수 있음 *(원문 미명시 — 추정, 설계 시 확인 필요)* |

**② Meta Agent (PPA) — A2A 협상 추론** ([04-FR.md §2.2 Negotiation](../04-FR.md))

| FR | 기능 | LLM 사용 |
|---|---|---|
| FR-MAF-02 | 1:1 협상 | 제안/역제안/수락·거절 판단 추론 |
| FR-MAF-03 | N-party 협상 | 다자 합의 추론 (참여자 수만큼 컨텍스트 증가) |
| FR-MAF-08 | 런타임 개입 반영 | 변경 지시 재반영 *(추론 관여 가능 — 원문 미명시)* |
| FR-MAF-11 | 이상 상황 처리 | 상태 판정 *(추론 관여 가능 — 원문 미명시)* |

> Orchestrator(서버)가 MA를 지시하지만(02-과제-개요.md 라인 41), **협상 메시지 단위의 추론은 온디바이스 MA가 수행**한다.

**③ Sub-Agent — 툴 호출** ([04-FR.md FR-MAF-09 등](../04-FR.md))

| FR | 기능 | LLM 사용 |
|---|---|---|
| FR-MAF-09 | 합의안 후속 실행 (Tool Calling) | 툴 선택·인자 구성 추론 |
| FR-MAF-04 | 협업 워크플로우 연동 | 외부 API 호출 조율 추론 |

> Sub-Agent는 도메인별로 **동적 생성**(02-과제-개요.md 라인 42)되며 상세 기능은 현재 스펙 아웃(04-FR §3)이나, 온디바이스 LLM 사용은 확정(라인 62).

### 1.3 본 결정에 미치는 함의

1. **단말 피크 메모리는 IDS·MA·SA 세 온디바이스 사용처로만 결정된다.** Orchestrator(서버 LLM)는 단말 메모리 예산과 무관하다.
2. **세 사용처는 활성 시점이 비대칭이다** — **IDS는 상시(always-on)**, MA·SA는 협상 에피소드 동안만 활성. 이 비대칭이 동시 추론(E7)의 근원이다(0.전제 3항).
3. **항상 풀(full) 추론이 도는 것은 아니다.** FR-IDS-05의 단순 디바이스 제어처럼 LLM이 불필요하거나 가벼운 경로가 존재하며, 이는 동시 추론의 *강도* 판단과 연결된다(→ 7절 가정·한계의 마지막 항).

## 3. 후보안

후보들은 두 개의 독립된 축 위에 놓인다 — **(A) 백본을 공유하는가**, **(B) 동시 추론을 어떻게 수용하는가**.

| 후보 | 백본 | 역할 특화 | 동시 추론(E7) 수용 방식 |
|---|---|---|---|
| **C1** | 역할별 독립 (3개) | 각자 별도 모델 | 물리적 분리 (각자 상주) |
| **C2** | 단일 공유 (1개) | 없음 | 직렬화 (불가에 가까움) |
| **C3** | 단일 공유 (1개) | LoRA, 활성 1개 | 직렬화/어댑터 thrash |
| **C4** | 단일 공유 (1개) | LoRA, **동시 다중 활성** | 멀티-LoRA 동시 추론 |
| **C5** | IDS 전용 + MA/SA 공유 (2개) | MA/SA만 LoRA | 물리적 분리(IDS) |

### 후보 C1. 역할별 독립 모델 (No-Sharing) — 기준선

- **구조**: IDS·Meta Agent·Sub-Agent가 각자 자신의 Compact 2B INT4 인스턴스를 별도 메모리 영역에 로드한다.
- **피크 메모리 추정**: 3 × ~1 GB ≈ **~3 GB** (가중치만, KV·상태 제외).
- **장점**
  - 역할 간 자원 격리가 단순함 → **E6 ★★★**.
  - 각자 상주하므로 IDS·MA·SA 동시 추론을 **그대로 수용** → **E7 ★★★**.
  - 역할별로 독립 모델이므로 품질 분화 자체는 자유 → **E3 ★★★**.
  - 한 인스턴스가 중단되어도 다른 역할에 영향 없음(장애 격리 유리).
- **단점**
  - 가중치가 N배로 복제되어 **피크 메모리가 가장 큼** → **E1 ★**.
- **위치**: 모든 역할이 추론급 모델을 요구한다는 전제(0.2)에서 "3×2B≈3GB"는 *부풀린 strawman이 아니라 정직한 기준선*이다. E1을 제외한 모든 축에서 강하나, 메모리 1차 목표에서 탈락. **다른 후보들이 얼마나 절감하는지 재는 baseline**으로 사용한다.

### 후보 C2. 단일 공유 Compact (특화·동시성 모두 없음)

- **구조**: 단말에 **Compact 2B INT4 인스턴스 1개만** 상주. 모든 역할이 동일 모델을 시간 분할로 공유한다. 역할별 차별화 수단 없음(프롬프트 차이만).
- **피크 메모리 추정**: 본체 ~1 GB + KV 캐시(가변) + 상태(가변).
- **장점**
  - 가중치 메모리 N배 → 1배 → **E1 ★★★**.
  - 구조 단순, 어댑터 자산 없음 → **E6 ★★★**.
  - 런타임 의존성 낮음(표준 단일 모델 추론기) → **E5 ★★★**.
- **단점**
  - **역할별 품질 보장 수단 없음**. 협상 품질(NFR-MAF-04)이 가장 큰 위험 → **E3 ★**.
  - **동시 추론 불가** — 단일 인스턴스를 시간 분할하므로 IDS 추론과 MA 협상이 겹치면 직렬화된다(0.전제 3항). IDS는 상시인데 협상 중 IDS 추론이 지연되면 FR-IDS-04(맥락 연속성)·NFR-IDS-01(2초)을 위배할 수 있음 → **E7 ★**.
- **위치**: 피크는 잘 줄이지만 *품질·동시성을 모두 포기* 하는 안. 사실상 기각 후보지만, C3/C4가 무엇을 추가로 해결하는지 보이기 위한 비교점으로 둔다.

### 후보 C3. 공유 백본 + 역할별 LoRA (활성 어댑터 1개)

> **요지**: C2와 동일한 본체 1개에 더해 역할별 LoRA 어댑터(본체의 1~3%)를 hot-swap한다. *역할 특화는 어댑터로 흡수* 하지만, **한 시점에 활성 어댑터는 1개**라는 가정을 둔다.

- **구조**: Compact 2B INT4 백본 1개 상주(~1 GB) + IDS/MA/SA용 LoRA 중 *활성 1개*만 본체에 결합. 전환은 hot-swap(수 ms).
- **피크 메모리 추정**: 본체 ~1 GB + 활성 어댑터 ~수~수십 MB + KV + 상태 ≈ **C2 + 어댑터 1개 분량**.
- **장점**
  - 가중치 메모리는 C2 수준 → **E1 ★★★**.
  - 역할 특화를 어댑터로 흡수 → 협상·의도추론·툴호출의 품질 분화 가능 → **E3 ★★**(어댑터 품질 의존).
  - 어댑터 swap이 본체 재로딩보다 압도적으로 가벼움.
- **단점**
  - **동시 추론을 수용하지 못한다** — "활성 어댑터 1개" 가정 때문에, IDS가 협상 도중 추론해야 하면(0.전제 3항) IDS↔MA 어댑터를 번갈아 swap해야 한다. 이는 (a) 추론을 직렬화하거나 (b) swap thrashing으로 지연을 키운다 → **E7 ★**, 그로 인해 **E4 ★★**.
  - 어댑터 소싱은 별도 운영 결정(6절).
- **위치**: 품질 분화는 얻지만 *동시성 문제를 풀지 못함*. **C4(멀티-LoRA)와 C5(IDS 분리)가 바로 이 약점을 푸는 두 갈래다.** C3는 그 출발점으로서 의미가 있다(동시 추론이 없는 워크로드라면 C3로 충분).

### 후보 C4. 공유 백본 + 동시 멀티-LoRA (S-LoRA 계열) ★ 권장 후보

> **요지**: C3와 동일한 단일 백본 위에, **여러 역할 어댑터를 동시에 활성화**하여 한 백본으로 IDS·MA·SA 추론을 병렬·배치 처리한다. C3의 "활성 1개" 제약을 풀어 동시성(E7)을 메모리 최소 비용으로 수용한다.

- **구조**: Compact 2B INT4 백본 1개 상주(~1 GB) + IDS/MA/SA LoRA를 **동시 상주·동시 활성**. 입력별로 적용 어댑터를 라우팅하여 배치 추론.
- **피크 메모리 추정**: 본체 ~1 GB + 동시 어댑터들(개당 수~수십 MB, 합쳐도 작음) + KV + 상태 ≈ **~1 GB + 어댑터 수개**. 가중치 측면 C2/C3와 사실상 동등.
- **장점**
  - 가중치 메모리 C2/C3 수준 → **E1 ★★★**.
  - **동시 추론을 단일 백본으로 수용** → **E7 ★★★**. IDS(상시) + MA/SA(협상)가 같은 백본을 동시에 사용.
  - 역할 특화 유지 → **E3 ★★**(어댑터 품질 의존).
  - 어댑터 thrashing 없음 → **E4 ★★★**.
- **단점**
  - **런타임 의존성이 가장 강함** — 단순 hot-swap(0.전제 5항에서 확인됨)을 넘어, *동시 다중 어댑터 추론*을 추론 엔진이 지원해야 한다. **이 기능은 아직 미검증** → **E5 ★**. (검증 실패 시 C5로 폴백.)
  - 동시 어댑터 라우팅·배치 스케줄링으로 구현 복잡도 상승 → **E6 ★**.
  - 동시 추론 시 단일 백본의 *연산* 경합은 여전 존재(메모리 문제와 별개, DP-D1 스케줄링 영역).

### 후보 C5. IDS 전용 백본 + MA/SA 공유 백본 + LoRA ★ 권장 후보(폴백)

> **요지**: IDS는 상시 동작이므로 *자기 백본을 따로 상주*시키고, 협상 시에만 활성화되는 MA/SA는 **공유 백본 1개 + LoRA**로 묶는다. 동시성을 *물리적 분리*로 해결하므로 특수 런타임 기능이 필요 없다.

- **구조**: IDS 전용 백본 1개(~1 GB, 상주) + MA/SA 공유 백본 1개(~1 GB) + MA/SA LoRA hot-swap. (MA/SA 사이의 동시성은 C3식 어댑터로, IDS와의 동시성은 백본 분리로 수용.)
- **피크 메모리 추정**: 협상 동반 모드에서 ~1 GB(IDS) + ~1 GB(MA/SA) ≈ **~2 GB + MA/SA 어댑터**. 독립 모드(MA/SA 유휴)에서는 ~1 GB로 내려갈 수 있음.
- **장점**
  - **동시 추론을 물리적으로 수용** → **E7 ★★★**. IDS·MA가 각자 백본을 가지므로 경합 없음.
  - 특수 런타임 기능 불필요(표준 추론기 2개 + 표준 LoRA hot-swap) → **E5 ★★★**.
  - IDS 장애 격리·전용 최적화 가능, IDS는 전용 백본이므로 품질 통제 용이 → **E3 ★★★**(IDS 전용 + MA/SA 어댑터).
- **단점**
  - 백본 2개 상주로 **피크 메모리가 C2/C3/C4보다 큼**(~2 GB) → **E1 ★★**. 단 C1(~3 GB)보다는 낮음.
  - 백본 2종 + MA/SA 어댑터 운영 → **E6 ★★**.

## 5. 권장안 — 조건부 (C4 우선, C5 폴백)

순수 C3는 **E7(동시 추론)에서 탈락**한다. IDS가 상시 동작하며 협상 중 추론이 필요하다는 사실(0.전제 3항)이 확인된 이상, "활성 어댑터 1개"는 성립하지 않는다. 따라서 권장은 동시성을 수용하는 두 안 사이의 **조건부 선택**이다:

> **런타임이 *동시 다중 LoRA 추론*을 지원하면 → C4 (멀티-LoRA).**
> **지원하지 않으면 → C5 (IDS 전용 백본 + MA/SA 공유).**

### 5.1 권장 근거

1. **메모리 최적 + 동시성 수용을 동시에 달성하는 유일한 안은 C4다.** 가중치 1배(~1 GB)를 유지하면서 IDS·MA·SA 동시 추론을 단일 백본으로 흡수한다. 다만 이는 *동시 다중 어댑터 추론*이라는 미검증 런타임 기능에 의존한다(E5 ★).
2. **C5는 그 런타임 위험을 제거한 안전한 폴백이다.** 표준 추론기 2개 + 표준 LoRA hot-swap만으로 동시성을 *물리적 분리*로 해결한다. 대가는 백본 2개로 인한 피크 ~2 GB(E1 ★★)지만, C1(~3 GB)보다 낮고 동시성·품질·런타임 안전성을 모두 확보한다.
3. **C1은 동시성·품질에서는 최상이나 메모리에서 탈락**, **C2는 동시성·품질에서 탈락**, **C3는 동시성에서 탈락**한다. 즉 세 기준(메모리·품질·동시성)을 동시에 충족하는 후보는 C4·C5뿐이다.
4. **결정의 분기점은 단 하나 — 동시 다중 LoRA 추론의 런타임 지원 여부다.** 이를 PoC에서 우선 검증해야 한다(6절). 검증 전까지는 **C5를 기본 가정**으로 설계하되, C4로 이행 가능하도록 백본·어댑터 자산을 공통화해 둔다.

### 5.2 권장안 구조 요약 (정성)

```
C4 (런타임 동시 LoRA 지원 시) — 메모리 최적
  ┌────────────────────────────────────────────┐
  │ Compact 2B INT4 백본 1개 (~1 GB) — 상주       │
  │   └ IDS · MA · SA LoRA 동시 활성 (배치 추론)   │
  ├────────────────────────────────────────────┤
  │ KV 캐시 (역할별 캡 + 양자화) / 상태(디스크 오프로드) │
  └────────────────────────────────────────────┘   피크 ≈ ~1 GB + 어댑터 수개

C5 (폴백) — 런타임 안전
  ┌──────────────────────┐   ┌──────────────────────────┐
  │ IDS 전용 백본 (~1 GB)  │   │ MA/SA 공유 백본 (~1 GB)     │
  │  상시 상주             │   │  + MA/SA LoRA hot-swap     │
  └──────────────────────┘   └──────────────────────────┘
        IDS 추론                 협상 추론 (MA·SA)
                         피크 ≈ ~2 GB + MA/SA 어댑터
```

### 5.3 직교(orthogonal) 보조 기법 — 어느 안에도 결합 가능

본 결정과 별개로, 다음 기법들은 후보와 무관하게 추가 적용 가능하며 권장안에 함께 채택한다.

- **가중치 양자화(INT4)**: 본 문서 모든 추정의 전제.
- **KV 캐시 양자화 + 컨텍스트 윈도 캡**: N-party·다라운드에서 가장 큰 보조 메모리 비용. 역할별 토큰 상한을 둔다. (IDS의 FR-IDS-04 긴 컨텍스트는 캡 설계 시 별도 고려.)
- **에이전트 상태 디스크 오프로드**: 이벤트 로그는 DP-A2(영속화)와 통합되어 디스크 보관. 메모리에는 최근 상태만.
- **mmap 기반 가중치 로딩** *(선택)*: 평상시 RSS 축소에 유효하나 워킹셋이 비결정적이라 *피크 보장(SLA)* 용도로는 부적합. 보조적 사용에 한정.

## 6. 후속 결정 (이 문서 다음에 풀어야 할 질문)

1. **★ 동시 다중 LoRA 추론의 런타임 지원 검증** (C4 vs C5 분기점): 단말 추론 엔진이 *동시 다중 어댑터 활성·배치 추론*을 지원하는지 PoC로 확인한다. 지원되면 C4, 아니면 C5. **이 검증이 본 결정의 최우선 후속 과제다.**
2. **어댑터 소싱 경로**: 역할별 LoRA를 (a) 직접 학습 / (b) 공개 어댑터 활용 / (c) 빈 어댑터(no-op)로 시작 중 어느 경로로 확보할지. *C4·C5 어느 쪽이든 공통으로 필요한 결정이다.*
3. **NFR 임계값 재정의**: 200/300MB는 Compact 2B INT4 단독 사용도 초과하므로, NFR-IDS-04·NFR-MAF-06의 임계값과 측정 범위(가중치 포함/제외, 백본 1개/2개)를 재정의해야 한다. C5(~2 GB)와 C4(~1 GB)는 임계값이 달라질 수 있다. → [05-NFR.md](../05-NFR.md), [07-QAS.md](../07-QAS.md) 동기화 (CLAUDE.md 7항).
4. **Meta Agent 협상 품질이 Compact + LoRA로 부족할 경우의 대응**: 서버 LLM 위임을 결정 공간에서 제외했으므로 단말 내 해결. 후보: (a) 협상 라운드 수 축소, (b) 협상 메시지 구조화 강화(DP-A3 연계), (c) MA에 한해 Compact보다 큰 단말 모델 도입 재검토(C5라면 MA/SA 백본만 키우면 되므로 비교적 국소적 변경).
5. **DP-D1과의 정렬**: 본 결정이 확정되면 DP-D1(자원 중재) 스케줄링 정책은 채택안(C4 동시 멀티-LoRA / C5 백본 2개)을 전제로 다시 작성되어야 한다. 특히 C4의 단일 백본 *연산* 경합, C5의 백본 2개 메모리 동시 상주는 DP-D1의 스케줄링 입력이 된다.

## 7. 참고 — 본 문서의 가정과 한계

- 모든 메모리 수치는 **정성적 추정**이다. 2B × 0.5 byte ≈ 1 GB는 표준 산식이며, 추론 엔진별 오버헤드(런타임·KV·활성값)는 별도이다.
- C1의 ~3 GB, C5의 ~2 GB 추정은 "가중치만, 해당 인스턴스 동시 상주 시"의 값이며, 실측에서는 KV·상태가 추가된다.
- LoRA 어댑터의 크기·hot-swap 지연·품질 효과, 그리고 **동시 다중 LoRA 추론의 지원 여부**는 모델 아키텍처·추론 엔진·학습 설정에 따라 변한다. "수~수십 MB", "수 ms swap"은 일반 LoRA 사례 기준이며 *우리 환경에서의 검증이 필요*하다(6절-1).
- IDS의 동작 모드(독립/협상 동반)와 동시 추론 빈도는 [04-FR.md](../04-FR.md)의 기능 정의에 근거한 *논리적 추론*이며, 실제 발생 빈도·동시 추론의 강도(IDS가 협상 중 전체 추론을 요구하는가, 경량 게이트로 충분한가)는 워크로드 측정으로 확인해야 한다. 이 강도에 따라 C4 단일 백본의 연산 경합 부담이 달라진다.

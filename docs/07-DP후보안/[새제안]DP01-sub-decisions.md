## DP-N1. Proposal Layer 데이터 구조

### 결정해야 할 것

Proposal Phase에서 각 단말이 자기 선호를 *어떤 CRDT 자료구조*에 기록할 것인가. 이 선택이 *gossip 메시지 크기 · 충돌 해소 의미론 · QAS-014 자원 부담*을 정한다.

### 후보안

**A. OR-Set (Observed-Removed Set)** — *추가는 누적, 삭제는 ID 기반.*

각 후보 시간 `add(actor, time)` 이벤트로 추가. 한 단말이 자기 후보를 *철회*하면 `remove(time_id)` 이벤트 추가. 동시에 한 시간을 추가·삭제하면 *추가가 이김* (Add-Wins).

- *장점*: 협상의 "후보 누적" 의미와 자연스럽게 맞음. 철회 의미론 명확. 충돌 자동 해소.
- *단점*: 삭제 메타데이터(tombstone)가 누적. 장기 협상 시 메모리 증가 (QAS-014 Memory에 부담).

**B. LWW-Map (Last-Writer-Wins Map)** — *키별로 마지막 쓴 사람 이김.*

`map[actor] = preference_set` 형태로 각 단말이 *자기 선호 집합 전체*를 한 키에 저장. 새 선호 추가 시 그 단말의 전체 집합을 다시 씀.

- *장점*: 단순. 한 단말의 선호 변경이 *원자적 단일 update*.
- *단점*: 매 변경에 *전체 집합 재전송* → gossip 메시지 크기 ↑ (QAS-014 통신 부담). 동시 변경 시 LWW로 정보 손실 가능.

**C. MV-Register (Multi-Value Register)** — *동시 쓰기 모두 보존, 상위 레이어가 해소.*

각 단말의 선호를 키별로 저장하되, 동시 쓰기 발생 시 *둘 다 보존*하고 *상위 레이어가 충돌 해소* 결정.

- *장점*: 정보 손실 없음. 충돌 해소를 협상 도메인 로직에 위임 (유연).
- *단점*: 상위 레이어 충돌 해소가 *비결정적*일 수 있어 QAS-015 (모든 노드 동일 상태)와 충돌 위험.

### 비교 (Trade-off Table)

| 항목 | A. OR-Set | B. LWW-Map | C. MV-Register |
|---|:---:|:---:|:---:|
| 의미론 적합도 (협상) | ★★★ (누적·철회 자연스러움) | ★★☆ (덮어쓰기) | ★☆☆ (충돌 해소 상위 위임) |
| 메시지 크기 | ★★★ (이벤트 단위) | ★☆☆ (전체 집합 재전송) | ★★☆ |
| 충돌 해소 결정론 | ★★★ (Add-Wins 자동) | ★★☆ (시간순) | ★☆☆ (상위 레이어 의존) |
| 메모리 누적 | ★☆☆ (tombstone) | ★★★ | ★★☆ |
| QAS-014 자원 | ★★★ | ★☆☆ | ★★☆ |
| QAS-015 일관성 | ★★★ | ★★★ | ★☆☆ |

**권고: A (OR-Set).** 협상의 누적·철회 의미와 가장 직접적이며 QAS-014·015 동시 만족. 메모리 누적(tombstone)은 *세션 종료 시 정리*하면 PoC 규모에서 문제 안 됨.

## DP-N2. Phase Transition 메커니즘

### 결정해야 할 것

Proposal Phase에서 Confirmation Phase로 *언제* 넘어가는가. 이 신호가 *모든 노드에 일관되게 도착*해야 Confirmation의 입력(snapshot)이 결정론적이 된다.

### 후보안

**A. Explicit Signal (사용자 명시적 완료)** — *"이제 결정하자" 버튼.*

한 명이 명시적으로 "협상 마감" 신호 → 모든 노드에 전파. 받은 노드는 *snapshot 잠금* 후 Confirmation 진입.

- *장점*: 의미가 사용자에게 명확. *의도된 전환*이라 잘못된 시점에 전환될 위험 적음.
- *단점*: 신호 *전파 지연* 동안 일부 노드는 Proposal, 일부는 Confirmation → split-phase 위험.

**B. Timeout (마지막 추가 후 N초)** — *quiet period 감지.*

마지막 `add` 이벤트로부터 N초간 추가 없으면 자동 Phase Transition. 각 노드가 *독립적으로* timer 운영.

- *장점*: 명시적 신호 불필요. 사용자 행위 부담 적음.
- *단점*: timer가 *각 노드마다 어긋남* (gossip 지연 때문). 어떤 노드는 일찍, 어떤 노드는 늦게 전환 → snapshot 어긋남.

**C. Quiescence Detection (gossip 정착 감지)** — *vector clock 기반.*

모든 노드의 vector clock이 안정됨을 감지 → 정확히 같은 시점에 전환. 분산 시스템 이론의 *snapshot 알고리즘* 응용.

- *장점*: snapshot이 *진정으로* 결정론적. 모든 노드 동일 입력 보장.
- *단점*: 구현 복잡. quiescence 자체를 감지하려면 추가 메시지 라운드 필요.

### 비교

| 항목 | A. Explicit | B. Timeout | C. Quiescence |
|---|:---:|:---:|:---:|
| Snapshot 결정론 | ★★☆ (전파 지연) | ★☆☆ (timer 어긋남) | ★★★ |
| 구현 단순성 | ★★★ | ★★★ | ★☆☆ |
| 사용자 부담 | ★☆☆ (명시 조작) | ★★★ | ★★★ |
| QAS-014 자원 | ★★★ | ★★★ | ★★☆ (추가 메시지) |
| QAS-015 (snapshot 일치) | ★★☆ | ★☆☆ | ★★★ |

**권고: A (Explicit) + B (Timeout) 결합.** Explicit이 1차, *발신자가 응답 못 받으면* Timeout으로 fallback. C는 PoC 범위에 비해 구현 비용이 크고, A+B 결합으로 *충분히 결정론적인 snapshot*을 얻을 수 있음. 단 "Explicit 신호 전파 지연 동안" 도착하는 변경은 *snapshot 잠금 시점 이전이면 포함, 이후면 거부*하는 규칙 필요.

## DP-N3. Confirmation Vote 정책 — 가장 결정적

### 결정해야 할 것

Confirmation Phase의 vote 수집에서 *얼마나 많은 노드의 동의가 필요한가*. 이 결정이 **QAS-016 "모든 ACK 수신 후 Action"의 정확한 의미**를 정한다.

### 후보안

**A. Unanimous (전원 합의)** — *N개 노드 모두 vote 필수.*

N개 vote를 모두 받아야만 commit. 한 명이라도 응답 안 하면 *무한 대기 또는 무효 전이*.

- *장점*: QAS-016의 "모든 ACK"를 *정확히* 만족. 합의 의미가 명확.
- *단점*: *지연자 한 명이 협상 전체를 멈춤.* 모바일 환경의 일시적 단절 시 매번 무효 전이 위험.

**B. Quorum (정족수, 보통 N/2+1)** — *과반 vote면 commit.*

과반 이상의 vote 수신 시 commit. 미응답자는 *나중에 sync*받음 (catch-up).

- *장점*: 일부 노드 일시 단절에도 협상 진행. *Liveness 보장.*
- *단점*: 미응답자가 *commit 사실을 모르는 순간*이 존재 → QAS-015 일관성을 *순간적으로* 위반. QAS-016의 "모든 ACK"와 의미론적으로 어긋남.

**C. Weighted Vote (가중치)** — *역할별 가중치.*

협상 개시자·우선 참여자 등에 가중치 부여, 가중치 합이 임계값 넘으면 commit.

- *장점*: 도메인 특화 (보호자-피보호자 등).
- *단점*: 가중치 정책 자체가 별도 결정. *고정 Coordinator의 변형*에 가까워질 위험.

### 비교

| 항목 | A. Unanimous | B. Quorum | C. Weighted |
|---|:---:|:---:|:---:|
| QAS-016 정의 충족 | ★★★ (정의 그대로) | ★☆☆ (의미론 어긋남) | ★★☆ |
| Liveness (진행 보장) | ★☆☆ (지연자 = 정지) | ★★★ | ★★☆ |
| QAS-015 일관성 | ★★★ | ★★☆ (catch-up 중 불일치) | ★★☆ |
| 단순성·공정성 | ★★★ | ★★★ | ★☆☆ |
| 모바일 환경 적합도 | ★☆☆ (일시 단절 취약) | ★★★ | ★★☆ |

### 솔직한 분석

이 결정이 **DP-N3의 본질적 갈등**이다. *QAS-016 정의*를 글자 그대로 따르면 A(Unanimous), *모바일 현실성*을 보면 B(Quorum). 두 요구가 정면 충돌.

세 가지 해결 경로:

1. **A 채택 + QAS-016 정의 유지**: 지연자 처리를 *명시적 무효 전이*로. 모바일에서는 협상이 자주 무효화될 수 있음 (PoC 검증 필요).
2. **B 채택 + QAS-016 의미 완화**: "모든 ACK"를 "정족수 ACK + 미응답자는 사후 sync"로 *QAS-016 자체를 수정*. 이건 QAS 측 변경이라 [`07-QAS.md`](../07-QAS.md) 갱신 필요.
3. **하이브리드 (A를 1차, B를 fallback)**: T1초 안에 모두 vote → commit; T1~T2 사이에 정족수 → commit + 미응답자 sync; T2 후 정족수 미달 → 무효.

**권고: 3 (하이브리드)** — QAS-016 의미를 최대한 유지하면서 모바일 현실성도 확보. 단 T1, T2 임계값이 새 미해결 결정으로 추가됨.

## DP-N4. 부분 ACK 처리 정책

### 결정해야 할 것

DP-N3가 Quorum 또는 하이브리드를 택했을 때, *정족수는 도달했지만 일부 미응답인* 상태를 어떻게 처리하는가.

### 후보안

**A. Wait (무한 대기)** — *모든 ACK 도달까지 commit 보류.*
- DP-N3=Unanimous면 자동 채택. QAS-016 정의 그대로.

**B. Retry (재시도)** — *미응답자에게 vote 재요청, K회까지.*
- Liveness와 정의 사이 절충. 일시 단절 회복 시 정합성 유지.

**C. Drop (미응답자 제외 commit)** — *정족수만으로 commit, 미응답자는 사후 무효.*
- Liveness 강함. 미응답자가 *재합류 시 새 협상 시작 필요*.

**D. Degrade (정족수 commit + 미응답자 catch-up)** — *commit 진행, 미응답자는 부활 시 결과만 통보.*
- 가장 실용적. 미응답자는 결과를 통보받지만 *Action 트리거는 그 단말에서 별도 처리*.

### 비교

| 항목 | A. Wait | B. Retry | C. Drop | D. Degrade |
|---|:---:|:---:|:---:|:---:|
| QAS-016 정의 충족 | ★★★ | ★★★ | ★☆☆ | ★★☆ |
| Liveness | ★☆☆ | ★★☆ | ★★★ | ★★★ |
| 미응답자 처리 | 없음 (대기) | 재시도 | 제외·재협상 | catch-up |
| 구현 복잡도 | ★★★ (단순) | ★★☆ | ★★★ | ★☆☆ |
| 도메인 적합도 | 동기적 합의 | 일반 협상 | 일회성 | 장기 협상 |

**권고: B (Retry) + D (Degrade) 단계적 적용.** K회 재시도로 일시 단절 회복 우선 (B), 그래도 미응답이면 정족수 commit + catch-up (D). 이게 DP-N3의 하이브리드 권고와 자연스럽게 결합됨.

## 종합 — 4개 sub-decision의 권고 묶음

| Sub-DP | 권고안 | 핵심 근거 |
|---|---|---|
| **DP-N1** | A. OR-Set | 협상 누적·철회 의미론 자연 매핑, QAS-014·015 동시 만족 |
| **DP-N2** | A+B (Explicit 1차, Timeout fallback) | 결정론적 snapshot + 사용자 부담 균형 |
| **DP-N3** | 하이브리드 (Unanimous→Quorum fallback) | QAS-016 정의 유지 + 모바일 현실성 |
| **DP-N4** | B+D (Retry→Degrade) | Liveness 확보 + QAS-016 의미 최대 유지 |

이 네 권고를 묶으면, DP01 v2의 2-Layer 구조가 *모든 QAS를 답하면서 PoC에서 구현 가능한 범위*에 들어온다.

## 새로 추가되는 미해결 결정

1. **DP-N3 하이브리드의 T1, T2 임계값** — Unanimous 대기 시간 T1, Quorum 허용 시점 T2의 구체 수치. QAS-015의 2초 한계와 조율 필요.
2. **DP-N4 Retry의 K값** — 재시도 횟수. 모바일 일시 단절 통계 기반 결정.
3. **DP-N4 Degrade 시 catch-up 메시지 비용** — 미응답자가 부활 시 받아야 할 누적 이벤트 양이 QAS-014 Memory에 영향.
4. **DP-N1 tombstone 정리 정책** — 세션 종료 시 OR-Set tombstone 삭제 시점. 세션 길이 가정에 따라 다름.

---

_본 sub-decision 문서는 DP01 v2의 결정 차원을 *명시적으로 분해*해 각 결정이 어느 QAS에 답하는지 추적성을 박는다. DP-N3의 하이브리드 권고가 가장 큰 미해결 결정(임계값)을 남기지만, QAS-016 정의 변경 없이 모바일 환경에서 의미를 살리는 유일한 경로._

# DP02 — 민감정보 처리 구조 설계
> 이전에 희정님이 작성하셨던, [DP06-협상 민감정보 처리 설계](<../_legacy/DP 후보안/DP06-협상 민감정보 처리 설계.md>)를 구성 형식만 수정한 것

---

## 1. 풀고자 하는 문제

협상이 성립하려면 단말은 상대에게 무언가를 전달해야 하는데, 그 "무언가"가 곧 사용자의 사생활이다(가능 시간대·예산·선호 장소, 그리고 그 근거가 되는 기기 컨텍스트). 따라서 풀어야 할 문제는, **민감정보 0% 유출(QAS-012)을 지키면서 협상 속도(QAS-009)를 어떻게 확보하는가** 이다.

핵심은 민감정보를 한 덩어리로 다루지 않고, 처리 방식이 다른 세 종류로 분리하는 것이다.

- **(a) 식별정보(PII):** 이름·전화번호·정확한 주소. 협상에 불필요하므로 완전히 가린다(토큰화/제거).
- **(b) 협상 사실값:** "토요일 저녁 가능", "1인 5만원 이하", "강남 인근". 협상에 필수이므로 드러내되 정밀도를 낮춰 최소화한다.
- **(c) 원본 컨텍스트:** 기기 원본("19:30 세탁 종료" 등). 출구로 나가면 안 되므로 단말에서 사실값으로 변환 후 폐기한다.

> 본 DP는 "정보를 공개한다"를 전제로, 그 정보를 **어떻게 안전하게 만들어 내보내는가**를 다룬다. "공개할지(제한 공개) 말지(비노출 계산)"의 결정은 [DP02-정보 공개 모드](<./DP02-정보 공개 모드.md>)가 다루며, 본 DP는 그 **제한 공개형을 택했을 때의 하위 설계**에 해당한다.

## 2. 아키텍처적 난제

| 난제 | 내용 | 관련 |
|---|---|---|
| **출구 통제** | 민감정보가 단말을 떠나는 출구(서버 LLM행, 상대 에이전트행)를 우회 불가능하게 통제해야 한다. 속도를 위해 게이트를 건너뛰면 0% 보장이 깨진다. | QAS-012 |
| **기밀성 vs 속도** | 안전화 비용이 매 협상 라운드의 임계 경로에 누적되면 협상이 느려진다. "어디서·언제 안전하게 만드나"가 둘을 동시에 좌우한다. | QAS-009, QAS-012 |
| **드러냄과 가림의 경계** | 협상 사실값은 가리면 협상이 불가능하고, 그대로 드러내면 노출된다. "강남 인근"까지 vs "강남역 도보 5분"까지의 **정밀도**가 노출 수준을 가른다. | QAS-012 |
| **권한 결속** | 사용자가 위임한 범위를 초과하는 정보는 생성·전송 단계에서 차단해야 한다. | FR-MAF-10 |

세 방안이 공유하는 전제:

1. **출구 단일화.** 외부로 나가는 모든 트래픽은 우회 불가능한 게이트를 통과한다. 0% 보장의 검증 지점을 단일화한다.
2. **전송 암호화.** TLS 1.3 이상(QAS-012)은 공통 필수이며 본 DP의 비교 대상이 아니다.
3. 본 DP의 결정축은 **민감정보를 안전하게 만드는 시점(When)·위치(Where)·방식(How)** 이다.

## 3. 방안 1 — 출구 제거 (Filter-at-Egress)

### 개념

협상 추론은 원본 민감정보를 자유롭게 쓰되, 단말을 떠나는 출구 직전에 동기적으로 제거·최소화한다. 위험한 원본을 들고 있다가 마지막 문턱에서 거른다.

### 장점

- 구조 변경이 가장 적다. 기존 출구 게이트에 필터만 더하면 된다.
- 출구가 독점되어 0% 보장의 검증·감사 지점이 자명하다(QAS-013 추적성 유리).
- 추론이 원본을 그대로 써서 제안 품질 손실이 없다(라운드 수 최소화에 유리).

### 위험

- 매 라운드 동기 필터가 임계 경로에 누적되어 협상이 느려진다(QAS-009 불리).
- 원본이 출구 직전까지 협상 경로 전체에 흐르므로, 0% 보장이 **필터의 신뢰성에 100% 의존**한다(필터 버그 = 유출).

## 4. 방안 2 — 사전 변환 (Transform-Upfront)

### 개념

협상 시작 시 원본을 단 한 번 안전한 형태로 변환한다 — PII는 토큰화, 원본 컨텍스트는 사실값으로 변환·폐기, 협상 사실값은 권한 범위에 맞춰 정밀도를 미리 낮춘다. 이후 협상 추론은 안전 형태만 보고, 원본은 협상 경로에 아예 들어오지 않는다.

### 필수 메커니즘

- **PrivacyMediator:** 세션 시작 시 (a) PII 토큰화 (c) 원본→사실값 변환·폐기 (b) 권한 기반 정밀도 사전 컴파일을 1회 수행한다.
- **PrivacyVault:** 토큰↔원본 매핑을 단말 내부에 보관한다. 합의 후 토큰을 원본으로 복원해 실행 단계(예약·기기 반영)에서만 사용한다.

### 장점

- 0%를 "필터로 거른다"가 아니라 "원본이 출구 경로에 애초에 없다"로 달성한다. 필터 버그가 나도 새어 나갈 원본 자체가 없다(방안 1보다 구조적으로 강함).
- 변환이 세션당 1회뿐이라 라운드당 프라이버시 비용이 0이다. 협상이 길수록(다라운드·교착) 속도 이득이 커진다(QAS-009 유리).
- 정밀도가 권한 범위에 결속되어, 권한 초과 사실값이 생성 단계에서 차단된다(FR-MAF-10) — 기밀성과 권한 통제를 한 메커니즘으로 묶는다.

### 위험

- 신규 컴포넌트(PrivacyMediator·PrivacyVault) 도입으로 복잡도·구현 비용이 는다.
- 토큰 매핑의 보관·만료·메모리 관리가 필요하다(세션 복구 QAS-008·온디바이스 자원 QAS-011과 연계 검토).
- 사전 컴파일된 안전 범위가 협상 중 바뀌어야 하면(사용자 직접 개입) 재컴파일 비용이 든다.

## 5. 방안 3 — 온디바이스 추론 격리 (On-device Inference Isolation)

### 개념

민감정보를 다루는 협상 추론 자체를 온디바이스 LLM에 가두고, 서버 LLM에는 민감 원본이 필요 없는 일반 작업만 위임한다. 데이터를 가공해 내보내는 대신, **민감 추론이 일어나는 위치**를 단말 밖으로 내보내지 않는다.

> 주의: 방안 1·2가 "안전화 시점·방식"을 정하는 데 비해, 방안 3은 "추론 위치"라는 다른 축이다. 서버 LLM행(출구1)만 닫으며 상대 에이전트행(출구2)은 커버하지 못하므로, 방안 1·2와 **결합**해야 완성된다.

### 장점

- 민감 추론이 단말 안에서만 일어나 서버 LLM행 유출 경로가 **구조적으로 닫힌다**(거름이 아니라 경로 부재).
- 민감/비민감 라우팅을 게이트가 강제하므로, 실수로 민감 데이터를 서버로 보내는 분기를 만들 수 없다.

### 위험

- 출구2(상대 에이전트행)의 정보 최소화·암호화는 여전히 별도 처리가 필요하다(방안 1·2와 결합 전제).
- 온디바이스 LLM은 서버 LLM보다 추론이 약하고 느려, 협상 품질(라운드 수)과 단말 자원에 부담을 준다(QAS-009·QAS-011 불리).

## 6. 종합 비교

> 척도: ★★★ 강함 · ★★☆ 보통 · ★☆☆ 취약

| 평가축 | 방안 1 출구 제거 | 방안 2 사전 변환 | 방안 3 온디바이스 격리 | 판단 |
|---|:---:|:---:|:---:|---|
| 기밀성 — 서버 LLM 유출 (QAS-012) | ★★☆ | ★★★ | ★★★ | 1은 늦게 거름(필터 의존), 2는 원본 미도달, 3은 서버행 경로 자체 부재 |
| 기밀성 — 제안 정보 최소화 (QAS-012) | ★★☆ | ★★★ | ★☆☆ | 3은 출구2를 못 막아 별도 보장 필요. 2는 권한 기반 정밀도로 안정적 |
| 권한 초과 차단 (FR-MAF-10) | ★★☆ | ★★★ | ★★☆ | 2는 생성 단계에서 차단(정밀도=권한 결속). 1·3은 게이트 별도 검증 |
| Communication 시간 (QAS-009) | ★☆☆ | ★★★ | ★☆☆ | 1은 라운드당 필터 누적, 2는 세션 1회, 3은 온디바이스 추론 지연 |
| 온디바이스 자원 (QAS-011) | ★★★ | ★★☆ | ★☆☆ | 3은 무거운 협상 추론을 단말에서 돌려 부담이 큼 |
| 추적성 (QAS-013) | ★★★ | ★★☆ | ★★☆ | 1은 출구 단일·동기라 추적이 단순 |
| 구현 복잡도 | ★★★ | ★☆☆ | ★★☆ | 1은 기존 게이트만 손댐, 2는 신규 컴포넌트 2개, 3은 라우팅 분리 |

**핵심 긴장:** 방안 1은 가장 단순하지만 원본을 들고 다녀 0%가 필터 신뢰에 의존하고 라운드당 필터 비용이 속도를 깎는다. 방안 2는 원본을 경로에 들이지 않아 기밀성이 구조적으로 강하고 세션 1회 비용이라 다라운드에서 속도도 유리하지만, 신규 컴포넌트의 복잡도를 부담한다. 방안 3은 서버 LLM 유출을 경로 부재로 가장 강하게 닫지만 출구2를 못 막고 온디바이스 추론이 속도·자원을 압박한다(방안 1·2와 결합 전제). 따라서 선택은 **기밀성을 "필터"로 달성할지 "원본 미도달"로 달성할지**, 그리고 **온디바이스 추론의 속도·자원 비용이 감당 가능한지**의 측정으로 좁혀진다. 한편 "정밀도를 어디까지 낮출지(강남 인근 vs 강남역 도보 5분)와 그 기준을 누가 정할지"는 세 방안 공통의 미결 변수로 남는다.

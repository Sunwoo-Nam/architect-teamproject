# DP03 — MAF 온디바이스 LLM 배치: 자원 PoC 계획 (Resource PoC)

> **검증 대상 결정**: [DP03-MAF-LLM-배치.md](./DP03-MAF-LLM-배치.md) — MAF용 추론 엔진을 후보 A(추론 유형별 모델 교체) vs B(단일 범용 상주)로 운용하는 결정
> **연관 전제·기준**: P2(IDS 1 + MAF 1, 동시 2모델) · M1/M2(메모리) · M4a(전환 지연) · M4b(스텝 추론 속도) — [DP03-MAF-LLM-배치.md §0·§1·§4](./DP03-MAF-LLM-배치.md)
> **Mapped NFR/QAS**: [QAS-011](../07-QAS.md) · [NFR-MAF-02](../05-NFR.md)(지연) · [NFR-MAF-06](../05-NFR.md)(메모리)
> **후속 결정 연계**: [DP03-MAF-LLM-배치.md §5.4·§6](./DP03-MAF-LLM-배치.md)
>
> 본 문서는 위 배치 결정을 검증하기 위한 **PoC 중 "자원·성립 가능성" 부분만** 다룬다. 실제 협상 data가 필요한 **품질(정확도) 검증은 범위 밖**이며 후속으로 분리한다(§6).

---

## 0. 이 PoC의 위치와 한계 (먼저 합의)

**이 PoC가 결정하는 것**
- ✅ **성립 가능성** — IDS 모델 + MAF 모델(동시 2모델, P2)이 실제 단말 RAM에 OS·App 위에서 올라가는가.
- ✅ **A의 비용 측면** — swap 지연(A의 약점)이 견딜 만한가, 작은 모델의 속도·메모리 이득(A의 강점)이 실재하는가.
- ✅ **비용만으로 A를 기각/강화** — swap이 과도하면 품질을 볼 것도 없이 B로 수렴(§4). 반대로 swap이 싸고 속도 이득이 크면 A의 비용 근거 확정.

**이 PoC가 결정하지 못하는 것**
- ❌ **A vs B 최종 결정** — 그것은 추론 유형별 품질(M3·[NFR-MAF-04](../05-NFR.md))이 가르며, 본 PoC 범위 밖이다.
- ❌ **절대적 실사용 품질** — 실데이터·실유저 필요.

> **설계 의도**: 가장 싸면서 가장 결정적인 실험을 *먼저* 한다. 품질 PoC(학습 data·평가 셋업이 필요한 비싼 일)에 들어가기 전에, **비용 근거만으로 A를 거를 수 있는 게이트**로 둔다.

## 1. 검증 항목

| # | 축 | 측정 대상 | 매핑 |
|---|---|---|---|
| R1 | **메모리** | 모델 적재 RAM, IDS+MAF 동시 상주 fit 여부 | M1·M2 · P2 · [QAS-011](../07-QAS.md) · [NFR-MAF-06](../05-NFR.md) |
| R2 | **모델 전환 지연** | unload→load→워밍업까지 swap 1회 비용 | M4a · [NFR-MAF-02](../05-NFR.md) |
| R3 | **추론 시간** | 프리필(TTFT)·decode 토큰/초·스텝 end-to-end | M4b · M2 · [NFR-MAF-02](../05-NFR.md) |
| R4 *(옵션)* | **에너지** | 추론당 J, 평균 전력 W | M2 |

> R4는 R2·R3와 *같은 실행*에서 전력만 추가 로깅하면 되므로 한계비용이 거의 0이다. 권장 포함.

## 2. 측정 정의 (해석이 갈리지 않도록 명시)

### R1 메모리
- **모델 상주 RSS / NPU 메모리**: idle(로드 직후) + **추론 중 peak**(프리필·decode 각각).
- **KV 캐시 포함 peak**: 타깃 컨텍스트 길이 기준.
- **동시 상주 시험**: IDS 모델 + MAF 모델을 *실제 OS·대표 App이 떠 있는 상태에서* 동시 적재 → LMK(Low Memory Killer) 발동 없이 유지되는가. (모델 단독 측정이 아니라 *실 환경 점유* 위에서 측정해야 의미 있음.)
- **양자화별**: 4-bit / 8-bit 각각.

### R2 모델 전환 지연
- swap 1회 = **(현 모델 unload) + (다음 모델 load/mmap) + (NPU 그래프 init/compile) + (워밍업: 첫 토큰까지)** 의 합. 각 단계 분해 기록.
- **cold vs warm 구분**: cold(플래시에서 최초 로드) / warm(OS 페이지캐시 적중). **cold를 최악값으로 보고**.
- **KV 캐시 재구성 비용**: swap 후 직전 컨텍스트 재적재 비용.
- 에피소드 환산: `swap 1회 비용 × 가정 swap 횟수(P6: 1~3회)` = 에피소드당 총 전환 오버헤드.

### R3 추론 시간
- **TTFT(Time-To-First-Token)**: 프롬프트 길이의 함수로.
- **Decode 처리량**: 토큰/초.
- **스텝 end-to-end**: 대표 스텝(예: 협상 1턴 N토큰 생성) 벽시계 시간.
- **모델 크기별 곡선으로**: 1B / 3B / 7B(Q4) 등 *크기 브래킷*으로 측정 → 최종 모델 미정이어도 결과가 전이되도록 (특정 모델 1개에 베팅하지 않음).

### R4 에너지(옵션)
- 추론당 J, 평균 전력 W. 가능하면 외부 전력계 또는 단말 배터리 통계 API.

## 3. 실험 셋업

| 항목 | 내용 |
|---|---|
| **단말** | NPU 탑재 플래그십 (예: Galaxy S25 / Snapdragon 8 Elite 급 — [DP03-MAF-LLM-배치.md §3.5](./DP03-MAF-LLM-배치.md) 참조) 또는 동급 |
| **런타임** | ⚠️ **최대 변수.** llama.cpp(CPU/GPU)는 셋업이 쉽지만 NPU 미사용 → 수치가 다름. NPU 정확 수치는 벤더 SDK(QAIRT/Genie, MediaTek 등) 필요. **"어느 백엔드로 측정했는지" 반드시 명시.** 1차는 llama.cpp로 ballpark, 확정치는 NPU SDK로. |
| **후보 모델** | 크기 브래킷: 작은(~1B, 툴/판정용 후보) + 큰(~3B, 협상용 후보), Q4 기준. *최종 선택 전이라 범위로 본다.* |
| **하네스** | load/unload·메모리 샘플링·tokens/sec·swap 루프·전력 로깅 자동화 스크립트 + 대표 길이 합성 프롬프트 (협상 *내용*은 무관, 길이만 대표값) |

## 4. 합격 기준 & A 기각/강화 판정 로직

> 절대 임계값은 NFR 재정의 후 확정(§5). 아래는 *판정 구조*.

- **R1 성립 가능성 (게이트)**: IDS+MAF 동시 상주가 단말 가용 RAM 안에서 LMK 없이 유지되면 **PASS**. 실패 시 → P2 자체 재검토(모델 더 줄이거나 양자화 강화 또는 서버 오프로드 재고).
- **A 비용 판정**:
  - swap 총 오버헤드(R2 × P6 횟수)가 **백그라운드·비동기 허용 범위**( [DP03-MAF-LLM-배치.md §1·§7](./DP03-MAF-LLM-배치.md) ) 안이고, 작은 모델의 R3(속도)·R1(메모리) 이득이 **유의미** → **A 비용 근거 확정** (남은 건 품질뿐).
  - swap 오버헤드가 허용 범위를 **초과** → 품질 검증 없이 **B로 수렴** ([DP03-MAF-LLM-배치.md §5.3-a](./DP03-MAF-LLM-배치.md) 정합).
- **속도-에너지 상관 주의**: R3와 R4는 *모델 크기 × 토큰*이라는 같은 뿌리라 강상관 → 이중 계산 금지. R3는 "지연", R4는 "배터리"로 *해석을 분리*해 본다.

## 5. 산출물

1. **3축 측정표** (모델 크기·양자화·cold/warm 별) + 크기→속도/메모리 곡선.
2. **메모리 예산 확정치** → 본 PoC의 핵심 산출. 현재 [QAS-011](../07-QAS.md)이 "MAF Memory 300MB"로 되어 있으나 2B 모델만 ~1.3GB로 **불일치**가 의심됨(300MB가 모델 제외 프레임워크 오버헤드인지 placeholder인지 모호). **실측으로 숫자를 못박아 QAS-011·[NFR-MAF-06](../05-NFR.md) 재매핑** ([DP03-MAF-LLM-배치.md §6-3](./DP03-MAF-LLM-배치.md), [CLAUDE.md 7항]).
3. **A 비용 판정 결과** (확정/강화 또는 기각) + 근거 수치.

## 6. 한계 및 후속

- **합성 프롬프트 ≠ 실사용 부하**: 길이·분포가 대표적이어야 수치가 전이됨. 토큰 길이 분포는 시나리오 설계에서 보수적으로 잡는다.
- **품질(정확도) PoC는 별도**: 협상 품질([NFR-MAF-04](../05-NFR.md))·swap 빈도(P6) 검증은 합성 시나리오 + (일정 조율 도메인의) 계산 가능한 최적해 + LLM-judge로 부트스트랩 가능하나, **본 PoC 범위 밖**. 본 자원 PoC가 A를 비용으로 기각하지 않은 경우에 한해 진행.
- **학습 data**: 후보 A의 *특화 모델*(또는 LoRA 어댑터 [§3.5](./DP03-MAF-LLM-배치.md))을 제대로 평가하려면 학습 data가 필요 — 가장 무거운 단계이므로, B가 품질 미달일 때만 투자 ([DP03-MAF-LLM-배치.md §5.3-c](./DP03-MAF-LLM-배치.md)).

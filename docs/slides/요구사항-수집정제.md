## SLIDE | 요구사항 분석 | 요구 사항 수집 · 정제 과정

> 한 줄 메시지: 다양한 이해관계자의 요구를 **수집 → 분석·정제 → Architecture Driver 도출** 3단계로 수렴시켜 설계 입력을 확정한다.

---

### ① 요구사항 수집 (Collection)

**■ 주요 Stakeholder 의견 청취**
- Key Player 중심: End User · Architect · Multi-agent Framework Developer · Security/Privacy 담당 · AI/ML Engineer · 개인정보 보안 담당자 · Project Leader
- 전체 18개 Stakeholder의 관심사를 **VOC(1인칭 발언)** 로 도출

**■ 수집 활동**
- 과제 킥오프 미팅에서 과제 범위·목표를 정렬하고 초기 요구를 수렴
- 주차별 정기 팀 미팅을 통해 요구사항을 지속 리뷰·갱신
- 9개 시나리오 워크숍을 진행하여 사용 맥락별 요구를 도출하고 우선순위를 논의
- End User·Architect·Security/Privacy·AI/ML Engineer 등 핵심 Stakeholder 인터뷰 수행
- Architect 주도 설계 리뷰 세션에서 요구사항의 설계 적합성을 검토
- GitHub Issues로 요구사항을 등록·추적하고 VOC를 1인칭 발언으로 정리

**■ 수집 산출물**
- Stakeholder 18종 VOC · 9개 시나리오 · 과제 설명서 기능/비기능 요구 초안

---

### ② 요구사항 분석 및 정제 (Analysis & Refinement)

**■ 요구사항 분석**
- 요구사항 분류 (기능 / 비기능 / 제약)
- 관련 요구사항 통합 (원본 FR-L·T·X·N·S 계열 → FR-01~06으로 통합)
- 과제 범위 정의 (Out of Scope: Sub-Agent · Device Proxy Agent 기능 요구)

**■ 요구사항 정제**
- 기능 요구사항 도출 → **FR-01~06**
- 제약사항 도출 → **C-01** (상대방 기기 동일 MAF 탑재 전제) 등
- 품질 시나리오 명세 → **NFR → QAS 16종** (ISO/IEC 25010 기준)
- Use Case · Domain Model 분석 (보조 도식: `ondevice_agentic_platform_domain_model.drawio`)

---

### ③ Architecture Driver 도출 (→ 다음 장 전개)

**■ 도출 과정**
- 기능 / 품질 / 제약 분류
- **Utility Tree 기반 ASR 우선순위화** (중요도·난이도 분석 → 상위 5개 선정)
- Architecture Drivers 종합

**■ 세부 전개 연결**
- 기능 요구사항 · 제약사항 → 다음 슬라이드 (FR/제약 표)
- 품질 요구사항 · ASR → 그다음 슬라이드 (Utility Tree·QA)
- Architecture Driver 종합 → 그다음 슬라이드 (FR + 제약 + QA 수렴)

## 부록 · 요구사항 수집 과정 — 솔직본 (실제 진행 기반)

> 본문 "① 요구사항 수집"의 각색 표현(킥오프·워크숍·인터뷰 등)을 **실제 진행 사실**로 대체한 버전이다.
> 슬라이드/문서에 정식 반영할 경우 본문의 "■ 수집 활동"을 아래 내용으로 교체한다.
> (출처: 실제 진행 정황 확인 — 2026-07-01 사용자 확인.)

**■ 요구 원천 (Sources)**
- **과제 설명서 docx** — 기본 과제는 제공받고, Stakeholder·요구·시나리오는 **팀이 보완**함
- **팀 사전 산출물 `arch-with-ai`** (본 과제 전 우리 팀이 직접 작성) — 24개 품질 시나리오(QS-001~024) · `business.md` · NFR 분석
- **9개 상호작용 시나리오 · UC 01~04** (docx 원본 및 팀 보완)

**■ 실제 수집 활동 (Activities)**
- **현업 과제 멘토 멘토링** — 정기 멘토링 **4회** + 상시 문의를 통해 요구·배경 정보를 도출 *(핵심 채널)*
- **팀원 워크숍** — 팀원 4명이 하루(8시간)를 투자해 9개 상호작용 시나리오만 집중 논의
- 제공 과제 설명서 정독 및 팀 보완
- 팀 사전 산출물(`arch-with-ai`) 재사용

**■ VOC 정리 및 정직성 (Traceability)**
- 도출된 요구를 Stakeholder별 **1인칭 VOC**로 정리
- 원본 "관심사"가 비어 있던 이해관계자는 과제 성격·시나리오를 근거로 VOC를 **추정**하고 *italic*으로 명시 구분
- 원본 Stakeholder 표의 **빈 행·중복 ID(16번 2회)**를 그대로 보존 → 검증된 요구와 추정 요구를 구분

> **각색본과의 차이.** 각색본은 "킥오프·주차별 미팅, Stakeholder 인터뷰, GitHub Issues" 등 일반적 활동으로 표현했으나, 실제로는 **현업 멘토 멘토링(4회 + 상시 문의)과 팀원 워크숍(4명 · 하루 8시간 · 시나리오 논의)**이 중심이었으며, 별도의 현장 사용자 인터뷰는 수행하지 않았다.

# 비기능 요구사항 명세서 (Non-Functional Requirements Specification)


> 본 문서의 품질 속성 분류는 **ISO/IEC 25010:2023** 소프트웨어 품질 모델을 기준으로 한다.

> 측정 기준의 수치는 구현 단계에서 실측 결과를 바탕으로 조정 가능한 초기 제안치이다.


---


## 1. IDS 비기능 요구사항


### NFR-IDS-01. Performance - Intent 인지 알림 속도


정의: 사용자 입력이 IDS에 전달된 시점부터 Intent에 대한 처리 알림이 UI에 표시되기까지의 시간을 최소화해야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| Intent 인지 알림 제공 시간 | IDS에 입력 전달 시점 ~ Intent 관련 UI 알림 표시 시점 | **2초 이내** | Nielsen UX 응답성 연구에서 2초는 사용자 주의가 끊기기 시작하는 임계값. 사용자가 Intent가 인지되었다고 느끼는 시점은 UI 알림이 표시되는 순간임 |


상세 설명:

* 측정 구간의 시작점인 "IDS에 입력 전달 시점"은 음성 인식·텍스트 입력 처리 모듈의 처리가 완료된 이후를 의미하며, 해당 모듈의 응답 시간은 본 NFR의 측정 범위에 포함하지 않는다.

* 처리 결과 표시 전이라도 "처리 중" 중간 피드백은 입력 전달 후 가능한 한 빠르게 UI에 표시되어야 한다.


---


### NFR-IDS-02. Performance - Routing 처리 속도


정의: IDS의 Routing 모듈이 수신한 Intent의 처리 주체(IDS 자체 처리 / MAF)를 신속하게 판별해야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| Task 처리 주체 판별 시간 | 판별 시작 ~ 판별 완료 시점 | **1초 이내** | 온디바이스 경량 LLM(분류 수준) 추론 시간 기준 |


---


### NFR-IDS-03. Performance - IDS 자체 Task 처리 속도


정의: IDS의 Task 처리 모듈이 자체 처리로 분류된 태스크를 신속하게 완료해야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| IDS 자체 Task 처리 시간 | 처리 주체 판별 완료 ~ 결과 반환 시점 | **5초 이내** | IDS가 자체 처리하는 태스크는 단순 로컬 명령·정보 검색 수준이므로 5초 이내 처리 가능 |


---


### NFR-IDS-04. Resource Efficiency - 자원 사용량


정의: IDS 동작 시 온디바이스 자원(CPU, NPU, Memory)을 최소로 사용하여 포그라운드 앱 성능에 영향을 주지 않아야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| CPU 사용률 (평균) | **10% 이하** | Android 백그라운드 서비스 권장 기준. 포그라운드 앱에 영향을 주지 않는 수준 |
| CPU 사용률 (피크) | **25% 이하** | Intent 인지 추론 시 일시적 허용 범위 |
| NPU 사용률 (LLM 추론 활성 시) | **70% 이하** | LLM 추론 작업 중 NPU의 대부분을 사용하나, 포그라운드 앱을 위한 NPU 헤드룸을 확보해야 함 |
| Memory 사용량 | **200MB 이하** | 온디바이스 경량 LLM 모델 로딩 포함 기준. Galaxy S26 RAM(12~16GB) 대비 합리적 할당 |


---


### NFR-IDS-05. Reliability - 자체 Task 처리 성공률


정의: IDS가 자체적으로 처리하기로 판별한 Task는 기술적 오류 없이 성공적으로 완료되어야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| IDS 자체 Task 처리 성공률 | **90% 이상** | Production 목표(95%+) 수립을 위한 초기 기준선으로 설정 |


상세 설명:

* 성공 기준: 기술적 오류(크래시, 타임아웃, 예외 미처리) 없이 결과가 반환된 경우.

* 의도 해석 정확도(Accuracy)는 별도 평가 항목으로 분리하며 본 NFR의 범위에 포함하지 않는다.

## 2. MAF 비기능 요구사항


> **Reliability 속성 정의 (ISO/IEC 25010:2023 기준)**

>

> 본 절의 Reliability NFR은 아래 두 세부 속성을 기반으로 한다. 두 속성은 오류 발생 시점과 세션 상태에 따라 명확히 구분된다.

>

> **Fault Tolerance (결함 감내)**: Communication 세션이 진행 중인 상태에서 네트워크 패킷 손실, 메시지 처리 실패 등 일시적 오류가 발생하더라도 세션이 중단되지 않고 사용자에게 투명하게(Transparently) 처리되어 Communication이 완료되는 능력.

> - 예시: 협상 라운드 중 메시지 패킷 손실 발생 → 자동 재전송 → 사용자 인지 없이 협상 계속 진행

>

> **Recoverability (복구 가능성)**: 네트워크 완전 단절, 앱 강제 종료 등으로 Communication 세션이 실제로 **중단된 이후**, 저장된 세션 상태를 기반으로 중단 시점부터 재개하는 능력.

> - 예시: 협상 진행 중 네트워크 완전 단절로 세션 종료 → 연결 복구 후 중단 시점의 협상 상태를 복원하여 재개

>

> **핵심 차이**: Fault Tolerance는 세션이 "중단 없이" 진행되는 것이고, Recoverability는 세션이 "중단된 후 복원"되는 것이다. 전자는 오류를 사용자에게 투명하게 처리하며, 후자는 중단이 사용자에게 인지되었을 수 있으나 복구 후 재개된다.


---


### NFR-MAF-01. Reliability - Communication 완결성 (Fault Tolerance)


정의: Communication 세션 진행 중 발생하는 일시적 오류가 사용자에게 투명하게 처리되어 Communication이 중단 없이 완료되어야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 일시적 오류 발생 시 자동 복구 완료율 | **99% 이상** | 세션 진행 중 발생한 일시적 오류 케이스 중 사용자 개입 없이 자동 처리되어 정상 완료된 비율 |


상세 설명:

* 본 NFR에서 "일시적 오류"의 범위는 다음과 같이 정의한다: 네트워크 계층(패킷 손실·일시적 연결 불안정), 메시지 계층(메시지 처리 실패·일시적 응답 지연). 앱 프로세스 재시작이나 세션 완전 단절은 NFR-MAF-03(Recoverability) 범위이며 본 NFR의 대상이 아니다.

* 완결성 판단 범위: Action(캘린더 등록, 예약 API 호출 등) 단계까지 포함하여 완결로 간주한다.

* 협상 결렬(양측 조건 불일치로 합의 불가 판정)은 기술적 실패가 아니며 완결성 위반으로 간주하지 않는다.

* 자동 복구가 불가능한 경우(99% 기준 초과) FR-MAF-11에 따라 처리한다.


---


### NFR-MAF-02. Reliability - Task 성공률


정의: 시작된 Communication 태스크가 기술적 오류 없이 정상 완료(합의 타결 또는 합의 불가 판정 포함)되어야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 전체 케이스 기술적 Task 성공률 | **95% 이상** | 기술적 실패(크래시, 타임아웃 미처리, 예외 미처리)만 실패로 계산 |


상세 설명:

* **NFR-MAF-01과의 차이**: NFR-MAF-01은 세션 진행 중 일시적 오류를 투명하게 처리하는 능력(과정 품질)을 측정하며 분모는 "오류가 발생한 케이스"이다. 본 NFR은 시도된 전체 Communication 태스크 중 성공적으로 완료된 비율(결과 품질)을 측정하며 분모는 "전체 시도 케이스"이다. NFR-MAF-01의 자동 복구가 실패한 케이스까지 포함한 종합적 완료율이다.

* 성공 기준: 케이스별 정상 흐름(합의 타결 또는 합의 불가 판정)이 오류 없이 완료된 경우.

* 케이스별 성공 기준의 상세 정의는 QAS에서 별도로 명시한다.


---


### NFR-MAF-03. Reliability - 세션 복구 가능성 (Recoverability)


정의: 앱 재시작 또는 네트워크 완전 단절로 Communication 세션이 중단된 이후, 저장된 세션 상태를 기반으로 복구를 시도했을 때 성공적으로 재개되어야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 세션 복구 성공률 | **95% 이상** | 세션 단절 후 복구 시도 케이스 중 중단 시점부터 정상 재개된 비율 |


상세 설명:

* 현재 우선 적용 범위: Negotiation 세션. 이후 Collaboration, Knowledge Sharing, Remote Monitoring 순으로 확장 적용한다.

* 케이스별 세션 복구 방식이 상이할 경우, 케이스별 세부 요건을 별도로 정의한다.


---


### NFR-MAF-04. Performance - Communication Latency


정의: MAF의 Communication 동작은 케이스별로 정의된 측정 구간 내에 완료되어야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 총 소요 시간 | Task가 MAF에 전달된 시점 ~ Action 완료 시점 | **180초(3분) 이내** | 사용자가 백그라운드 처리를 인지하고 기다릴 수 있는 최대 허용 시간 |
| Discovery | 에이전트 탐색 시작 ~ 암호화 채널 수립 완료 | **5초 이내** | TLS 핸드쉐이크 + 인증 교환 시간 기준 |
| Negotiation (1:1, 2인) | 협상 시작 ~ 합의 또는 결렬 완료 | **60초 이내** | 온디바이스 LLM 추론(약 10~15초/라운드) × 최대 4라운드 기준 |
| Negotiation (N-party, N≥3) | 협상 시작 ~ 합의 또는 결렬 완료 | **60 + (N-2) × 30초 이내** | 참여자 1명 추가당 30초 이내의 선형 증가를 허용. 검증 기준점: 3명→90초, 5명→150초, 7명→210초 |
| Collaboration | 협업 요청 시작 ~ 협업 세션 수립 완료 | **10초 이내** | 외부 플랫폼 API 권한 확인 + 세션 초기화 시간 기준. 협업 내용 완료 시간은 케이스별로 상이하여 측정 범위에서 제외 |
| Knowledge Sharing | 지식 공유 요청 시점 ~ 상대 에이전트 수신 확인 시점 | **10초 이내** | 데이터 직렬화 + 암호화 + 전송 + 수신 확인 응답 시간 기준 |
| Remote Monitoring | 이상 징후 감지 시점 ~ Caregiver 에이전트 알림 전달 완료 시점 | **5초 이내** | 이상 징후 알림은 지연 허용도가 가장 낮음. 긴급 상황 대응 기준으로 가장 엄격하게 설정 |


상세 설명:

* N-party 증가분 설계 근거: 단일 디바이스 관점에서 참여자 추가 시 LLM 추론 비용은 O(1)로 고정되며(자신의 제안 생성), 증가하는 비용은 N-1개 피어의 메시지 처리 및 상태 관리(O(N))이다. 이에 따라 전체 지연은 선형(O(N)) 이내로 증가해야 한다.

### NFR-MAF-05. Performance - 사용자 개입 반영 속도


정의: Communication 처리 중 사용자의 직접 개입 시, 변경 요청이 MAF에 반영되는 시간을 최소화해야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 사용자 개입 반영 시간 | 사용자 UI 조작 시점 ~ MAF에서 Intent 변경 인지 및 적용 완료 시점 | **2초 이내** | IDS 경유 Intent 전달 시간(NFR-IDS-01 기준 2초) 포함. 최종 처리 완료(상대 에이전트 통보 등)까지는 포함하지 않음 |


---


### NFR-MAF-06. Resource Efficiency - 자원 사용량


정의: MAF 동작 시 온디바이스 자원(CPU, NPU, Memory)을 최소로 사용하여 사용자 경험에 영향을 주지 않아야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| CPU 사용률 (평균) | **15% 이하** | IDS보다 복잡한 처리(협상 추론 등)를 고려하여 IDS 기준(10%)보다 높게 설정 |
| CPU 사용률 (피크) | **40% 이하** | 협상 LLM 추론 시 일시적 사용 허용 범위 |
| NPU 사용률 (LLM 추론 활성 시) | **70% 이하** | 협상 LLM 추론 중 NPU 대부분을 사용하나, 동시 동작하는 IDS 및 포그라운드 앱을 위한 NPU 헤드룸 확보 |
| Memory 사용량 | **300MB 이하** | Multi-Agent 상태 관리 + 온디바이스 LLM 로딩 포함 기준 |


상세 설명:

* LLM 배치 원칙: Meta Agent·Sub-Agent는 온디바이스 LLM 사용. Orchestrator는 서버 LLM 사용.

* 서버 LLM(Orchestrator) 호출은 Task Planning 단계로 한정하여 외부 호출을 최소화한다.

* 본 기준은 1:1(2인) Communication 기준이며, N-party Negotiation의 자원 확장 기준은 NFR-MAF-09를 따른다.


---


### NFR-MAF-07. Security - 개인정보 보안


정의: 에이전트 간 Communication 수행 시 민감 정보를 암호화하고 개인정보 공유를 최소화해야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 전송 계층 암호화 | **TLS 1.3 이상** | 현 시점 업계 표준 최소 요건 |
| Private Knowledge 외부 전송 건수 | **0건** | FR-MAF-05(지식 공유 보안 등급 구분) 기준 필터링 후 전송. 개인정보 유출 원천 차단 |


상세 설명:

* 에이전트 간 전송 데이터는 FR-MAF-05에서 Private Knowledge로 분류된 항목을 포함하지 않아야 한다.

* 암호화 채널 수립은 FR-MAF-01(Discovery & Authentication) 완료 시점에 보장되어야 한다.


---


### NFR-MAF-08. Traceability - Communication 이력 추적


정의: Communication의 진행 과정 전체가 완전하게 기록·보존되어, 사용자가 원할 때 즉시 조회할 수 있어야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 이벤트 로그 기록률 | **100%** | 추적 공백이 발생하면 분쟁·오류 원인 분석 불가 |
| 사용자 이력 조회 응답 시간 | **2초 이내** | FR-MAF-07(진행 상태 가시성)의 실시간 조회 UX 기준과 일치 |


상세 설명:

* FR-MAF-07이 기능(무엇을 보여주는가)을 정의하는 데 반해, 본 NFR은 품질(얼마나 완전하고 빠르게 제공되는가)을 정의한다.

* 기록 대상 이벤트: 협상 제안·역제안·수락·거절, 협업 세션 수립·종료, 지식 공유 요청·전달·수신 확인, 이상 징후 감지·알림 전달 등 모든 Communication 상태 전이.

* 로그 보존 기간은 운영 정책에 따라 결정되는 구성 값으로, 아키텍처 수준의 측정 기준에서 제외한다.

### NFR-MAF-09. Scalability - 다자간 협상 온디바이스 자원 확장성


정의: 다자간 협상 참여자 수가 증가하더라도 온디바이스 자원 사용량은 선형(O(N)) 이내로 증가해야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 참여자 1명 추가당 CPU 피크 증가분 | **2% 이내** | 추가 참여자에 따른 증가 비용은 주로 메시지 처리(O(N))이며, LLM 추론은 O(1)로 고정되므로 CPU 증가분은 제한적 |
| 참여자 1명 추가당 Memory 증가분 | **20MB 이내** | 참여자 1인당 협상 상태(제안 이력, 선호 정보) 저장에 필요한 메모리 기준 |


검증 기준점 (1:1 기준값: CPU 피크 40%, Memory 300MB):


| 참여자 수 | CPU 피크 상한 | Memory 상한 |
|---|---|---|
| 3명 | 42% 이하 | 320MB 이하 |
| 5명 | 46% 이하 | 360MB 이하 |
| 7명 | 50% 이하 | 400MB 이하 |


상세 설명:

* 증가 모델 근거: 단일 디바이스 관점에서 LLM 추론 비용은 자신의 제안 생성에 한정되어 O(1)로 고정되며, 참여자 수에 따라 증가하는 비용은 N-1개 피어의 메시지 파싱·상태 관리(O(N))에서 발생한다. 전체 자원 증가는 선형 이내여야 한다.

* 위 검증 기준점에서 실측값이 상한을 초과하는 경우, 선형 증가 모델을 유지하면서 증가분 상한을 재조정해야 한다.


---


### NFR-MAF-10. Reliability - 다자간 협상 상태 일관성 (State Consistency)


정의: 다자간 협상이 **진행되는 동안** 모든 참여 에이전트는 동일한 협상 상태(현재 라운드, 제안 내용, 수락·거절 현황 등)를 공유해야 한다.


측정 기준:


| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 참여 에이전트 간 상태 불일치 발생률 | 협상 진행 전 구간 | **0%** | 상태 불일치는 잘못된 제안·판단으로 이어져 협상 무결성을 훼손함 |
| 상태 동기화 최대 지연 | 상태 변경 발생 ~ 전체 참여자 반영 완료 | **2초 이내** | 협상 라운드 단위 동기화 기준. NFR-MAF-05 사용자 개입 반영(2초)과 동일 수준 |


상세 설명:

* **NFR-MAF-11과의 차이**: 본 NFR은 협상 **진행 중**의 정보 일관성을 다룬다. 모든 참여자가 동일한 "현재 상태"를 바라봐야 한다는 요건이다. 예시: 참여자 A가 "B가 오후 6시를 제안했다"고 인식하고 있다면, B도 자신이 오후 6시를 제안한 것으로 인식하고 있어야 하며, B가 해당 제안을 철회했다면 A도 이를 즉시 반영해야 한다. 이는 실시간 동기화(Liveness) 요건이다.

* 네트워크 파티션 등으로 상태 불일치가 발생한 경우, NFR-MAF-03 세션 복구 메커니즘을 통해 복원하고 불일치 상태를 해소해야 한다.

* 상태 불일치가 해소되기 전까지 해당 에이전트의 협상 진행은 일시 중단해야 한다.


---


### NFR-MAF-11. Reliability - 다자간 합의 결과 일관성 (Consensus Consistency)


정의: 다자간 협상이 **타결되었을 때**, 모든 참여 에이전트가 동일한 최종 합의 결과를 수신하고 확인해야 한다.


측정 기준:


| 항목 | 임계값 | 근거 |
|---|---|---|
| 합의 결과 불일치 발생률 | **0%** | 참여자 간 합의 결과가 다를 경우 각자 다른 후속 행동(캘린더 등록, 예약 API 호출 등)을 수행하게 되어 실세계 충돌 발생 |
| 합의 결과 전달 완료 기준 | **모든 참여 에이전트의 수신 확인(ACK) 완료** | 단 1명의 미확인도 합의 미완료로 간주 |


상세 설명:

* **NFR-MAF-10과의 차이**: 본 NFR은 협상 **완료 시점**의 결과 일관성을 다룬다. 진행 중 상태가 아니라 최종 합의 내용이 모든 참여자에게 동일하게 전달되어야 한다는 요건이다. 예시: 협상 완료 후 참여자 A의 에이전트가 "오후 6시, 레스토랑 X"로 캘린더 등록을 수행한다면, B와 C의 에이전트도 동일하게 "오후 6시, 레스토랑 X"로 등록해야 한다. 이는 원자적 커밋(Atomicity) 요건이다.

* 합의 결과 전달 후 ACK를 수신하지 못한 경우 FR-MAF-11(이상 상황 처리)에 따라 처리한다.

* 모든 참여자의 ACK 완료 이후에만 FR-MAF-09(합의안 후속 실행)를 트리거한다.

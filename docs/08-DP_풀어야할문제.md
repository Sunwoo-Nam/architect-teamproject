# 핵심 Decision Point (Critical Architectural Decisions)

> 본 문서는 FR/NFR/Constraints에서 도출한 **아키텍처적으로 결정적인** 설계 문제만 선별한 것이다.
> 선별 기준: (1) 되돌리기 비용이 큰가, (2) 여러 NFR이 충돌하는가, (3) 분산 시스템 이론·LLM 비결정성 등 근본적 한계와 부딪히는가, (4) 다른 결정을 연쇄 제약하는 keystone인가.
> 단순 구현 선택(Discovery 방식, 재시도 위치, 로깅 방식 등)은 본 문서의 하위 결정으로 보고 제외한다.

---

## 문제 지형도 (왜 이 시스템이 어려운가)

이 시스템의 본질적 난이도는 세 가지가 동시에 겹친다는 데서 온다.

첫째, **분산 합의 문제**다. 개인 단말 간 P2P에서 여러 Agent가 하나의 결론에 도달해야 하는데, 네트워크는 불안정하고 NFR은 불일치 0%를 요구한다. 이것은 분산 시스템에서 가장 오래되고 어려운 문제다.

둘째, **비결정적 추론 엔진(LLM)을 신뢰 경계에서 사용**한다. LLM은 출력이 비결정적인데, 시스템은 "민감정보 유출 0건", "상태 일관성 0% 불일치" 같은 결정론적 보장을 요구한다. 비결정적 컴포넌트로 결정론적 보장을 만드는 문제다.

셋째, **제약된 단말 위에서 이 모든 것을 한다**. 클라우드라면 확장으로 풀 문제를 단일 NPU·제한 메모리에서 풀어야 한다.

아래 DP들은 이 세 겹의 난이도가 구체적 결정으로 응축되는 지점들이다.

---

## A. 분산 협상의 정확성 (Correctness of Distributed Negotiation)

이 클러스터가 시스템에서 가장 어렵다. 세 결정이 강하게 얽혀 있으며, DP-A2(상태 관리)가 나머지를 떠받치는 keystone이다.

### DP-A1. 분산 협상의 정합성 모델과 CAP 트레이드오프

**풀어야 할 문제**: NFR-MAF-10(상태 불일치 0%)과 NFR-MAF-11(합의 결과 불일치 0%)을 요구하면서, 동시에 네트워크 장애 시 세션을 중단·재개해야 한다. 어떤 일관성 모델과 합의 프로토콜로 이 "0%"를 달성할 것인가?

**왜 어려운가**: CAP 정리상 네트워크 파티션(P)이 발생하면 일관성(C)과 가용성(A)을 동시에 가질 수 없다. FLP 불가능성 정리는 비동기 네트워크에서 한 노드라도 실패할 수 있으면 결정론적 합의의 종료를 보장할 수 없음을 증명한다. 즉 "불일치 0%"는 순수 분산 합의로는 달성 불가능에 가깝다. 두 단말이 서로 다른 합의 결과로 각자 예약을 확정하면 실세계 충돌(한쪽 7시, 다른쪽 8시)이 발생한다. 여기에 LLM 추론의 비결정성까지 더해진다.

**핵심 결정 축**:
- 일관성 모델: Strong Consistency(원자적·느림) / Eventual Consistency(빠름·일시 불일치 허용) / Causal Consistency
- 파티션 시 선택: **CP**(일관성 우선, 합의 보류) / **AP**(가용성 우선, 사후 조정)
- 합의 프로토콜: 2PC(블로킹·단순) / 3PC(논블로킹·복잡) / Paxos·Raft(견고·무거움) / 협상 특화 commit
- **"0%"의 운영적 재정의**: "불일치가 절대 발생하지 않는다"는 불가능. "불일치 상태에서는 누구도 commit하지 않는다(All-or-Nothing, 원자적 커밋)"로 재정의하는 것이 현실적

**통찰**: NFR-MAF-10 상세 설명의 "불일치 시 협상 일시 중단"은 이미 **CP 선택**을 암시한다. 이것이 의도된 트레이드오프임을 문서에 명시하고, 0%를 "원자적 커밋으로 부분 합의를 방지한다"로 재정의해야 한다.

**연쇄 관계**: DP-A2(상태 표현)와 DP-A3(메시지 표현)의 선택을 제약한다.

---

### DP-A2. 세션 상태 표현 및 영속화 전략 ★ Keystone

**풀어야 할 문제**: (a) 네트워크 장애 후 정확한 지점부터 재개, (b) 30초 내 세션 복구(NFR-MAF-03), (c) 100% 이벤트 로그(NFR-MAF-08), (d) 다자간 상태 일관성(NFR-MAF-10) — 이 넷은 모두 "협상 상태를 어떻게 표현하고 저장하는가"라는 단일 결정에 의존한다.

**왜 어려운가**: 이것은 keystone 결정이다. 잘못 정하면 복구·일관성·추적·재개가 연쇄적으로 무너진다. 그리고 온디바이스 저장 용량과 쓰기 지연이라는 물리적 제약 안에서 정해야 한다.

**핵심 결정 축**:
- **Event Sourcing**: 모든 협상 이벤트를 append-only 로그로 저장하고 재생으로 상태 복원. NFR-MAF-08의 100% 로그와 자연 통합되고 완전한 재현이 가능하다. 단, 저장 증가와 재생 시간이 비용.
- **Snapshot**: 현재 상태만 주기 저장. 복구가 빠르다. 단, 스냅샷 사이 이벤트가 유실되어 100% 로그를 별도 구현해야 한다.
- **Hybrid(Snapshot + Event Log)**: 스냅샷으로 빠른 복구 + 이벤트 로그로 완전성. 복잡도가 비용.

**통찰**: NFR-MAF-08의 "이벤트 로그 100%" 요구가 사실상 Event Sourcing을 강하게 시사한다. 별도 로깅 체계를 두면 협상 상태와 로그가 이중 기록되어 둘 사이의 불일치 위험이 생긴다. 즉 이 NFR이 결정 공간을 이미 상당히 좁히고 있다. 이 점을 인지하고 결정해야 한다.

**연쇄 관계**: DP-A1(일관성), 세션 복구, 추적성 전체가 이 결정 위에 선다. **가장 먼저 결정해야 한다.**

### DP-A3. 협상 메시지의 표현 방식 — Structured vs LLM-native ★ Fork in the Road

**풀어야 할 문제**: Agent는 온디바이스 LLM으로 협상한다(CON-06). 협상 메시지를 (a) 구조화된 스키마(필드 기반)로 정의할지, (b) LLM이 생성하는 자연어로 할지. 이것은 단순 포맷 선택이 아니라 나머지 거의 모든 NFR의 난이도를 결정하는 분기점이다.

**왜 어려운가**:
- 자연어 협상은 표현력·유연성이 높지만 상태 일관성(NFR-MAF-10)을 검증하기 어렵다. "현재 협상 상태"를 자연어 메시지에서 결정론적으로 추출할 수 있는가?
- 보안(NFR-MAF-07): 자연어 메시지에 Private Knowledge가 새지 않았음을 어떻게 보장하는가? LLM 출력은 비결정적이다.
- 추적(NFR-MAF-08): 자연어 협상 로그를 어떻게 구조적으로 기록·분석·재생하는가?
- 구조화는 위 셋을 모두 쉽게 하지만 협상의 뉘앙스와 창의적 절충안을 제한한다.

**핵심 결정 축**:
- Pure Structured: JSON 스키마로 의도·조건·제약을 필드화. 검증·필터·로깅 용이, 표현력 제한.
- Pure LLM-native: 자연어 제안·해석. 유연하나 검증·보안·추적이 난해.
- Hybrid: 구조화 필드(상태 전이·핵심 조건) + 자연어 설명(근거·맥락). 상태/보안은 구조화 부분으로 통제하고 유연성은 자연어 부분으로 확보.

**통찰**: NFR-MAF-10의 0% 일관성 + NFR-MAF-07의 0건 유출이 Pure LLM-native를 사실상 배제한다. 적어도 **상태 전이와 보안 경계는 구조화되어야 기계적 검증이 가능**하다. Hybrid가 유력하지만, "어디까지 구조화하고 어디부터 자연어인가"의 경계 설정이 실제 난제다.

**연쇄 관계**: DP-B2(LLM 유출 방지)가 이 결정에 직접 의존한다.

## B. 신뢰와 프라이버시 (Trust & Privacy)

A2A 프로토콜 강제가 제약에서 해제되면서, 신뢰·통신 보안 설계가 **열린 결정**이 되었다. 이 클러스터는 그래서 더 중요해졌다.

### DP-B1. P2P 환경의 신뢰·신원 모델

**풀어야 할 문제**: FR-MAF-01은 상호 인증을 요구한다. 시스템은 개인 단말 간 P2P이고 중앙 신원 기관을 하드 요구사항으로 두지 않았다. 두 사용자의 Agent가 — 서로 모르는 사이일 수도 있는 — 상대가 사칭이 아님을 어떻게 증명하고 신뢰를 수립하는가?

**왜 어려운가**: 중앙 IdP(Identity Provider) 의존을 가정하지 않으면 신뢰 수립은 분산 신원 문제가 된다. 협상 상대가 사칭이면 잘못된 합의와 정보 유출로 직결된다. MITM, replay, Sybil 공격을 모두 방어해야 한다. 그리고 오프라인 동작(CON-04 계열 요구)과도 충돌한다 — 중앙 인증에 의존하면 오프라인에서 신뢰 수립이 불가능하다.

**핵심 결정 축**:
- 중앙 PKI / 계정 기반 federation: 단순하나 중앙 서버 종속, 오프라인 인증 불가
- 분산 신원(DID) / Web-of-Trust: 탈중앙이나 복잡하고 부트스트랩이 어려움
- TOFU(Trust On First Use) + 연락처 기반: 최초 사용자 확인 후 공개키 고정. 아는 사람과 협상하는 일상 시나리오에 적합. 최초 연결 시점이 취약점.

**통찰**: CON-02(동일 MAF 전제)는 이 문제를 일부 단순화한다 — 양측이 동일한 신뢰 부트스트랩 메커니즘을 공유한다. 그러나 "동일 MAF"가 "동일 신뢰"를 의미하지는 않는다. MAF가 같아도 상대가 사칭일 수 있다.

---

### DP-B2. LLM 출력을 통한 Private Knowledge 유출 방지

**풀어야 할 문제**: NFR-MAF-07은 Private Knowledge 외부 전송 0건을 요구한다. 그러나 Meta Agent는 LLM으로 협상 메시지를 생성하며, 그 LLM 컨텍스트에는 협상에 필요한 사용자 데이터가 들어간다. LLM 출력이 비결정적인데, 생성된 메시지가 민감 정보를 포함하지 않음을 어떻게 보장하는가?

**왜 어려운가**: "유출이 없음"은 부정의 증명이며, 비결정적 LLM에서는 본질적으로 입증이 어렵다. 단순 정책 매핑(FR-MAF-05)으로는 LLM이 민감 정보를 의역·재구성해 출력하는 경우를 막지 못한다. 예를 들어 "사용자의 집 주소"를 직접 출력하지 않아도 "강남역에서 도보 5분, 3번 출구 앞 아파트"처럼 재구성하면 정책 필터를 우회한다.

**핵심 결정 축**:
- 컨텍스트 격리: 외부 통신용 LLM에는 애초에 Private Knowledge를 주입하지 않는다(내부용/외부용 컨텍스트 분리). 가장 강력하나 협상 품질 저하 가능.
- 출력 필터링: LLM 출력을 전송 전 별도 검사기(규칙/분류기)로 스캔. 사후 방어, 누락 위험.
- 이중 모델: 내부 추론 모델과 외부 발화 모델을 분리, 외부 모델은 정제된 입력만 수신.
- 구조화 강제(DP-A3 연계): 외부로 나가는 것은 구조화 필드만 허용, 자유 텍스트 금지.

**통찰**: DP-A3와 강하게 연계된다. 외부 메시지를 구조화 필드로 제한하면 유출 표면(attack surface)이 극적으로 줄어든다. "민감정보 0건 유출"이라는 결정론적 보장은 **비결정적 LLM 출력을 외부 경계에서 차단하고 구조화된 것만 내보내는** 구조로만 현실적으로 달성 가능하다.

## C. 자율성과 통제 (Autonomy & Control)

시스템의 핵심 가치(자율성)와 안전장치(통제권)가 정면충돌하는 영역이다.

### DP-C1. Agent 자율성의 경계 결정 메커니즘

**풀어야 할 문제**: 시스템의 핵심 가치는 "사용자가 신경 쓰지 않아도 Agent가 알아서 합의"하는 자율성이다. 동시에 FR-MAF-10은 위임 범위 초과 시 사용자 승인을, FR-MAF-08은 실시간 개입을 요구한다. 본질적 질문: **Agent는 어떤 행위를 실행하기 전에 그것이 위임 범위를 벗어나는지를 어떻게 판단하는가?**

**왜 어려운가**: 이것은 정책 저장 위치 문제가 아니라 의사결정 제어 흐름의 근본 설계다. Agent가 행위의 결과를 사전에 예측해야 "이건 승인이 필요하다"를 판단할 수 있다. 너무 자주 물으면 자율성의 가치가 사라지고, 너무 안 물으면 통제권을 잃는다. 더 어려운 점은 LLM 기반 Agent가 자신의 행위 경계를 스스로 정확히 인식하기 어렵다는 것이다(과신 경향).

**핵심 결정 축**:
- 사전 정책 평가(Pre-flight check): 모든 행위 전 위임 범위를 결정론적 규칙으로 검사. 안전하나 경직.
- LLM 자기판단: Agent가 추론으로 "승인 필요" 여부를 판단. 유연하나 신뢰성이 낮음.
- 단계적 권한(Capability tiers): 행위를 위험도 등급으로 분류하고 등급별 자동/승인을 사전 정의.
- **가역성 기반**: 되돌릴 수 있는 행위는 자율, 비가역(결제·예약 확정)은 승인.

**통찰**: "가역성"이 강력한 설계 원칙이 될 수 있다. 협상 과정 자체(가역적, 언제든 취소 가능)는 자율로 두고, 최종 commit(비가역적, FR-MAF-09의 예약·결제)은 승인 게이트를 둔다. 이렇게 하면 자율성과 통제권의 충돌이 "가역성 경계"라는 명확한 선으로 정리된다.

**연쇄 관계**: DP-C2(묵시적 의도)와 연결 — 신뢰도 낮은 묵시적 의도일수록 승인 게이트를 앞당겨야 한다.

---

### DP-C2. IDS 묵시적 Intent 감지의 정밀도·프라이버시 균형

**풀어야 할 문제**: FR-IDS-02는 화면·디바이스 상태를 상시 분석해 묵시적 의도를 추론한다. 이는 (a) 거짓 양성(쓸데없이 협상 시작) vs 거짓 음성(의도 놓침)의 균형, (b) "시스템이 항상 화면을 본다"는 프라이버시 부담, (c) 상시 추론의 배터리 비용을 동시에 안고 있다.

**왜 어려운가**: 묵시적 감지는 시스템 유용성의 핵심이자 사용자 신뢰의 급소다. 너무 적극적이면 사용자를 귀찮게 하고 감시 우려를 키운다. 너무 소극적이면 "알아서 해주는" 가치가 사라져 그냥 또 하나의 명령형 비서가 된다. 온디바이스 처리는 프라이버시를 돕지만 자원 비용을 키운다(NFR-IDS-03과 충돌).

**핵심 결정 축**:
- 감지 적극성: 보수적(명확한 신호만) vs 적극적(약한 신호도 추론).
- 확인 단계(Confirmation gate): 묵시적 의도는 항상 사용자 확인 후 MAF 진입 vs 일부 자동 진행.
- 거짓 양성/음성 비용 가중: 어느 쪽 오류를 더 회피할 것인가(보수적 설계 = 거짓 음성 감수).
- 개인화 학습: 사용자 피드백으로 감지 임계값을 점진 조정.

**통찰**: DP-C1과 묶어서 보면, 묵시적 의도는 신뢰도가 본질적으로 낮으므로 자율 실행 대신 "감지 → 사용자 확인 → 실행"의 확인 게이트를 두는 것이 안전 기본값이다. 명시적 의도(FR-IDS-01)와 묵시적 의도를 자율성 등급에서 다르게 다뤄야 한다.

## D. 자원과 분산 토폴로지 (Resource & Distribution)

온디바이스라는 물리적 제약과 서버-디바이스 분할이라는 구조적 제약이 만드는 결정들이다.

### DP-D1. 온디바이스 LLM 자원 중재 및 스케줄링

**풀어야 할 문제**: 단일 단말에서 IDS(30초 주기 상시 모니터링), Meta Agent(다라운드 협상 추론), Sub-Agent(실행)가 모두 온디바이스 LLM을 필요로 한다. CPU/Memory 예산은 빠듯하다(NFR-IDS-03 평균 10%, NFR-MAF-06 평균 15%). 협상이 진행되는 동안 IDS 모니터링이 굶지(starvation) 않게 어떻게 자원을 중재하는가?

**왜 어려운가**: 온디바이스 멀티에이전트 고유의 문제다. 클라우드라면 수평 확장으로 풀지만 단말에서는 단일 NPU와 제한 메모리를 공유한다. 무거운 협상 추론이 LLM을 점유하면 IDS의 실시간 Intent 인지(NFR-IDS-01의 2초)가 깨진다. N명 협상(DP-A 계열, 최대 5명)이면 부하가 가중된다. 게다가 단말은 사용자의 다른 앱도 돌리고 있다.

**핵심 결정 축**:
- 단일 공유 모델 인스턴스 + 스케줄러: 메모리 절약, 그러나 경합과 대기 발생.
- 다중 모델 인스턴스: 동시성 향상, 메모리 압박.
- 우선순위 스케줄링: IDS 모니터링을 최우선, 협상은 best-effort.
- 모델 변종 활용(과제 개요의 Compact/Balanced/Supreme): 작업별로 경량/중량 모델을 분배.

**통찰**: IDS의 실시간성(NFR-IDS-01)과 MAF의 처리량(NFR-MAF-04)이 단일 자원을 두고 경쟁한다. 우선순위와 격리 없이는 둘 중 하나의 NFR이 반드시 깨진다. 이것은 측정 기준만으로는 드러나지 않는, 동시 동작 시에만 터지는 문제다.

---

### DP-D2. 서버-디바이스 책임 분할과 상태 권한(Authority)

**풀어야 할 문제**: LLM 배치가 고정이다 — Orchestrator는 서버 LLM, Meta Agent는 온디바이스(NFR-MAF-06). 하나의 Task가 진행되는 동안 계획(서버)과 실행·협상(디바이스)이 네트워크를 가로질러 분할된다. 이 분할을 어떻게 조율하고, 상태와 트랜잭션의 권한(Single Source of Truth)을 어디에 둘 것인가?

**왜 어려운가**: 단일 논리적 작업이 신뢰 경계와 네트워크 경계를 동시에 가로지른다. 서버가 계획 중인데 디바이스가 오프라인되면 누가 진행하는가? 서버가 SSOT인가 디바이스인가? 프라이버시(서버에 무엇을 보내는가, NFR-MAF-07)와 가용성(오프라인 시 누가 상태를 쥐는가, CON-04 계열)이 정면충돌한다.

**핵심 결정 축**:
- Device-authoritative: 디바이스가 상태의 주인, 서버는 stateless planner. 오프라인 복원력과 프라이버시가 높다. 서버 활용이 제한된다.
- Server-authoritative: 서버가 오케스트레이션 상태를 보유. 조율이 쉽지만 오프라인에 취약하고 프라이버시 위험.
- 계획-실행 분리 계약: 서버는 "계획"만 1회 산출하고, 이후 실행·상태는 전적으로 디바이스가 보유. 경계가 명확.

**통찰**: 프라이버시(NFR-MAF-07)와 오프라인 재개(CON-04 계열, DP-A2)가 **Device-authoritative**를 강하게 지지한다. 서버는 "한 번 계획해주고 빠지는" stateless 역할로 한정하는 것이 일관성·프라이버시·오프라인 요구를 모두 만족시키는 방향으로 보인다. 다만 이 경우 서버 LLM의 활용 가치가 제한되므로, 서버를 정말 둘 필요가 있는지 자체를 재검토할 여지가 있다.

## E. 프레임워크 구조 (Framework Structure)

### DP-E1. 4대 케이스의 공통성/가변성 구조 (Commonality/Variability)

**풀어야 할 문제**: MAF는 Negotiation / Collaboration / Knowledge Sharing / Remote Monitoring 4대 케이스를 처리한다. 이들은 공통 요소(Discovery·인증·사용자 제어·후속 실행·복구·로깅)와 고유 요소(협상 사이클 vs 협업 워크플로 vs 데이터 교환 vs 모니터링)를 동시에 갖는다. 단일 통합 엔진으로 설계할지, 케이스별 전용 엔진 + 공통 플랫폼으로 설계할지.

**왜 어려운가**: 프레임워크의 골격을 정하는 결정이다. 과도한 통합은 각 케이스의 고유 요구를 억지로 끼워맞춰 추상화 누수(leaky abstraction)를 부른다. 과도한 분리는 공통 로직(인증·복구·로깅)을 중복시킨다. 잘못 정하면 향후 케이스 추가 시 전체 구조를 갈아엎어야 한다.

**핵심 결정 축**:
- 단일 통합 Communication 엔진: 4 케이스를 파라미터·전략으로 처리. DRY하나 추상화 누수 위험.
- 케이스별 전용 엔진 + 공통 코어(인증·세션·복구·로깅): 변경 격리가 좋으나 공통 코어 설계가 핵심 난제.
- 플러그인 아키텍처: 공통 파이프라인 + 케이스별 플러그인.

**통찰**: NFR을 보면 대부분이 공통(복구·일관성·추적·자원)이고 FR이 케이스별로 갈린다. 이 비대칭은 **공통 코어(NFR을 책임) + 케이스별 플러그인(FR을 책임)** 구조를 시사한다. 다만 Negotiation만 N-party 일관성(NFR-MAF-09~11)이라는 고유 NFR을 추가로 갖는다는 점에서, 완전한 대칭 플러그인은 아니다.

## 의존 구조와 결정 순서

이 DP들은 독립적이지 않다. 결정 순서가 중요하다.

```
[1순위 — Keystone]
  DP-A2 (상태 표현/영속화)  ← 가장 먼저. 복구·일관성·추적이 모두 의존
  DP-E1 (공통성/가변성)     ← 프레임워크 골격. 다른 DP들이 놓일 자리를 정함

[2순위 — Fork / 근본 선택]
  DP-A3 (메시지 표현)       ← DP-A2 위에서. DP-B2를 직접 제약
  DP-D2 (서버-디바이스 권한) ← DP-A2와 연동. 상태 주인을 정함

[3순위 — 위 결정에 종속]
  DP-A1 (정합성 모델)       ← DP-A2, DP-A3 위에서 결정
  DP-B2 (LLM 유출 방지)     ← DP-A3에 직접 의존
  DP-B1 (신뢰/신원)         ← DP-D2(오프라인 권한)와 정합 필요
  DP-C1 (자율성 경계)       ← 제어 흐름. DP-C2와 묶임
  DP-C2 (묵시적 의도)       ← DP-C1과 묶임
  DP-D1 (자원 중재)         ← 다른 결정 후 실측으로 조정
```

**핵심 메시지**: DP-A2(상태 관리)와 DP-A3(메시지 표현)를 먼저 못 박지 않으면, 일관성·복구·보안·추적 결정이 모두 공중에 뜬다. 이 둘이 본 아키텍처의 진짜 출발점이다.

## 비판적으로 남겨두는 질문

마지막으로, 결정 이전에 팀이 합의해야 할 근본 질문 세 가지를 남긴다.

첫째, **NFR의 "0%"는 정말 0%여야 하는가?** 분산 환경에서 진짜 0%는 원자적 커밋(All-or-Nothing)으로만 가능하고, 이는 가용성을 희생한다. "부분 합의를 방지한다"가 목표라면 그렇게 다시 써야 한다. 이 재정의 없이는 DP-A1이 풀리지 않는다.

둘째, **서버(Orchestrator)는 정말 필요한가?** DP-D2의 통찰대로 프라이버시·오프라인 요구가 Device-authoritative를 지지한다면, 서버 LLM의 역할은 "1회성 계획"으로 축소된다. 그 정도 가치를 위해 서버-디바이스 분할의 복잡도를 감수할지는 별도 판단이 필요하다.

셋째, **이 시스템은 본질적으로 "신뢰할 수 없는 컴포넌트(LLM)로 신뢰할 수 있는 보장을 만드는" 문제인가?** 그렇다면 핵심 설계 원칙은 "LLM은 제안하고, 결정론적 계층이 검증·집행한다"가 되어야 한다. DP-A3(구조화 경계), DP-B2(출력 차단), DP-C1(가역성 게이트)이 모두 이 한 원칙의 다른 표현일 수 있다.

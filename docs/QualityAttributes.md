### **보안성 (Security)**

**1. 보안성 - 기밀성 (Confidentiality) : 개인 정보 노출 및 외부 전송 최소화**
 * **정의:** 단말 내의 민감한 컨텍스트(DPA 데이터, 사용자 성향 등)가 인가되지 않은 외부(서버 Orchestrator, 외부 SA 등)로 유출되는 것을 차단하는 능력입니다.
 * **평가 지표:** 외부 네트워크 호출 시 페이로드(Payload) 내 원본 PII(개인식별정보) 노출 비율 (목표치: 0%).
 * **상세 설명:** 온디바이스 플랫폼의 존재 이유를 증명하는 핵심 제약입니다. 이를 위해 데이터를 날것(Raw)으로 전송하지 않고 의도(Intent)만 추상화하여 전달하는 데이터 마스킹 기법 및 PII 필터링 프록시 레이어(Privacy Filter) 아키텍처 도입이 필수적입니다.

**2. 보안성 - 무결성 (Integrity) : 권한 범위 통제 및 의도 위변조 방지**
 * **정의:** 외부 통신 시 위임받은 '권한 범위' 내에서만 데이터가 전송되고, 그 값이 전송 과정이나 대리인(Proxy Agent)에 의해 임의로 위변조되지 않았음을 보장하는 능력입니다.
 * **평가 지표:** 부여된 권한 범위를 초과한 데이터 접근 및 전송 시도에 대한 시스템 차단 성공률 (목표치: 100%).
 * **상세 설명:** 대리인 역할을 수행하는 에이전트의 신뢰를 기술적으로 담보하기 위해, 서명된 권한 토큰(Signed Delegation Token) 기반의 검증 및 Intent 추상화 계층 설계가 강제됩니다.

### **신뢰성 (Reliability) 및 기능 적합성 (Functional Suitability)**

**3. 신뢰성 - 가용성 (Availability) : 다중 Task 동시성 제어 및 교착 상태 방지**
 * **정의:** 단일 디바이스 내에서 다중 Task가 캘린더나 예약 모듈 등 단일 자원에 동시 접근할 때, 교착 상태(Deadlock)나 경합(Race Condition)을 방지하여 시스템의 가용한 상태를 유지하는 능력입니다.
 * **평가 지표:** 동시 다발적 에이전트 요청 하에서 리소스 접근 충돌로 인한 데드락(Deadlock) 발생 횟수.
 * **상세 설명:** 시스템 자원이 동시에 잠기는 현상을 막기 위해 로컬 리소스에 대한 글로벌 락(Global Lock) 관리 메커니즘과 Task 큐잉(Queuing) 시스템 설계가 요구됩니다.

**4. 기능 적합성 - 기능적 정확성 (Functional Correctness) : 분산 환경의 데이터 정합성 유지**
 * **정의:** N-party 그룹 협상 등 분산 환경에서 다수의 참여자 간 합의 상태가 엇갈리지 않고, 최종적으로 완벽하게 동일한 상태(트랜잭션 원자성)를 유지하여 정확한 결과를 제공하는 능력입니다.
 * **평가 지표:** 다자간 협상 완료 후 참여 노드 간 데이터(예약, 스케줄 등) 불일치 발생 횟수 (목표치: 0).
 * **상세 설명:** 네트워크 분산 환경 특유의 뇌절(Split-brain) 현상을 막기 위해, 2-Phase Commit이나 Saga 패턴 기반의 엄격한 분산 상태 동기화 엔진(State Machine)이 구축되어야 합니다.

**5. 신뢰성 - 결함 허용성 (Fault Tolerance) : 예외 상황의 장애 격리 및 시스템 생존**
 * **정의:** 노드 이탈, 네트워크 단절 등 예외 상황이 발생했을 때 코어 프레임워크(MAF) 전체가 다운되지 않고, 오염된 스레드나 프로세스만을 격리하는 능력입니다.
 * **평가 지표:** 개별 Sub-Agent의 Crash 발생 시, 해당 장애가 Orchestrator 및 타 Agent로 전파되는 장애 전파율 (목표치: 0%).
 * **상세 설명:** 장애가 연쇄적으로 파급되는 도미노 현상을 막기 위해 서킷 브레이커(Circuit Breaker) 패턴과 에이전트의 생명주기를 독립적으로 감시하고 재시작하는 Supervisor 트리 구조가 필수적입니다.

**6. 신뢰성 - 복구 용이성 (Recoverability) : 실패한 트랜잭션의 논리적 롤백**
 * **정의:** 협상 진행 중단이나 외부 API 호출 실패 시, 시스템을 데이터 오염 이전의 '유효한 기준 상태'로 완벽히 되돌려 놓는 능력입니다.
 * **평가 지표:** 예외 상황 발생 후 보상 트랜잭션을 통한 시스템 상태 복구 성공률 (목표치: 100%).
 * **상세 설명:** DB의 물리적 롤백이 아닌, 잠정 합의안을 무효화하고 기존 캘린더 점유를 해제하는 등 비즈니스 논리 차원의 보상 트랜잭션(Compensation Transaction) 아키텍처가 필요합니다.

### **성능 효율성 (Performance Efficiency)**

**7. 성능 효율성 - 자원 활용성 (Resource Utilization) : 온디바이스 메모리 및 배터리 점유 통제**
 * **정의:** 제한된 온디바이스 환경에서 수많은 동적 에이전트가 생성될 때, OOM(Out of Memory)과 극심한 배터리 고갈을 방지하기 위해 시스템 자원 점유를 통제하는 능력입니다.
 * **평가 지표:** 백그라운드 활성/대기 에이전트의 피크 메모리(Peak Memory) 점유량 및 시간당 배터리 소모율.
 * **상세 설명:** 가용 메모리 임계치 도달 시, 활성 상태가 아닌 에이전트의 컨텍스트를 디스크로 내리거나 상태만 저장하고 실행 코드를 소멸시키는 스와핑(Eviction/Caching) 계층 설계가 수반되어야 합니다.

**8. 성능 효율성 - 수용성 (Capacity) : 제한된 NPU 환경의 LLM 토큰 버짓 관리**
 * **정의:** 단말 NPU의 제한된 컴퓨팅 파워 안에서, 다중 턴(Turn) 협상이 지속되더라도 LLM 연산의 컨텍스트 윈도우 한계를 초과하지 않고 요청을 수용하는 능력입니다.
 * **평가 지표:** 단일 협상 세션당 최대 허용 토큰 소모량 초과 빈도 및 그로 인한 처리 거부율.
 * **상세 설명:** 무한 루프나 긴 협상으로 인한 자원 고갈을 막기 위해, 매 턴마다 소모되는 토큰 버짓(Token Budget)을 엄격히 할당하고, 과거 대화를 동적으로 압축하는 대화 요약(Summarization) 파이프라인 설계가 요구됩니다.

**9. 성능 효율성 - 시간 반응성 (Time Behavior) : 상태 동기화 및 지연 시간 최소화**
 * **정의:** 비동기 네트워크 통신 및 온디바이스 LLM의 추론 지연이 발생하는 환경에서도, 시스템의 상태 변화를 사용자 UI에 단절감 없이 즉각적으로 피드백하는 능력입니다.
 * **평가 지표:** 이벤트 발생(예: 상대방 제안 수신) 시점부터 사용자 UI에 결과가 반영되기까지의 End-to-End 대기 시간(Latency).
 * **상세 설명:** 메인 스레드가 블로킹되어 단말이 멈추는 현상을 방지하기 위해, 철저한 Event-driven 방식의 비동기 메시지 통신(Message Queue) 구조로 아키텍처를 설계해야 합니다.

### **호환성 (Compatibility)**

**10. 호환성 - 상호운용성 (Interoperability) : 이기종 에이전트 및 레거시 서비스 적응성**
 * **정의:** A2A 표준 프로토콜을 사용하는 타 에이전트뿐만 아니라, 통신 규약이 전무한 구형 레거시(Legacy) 서비스의 비정형 데이터까지 시스템이 수용하여 상호 소통할 수 있는 능력입니다.
 * **평가 지표:** 코어 협상 엔진의 코드 수정 없이, 새로운 외부 서비스 인터페이스(Adapter)를 연동하는 데 소요되는 아키텍처 수정 비용(Man-Month 또는 시간).
 * **상세 설명:** 플랫폼의 고립을 방어하기 위해 헥사고날 아키텍처(Port & Adapter 패턴)를 적용하여 외부 세계와의 결합도를 낮추고, RAG 기반의 비정형 데이터 파싱 모듈을 단말 내/외부 경계에 배치해야 합니다.

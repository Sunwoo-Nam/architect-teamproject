# 비기능 요구사항 명세서 (Non-Functional Requirements Specification)

> 본 문서의 품질 속성 분류는 **ISO/IEC 25010:2023** 소프트웨어 품질 모델을 기준으로 한다.
> 측정 기준의 수치는 PoC 단계 기준의 제안치이며, 구현 단계에서 실측 결과를 바탕으로 조정 가능하다.

---

## 1. IDS 비기능 요구사항

### NFR-IDS-01. Performance - Intent 인지 및 알림 속도

정의: IDS는 사용자의 Intent를 빠른 시간 내에 인지하고, 인지 여부를 사용자에게 즉각적으로 알려야 한다.

측정 기준:

| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 명시적 Intent 인지 시간 | 입력 시점 ~ 인지 완료 시점 | **2초 이내** | Nielsen UX 응답성 연구에서 2초는 사용자 주의가 끊기기 시작하는 임계값. 상용 음성 명령 처리 시스템(Siri, Google Assistant)의 실측 기준과 동일 |
| 비명시적 Intent 분석 주기 | 반복 분석 간격 | **30초 이하** | 백그라운드 모니터링 서비스의 배터리 효율과 반응성의 균형점 |
| Intent 인지 알림 제공 시간 | 인지 시작 시점 ~ UI 알림 표시 시점 | **500ms 이내** | 사용자에게 "처리 중" 피드백이 1초 이내에 제공되어야 흐름이 끊기지 않음(Nielsen). 알림은 처리 완료 전에 선제적으로 제공되어야 하므로 500ms로 설정 |

---

### NFR-IDS-02. Performance - Task 처리 속도

정의: IDS가 수신한 Task의 처리 주체 판별과 자체 처리를 신속하게 완료해야 한다.

측정 기준:

| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| Task 처리 주체 판별 시간 | 판별 시작 ~ 판별 완료 시점 | **1초 이내** | On-device 경량 LLM(분류 수준) 추론 시간 기준. Gemma 2B 수준의 단순 분류 작업 기준 0.5~1초 |
| IDS 자체 Task 처리 시간 | 판별 완료 ~ 결과 반환 시점 | **5초 이내** | IDS가 자체 처리하는 태스크는 단순 로컬 명령·정보 검색 수준이므로 5초 이내 처리 가능 |

---

### NFR-IDS-03. Resource Efficiency - 자원 사용량

정의: IDS 동작 시 On-Device 자원(CPU, Memory, NPU)을 최소로 사용하여 포그라운드 앱 성능에 영향을 주지 않아야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| CPU 사용률 (평균) | **10% 이하** | Android 백그라운드 서비스 권장 기준. 포그라운드 앱에 영향을 주지 않는 수준 |
| CPU 사용률 (피크) | **25% 이하** | Intent 인지 추론 시 일시적 허용 범위 |
| Memory 사용량 | **200MB 이하** | On-device 경량 LLM 모델 로딩 포함 기준. Galaxy S26 RAM(12~16GB) 대비 합리적 할당 |
| 외부 LLM API 호출 비율 | **전체 IDS 처리의 20% 이하** | On-device 우선 처리 원칙. 외부 LLM 호출은 토큰 비용·응답 지연을 발생시키므로 On-device 처리 불가 케이스로 한정 |

---

### NFR-IDS-04. Reliability - 자체 Task 처리 성공률

정의: IDS가 자체적으로 처리하기로 판별한 Task는 기술적 오류 없이 성공적으로 완료되어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| IDS 자체 Task 처리 성공률 | **90% 이상** | PoC 단계 기준치. Production 목표(95%+) 수립을 위한 기준선으로 설정 |

상세 설명:
* 성공 기준: 기술적 오류(크래시, 타임아웃, 예외 미처리) 없이 결과가 반환된 경우.
* 의도 해석 정확도(Accuracy)는 별도 평가 항목으로 분리하며 본 NFR의 범위에 포함하지 않는다.

## 2. MAF 비기능 요구사항

> **Reliability 속성 정의 (ISO/IEC 25010:2023 기준)**
>
> 본 절의 Reliability NFR은 아래 두 세부 속성을 기반으로 한다.
> - **Fault Tolerance**: 결함이 발생하더라도 의도한 동작을 유지하는 능력 (→ 재시도, 오류 시 정상 동작)
> - **Recoverability**: 중단 또는 장애 후 원하는 상태로 복원하는 능력 (→ 세션 복구)

---

### NFR-MAF-01. Reliability - Communication 완결성 (Fault Tolerance)

정의: 시작된 Communication 동작(Negotiation, Collaboration, Knowledge Sharing, Remote Monitoring)은 일시적 오류가 발생하더라도 자동 복구를 통해 정상적으로 완료되어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 일시적 오류 시 자동 재시도 횟수 | **최대 3회** | 표준 네트워크 재시도 정책 기준. Exponential backoff 적용(1초 → 2초 → 4초) |
| 3회 재시도 후에도 실패 시 | FR-MAF-08에 따라 사용자 에스컬레이션 | - |

상세 설명:
* 완결성 판단 범위: Action(캘린더 등록, 예약 API 호출 등) 단계까지 포함하여 완결로 간주한다.
* 협상 결렬(양측 조건 불일치로 합의 불가 판정)은 기술적 실패가 아니며 완결성 위반으로 간주하지 않는다.

---

### NFR-MAF-02. Reliability - Task 성공률

정의: MAF가 처리하는 Communication 태스크는 기술적 오류 없이 정상 완료되어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 전체 케이스 기술적 Task 성공률 | **95% 이상** | PoC 단계 기준. 기술적 실패(크래시, 타임아웃 미처리, 예외 미처리)만 실패로 계산 |

상세 설명:
* 성공 기준: 케이스별 정상 흐름(합의 타결 또는 합의 불가 판정)이 오류 없이 완료된 경우.
* 케이스별 성공 기준의 상세 정의는 QAS에서 별도로 명시한다.

---

### NFR-MAF-03. Reliability - 세션 복구 가능성 (Recoverability)

정의: 앱 재시작 또는 일시적 네트워크 장애 후에도 진행 중이던 Communication 세션을 복구할 수 있어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 세션 복구 소요 시간 | **30초 이내** | 30초 초과 시 사용자가 자발적으로 재시작을 선택하는 임계값 기준 |
| 세션 복구 성공률 | **95% 이상** | PoC 단계 기준 |

상세 설명:
* 현재 우선 적용 범위: Negotiation 세션. 이후 Collaboration, Knowledge Sharing, Remote Monitoring 순으로 확장 적용한다.
* 케이스별 세션 복구 방식이 상이할 경우, 케이스별 세부 요건을 별도로 정의한다.

---

### NFR-MAF-04. Performance - Communication Latency

정의: MAF의 Communication 동작은 케이스별로 정의된 측정 구간 내에 완료되어야 한다.

측정 기준:

| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 총 소요 시간 | Task가 MAF에 전달된 시점 ~ Action 완료 시점 | **180초(3분) 이내** | 사용자가 백그라운드 처리를 인지하고 기다릴 수 있는 최대 허용 시간. 협상 라운드 포함 |
| Discovery | 에이전트 탐색 시작 ~ 암호화 채널 수립 완료 | **5초 이내** | TLS 1.3 핸드쉐이크 + Agent Card 교환 시간 기준 |
| Negotiation (1:1) | 협상 시작 ~ 합의 또는 결렬 완료 | **60초 이내** | On-device LLM 추론(약 10~15초/라운드) × 최대 4라운드 기준 |
| Negotiation (N-party) | 협상 시작 ~ 합의 또는 결렬 완료 | **120초 이내** | 참여자 수 증가에 따른 추론 및 조율 시간 추가 반영 |
| Collaboration | 협업 요청 시작 ~ 협업 세션 수립 완료 | **10초 이내** | 외부 플랫폼 API 권한 확인 + 세션 초기화 시간 기준. 협업 내용 완료 시간은 케이스별로 상이하여 측정 범위에서 제외 |
| Knowledge Sharing | 지식 공유 요청 시점 ~ 상대 에이전트 수신 확인 시점 | **10초 이내** | 데이터 직렬화 + 암호화 + 전송 + 수신 확인 응답 시간 기준 |
| Remote Monitoring | 이상 징후 감지 시점 ~ 자녀 에이전트 알림 전달 완료 시점 | **5초 이내** | 이상 징후 알림은 지연 허용도가 가장 낮음. 긴급 상황 대응 기준으로 가장 엄격하게 설정 |

---

### NFR-MAF-05. Performance - 사용자 개입 반영 속도

정의: 자율 처리 진행 중 사용자의 직접 개입 시, 변경 요청이 MAF에 반영되는 시간을 최소화해야 한다.

측정 기준:

| 항목 | 측정 구간 | 임계값 | 근거 |
|---|---|---|---|
| 사용자 개입 반영 시간 | 사용자 UI 조작 시점 ~ MAF에서 Intent 변경 인지 및 적용 완료 시점 | **2초 이내** | IDS 경유 Intent 전달 시간(NFR-IDS-01 기준 2초) 포함. 최종 처리 완료(상대 에이전트 통보 등)까지는 포함하지 않음 |

---

### NFR-MAF-06. Resource Efficiency - 자원 사용량

정의: MAF 동작 시 On-Device 자원(CPU, Memory)을 최소로 사용하여 사용자 경험에 영향을 주지 않아야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| CPU 사용률 (평균) | **15% 이하** | IDS보다 복잡한 처리(협상 추론 등)를 고려하여 IDS 기준(10%)보다 높게 설정 |
| CPU 사용률 (피크) | **40% 이하** | 협상 LLM 추론 시 일시적 사용 허용 범위 |
| Memory 사용량 | **300MB 이하** | Multi-Agent 상태 관리 + On-device LLM 로딩 포함 기준 |

상세 설명:
* LLM 배치 원칙: Meta Agent·Sub-Agent는 On-device LLM 사용. Orchestrator는 서버 LLM 사용.
* 서버 LLM(Orchestrator) 호출은 Task Planning 단계로 한정하여 외부 호출을 최소화한다.

---

### NFR-MAF-07. Security - 개인정보 보안

정의: 에이전트 간 Communication 수행 시 민감 정보를 암호화하고 개인정보 공유를 최소화해야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 전송 계층 암호화 | **TLS 1.3 이상** | 현 시점 업계 표준 최소 요건 |
| 민감 데이터 외부 전송 | FR-MAF-05(지식 공유 보안 등급 구분) 기준 필터링 후 전송, **Private Knowledge 외부 전송 0건** | 개인정보 유출 원천 차단 |

상세 설명:
* 에이전트 간 전송 데이터는 FR-MAF-05에서 Private Knowledge로 분류된 항목을 포함하지 않아야 한다.
* 암호화 채널 수립은 FR-MAF-01(Discovery & Authentication) 완료 시점에 보장되어야 한다.

---

### NFR-MAF-08. Traceability - 협상 이력 추적

정의: Communication의 진행 과정 전체가 완전하게 기록·보존되어, 사용자가 원할 때 즉시 조회할 수 있어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 이벤트 로그 기록률 | **100%** | 추적 공백이 발생하면 분쟁·오류 원인 분석 불가 |
| 로그 보존 기간 | **30일 이상** | 일반적인 서비스 이슈 대응 및 사용자 분쟁 처리 기간 기준 |
| 사용자 이력 조회 응답 시간 | **2초 이내** | FR-MAF-07(협상 상태 가시성)의 실시간 조회 UX 기준과 일치 |

상세 설명:
* FR-MAF-07이 기능(무엇을 보여주는가)을 정의하는 데 반해, 본 NFR은 품질(얼마나 완전하고 빠르게 제공되는가)을 정의한다.
* 기록 대상 이벤트: 협상 제안·역제안·수락·거절, 협업 세션 수립·종료, 지식 공유 요청·전달·수신 확인, 이상 징후 감지·알림 전달 등 모든 Communication 상태 전이.

### NFR-MAF-01. Reliability - Communication 완결성 (Fault Tolerance)

정의: 시작된 Communication 동작(Negotiation, Collaboration, Knowledge Sharing, Remote Monitoring)은 일시적 오류가 발생하더라도 자동 복구를 통해 정상적으로 완료되어야 한다. 다자간(N-party) Communication에서는 일부 참여자 결함 발생 시에도 시스템 전체가 무한 대기 없이 합의 또는 명시적 실패 종료에 도달해야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 일시적 오류 시 자동 재시도 횟수 | **최대 3회** | 표준 네트워크 재시도 정책 기준. Exponential backoff 적용(1초 → 2초 → 4초) |
| 3회 재시도 후에도 실패 시 | FR-MAF-08에 따라 사용자 에스컬레이션 | - |
| **(N-party) 부분 결함 시 진행 보장** | 참여자 중 **정족수(과반) 이상 응답** 시 **100% 합의 또는 명시적 실패 종료** | 분산 시스템 표준 정족수 원칙. 일부 참여자 응답 지연·종료로 인한 무한 대기 차단 |
| **(N-party) 정족수 미만 시 처리** | **명시적 "협상 무효" 상태 전이 100%** | 의사 합의(살아남은 일부만의 합의)로 인한 분쟁 방지 |

상세 설명:
* 완결성 판단 범위: Action(캘린더 등록, 예약 API 호출 등) 단계까지 포함하여 완결로 간주한다.
* 협상 결렬(양측 조건 불일치로 합의 불가 판정)은 기술적 실패가 아니며 완결성 위반으로 간주하지 않는다.
* **N-party 정족수 기준**: PoC 단계에서는 단순 과반(N/2 + 1)을 적용한다. 협상 시작 시점에 참여자 수 N이 확정되며, 협상 중 합류·이탈은 본 PoC 범위에서 제외한다.
* **"무한 대기" 판정 기준**: NFR-MAF-04의 N-party 협상 시간 한계(120초)를 초과한 응답 미수신 참여자는 결함으로 간주한다.

### NFR-MAF-03. Reliability - 세션 복구 가능성 (Recoverability)

정의: 앱 재시작 또는 일시적 네트워크 장애 후에도 진행 중이던 Communication 세션을 복구할 수 있어야 한다. 다자간(N-party) Communication에서는 복구 후 모든 살아있는 참여자의 세션 상태가 일관되어야 한다.

측정 기준:

| 항목 | 임계값 | 근거 |
|---|---|---|
| 세션 복구 소요 시간 | **30초 이내** | 30초 초과 시 사용자가 자발적으로 재시작을 선택하는 임계값 기준 |
| 세션 복구 성공률 | **95% 이상** | PoC 단계 기준 |
| **(N-party) 복구 후 사본 간 상태 일관성** | 살아있는 참여자 사본 간 **부분 불일치 발생률 0%** | 분산 환경에서 일부 사본만 stale·다른 상태로 복구되면 의사 합의·중복 실행 위험. 정합성은 협상 신뢰성의 전제 조건 |
| **(N-party) 사본 수렴 시간 (T_reconcile)** | 사건 해소 후 **10초 이내** 전체 사본 수렴 | NFR-MAF-04 N-party 협상 시간(120초)의 약 8% 수준으로 설정하여 협상 진행에 미치는 영향을 최소화 |

상세 설명:
* 현재 우선 적용 범위: Negotiation 세션. 이후 Collaboration, Knowledge Sharing, Remote Monitoring 순으로 확장 적용한다.
* 케이스별 세션 복구 방식이 상이할 경우, 케이스별 세부 요건을 별도로 정의한다.
* **"사본 일관성"의 정의**: 살아있는 모든 참여자의 협상 상태가 다음 세 가지 중 하나로 동일하게 수렴해야 한다.
  - (a) 동일한 합의 상태 — 합의 도달 케이스
  - (b) 동일한 실패 상태 — 합의 불가능 케이스, 동일 사유로 종료
  - (c) 동일한 진행 상태 — 사건 이전과 같은 라운드/제안 컨텍스트로 협상 재개
* **"살아있는 참여자"의 정의**: NFR-MAF-01의 정족수 판정 기준에 따라 살아있는 것으로 식별된 참여자만 일관성 측정 대상이 된다. 죽은 단말의 사본은 제외한다.
* **본 NFR과 NFR-MAF-01의 관계**: 본 NFR(Recoverability)은 "복구 후 N개 사본이 같은 상태인가"(Safety)를 다루고, NFR-MAF-01(Fault Tolerance)은 "결함 발생 시 진행이 보장되는가"(Liveness)를 다룬다. 분산 시스템에서 두 속성은 짝으로 충족되어야 한다.

# 기능 요구사항 명세서 (Functional Requirements Specification)

---

## 1. Intent Driven Secretary (IDS) 기능 요구사항

### FR-IDS-01. 명시적 의도 추출

정의: 사용자가 IDS와 직접 상호작용하는 인터페이스(음성 발화, 텍스트 입력, UI 조작 등)를 통해 명시적으로 지시한 명령으로부터 제어 의도를 도출해야 한다.

상세 설명:
* 명시적 의도의 예시는 다음과 같다: 음성 또는 텍스트로 직접 지시("○○와 이번 주말 저녁 식사 일정을 잡아줘"), IDS가 제공한 추천 알림에 대한 수락 응답("일정을 잡아드릴까요?" 알림에 동의 선택), 진행 중인 Communication 작업에 대한 조건 변경 또는 취소 지시.
* 메시징 앱에서의 일반 대화 내용(예: "우리 이번 주말에 만날까?")은 명시적 의도가 아니며, FR-IDS-02의 비명시적 의도 분석 대상이다.

---

### FR-IDS-02. 비명시적 의도 도출

정의: 디바이스의 다양한 컨텍스트 데이터를 주기적으로 수집·분석하여 사용자가 명시적으로 명령하지 않은 잠재적 의도를 추론해야 한다.

상세 설명:
* 수집·분석 대상 데이터의 예시는 다음과 같다.
  * 화면 컨텍스트(Screen Context): 현재 사용 중인 앱, 소비 콘텐츠(YouTube 시청 영상 주제, 검색 키워드, 열람 중인 웹페이지 등)
  * 커뮤니케이션 컨텍스트: 메시징 앱(카카오톡 등) 대화 내 키워드 및 맥락(특정 장소·인물·활동 언급 등)
  * 디바이스 상태: GPS 위치, 배터리 수준, 네트워크 연결 상태
  * 일정 및 연락처 컨텍스트: 캘린더 일정 현황, 자주 연락하는 상대 및 커뮤니케이션 빈도
  * 행동 패턴: 특정 앱·콘텐츠의 반복 소비 패턴, 특정 연락처와의 상호작용 빈도 변화
* 비명시적 의도 추론의 예시는 다음과 같다.
  * 메시징 앱에서 특정 지인과 "이번 주말에 만날까?"라는 대화가 감지된 경우 → 해당 지인과의 일정 조율 추천 알림 제공
  * YouTube에서 특정 식당 관련 콘텐츠를 반복 소비하고, 메시징 앱에서 지인과 해당 식당을 언급한 경우 → 해당 지인과 해당 식당 방문 일정 조율 추천 알림 제공
  * 패션 관련 영상을 반복 시청 중인 경우 → 관련 쇼핑 검색 또는 상품 추천 제공

---

### FR-IDS-03. 처리 주체 식별 및 태스크 매핑

정의: 추출된 의도의 성격을 분석하여, 사전에 정의된 역할 분리에 따라 IDS 자체 처리 고유의 태스크인지, Multi-Agent Framework(MAF) 고유의 태스크인지 명확히 식별하고 매핑해야 한다.

상세 설명:
* IDS 자체 처리 대상: 단일 사용자 컨텍스트 내에서 완결되는 태스크로, 외부 에이전트와의 상호작용이 불필요한 작업이 해당된다. 디바이스 직접 제어, 정보 검색, 프라이버시 민감 작업 등이 포함된다.
* MAF 처리 대상: 타 에이전트와의 Negotiation·Collaboration·Knowledge Sharing·Remote Monitoring 등 외부 에이전트와의 복합 상호작용이 필요한 태스크, 또는 Sub-Agent의 활용이 필요한 태스크가 해당된다.
* 처리 주체 분리는 단순히 로컬 처리 여부가 아니라, 태스크의 본질적 목적과 필요한 컴포넌트 구성에 따라 결정해야 한다.

---

### FR-IDS-04. 컨텍스트 연속성 유지 및 맥락적 의도 관리

정의: 이전 대화나 수행 중인 작업 맥락(Context)을 유지하며, 기존 맥락과 연계된 사용자의 추가 의도를 분석하고 처리해야 한다.

상세 설명:
* 사용자의 새로운 발화가 기존에 IDS가 인식하여 자체 처리 중이거나 MAF로 전달한 작업과 맥락적으로 이어지는지 대조 분석해야 한다.
* 기존 맥락의 유효 기간(Session Timeout) 내에 발생한 연관 의도는 기존 세션에 매핑하여 연속적인 하위 태스크로 IDS 자체 처리 엔진 또는 MAF에 전달해야 한다.
* 참고: 맥락이 전혀 없는 완전히 새로운 독립적 의도는 FR-IDS-01 및 FR-IDS-02에 의해 초기 의도 추출 프로세스로 처리된다.

---

### FR-IDS-05. IDS 자체 처리를 통한 태스크 수행

정의: FR-IDS-03에서 IDS 자체 처리로 분류된 태스크를 IDS 자체 처리 엔진을 통해 실행하고, 처리 결과를 사용자에게 반환해야 한다.

상세 설명:
* IDS 자체 처리 대상 태스크의 유형은 다음과 같다: 디바이스 직접 제어(알림 설정, 시스템 설정 변경 등), 단일 사용자 컨텍스트 내 정보 검색(캘린더 조회, 연락처 검색 등), 프라이버시 민감 작업(디바이스 외부로 전송하지 않고 처리해야 하는 데이터 처리).
* 태스크 처리 완료 후 결과를 사용자에게 피드백으로 제공해야 한다.
* 태스크 처리 중 오류가 발생한 경우 사용자에게 즉시 통보해야 한다.

---

## 2. Multi-Agent Framework (MAF) 기능 요구사항

MAF는 사용자 에이전트 간 상호작용을 **Communication**이라는 상위 범주 내에서 **Negotiation(협상)**, **Collaboration(협력)**, **Knowledge Sharing(지식 공유)**, **Remote Monitoring(원격 모니터링)** 4대 케이스로 처리한다. 아래 요구사항은 이 구조를 기준으로 구분하며, 모든 케이스에 공통 적용되는 요구사항은 앞에 별도로 명시한다.

---

### 2.1 공통

#### FR-MAF-01. 에이전트 디스커버리 및 인증 (Discovery & Authentication)

정의: 협업 목적에 부합하는 타 사용자의 에이전트 엔드포인트를 탐색(Discovery)하고, 에이전트 간 상호 인증(Mutual Authentication) 및 암호화 채널을 수립하여 통신이 가능한 보안 상태를 구축해야 한다.

---

#### FR-MAF-07. 진행 상태 가시성 (UI Visibility)

정의: 백그라운드에서 진행 중인 에이전트 간의 Communication 상태를 사용자에게 직관적으로 시각화해야 한다.

상세 설명:
* 현재 진행 턴(Turn), 남은 쟁점 사항, 에이전트 간 제안 타임라인 등을 사용자가 원할 때 언제든 실시간 UI를 통해 모니터링할 수 있는 가시성을 제공해야 한다.

---

#### FR-MAF-08. 런타임 직접 개입 (Human-in-the-Loop)

정의: Communication 처리 중 사용자가 실시간으로 개입하여 에이전트의 행위를 직접 제어할 수 있어야 한다.

상세 설명:
* 에이전트의 처리 프로세스 중 사용자가 명시적인 UI 조작이나 신규 의도 발화(IDS 연동)를 통해 처리 조건을 즉시 변경, 강제 수락, 진행 취소 등을 지시할 경우 이를 즉각 반영하고 상대 에이전트에게 통보해야 한다.

---

#### FR-MAF-09. 합의안 후속 실행 (Tool Calling)

정의: 에이전트 간의 Communication이 최종 타결되었을 때, 도출된 결과를 디바이스 및 외부 서비스에 실질적으로 반영해야 한다.

상세 설명:
* 최종 합의 데이터를 기반으로 디바이스 내의 시스템 API(캘린더 등록, 알림 설정 등) 및 외부 서드파티 예약 API를 호출하여 상태 변경을 완료해야 한다.

---

#### FR-MAF-10. 권한 위임 정책 관리 (Authorization Policy Management)

정의: 사용자가 자신의 에이전트에 위임할 수 있는 처리 범위를 정의하고, 에이전트는 그 범위 내에서만 동작하며 범위를 초과하는 경우 사용자 승인을 요청해야 한다.

상세 설명:
* 사용자는 케이스 유형(Negotiation / Collaboration / Knowledge Sharing / Remote Monitoring)별로 처리 허용 범위를 사전 설정할 수 있어야 한다.
* 에이전트는 실행 전 해당 행위가 위임 범위 내인지 판단하고, 범위를 초과하는 경우 사용자에게 승인 요청 후 대기해야 한다.
* 위임 범위 및 정책은 사용자가 언제든 수정·철회할 수 있어야 한다.

---

#### FR-MAF-11. 이상 상황 처리 (Abnormal Situation Handling)

정의: Communication 동작 중 이상 상황이 발생하는 경우, 이를 감지하고 정의된 처리 흐름에 따라 처리되어야 한다.

상세 설명:
* 감지 대상 이상 상황의 예시는 다음과 같다: 상대 에이전트 무응답, 협상 타임아웃, 합의 불가 판정, 네트워크 장애 등.
* 이상 상황 발생 시 Communication이 불확정 상태(Undefined State)에 빠지지 않아야 한다.
* 이상 상황은 정의된 처리 흐름을 통해 완결(성공 또는 실패)되어야 한다.

---

### 2.2 Negotiation (협상)

사용자의 에이전트가 1:1 또는 다수 상대방 에이전트와 일정 예약, 장소 선택 등의 의사결정을 협상하여 합의에 도달하는 케이스.

#### FR-MAF-02. 1:1 협상

정의: 단일 상대방 에이전트와 일정 조율, 조건 합의 등의 의사결정을 수행해야 한다.

상세 설명:
* 사용자가 설정한 정책 범위 내에서 단일 에이전트와 제안(Propose), 역제안(Counter-propose), 수락/거절의 협상 사이클을 수행해야 한다.

---

#### FR-MAF-03. 다자간 합의 및 교착 상태 방지 (N-Party Negotiation)

정의: 3인 이상의 에이전트가 참여하는 환경에서 다자간 합의 프로토콜을 통해 협상을 타결해야 한다.

상세 설명:
* 다자간 합의 프로토콜(예: 다자간 계약 가이드라인 또는 분산 합의 알고리즘)을 구동하여 합의를 도출해야 한다.
* 무한 루프, 메시지 폭주, 교착 상태(Deadlock)에 빠지지 않도록 타임아웃 및 협상 상태 전이 제어 메커니즘을 포함해야 한다.

---

### 2.3 Collaboration (협력)

사용자의 에이전트가 1:1 또는 다수 상대방 에이전트와 공동 문서 작성 등 협업 워크플로우를 실행하는 케이스.

#### FR-MAF-04. 외부 협업 워크플로우 연동 (Collaboration Integration)

정의: 에이전트 간 공동 문서 작성 등의 협업 워크플로우 수행 시, 기존 상용 협업 플랫폼의 API를 호출하여 작업을 조율하고 대행해야 한다.

상세 설명:
* 외부 협업 플랫폼(예: 서드파티 문서 편집기 등)에 대한 권한 인증 상태를 관리하고, 에이전트 간 작업 순서에 맞춰 해당 플랫폼의 API를 호출하여 사용자 대신 워크플로우를 진행할 수 있어야 한다.

---

### 2.4 Knowledge Sharing (지식 공유)

사용자의 에이전트가 1:1 또는 다수 상대방 에이전트와 지식을 교환하는 케이스. 공유 데이터의 보안 등급 제어가 핵심 요건이다.

#### FR-MAF-05. 지식 공유 보안 등급 구분 (Privacy Segregation)

정의: 외부 에이전트와 지식을 공유할 때, 데이터의 보안 등급을 식별하고 제어해야 한다.

상세 설명:
* 외부로 공유 가능한 공용 데이터(Shared/Public Knowledge)와 절대 공유할 수 없는 고유 민감 데이터(Private Knowledge)를 명확히 구분할 수 있어야 한다.
* 데이터 자산별 프라이버시 정책 매핑을 통해 외부 에이전트 요청 시 자동으로 필터링 및 익명화 처리가 가능해야 한다.

---

### 2.5 Remote Monitoring (원격 모니터링)

Care Recipient(피케어자) 기기의 에이전트가 Caregiver(케어 제공자) 에이전트에게 권한을 위임하고, Caregiver 에이전트가 위임받은 권한 범위 내에서 Care Recipient의 데이터를 확인·처리하는 케이스.

#### FR-MAF-06. 동의 기반 원격 케어 (Remote Care / Remote Monitoring)

정의: Care Recipient 기기와 Caregiver 기기 간의 에이전트 연동을 통해, Care Recipient의 독립성을 존중하면서 이상 징후를 모니터링하고 권한을 위임받아 처리해야 한다.

상세 설명:
* Care Recipient 기기 에이전트의 명시적 동의 및 권한 위임(Delegation)을 바탕으로 구동되어야 한다.
* Care Recipient의 사생활을 침해하는 실시간 화면 감시가 아닌, 에이전트가 로컬에서 '이상 행동 패턴 분석', '오랫동안 디바이스 미사용', '긴급 알림 발생' 등을 감지했을 때 Caregiver 에이전트에게 보안 통지 및 데이터를 전송하는 방식으로 원격 모니터링을 수행해야 한다.
* Care Recipient의 요청 시, Caregiver 에이전트가 Care Recipient 기기의 특정 복잡 설정이나 예약 대행 등의 태스크 권한을 안전하게 위임받아 실행할 수 있어야 한다.

---

## 3. 논의 제외 사항 (Out of Scope)

* **Sub-Agent 기능 요구사항**: 아키텍처 중간점검 단계까지는 본 논의 및 스펙 아웃으로 처리하며, 추후 요구사항 확장 시 반영한다.
* **Device Agent 기능 요구사항**: 중간점검 완료 시점까지 논의 대상에서 제외한다. 단, 사용자 주단말(스마트폰)의 Device Proxy Agent(DPA)는 현재 범위 내로 포함된다.
* **웨어러블 디바이스 연동**: 현재 범위는 스마트폰 단말 중심으로 한정하며, 웨어러블 디바이스와의 연동은 향후 확장 고려 사항으로 분류한다.

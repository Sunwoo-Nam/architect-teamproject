## 2.2 핵심 개념

| 용어 | 설명 |
|---|---|
| **NPN** (Neural Proxy Network) | 사용자 간의 이해관계를 각자의 AI Agent가 대리하여 조율·확정하는 Agent 간 자율 협상 네트워크. 본 플랫폼이 실현하려는 핵심 개념임 |
| **PPA** (Personal Proxy Agent) | 사용자(나)를 대표하는 Agent. 안드로이드 스마트폰에 탑재되며, 권한과 선호를 위임받아 상대방 PPA와 A2A 협상을 수행함. 시스템 내 **Meta Agent**가 PPA 역할을 담당함 |
| **DPA** (Device Proxy Agent) | 사용자 소유의 개별 스마트 기기(세탁기·TV·냉장고 등)에 탑재된 Agent. 해당 기기의 상태·스케줄 정보를 제공함. 외부 협상은 수행하지 않음 |
| **A2A 프로토콜** | 두 기기의 PPA(Meta Agent) 간 협상 메시지를 주고받는 통신 프로토콜. 디스커버리 → 연결 → 협상 → 합의의 흐름으로 동작함 |
| **IDS** (Intent Driven Secretary) | 안드로이드 기기 활동을 모니터링하여 사용자 Intent를 감지·분류하는 컴포넌트. Zero-claw 프레임워크 기반으로 구현됨 |

용어 정의의 약어 전체 목록은 [`annex/약어.md`](annex/약어.md)에 있음.
아키텍처 차원의 전문 용어(트랜잭션 원자성·비동기 협상·RAG·DAG)는 [`annex/아키텍처-주요-용어.md`](annex/아키텍처-주요-용어.md)에 있음.

## 2.3 시스템 구성 (Layered View)

본 플랫폼은 안드로이드 스마트폰을 중심으로 **4개 계층의 Agent**로 구성됨.

| 계층 | 컴포넌트 | 역할 | LLM 위치 |
|---|---|---|---|
| 1 | **IDS** | 기기 활동 전체 모니터링, Intent 감지·라우팅. 단순 Intent는 자체 해결, 복합 Intent는 Orchestrator에 전달함 | 온디바이스 |
| 2 | **Orchestrator** | IDS로부터 Task 수신 후 Planning, 필요한 Agent/Tool 동적 생성 프롬프팅을 수행함 | 서버 |
| 3 | **Meta Agent (= PPA)** | Orchestrator 지시에 따라 Agent/Tool 호출, 상대방 PPA와 A2A 협상, 필요 시 DPA에 정보 요청을 수행함 | 온디바이스 |
| 4 | **Sub-Agent** | 실제 Tool 호출(API·캘린더 등)을 수행함. 도메인별로 동적 생성됨 | 온디바이스 |

## 2.4 Context Diagram (전체 그림)

본 시스템의 외부 액터와의 관계를 표현한 시스템 컨텍스트 다이어그램은 다음 파일에서 확인할 수 있음.

- 📐 **drawio 원본**: [`02-context-diagram.drawio`](02-context-diagram.drawio)
  *draw.io ([app.diagrams.net](https://app.diagrams.net/)) 에서 열어 직접 수정 가능함.*

### 2.4.1 다이어그램 요소

| 요소 | 종류 | 설명 |
|---|---|---|
| **사용자 (User)** | Person | Android 폰 소유자. 권한 위임 / 묵시적 Intent 표출 / 협상 결과 승인을 수행함 |
| **Ondevice Agentic Platform (Android)** | Focus System | 본 과제의 대상 시스템. 내부에 IDS와 Multi-Agent Framework(Orchestrator → Meta Agent → Sub-Agent)을 포함함 |
| **상대방 PPA** | External System | 상대방 기기의 Personal Proxy Agent — A2A 협상의 카운터파트 |
| **Device Proxy Agent (DPA)** | External System | 사용자 소유 가전. 상태·스케줄 정보 제공 |
| **외부 서비스 API** | External System | 숙박·교통·식당·캘린더·결제 등. Sub-Agent가 Tool로 호출함 |
| **온디바이스 LLM** | External System | 경량 추론 모델. IDS·Meta Agent·Sub-Agent가 사용함 |
| **서버 LLM** | External System | 고성능 추론 모델. Orchestrator의 Task Planning·Agent 생성에 사용됨 |

### 2.4.2 주요 상호작용

| 상호작용 | 양측 | 설명 |
|---|---|---|
| 권한 위임 / Intent 표출 | 사용자 → Platform | 기기 사용 행동을 통해 묵시적 Intent를 전달함 |
| 결과 제시 / 승인 요청 | Platform → 사용자 | Intent 감지 알림, 협상 결과를 UI로 제시함 |
| **A2A 협상** | Platform ↔ 상대방 PPA | 협상 메시지 송수신 (NPN의 핵심) |
| 컨텍스트 수집 | Platform ↔ DPA | 가전의 스케줄·상태 정보 요청·응답 |
| 외부 실행 | Platform ↔ 외부 서비스 API | 예약·결제·조회 API 호출 |
| LLM 추론 | Platform ↔ 온디바이스 LLM / 서버 LLM | Intent 분류, Task Planning, Agent 생성 등 |

## 2.5 시스템 범위

### 2.5.1 범위 내 (In Scope)

- Intent 감지 및 Task 생성 (IDS)
- 동적 Agent/Tool 생성 (Orchestrator)
- A2A 자율 협상 (Meta Agent ↔ 상대방 PPA — NPN의 핵심)
- 계층적 Multi-Agent Orchestration (IDS → Orchestrator → Meta Agent → Sub-Agent)
- 기기 생태계 연동 (DPA로부터 정보 수집 및 결과 반영)
- 사용자 권한 위임 관리

### 2.5.2 범위 외 (Out of Scope)

- 사용자 계정 인증·회원 관리 (공통 인프라)
- 카카오톡 등 개별 앱 자체 기능 개발
- 클라우드 전용 Agent 서비스 (본 과제는 온디바이스 중심임)
- 범용 IoT 플랫폼 기능 (기기 제조·펌웨어 관리 등)
- iOS 지원 및 B2B 기업용 Agent 연동 (향후 확장 고려 사항)

## 2.6 대표 시나리오 — 여행 일정 조율

A와 B가 카카오톡으로 여행 이야기를 나누는 상황에서 시스템은 다음과 같이 동작함.

1. A의 **IDS**가 대화에서 여행 계획 Intent를 감지하고 Orchestrator에 Task를 전달함
2. **Orchestrator**가 여행 Task 매니저를 생성함 (캘린더 Sub-Agent, 숙박 Sub-Agent 등 동적 그룹핑)
3. **Meta Agent (A의 PPA)** 가 B의 기기를 디스커버리하고 A2A 연결을 수립함
4. A의 Meta Agent가 자신의 DPA들(세탁기·TV·냉장고)에 스케줄 정보를 요청하여 컨텍스트를 수집함
5. A의 Meta Agent ↔ B의 Meta Agent 간 자율 협상이 진행됨 (일정·숙박 조건 등)
6. 협상 완료 후 합의안을 A 사용자에게 제시함 → 사용자가 승인함
7. Sub-Agent가 캘린더 등록·숙박 예약 등을 실행함

본 시나리오 외 9종의 상호작용 모델(P2P / Hierarchical & Resource Control / B2C & Legacy / Delegation & Orchestration)은 [`annex/상호작용-시나리오-9종.md`](annex/상호작용-시나리오-9종.md)에 있음.

# Design Decision 후보안 — 과제 Overview

> 본 문서는 `docs/DP 후보안/`의 Design Decision(DP) 후보들을 묶어, 과제의 **개요·배경·목적·범위**를 정리한다.
> 개별 DP의 상세 해결 방안·UML·평가는 각 DP 문서를 참조하고, 본 문서는 전체를 조망하는 상위 관점을 제공한다.

---

## 1. Overview (개요)

본 과제는 **Ondevice Agentic Platform**의 핵심 아키텍처 결정들을 다룬다. 이 플랫폼은 사용자를 대리하는 개인 대리 에이전트(PPA, Personal Proxy Agent = Meta Agent)들이 서로 자율적으로 협상하여, 일정 조율·예약·조건 합의 같은 일상의 미세 의사결정을 사람 개입 없이 처리하는 것을 목표로 한다(Neural Proxy Network 개념).

DP 후보안은 이 플랫폼을 실제로 구현할 때 마주치는 **5개의 핵심 설계 결정(Design Decision)** 으로 구성된다. 각 DP는 "풀고자 하는 문제 → 아키텍처적 난제 → 해결 방안 후보 → 품질 속성 Trade-off 평가" 구조로 작성되며, 자율 협상이라는 비결정적(LLM)·자율적(무인)·온디바이스(자원 제약) 환경에서 발생하는 서로 다른 난제를 분담한다.

| DP | 제목 | 핵심 관심사 | 대표 QAS |
|----|------|-------------|----------|
| **DP06** | 협상 시 민감정보 처리 | 보안 — 프라이버시·권한 | QS-009 (서버 LLM PII 전송) |
| **DP07** | 비동기 턴제 협상 중 네트워크 단절 복구 | 신뢰성 — 연결 복구·상태 일치 | QS-006 (네트워크 단절 복구) |
| **DP08** | A2A 다자간 협상 아키텍처 | 성능·정합성 — 협상 효율·합의 수렴 | QS-002 (협상 완료 시간) |
| **DP09** | 인텐트 변경·취소 시 동시성 제어 | 동시성 — in-flight 트랜잭션 방향 전환 | QS-016 (상태 복구) |
| **DP11** | 다중 Intent 동시 발생·변경·실패 처리 | 가용성 — 실행 상태 관리·보상 | QS-024 (실행 실패 처리) |

## 2. 배경 (Background)

자율 협상 플랫폼은 일반적인 클라우드 챗봇과 본질적으로 다른 세 가지 어려운 성질을 동시에 갖는다. 이 세 성질의 교차점에서 DP들이 도출되었다.

- **비결정적(Non-deterministic):** 협상·의도 판단·동적 생성이 LLM 추론에 의존한다. LLM은 본질적으로 실패·환각 가능성이 있어, 결정적 제어 흐름과 비결정적 추론을 분리하지 않으면 재현성과 검증이 불가능해진다.
- **자율적(Autonomous):** 사람이 보지 않는 백그라운드에서, 상대방 에이전트와 무인으로 협상이 길게 진행된다. 이 과정에서 사용자의 의도가 바뀌거나(DP09), 여러 의도가 동시에 발생하거나(DP11), 네트워크가 끊기거나(DP07), 합의 후 외부 실행이 실패할 수 있다(DP11). 무인이기 때문에 모든 예외를 시스템이 스스로 안전하게 처리해야 한다.
- **온디바이스(On-device):** 안드로이드 단말의 제한된 메모리·배터리·NPU 안에서 동작하며, OS가 언제든 프로세스를 강제 종료할 수 있다. 데이터 주권을 단말에 두기 위해 민감 정보가 기기 밖으로 나가면 안 된다(DP06).

또한 중앙 서버가 모든 것을 통제하지 않는 **Federated(연합) 토폴로지**가 전제된다. 각 단말이 자기 데이터·권한·협상 상태를 보유하고(sovereign), 서로 대등하게(P2P) 협상하며, 메시지 분배 Broker는 relay만 담당한다. 이 분산 구조가 상태 정합성·복구·동시성 난제(DP07/08/09/11)를 만들어낸다.

## 3. 목적 (Purpose)

각 DP의 목적은 위 세 성질이 만들어내는 구체적 난제를 안전하게 해소하는 것이다.

- **DP06 — 민감정보 0% 유출과 권한 100% 통제를 보장하면서 협상 속도를 확보.** 민감정보를 식별정보(PII)·협상 사실값·원본 컨텍스트 세 종류로 분리하고, 출구(서버 LLM·상대 PPA·클라우드 빌드) 통제와 사전 변환으로 데이터 주권을 단말에 유지한다.
- **DP07 — 비동기 통신에서 메시지 유실 시 양측 협상 상태를 인간 개입 없이 일치.** 비대칭 단절·재시도 중복으로 인한 상태 불일치(split-brain)를 멱등성·상태 동기화로 해소한다.
- **DP08 — 다자간(N-party) 협상에서 어느 단말의 LLM이 결정하고, PII 노출 없이 합의를 일관되게 수렴시킬지 결정.** 중앙 LLM 집중의 프라이버시·토큰·일관성 문제를 분산 협상 구조로 해결한다.
- **DP09 — 협상 진행 중 사용자의 의도가 바뀔 때, 이미 돌아가는 분산 트랜잭션을 꼬임 없이 방향 전환.** 경합 상태(Race Condition)와 채널 비동기화를 동시성 제어(비관적 락 vs 낙관적 주입)로 다룬다.
- **DP11 — 여러 Intent가 동시에 발생·변경·실패할 때 실행 상태를 통합 또는 격리 관리하고, 이미 발생한 외부 효과를 어디까지 보상.** 계획 수립(서버)과 실행 상태 소유권(단말)을 분리한다.

공통 목적은, 자율 협상이 **신뢰할 수 있는가**라는 PoC의 핵심 가설을 — 빠르면서도(QA-001) 안전하고(NFR) 복구 가능한 형태로 — 입증하는 것이다.

## 4. 범위 (Scope)

### 4.1 포함 범위 (In Scope)

- 단말 내부 협상 추론·민감정보 처리 구조 (DP06)
- 1:1 및 N-party A2A 협상의 메시지 교환·합의 수렴 구조 (DP07, DP08)
- 협상 in-flight 상태에서의 변경·취소·네트워크 단절·프로세스 종료에 대한 복구 (DP07, DP09)
- 다중 Intent의 동시 실행·관계 해소·실패 보상 (DP11)
- 위 결정들이 단말·Broker·서버·외부 서비스의 배포 구조(인프라)에 미치는 영향

### 4.2 제외 범위 (Out of Scope)

- 동적 생성된 Agent/Tool의 코드 빌드 세부(클라우드 Capability Builder 내부 구현)는 별도 결정으로 다루며 본 DP 묶음에서는 출구·배포 관점에서만 참조
- LLM 모델 자체의 학습·튜닝, NPU 하드웨어 최적화 등 구현 튜닝 영역
- 06-QAS.md 6.5에서 "영속화 설계에서 파생" 또는 "구현 패턴 수준"으로 분류되어 미선정된 QAS의 정량 허용치 설정

### 4.3 전제 (Assumptions)

- 안드로이드 온디바이스 중심 PoC, 에이전트 간 통신은 비동기(Async) 가정 (DP07)
- Federated 토폴로지: 각 단말이 협상 canonical state 사본을 보유, Broker는 relay만 담당
- 데이터 주권 원칙: 민감 원본은 단말 내부에만 존재, 서버로는 비민감·PII 제거 데이터만 전송

## 5. DP 간 관계 (How DPs Connect)

DP들은 독립적이지 않고 자율 협상의 생애주기를 따라 맞물린다.

- **정상 흐름:** Intent 감지(DP11) → 협상 시작 시 민감정보 안전화(DP06) → 1:1/다자 협상 진행(DP08) → 합의 → 외부 실행
- **예외 흐름:** 협상 중 네트워크 단절(DP07) · 사용자 의도 변경(DP09) · 다중 Intent 충돌·실행 실패(DP11)
- **횡단 관심사:** QA-001(협상 완료 시간, 1순위)은 DP06·08·09에 공통으로 걸리고, 상태 복구·정합성(QS-016/006/024)은 DP07·09·11이 분담한다.

상세 DP↔QAS 매핑은 [`DP별 대표 QA.md`](DP별 대표 QA.md), 인프라 관점의 전체 커버리지는 [`DP-Coverage-View.drawio`](DP-Coverage-View.drawio) 참조.

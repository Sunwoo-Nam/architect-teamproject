## DD-01. Orchestrator가 서버 LLM에 요청할 때, 민감 정보를 어떻게 차단할 것인가?

### 맥락

Orchestrator는 [`02-과제-개요.md`](02-과제-개요.md) §2.3에 따라 **서버 LLM**을 사용하여 Task Planning과 동적 Agent 생성을 수행함. 서버 LLM에 입력되는 프롬프트에는 IDS가 감지한 Intent의 원문 컨텍스트(메시지, 일정, 기기 정보 등)가 포함될 가능성이 높음. 그러나 NFR-005는 `R_pii_leak = 0%`를 요구하므로, **서버 LLM 호출 경로는 구조적으로 PII가 새지 않아야 함**.

### 핵심 질문

> 서버 LLM의 Planning 품질을 유지하면서, 민감 정보(PII·기기 식별자·메시지 원문 등)의 외부 유출을 어떻게 **구조적으로 0%로 보장**할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)**

- 온디바이스에서 Data sanitize 후 전송 *(원본)*
- 온디바이스 LLM 사용 (서버 LLM과 역할 분리) *(원본)*
- 데이터를 Layer화하여, 민감도별로 도달 경로를 분리 *(원본)*
- 토큰화·플레이스홀더 치환 후 전송 — 원문 결합은 온디바이스에서만 수행 *(정제 시 추가, [`docs/DP 후보안/QS-002-프라이버시-보존-협상-속도.md`](DP%20후보안/QS-002-프라이버시-보존-협상-속도.md) DP-2 참조)*

**하위 질문 (Sub-questions)**

- Sanitize 후 잔여 컨텍스트만으로 서버 LLM이 의미 있는 Plan을 생성할 수 있는가? *(정제 시 추가)*
- 어떤 정보가 "Plan 생성에 필수"이고 어떤 정보가 "전송 없이도 가능"인지의 경계는 무엇인가? *(정제 시 추가)*

**고려사항 (Considerations)**

- 검증 가능성 — 0% 보장을 어떻게 측정·감사할 것인가 *(정제 시 추가)*
- 우회 불가능성 — 속도·편의를 위한 우회 경로가 존재하면 0% 보장이 깨짐 *(정제 시 추가)*

### 관련 NFR·QAS

- **NFR-005** (서버 LLM PII 전송 차단, `R_pii_leak = 0%`) — [`06-QAS.md`](06-QAS.md) §6.3
- **NFR-9** (원본 메시지·일정·민감 metadata 외부 노출 금지) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR12** (최소 데이터 보관 + 보관 기간 정책) — [`05-NFR.md`](05-NFR.md) 버전 B
- **QS-009** (민감 데이터 서버 LLM 전송 최소화)
- 참고 — DD-06과 강하게 연결됨(둘 다 PII 통제 결정이나, 도달 경로가 다름: DD-01은 서버 LLM행, DD-06은 상대 PPA행). 단일 강제 게이트 설계(DP-1)는 두 결정 모두에 영향.

### Open Questions

- 서버 LLM이 정말 필요한가? On-device LLM의 발전 추세를 고려할 때, 서버 LLM 의존을 시간이 지남에 따라 감축할 로드맵을 둘 것인가?
- Sanitize 실패(예: 미처 식별 못 한 PII 패턴) 시의 fail-safe 동작은? 차단·로그·알림·서비스 fallback 중 어디까지 정의할지.

## DD-02. Agent / Tool을 동적으로 생성한다는 것은 무엇을 어디까지 의미하는가?

### 맥락

[`02-과제-개요.md`](02-과제-개요.md) §2.3의 Orchestrator 역할 정의에 "필요한 Agent/Tool 동적 생성 프롬프팅"이 포함되어 있고, [`06-QAS.md`](06-QAS.md) QS-007은 동적 생성 Agent/Tool의 first-run 성공률을 정량 측정 대상으로 두고 있음. 그러나 "동적 생성"의 의미가 (a) 사전 정의된 풀에서의 선택·조합, (b) 템플릿 + LLM 생성 config, (c) 코드까지 LLM 생성, 중 어디인지는 결정되지 않음. 의미 정의에 따라 보안·성공률·확장성·구현 난도가 크게 달라짐.

### 핵심 질문

> 동적 Agent/Tool 생성을 **어느 수준**(선택·조합 / config 생성 / 코드 생성)까지 허용할 것이며, 그 수준이 시스템의 비전을 만족하기에 충분한가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — 원본 하위 질문에서 도출)*

- 사전 정의된 Agent/Tool 풀 + LLM은 **선택·조합**만 담당
- 사전 정의된 템플릿 + LLM은 **파라미터·config**만 생성
- Agent/Tool의 **코드 자체를 LLM이 생성**하고 sandbox에서 실행
- 위의 단계적 적용 — 초기에는 (1)·(2)만, 검증되면 (3)으로 확장

**하위 질문 (Sub-questions)**

- 동적으로 생성하는 게 정말 가능한가? *(원본)*
- 동적 생성이 반드시 필요한가? *(원본)*
- 코드까지 생성하는 게 필요한가? *(원본)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 신뢰성 — 코드 생성을 허용하면 first-run 성공률이 낮아질 수 있음 (QS-007 측정 기준의 부담)
- 보안 — 코드 생성은 NFR-11(동적 생성 tool/agent의 sandbox·policy enforcement) 요구를 강화시킴
- 확장성·차별화 — "사전 풀 선택"만으로는 PoC의 차별화 가치(NPN 비전)가 약해질 수 있음

### 관련 NFR·QAS

- **NFR-11** (동적 생성 tool/agent의 sandbox·policy enforcement) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR16** (Agent 확장 용이성) — [`05-NFR.md`](05-NFR.md) 버전 B
- **NFR-6** (동적 생성 agent 수·수명 제어) — [`05-NFR.md`](05-NFR.md) 버전 C
- **QS-007** (동적 생성 Agent/Tool 실행 성공률), **QA-002** (동적 생성 Agent/Tool 실행 성공률 최대화, 우선순위 2)

### Open Questions

- PoC 단계에서 "코드 생성"을 검증 대상으로 포함하지 않으면 본 과제의 차별화가 약해지는가, 아니면 충분한가?

## DD-03. Multi-Agent 대신 Single Agent + Skill 구조로 가능한가?

### 맥락

[`02-과제-개요.md`](02-과제-개요.md) §2.3은 **4계층 Multi-Agent**(IDS → Orchestrator → Meta Agent → Sub-Agent) 구조를 전제하고 있음. 그러나 IDS는 ZeroClaw 프레임워크 기반으로([`05-NFR.md`](05-NFR.md) CON·NFR-1 맥락), ZeroClaw 류는 본질적으로 **Single Agent + Skill 플러그인**으로 도메인 기능을 확장하는 구조임. 따라서 "Multi-Agent를 정말 다 깔아야 하는가, ZeroClaw 류의 Single Agent를 Skill로 확장하는 것으로 동등한 가치를 줄 수 있는가"는 아키텍처 근간을 흔드는 결정임.

### 핵심 질문

> Multi-Agent 4계층 구조를 채택하지 않고, **Single Agent(ZeroClaw 류) + 도메인별 Skill 모듈** 만으로 본 시스템의 비전(NPN — Agent 간 자율 협상)을 실현할 수 있는가? 못 한다면 어떤 부분에서 본질적으로 Multi-Agent가 필요한가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — 원본은 단순 "VS"만 제시)*

- **(A) 완전 Multi-Agent** — 현재 문서 전제대로 IDS / Orchestrator / Meta Agent / Sub-Agent 4계층 모두 분리
- **(B) 완전 Single Agent + Skill** — ZeroClaw 류 1개 Agent에 도메인 Skill을 장착. A2A 협상도 "외부 통신 Skill"로 처리
- **(C) 혼합** — 내부 Task 분해·실행은 Single Agent + Skill로, 외부 A2A 협상이 발생하는 시점에만 Meta Agent 인스턴스를 spawn

**하위 질문 (Sub-questions)** *(정제 시 추가)*

- "Agent를 분리한다"는 것이 본 시스템에서 본질적으로 가져다 주는 가치는 무엇인가? — 위임/권한/추적성/확장성 중 어떤 것이며, Skill 모듈로는 그 가치를 줄 수 없는가?
- 외부 PPA와의 A2A 협상은 본질적으로 "타 Agent와의 대화"인데, 우리 쪽이 Single Agent라도 프로토콜 차원에서는 문제가 없는가? (즉 상대편이 우리를 "Agent"로 본다면 내부 구현은 Single이어도 무방한가?)
- 동적 Agent 생성(DD-02)이 본질적으로 필요한 시나리오가 있는가, 아니면 Skill 동적 로딩으로 충분한가?

**고려사항 (Considerations)**

- Latency *(원본)* — Single Agent는 inter-agent 통신 오버헤드 없음, 그러나 Skill 직렬 호출 시 병렬성 잃을 수 있음
- Accuracy *(원본)* — Multi-Agent의 역할 분리가 추론 품질에 유리한 task가 있음(예: Planner vs Executor 분리)
- Cost *(원본)* — Multi-Agent는 동시 다중 LLM 호출로 token 사용량 증가, Single Agent는 압축적이나 한 번에 더 큰 컨텍스트 필요
- 확장성·표준 적합성 *(정제 시 추가)* — A2A 프로토콜은 Agent 간 통신을 가정하므로, 외부 인터페이스만큼은 "Agent" 추상이 보장되어야 함

### 관련 NFR·QAS

- **NFR04** (A2A 메시지 규약 준수) — [`05-NFR.md`](05-NFR.md) 버전 B
- **NFR17** (Intent / Orchestration / Negotiation / Memory / Security 모듈 경계 명확성) — [`05-NFR.md`](05-NFR.md) 버전 B
- **QA-001** (A2A 협상 완료 시간 최소화, 우선순위 1), **QA-002** (동적 Agent/Tool 실행 성공률), **QS-002**, **QS-017**

### Open Questions

- 본 결정은 다른 모든 DD에 영향을 주는 **루트 결정**임. 다른 DD를 먼저 결정한 뒤 본 DD로 돌아오면 모순이 생길 수 있으므로, 본 DD를 먼저 잠정 결정한 뒤 진행하는 것이 옳은가?
- "Single Agent + Skill"을 채택할 경우, 본 과제가 차별화 가치로 내세운 "NPN 비전"(여러 Agent의 자율 협상)이 외부 단일 Agent 입장에서는 표면상 같아 보일 수 있음 — 비전 서사가 약화되지 않는가? (전략적 판단 필요)

## DD-04. 양측 IDS가 동시에 협상을 open했을 때, 두 세션을 하나로 머지할 수 있는가?

### 맥락

상호 협상이 필요한 컨텍스트(예: 같은 카카오톡 대화에서 여행 계획 논의)에서 A의 IDS와 B의 IDS가 **동시에** 협상 필요성을 감지하면, 각자 상대방을 향해 협상 세션을 open하려 함. 이때 두 개의 세션이 생기면 (a) 자원 낭비, (b) 상태 불일치, (c) 합의안 충돌 위험이 발생함. 따라서 두 세션을 어떻게 동일 협상으로 식별·머지할지, 혹은 애초부터 하나만 생성되게 할지가 결정되어야 함.

### 핵심 질문

> A·B 두 사용자의 IDS가 동시에 동일 컨텍스트로 Intent를 발생시켰을 때, 양측이 각자 open한 협상 세션을 **하나의 협상으로 식별·머지**할 수 있는 구조를 어떻게 설계할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — 원본 하위 질문에서 도출)*

- **컨텍스트 키 기반 dedup** — 양측이 협상 open 시 컨텍스트 fingerprint(예: 대화 ID 해시, intent 카테고리, 시간 window)를 advertise하고 동일 키 검출 시 후-open한 쪽이 자동 join
- **우선권 규칙(Master 지정)** — 먼저 open한 쪽이 master, 늦게 open한 쪽은 자동으로 join (timestamp tie-break 규칙 필요)
- **N:N 대등 모델** — 두 세션을 그대로 두되, 합의안 단계에서 충돌 감지·해소

**하위 질문 (Sub-questions)**

- 각 Device에서 Intent를 각자 발생시킬 텐데, 어떻게 하나로 머지할 것인가? *(원본)*
- Master 역할자를 지정할 것인가, 아니면 N:N 대등으로 갈 것인가? *(원본)*
- 머지를 위한 "동일 협상"의 정의(컨텍스트 키)는 무엇인가? *(정제 시 추가)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 프라이버시 — 컨텍스트 키 자체가 PII가 되어서는 안 됨 (예: 대화 원문 해시 대신 의미 카테고리 해시)
- 표준 적합성 — A2A 프로토콜에 dedup 핸드셰이크 단계가 존재하는지, 없다면 우리 쪽 확장이 필요한지
- 실패 모드 — dedup 실패로 두 세션이 동시에 진행되는 경우의 일관성 보장

### 관련 NFR·QAS

- **NFR04** (A2A 메시지 규약 준수), **NFR-18** (협상 프로토콜·capability schema 표준화·버전관리) — [`05-NFR.md`](05-NFR.md) 버전 B/C
- **NFR08** (결과 일관성 — 중복·누락 없는 처리)
- **QS-002** (A2A 협상 완료 시간)

### Open Questions

## DD-05. 가전기기(DPA)의 데이터를 누가·어디서·어떻게 처리할 것인가?

### 맥락

[`02-과제-개요.md`](02-과제-개요.md) §2.3에서 DPA는 "해당 기기의 상태·스케줄 정보를 제공"하는 역할만 명시되어 있음. 그러나 DPA가 보유한 데이터(세탁기 사용 시간 패턴, TV 시청 이력, 냉장고 식재료 목록 등)는 모두 민감 정보임. 이 데이터의 raw form을 **누가** 보유하고, **누구의 요청**에 응답하며, **어떤 가공**을 거쳐 외부(서버 LLM·상대 PPA)로 흘러나갈지의 경계가 모호함.

### 핵심 질문

> DPA가 보유한 raw data의 **저장 위치**(가전 자체 / 폰의 PPA / 서버), **추상화 시점**(가전 측 / 폰 측 / 서버 측), **노출 범위**(PPA만 / 협상 메시지 / 서버 LLM)를 어떻게 분리·설계할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — 원본은 고려사항만 제시)*

- **(A) DPA-Owned Raw, PPA가 Pull** — Raw data는 가전이 보유, PPA가 협상 시점에 요청하여 가져옴. 가공은 PPA에서 수행.
- **(B) DPA Side 추상화 + Push** — 가전이 미리 요약·추상화한 컨텍스트를 PPA로 push, raw data는 가전 외부로 나가지 않음.
- **(C) PPA 측 캐시 + DPA는 Source of Truth** — 폰 측에 PPA가 가전 컨텍스트의 캐시를 보유, 동기화는 주기적·이벤트 기반.

**하위 질문 (Sub-questions)** *(정제 시 추가)*

- 가전 측의 컴퓨팅 자원이 추상화 처리를 감당할 수 있는가? (저성능 임베디드 가전 케이스)
- 가전이 OFF·offline 상태일 때 PPA가 협상에 필요한 정보를 어떻게 보장할 것인가? (캐시 vs Pull 차이)

**고려사항 (Considerations)**

- 개인정보·보안 — raw data의 도달 범위 최소화 *(원본)*
- 응답 시간 — Pull은 가전 응답 지연이 협상 critical path에 들어옴 *(정제 시 추가)*
- 일관성 — 캐시 모델은 stale data 위험 *(정제 시 추가)*
- 가전 H/W 제약 — 가전마다 자원 수준이 다르므로 일률적 위임이 불가 *(정제 시 추가)*

### 관련 NFR·QAS

- **NFR11** (최소권한 접근제어), **NFR12** (최소 데이터 보관 + 보관 기간 정책) — [`05-NFR.md`](05-NFR.md) 버전 B

- **QS-019** (협상 제안 내 개인/기기 정보 최소화), **NFR-006** (협상 제안 내 불필요 정보 노출 차단, `R_unnecessary_info = 0%`)

## DD-06. 협상 메시지에 각 device의 민감정보가 노출되지 않도록 어떻게 처리할 것인가?

### 맥락

A2A 협상은 본질적으로 양측이 "협상에 필요한 정보"를 교환하는 행위임. 그러나 협상에 도움이 되는 정보일수록 민감한 정보일 가능성이 높음(예: 일정·위치·선호·구매 한도 등). NFR-006은 협상 메시지 내 불필요 정보 노출을 `R_unnecessary_info = 0%`로 요구함. 따라서 "협상이 성립하기 위한 최소 정보"와 "노출되어서는 안 되는 정보"의 경계를 어떻게 그을지, 그리고 그 경계를 매 라운드 어떻게 강제할지가 결정 대상임.

본 결정은 [`docs/DP 후보안/QS-002-프라이버시-보존-협상-속도.md`](DP%20후보안/QS-002-프라이버시-보존-협상-속도.md)에서 이미 상세 분석이 진행 중이므로, **본 항목은 그 분석의 결정 요약 위치**로 사용함.

### 핵심 질문

> 협상이 성립하기 위한 정보 교환을 보장하면서, 각 device의 민감 정보(원문·기기 식별자·세부 조건)가 상대 PPA에 노출되지 않도록 **구조적으로 0%** 보장하는 게이트를 어디에·어떤 형태로 둘 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — DP 후보안 문서의 CA-DP1/2/3 후보 구조에서 도출)*

- **인라인 동기 게이트** — A2AProtocolAdapter 내부에 동기 프라이버시 필터를 두어 매 호출 시 강제 통과
- **사전 컴파일된 안전 범위 + 가벼운 검증** — 협상 시작 시 노출 가능한 변수 집합을 한 번에 추출, 라운드 중에는 그 범위 안에서만 제안 생성 (DP-4)

**하위 질문 (Sub-questions)** *(정제 시 추가)*

**고려사항 (Considerations)**

## DD-07. 전체 Task가 신뢰성 있게 완료되도록 어떤 구조로 보장할 것인가?

### 맥락

사용자 Intent 하나는 IDS → Orchestrator → Meta Agent → Sub-Agent → 외부 API의 다단계 비동기 흐름으로 처리됨. 어느 단계에서든 실패가 가능함 — LLM 추론 실패, 네트워크 단절, 상대 PPA 미응답, 외부 API 오류, 프로세스 종료. 이때 (a) 부분 진행 상태의 일관성, (b) 실패 격리, (c) 재개 가능성, (d) 외부 부작용의 idempotency를 어떻게 보장할지가 결정 대상임.

### 핵심 질문

> 다단계 비동기 Task가 어느 단계에서 실패하더라도, **부분 완료 상태의 일관성**·**재개 가능성**·**보상(rollback) 가능성**을 어떻게 구조적으로 보장할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가)*

- **단계별 Checkpoint + Persistent Session State** — 각 단계 경계에서 상태를 영속화, 실패 시 마지막 checkpoint에서 재개
- **Saga 패턴** — 각 단계의 보상 트랜잭션(compensation)을 정의, 실패 시 역방향으로 보상 실행
- **Idempotent 외부 호출 + 재시도** — 모든 외부 API 호출에 idempotency key 부여, 실패 시 안전한 재시도
- 위 세 가지의 단계별 조합 — 협상 단계는 Checkpoint, 외부 실행 단계는 Saga 또는 Idempotent retry

**하위 질문 (Sub-questions)**

- 신뢰성·장애처리 등의 고려가 필요함 *(원본 — 어떤 실패 모드를 다룰지의 명세가 필요)*
- "Task 단위"의 경계는 어디인가? — Intent 1건 = Task 1건인가, 아니면 협상 1건·실행 1건으로 분리되는가? *(정제 시 추가)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 외부 부작용의 비가역성 — 결제·예약 같이 보상이 어려운 행동의 시점은?
- Resume의 데이터 신선도 — 오래된 세션을 재개할 때 외부 컨텍스트가 변경되었을 수 있음
- 사용자 인지 — 장시간 진행 중인 Task의 진행 상태·실패 알림 방식

### 관련 NFR·QAS

- **NFR06** (세션 복구 가능성), **NFR07** (장애 격리성), **NFR08** (결과 일관성) — [`05-NFR.md`](05-NFR.md) 버전 B

## DD-08. 협상에서 Token 사용량을 어떻게 효율화할 것인가?

### 맥락

A2A 협상은 매 라운드 LLM 추론을 수반하므로 token 사용량이 N(라운드 수) × 라운드당 추론 호출 수로 누적됨. [`05-NFR.md`](05-NFR.md) NFR26은 서버 LLM 호출의 token 사용량을 사전 정의된 예측 모델·상한선 안에서 동작시키도록 요구함. 동시에 token 효율화는 협상의 응답 시간(QA-001 최우선)과 trade-off 관계임.

### 핵심 질문

> 협상 품질·응답 시간을 만족하면서, server LLM 호출 빈도와 token 사용량을 NFR26의 상한선 안에서 어떻게 관리할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가)*

- **라운드 수(N) 최소화** — 라운드당 추론을 약간 무겁게 하더라도 한 번에 합의 가능성이 높은 제안 생성 (DP 후보안 DP-3)
- **컨텍스트 압축·요약** — 라운드 history를 누적 전송하지 않고 직전 결정 요약만 컨텍스트로 사용
- **Server LLM ↔ On-device LLM 분배** — 단순 라운드는 on-device, 복잡 합의 추론만 server로 escalate
- **결과 캐싱** — 이전 협상에서 학습된 사용자 선호·합의 패턴을 캐시하여 신규 협상의 초기 제안 품질 향상
- **Prompt template 표준화** — 협상 메시지 schema를 정형화하여 매 라운드 자유 자연어 대신 구조화된 표현으로 token 절약

**하위 질문 (Sub-questions)** *(정제 시 추가)*

- Token 사용량의 사전 예측 모델은 무엇을 입력으로 받아 작동하는가? (도메인·라운드 수·사용자 컨텍스트)
- 임계치 초과 시 throttle / escalation 중 어느 동작이 사용자 경험을 덜 해치는가?

**고려사항 (Considerations)**

- Latency 도 함께 고려해야 함 *(원본)*
- 협상 품질 — 압축·캐싱이 합의 품질에 영향을 줄 수 있음 *(정제 시 추가)*
- 비용·품질·속도의 3원 trade-off *(정제 시 추가)*

### 관련 NFR·QAS

- **NFR26** (서버 LLM 호출 비용 통제) — [`05-NFR.md`](05-NFR.md) 버전 B
- **QA-001** (A2A 협상 완료 시간 최소화, 우선순위 1)
- **QS-002**, **QS-017** (Orchestrator 계획 수립 지연)

## DD-09. 협상이 취소·변경될 때 어떻게 처리할 것인가?

### 맥락

협상은 [`02-과제-개요.md`](02-과제-개요.md) §2.6 시나리오 7단계 중 어디서든 사용자에 의해 취소·조건 변경될 수 있음. 또한 시스템 측 사유(상대 미응답, 네트워크 단절 등)로도 중단됨. 이때 (a) 진행 중인 라운드를 어떻게 안전히 멈출지, (b) 부분 발생한 외부 부작용을 어떻게 처리할지, (c) 중간 상태 정보를 보관할지 삭제할지, (d) 재시작 시 어디까지 복원할지를 결정해야 함.

본 DD는 DD-07(전체 Task 신뢰성)의 부분 집합이지만, "사용자 의도로 인한 중단"이라는 특수성 때문에 별도 결정으로 분리되어 있음.

### 핵심 질문

> 협상이 사용자·시스템 사유로 중단되었을 때, **중단 메시지의 전달**·**부분 부작용의 정리**·**상태 정보의 보관/삭제**·**재시작 시 복원 범위**를 어떻게 정책으로 정의할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가 — 원본 하위 질문의 답안 후보 형태)*

- **즉시 강제 중단 + 상대에 cancellation 메시지 의무화** — 진행 중인 라운드는 폐기, 상대측 자원도 즉시 해제
- **부분 commit + 트랜잭션 경계 분리** — 단순 조회·discovery는 즉시 commit, 비가역 행동(결제·예약)은 final approval 전까지 gated
- **세션 스냅샷 + 만료 정책** — 중단 시 세션 상태를 일정 시간 보관(예: 24h), 사용자가 재개 의사 표시 시 복원, 만료 후 자동 삭제
- **변경 시 재협상 vs 부분 갱신** — 조건 변경 시 처음부터 재협상 / 변경된 부분만 patch 전송

**하위 질문 (Sub-questions)**

- 어떻게 중단시킬 수 있고, 중단 후 정보의 삭제/보관은 어떻게 할 것인지? *(원본)*
- 재시작한다면 복원을 어떻게 할 것인지? *(원본)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 상대 PPA 측 자원 회수 — 우리 쪽만 중단해서는 상대 자원이 남을 수 있음
- 보관 기간 정책 — NFR12와의 정합 (중간 상태도 민감 정보)
- 사용자 UX — 중단·재개 가능성을 사용자가 인지·제어할 수 있는 UI 동선

### 관련 NFR·QAS

- **NFR-12** (graceful fallback), **NFR-13** (partial result + resume), **NFR-15** (task cancellation·rollback) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR12** (최소 데이터 보관 + 보관 기간 정책) — [`05-NFR.md`](05-NFR.md) 버전 B
- **QS-016** (Android 프로세스 종료 후 복구), **QS-023** (상대방 PPA 장기 미응답 세션 만료)
- DD-07과 강하게 결합됨 — 함께 결정해야 일관성 확보 가능.

### Open Questions

- "조건 변경 시 재협상 vs 부분 갱신"은 A2A 프로토콜 표준에 명세된 동작이 있는가, 아니면 우리 쪽 확장이 필요한가?
- 사용자가 명시적으로 취소하지 않고 단순히 "오랜 시간 응답 없음" 상태일 때, 시스템이 자동 취소하는 임계 시간을 사용자 설정 가능한 항목으로 둘 것인가, 고정값으로 둘 것인가?

## DD-10. IDS의 백그라운드 monitoring을 어떻게 효율적으로 수행할 것인가?

### 맥락

IDS는 [`02-과제-개요.md`](02-과제-개요.md) §2.3에 따라 "기기 활동 전체를 모니터링하여 Intent를 감지·라우팅"하는 역할임. 즉 사용자가 명령하지 않아도 백그라운드에서 지속적으로 동작해야 함. 그러나 NFR-003(`B_delta < 2%/h`)·NFR-004(`M_idle < 500MB`)는 백그라운드 자원 사용에 엄격한 상한을 둠. 또한 NFR-1은 intent detection의 실시간/준실시간 동작을 요구함 — 즉 "충분히 자주 보면서도 충분히 가벼워야" 한다는 양립 조건임.

### 핵심 질문

> 실시간/준실시간 Intent 감지(NFR-1)를 만족하면서, 백그라운드 메모리·배터리·CPU를 NFR-003·004의 상한 안에서 어떻게 통제할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가)*

- **이벤트 기반 sampling** — 전수 모니터링 대신 OS의 활동 이벤트(앱 전환, 메시지 수신, 캘린더 변경 등)에 hook을 걸어 trigger 시점만 추론
- **2단계 분류기 (Cascade)** — 1단계는 lightweight on-device 분류기(rule·작은 모델)로 의심 활동만 필터, 2단계만 full LLM 호출
- **Foreground service의 batching·sleep** — 일정 시간 활동이 없으면 sleep, 누적된 이벤트를 batch로 처리
- **사용자 컨텍스트 기반 적응형 sampling** — 사용자가 활동 중일 때만 frequency 증가, 야간·휴면 시간은 frequency 감소

**하위 질문 (Sub-questions)** *(정제 시 추가)*

- "Intent 감지"는 모든 사용자 활동에 대해 수행되어야 하는가, 아니면 특정 트리거 시점에만 수행되면 충분한가?
- On-device LLM 호출과 lightweight 분류기를 어떻게 분리할 것인가? — DD-03(Single vs Multi-Agent)과의 연관.

**고려사항 (Considerations)**

- 전력, 메모리 등을 고려 *(원본)*
- 응답 시간 — sampling 간격이 길수록 Intent 감지 지연이 커짐
- 사용자 프라이버시 — 모니터링 범위와 대상의 명시적 동의 *(정제 시 추가)*

### 관련 NFR·QAS

- **NFR-1** (실시간/준실시간 intent detection), **NFR-3** (협상 세션 latency), **NFR-4** (CPU/NPU/메모리), **NFR-5** (배터리 소모) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR-003** (`B_delta < 2%/h`), **NFR-004** (`M_idle < 500MB`) — [`06-QAS.md`](06-QAS.md) §6.3
- **QS-001** (Intent 감지 지연 시간), **QS-003** (백그라운드 모니터링 메모리 사용량), **QS-021** (백그라운드 배터리 소비)
- **QA-003** (백그라운드 배터리 소비 최소화, 우선순위 3), **QA-005** (Intent 감지 지연 최소화, 우선순위 5)

### Open Questions

- "전수 모니터링"에 대한 사용자 동의·인지 모델은 어떻게 둘 것인가? — 일반 사용자가 명확히 이해하기 어려운 영역.
- Galaxy S26의 온디바이스 LLM 3종(Compact/Balanced/Supreme) 중 IDS에 적합한 변종은 무엇인가? — H/W·전력 곡선과의 정합.

## DD-11. 다중 Intent가 동시에 발생할 때 어떻게 처리할 것인가?

### 맥락

사용자는 동시에 여러 Intent를 발생시킬 수 있음(예: 카카오톡 여행 일정 논의 + 일정 등록 알림 + 가전 A/S 예약 동시 발생). 또한 한 Intent를 처리하는 중간에 새 Intent가 들어올 수도 있음. IDS·Orchestrator가 이를 어떻게 큐잉·우선순위 부여·병렬 실행할지, 자원 한도를 초과하지 않게 어떻게 조절할지가 결정 대상임.

### 핵심 질문

> 다중 Intent의 **큐잉 정책**·**우선순위 결정 기준**·**병렬 실행 한도**·**자원 충돌 감지·해소**를 어떻게 정의할 것인가?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가)*

- **FCFS (First-Come-First-Served)** — 단순 큐잉, 도착 순서대로 직렬 처리
- **우선순위 큐** — 위급도·사용자 명시·시간 민감도 기반 정렬
- **병렬 실행 + 자원 한도** — 동시 처리하되 Sub-Agent 수의 상한·메모리 상한 강제 (DD-12와 연계)
- **Pre-emption 허용** — 새 Intent가 더 높은 우선순위면 진행 중 task를 일시 정지

**하위 질문 (Sub-questions)**

- 동시에 여러 Intent가 들어올 때, 하나의 Intent 처리 중에 다른 것이 들어올 때 등 *(원본)*
- 우선순위 기준을 시스템이 추론할 것인가, 사용자에게 묻을 것인가? *(정제 시 추가)*
- 동일 외부 자원(예: 같은 캘린더 슬롯)을 두 Intent가 동시에 점유하려 할 때 어떻게 해소할 것인가? *(정제 시 추가)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 사용자 인지 — 백그라운드 다중 처리 결과가 한꺼번에 알림으로 쏟아지면 침습적 (NFR15 비침습성)
- 자원 한도 — DD-12와 결합 시 Sub-Agent life cycle 정책이 다중 처리 throughput에 영향

### 관련 NFR·QAS

- **NFR-6** (동적 생성 agent 수·수명 제어), **NFR-14** (agent runaway·deadlock·infinite negotiation 방지) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR19** (Agent 수 증가 대응) — [`05-NFR.md`](05-NFR.md) 버전 B
- **NFR15** (선제 제안의 비침습성)
- **QS-004** (동시 Agent 실행 최대 메모리 사용량)

### Open Questions

- 사용자 명시 Intent(직접 명령)와 IDS 추론 Intent(묵시적 감지) 간에 기본 우선순위 차이를 둘 것인가?
- "Pre-emption"은 일시 정지된 task의 상태 일관성을 깨뜨릴 위험이 있음 — 어느 단계에서만 pre-emption을 허용할 것인가?

## DD-12. Sub-Agent의 Life Cycle을 어떻게 관리할 것인가?

### 맥락

Sub-Agent는 도메인별로 동적 생성되어 Tool 호출을 수행함([`02-과제-개요.md`](02-과제-개요.md) §2.3). Task가 끝난 뒤 Sub-Agent를 즉시 해제할지, 일정 시간 메모리에 유지하여 재사용할지의 정책이 (a) 자원 사용량(NFR-004), (b) 후속 task의 응답 시간, (c) 다중 Intent 처리의 throughput에 모두 영향을 줌. DD-11과 결합되어 시스템 전체의 자원·반응성 특성을 결정함.

### 핵심 질문

> Sub-Agent를 **언제** 생성하고, **얼마나** 유지하며, **어떤 기준**으로 해제할 것인가? 풀(pool)을 둔다면 풀 크기·evict 정책은?

### 식별된 선택지 / 하위 질문 / 고려사항

**선택지 (Options)** *(정제 시 추가)*

- **즉시 해제 (No-Pool)** — Task 완료 즉시 Sub-Agent 자원 회수. 메모리 우선, 재사용성 없음
- **Idle Timeout 후 해제** — Task 완료 후 N분 idle 상태로 유지, 그 시간 안에 같은 도메인 task가 오면 재사용
- **LRU Pool** — 최대 K개 Sub-Agent를 유지하는 풀, 새 도메인 task가 오면 가장 오래 idle한 것을 evict
- **협상-aware 정책** — 협상 중인 Sub-Agent는 협상 종료 + N분까지 유지, 그 외는 즉시 해제

**하위 질문 (Sub-questions)**

- Sub Agent를 언제까지 메모리에 올려둘지 *(원본)*
- "재사용"이 성립하기 위한 동일성 기준은 무엇인가? — 동일 도메인이면 충분한가, 동일 사용자 컨텍스트도 필요한가? *(정제 시 추가)*

**고려사항 (Considerations)** *(정제 시 추가)*

- 메모리 vs 응답 시간 trade-off
- Sub-Agent의 컨텍스트 상태가 누적될 경우의 보안 — 이전 task의 정보가 다음 task에 누출되지 않도록 격리
- 시스템 stress 시의 동작 — 메모리 부족 시 풀이 자동 축소되는가

### 관련 NFR·QAS

- **NFR-6** (동적 생성 agent 수·수명 제어), **NFR-4** (CPU/메모리), **NFR-5** (배터리 소모) — [`05-NFR.md`](05-NFR.md) 버전 C
- **NFR-004** (`M_idle < 500MB`) — [`06-QAS.md`](06-QAS.md) §6.3
- **QS-004** (동시 Agent 실행 최대 메모리 사용량)
- DD-11과 강하게 결합됨 — 동시 Intent 처리량 상한이 Sub-Agent 풀 크기의 상한과 직접 연결됨.

### Open Questions

- 재사용 시 Sub-Agent 내부의 LLM 컨텍스트(memory)를 reset할 것인가, 유지할 것인가? — 유지하면 성능 향상, reset하면 안전성 향상.
- "협상-aware 정책"을 채택한다면, 협상 종료 후 유지 시간 N은 어떻게 결정할 것인가? — 사용자 별·도메인 별로 다를 수 있음.

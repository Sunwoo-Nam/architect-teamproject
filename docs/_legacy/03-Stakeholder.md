> **VOC(Voice of Customer)** 는 각 stakeholder가 **현실에서 제기할 수 있는 발언**을 1인칭으로 기록한 것임.
> 원본의 "관심사" 항목이 있는 경우는 그것을 VOC 형태로 풀어낸 것이며, 추가 항목은 비즈니스 드라이버·과제 성격을 근거로 합리적으로 추정한 것임. 추정 항목은 *italic 표시*로 구분함.

## 3.2 Stakeholder별 설명 및 VOC

### 3.2.1 End User

**역할.** 자신의 일상·업무 조율을 Agent에 위임하는 본인 사용자. 본 플랫폼이 만들어내는 가치의 1차 소비자임.

**VOC.**
- "내가 일정 잡으려고 메시지를 다섯 번 주고받는 그 시간이 너무 아까움."
- "내 폰에서 다 처리되면 좋겠음. 내 일정·내 메시지가 외부 서버로 새는 건 절대 싫음."
- "Agent가 마음대로 결정해 버리는 게 제일 무서움. 마지막엔 내가 승인하고 끝낼 수 있어야 함."
- "결과만 좋다고 다는 아님. 왜 그 안을 골랐는지 한 줄이라도 이해되어야 신뢰가 감."
- *"오늘 컨디션이 안 좋으니 잠시 자율 모드는 꺼 두고 싶음."*

> 근거: 원본 표 (편의성, 결과 만족도, 개인정보 보호, 최종 통제권)

---

### 3.2.2 Counterparty User

**역할.** 상대편 사용자. 내 Agent와 협상하는 상대방의 Agent를 가진 사람임. 본 시스템의 직접 사용자는 아니지만, 시스템 가치는 양쪽 모두가 안전하다고 느낄 때 성립함.

**VOC.**
- "내 일정을 통째로 보여주고 싶지 않음. 협상에 필요한 만큼만 공유되어야 함."
- "내가 원치 않는 결정을 상대 Agent가 일방적으로 밀어붙이는 일은 없어야 함."
- "협상이 어떻게 흘러갔는지 사후에 확인할 수 있어야 마음이 놓임."
- *"내가 거절하거나 조건을 바꾸겠다고 할 때 즉시 반영되어야 함."*

> 근거: 원본 표 (자신의 일정 및 선호 노출 최소화, 협상 과정 신뢰성, 수정/거절 권한 보장)

---

### 3.2.3 Service Proxy Agent Owner (서비스 사업자)

**역할.** 예약·구매 대상 서비스를 제공하는 사업자(미용실·식당·숙소·커머스 등). SA의 실 운영자임.

**VOC.**
- "신원이 검증된 요청만 받고 싶음. 봇이 무차별로 들어오면 운영이 마비됨."
- "현재 우리 가게의 정책(가격·시간·노쇼 페널티 등)이 그대로 반영되어야 함."
- "Agent를 통해 들어오는 예약이 실제 결제와 방문으로 이어져야 의미 있음."
- *"기존에 쓰는 예약 시스템과 충돌 없이 연동되어야 함. 우리 시스템을 갈아엎으라는 요구는 받기 어려움."*

> 근거: 원본 표 (인증된 요청만 수신, 운영 조건 반영, 예약 전환율, 기존 시스템 연동)

---

### 3.2.4 Project Leader

**역할.** 본 연구 과제의 방향·일정·예산을 책임짐. 과제 외부에 가치와 성과를 설명해야 함.

**VOC.**
- "이 연구가 끝났을 때 외부에 자랑할 수 있는 새로운 사용자 가치가 있어야 함."
- "9개 시나리오를 모두 다 만들 수는 없음. 어느 것이 우리 차별화의 핵심인지부터 정해야 함."
- "기술적으로 멋있는 것 말고, 사용자가 정말 다르게 느끼는 UX가 있어야 함."
- *"비용 대비 검증 가치가 명확해야 의사결정자에게 설명할 수 있음."*

> 근거: 원본 표 (신규 사용자 가치, 시나리오 실현성, UX 차별화, 시장성, 비용)

---

### 3.2.5 Architect

**역할.** 시스템 구조를 설계하는 책임자. 본 저장소의 1차 사용자이기도 함.

**VOC.**
- "Intent / Orchestration / Negotiation / Memory / Security 다섯 모듈 사이의 경계가 흐릿하면 PoC도 운영 단계로 못 감."
- "Agent 종류·도메인이 늘어났을 때 기존 코드에 손을 거의 안 대고도 확장될 수 있어야 함."
- "PoC라도 변경 영향이 추적 가능해야 함. 한 군데 고쳤더니 다른 곳이 깨지는 상황을 만들면 안 됨."
- *"Mock으로 협상 흐름 전체를 재현할 수 있어야 검증과 비교 실험이 가능함."*

> 근거: 원본 표 (모듈화, 확장성, 유지보수성, 테스트 용이성)

---

### 3.2.6 Multi-agent Framework Developer

**역할.** Orchestrator·Meta Agent·Sub-Agent의 실제 구현 담당자.

**VOC.**
- *"동적 생성된 Agent와 Tool이 sandbox 안에서 안전하게 격리 실행되어야 함."*
- *"실패한 LLM 호출·Tool 호출에 대해 재시도·대체·escalation 정책이 명확히 정해져야 코드를 단순하게 짤 수 있음."*
- *"Agent 간 메시지 형식과 lifecycle 상태 전이가 표준화되어야 함."*
- *"디버깅이 가능한 추적 로그가 필요함. LLM 비결정성 때문에 재현이 안 되면 손을 못 댐."*

> 근거: 원본 표(관심사 비어있음) + 본 과제 docx의 FR-L01~L07, FR-T01~T05, FR-X01~X03에서 추정

---

### 3.2.7 Third-party Sub-Agent Developer

**역할.** 외부에서 Sub-Agent를 만들어 본 플랫폼에 등록하는 개발자.

**VOC.**
- *"Agent 등록 절차와 권한 모델이 명확해야 함. 어떤 권한까지 줄 수 있는지 모르면 위험해서 못 만듦."*
- *"SDK가 안정적이고 버전 관리가 되어야 함. 매번 깨지는 SDK는 사용 불가."*
- *"내 Sub-Agent가 다른 Agent와 충돌·간섭 없이 격리되어 동작해야 함."*

> 근거: 원본 표(관심사 비어있음) + FR-L05(Sandbox), FR-S01(권한 위임)에서 추정

---

### 3.2.8 가전사 담당자

**역할.** DPA가 탑재되는 가전(세탁기·TV·냉장고 등)의 사업 책임자.

**VOC.**
- *"가전이 보유한 raw 데이터(에러 코드·사용 이력)가 외부로 그대로 새 나가면 안 됨."*
- *"가전 단위로 세분화된 권한 통제가 가능해야 함."*
- *"PPA와 가전 사이의 보안 채널이 표준화되어야 가전사들이 같은 모델을 따라갈 수 있음."*

> 근거: 원본 표(관심사 비어있음) + 시나리오 3·4(공용 가전 중재, A/S 협상)에서 추정

---

### 3.2.9 Security / Privacy 담당

**역할.** 권한·암호화·감사 정책 책임자.

**VOC.**
- "Agent는 위임받은 권한 범위 안에서만 행동해야 함. 단 한 건의 초과 행동도 용납 불가."
- "Agent 간 메시지는 암호화되어야 하고 변조도 감지·차단되어야 함."
- "누가 어떤 권한으로 무엇을 했는지 사후 감사가 가능해야 함."
- "민감 데이터는 가능한 한 온디바이스에서 처리되어야 함. 외부 서버로 보낼 때는 PII가 빠져야 함."
- *"동적 생성된 Agent도 동일한 보안 정책 검사 경로를 통과해야 함."*

> 근거: 원본 표 (최소 권한 원칙, 데이터 보호, 인증/인가, 감사 로그) + arch-with-ai의 NFR-001/002/005/006

---

### 3.2.10 AI/ML Engineer

**역할.** Intent 인식, 온디바이스 추론, Memory Personalization, Self-improving 모델을 책임짐.

**VOC.**
- "Intent 분류 품질이 떨어지면 그 뒤의 모든 Agent가 헛수고함. 정확도와 confidence score가 안정되어야 함."
- "온디바이스 LLM 모델을 다른 모델로 빠르게 교체할 수 있어야 함. 시장 변화 속도가 너무 빠름."
- "사용자 메모리·협상 이력이 모델 개선에 다시 들어가는 closed-loop가 있어야 self-improving이 됨."
- "메모리·CPU·NPU 사용량이 OS 정책 안에 들어가야 모델을 실제 폰에 올릴 수 있음."

> 근거: 원본 표 (intent 인식 품질, on-device 추론 가능성, memory personalization, self-improving 구조) + arch-with-ai QA-005, QA-006

---

### 3.2.11 UX 디자인 팀

**역할.** 사용자에게 협상 과정과 결과를 보여주는 화면, 승인 흐름을 설계함.

**VOC.**
- *"승인·거절·조건 수정이 최소 단계로 끝나야 함. 클릭 5번이면 사용자가 떠남."*
- *"협상 진행을 사용자가 보고 싶을 때 보고, 보기 싫을 때 안 봐도 되도록 해야 함."*
- *"비명시적 Intent 기반 선제 제안이 너무 자주 뜨면 침습적으로 느껴짐."*

> 근거: 원본 표(관심사 비어있음) + arch-with-ai NFR(사용성), QS-019/020에서 추정

---

### 3.2.12 운영 / SRE

**역할.** 운영 환경에서 장애 대응·세션 복원·모니터링 책임.

**VOC.**
- "장애가 나도 진행 중이던 협상 세션은 복구 가능해야 함. 사용자 입장에서 '갑자기 사라진' 경험은 치명적임."
- "한 Agent의 장애가 시스템 전체로 번지면 안 됨."
- "agent·task·tool 실행 상태를 실시간으로 관찰할 수 있어야 함."
- "로그가 privacy-safe 형태로 저장되어야 운영자가 들여다볼 수 있음."

> 근거: 원본 표 (장애 복구, 세션 복원, 모니터링, 로그 추적성) + arch-with-ai NFR-007

---

### 3.2.13 품질 검증팀

**역할.** 정량 지표 측정·테스트 자동화·재현 가능한 실험 환경 구축.

**VOC.**
- *"협상 성공률·평균 협상 턴 수·평균 응답시간·approval 전환율을 정량으로 측정할 수 있어야 함."*
- *"실제 상대 기기 없이 Mock PPA만으로 End-to-End 협상이 재현되어야 같은 실험을 반복할 수 있음."*
- *"실패 케이스(네트워크 단절·타임아웃·합의 실패)도 자유롭게 주입할 수 있어야 함."*

> 근거: 원본 표(관심사 비어있음) + arch-with-ai NFR-007(시뮬레이션), QS-014에서 추정

---

### 3.2.14 Legal / Compliance

**역할.** OSS 라이선스 적합성·개인정보 처리 적법성·예약/대행 행위의 책임 범위를 검토함.

**VOC.**
- "도입하는 OSS와 솔루션은 사내 사용 가능 라이선스 안에 있어야 함."
- "Agent가 사용자를 대신해 예약·결제 같은 행위를 할 때, 그 책임 소재가 약관과 시스템에 명확히 기록되어야 함."
- "개인정보를 어디서·얼마나·왜 처리하는지가 사용자에게 사전 고지되어야 함."
- *"고령자 등 민감 사용자에 대한 대리 권한 위임은 추가 검토가 필요함."*

> 근거: 원본 표 (OSS 라이선스, 개인정보 처리 적법성, 예약/대행 책임 범위) + 시나리오 8(부모-자녀 위임)

---

### 3.2.15 MX H/W 담당자

**역할.** Android 단말의 NPU·RAM·발열·배터리 자원을 책임짐.

**VOC.**
- *"백그라운드 모니터링이 Android Doze·OOM Killer를 정상적으로 통과해야 함. 안 그러면 OS가 우리를 끝장냄."*
- *"동시에 떠 있는 Agent 수와 메모리 점유에 상한이 있어야 함."*
- *"발열·배터리 소비가 OS 정책 안에 들어와야 사용자가 우리를 비활성화하지 않음."*

> 근거: 원본 표(관심사 비어있음) + arch-with-ai NFR-003/004(배터리·메모리 허용치)에서 추정

---

### 3.2.16 MX 상품화 의사결정 담당자 (ID 16, 첫 번째)

**역할.** 본 PoC 결과를 상품화로 이어갈지·범위를 어디까지 할지 결정함.

**VOC.**
- *"검증된 가설과 미검증 가설이 명확히 구분되어야 상품화 결정에 쓸 수 있음."*

## 3.3 핵심 관심사 매핑

영향도와 관심도가 동시에 높은 **Key Player**는 다음과 같음.

| 분면 | Stakeholder |
|---|---|
| 영향도 ↑ / 관심도 ↑ (Key Player) | End User · Architect · Multi-agent Framework Developer · Security/Privacy 담당 · AI/ML Engineer · 개인정보 보안 담당자 · Project Leader |
| 영향도 ↑ / 관심도 ↓ | Legal/Compliance · MX H/W 담당 · MX 상품화 의사결정 · Cloud 담당 |
| 영향도 ↓ / 관심도 ↑ | Counterparty User · Service Proxy Agent Owner · Third-party Sub-Agent Developer · 가전사 담당자 · UX 디자인 팀 · 운영/SRE · 품질 검증팀 |

설계 단계에서는 **Key Player의 VOC를 우선 만족**시키되, 다른 분면의 VOC도 트레이드오프 분석 시 누락하지 않도록 함.

## 0. 풀고자 하는 문제

서로 다른 사용자의 디바이스에 있는 에이전트들이 약속 일정 조율, 식사 장소 선정 등 **공동 합의가 필요한 태스크**를 수행할 때:

1. **"어느 디바이스의 LLM이 결정을 내려야 하는가"** — 중앙 LLM에 모든 정보를 던지면 프라이버시 침해 + 토큰 낭비 + 일관성 저하
2. **"원본 개인 데이터를 노출하지 않고 합의에 도달할 수 있는가"** — 선호도·일정 등 PII를 상대 디바이스에 그대로 전송하면 NFR12, QS-019 위반
3. **"N명이 참여할 때 합의 상태가 분산 환경에서 일관되게 수렴하는가"** — split-brain, 교착, 무한 루프 방지 (NFR-14)

---

## 1. 핵심 설계 목표

- **LLM 추론은 로컬 우선**: 개인 데이터 해석·판단은 각 디바이스 온디바이스 LLM이 담당
- **외부로 나가는 것은 추상화된 결과물만**: 원본 데이터가 아닌 제약 벡터, yes/no/counter 응답만 A2A 전송
- **Coordinator 역할 최소화**: 협상 개시자 디바이스가 Coordinator. 가능하면 LLM 미사용·결정론적 연산 권장 (Sol.2는 예외적으로 Coordinator도 LLM 사용)

---

## 2. 핵심 긴장 관계 (Trade-off Map)

| 긴장 축 | 한 쪽 | 반대 쪽 |
|---|---|---|
| 정확도 vs 프라이버시 | 더 많은 데이터 공유 → 더 나은 합의 | 원본 PII 노출 최소화 (NFR09·NFR12·QS-019) |
| 지연 vs 일관성 | 라운드 수 줄이기 | 분산 환경 합의 정합성 (QA #4) |
| 유연성 vs 토큰 효율 | LLM에게 자유 협상 위임 | 토큰 버짓 통제 (NFR26) |
| 단순성 vs 장애 내성 | 단일 Coordinator | SPOF 없는 구조 (NFR07) |

### Sol.1 — Structured Preference Vector + Deterministic Solver

각 디바이스가 로컬 LLM을 통해 사용자 선호를 정규화된 벡터로 변환하고, Coordinator가 LLM 없이 결정론적 알고리즘(Borda Count, 가중합)으로 최적 결과를 산출하는 방식이다. 단 1라운드 통신으로 협상이 완료되어 지연과 토큰 비용이 최소화된다. 대신 참여자에게 결과 거부권이 없고, 선호 벡터 스키마로 표현할 수 없는 맥락적 조건("이번 주만 특별히...")은 협상에서 누락된다.

> 📐 설계도: [DP08-sol1.drawio](참고자료/DP08-sol1.drawio)

**구조:**
```
[각 디바이스 Local LLM]
  원본 선호도 → 정규화된 선호 벡터 생성
  예: {시간대: [0.9, 0.3, 0.8], 음식: {한식: 0.8, 양식: 0.4}}
      ↓ A2A (벡터만 전송)
[Coordinator — LLM 미사용]
  결정론적 알고리즘으로 최적 후보 선택
  예: Borda Count, 가중합 스코어링
      ↓
[전체 참가자에게 결과 브로드캐스트]
```

**워크플로우 상세:**

1. **협상 개시**: Initiator 디바이스의 Personal Proxy Agent가 협상 주제(도메인, 선택지 목록, 참여자 목록)를 A2A로 브로드캐스트
2. **선호 벡터 추출**: 각 참여자 디바이스에서 Local LLM이 사용자의 원본 선호도·일정을 읽어 도메인 스키마에 맞는 정규화된 선호 벡터 생성 — 원본 데이터는 기기 밖으로 나가지 않음
3. **벡터 전송**: A2A Transport가 선호 벡터만 Coordinator(Initiator 디바이스)로 암호화 전송
4. **결정론적 집계**: Coordinator의 Deterministic Solver가 수신된 벡터들에 Borda Count 또는 가중합 알고리즘 적용 → 최적 후보 산출 — LLM 미호출
5. **결과 브로드캐스트**: 산출된 결과를 모든 참여자에게 전송
6. **통보**: 각 디바이스 Personal Proxy Agent가 결과를 사용자에게 제시 — 참여자에게 별도 거부권 없음

**예시 (3인 식사 약속 — Alice 협상 개시, 참여자: Bob·Carol):**

```
[Step 1] Alice Personal Proxy Agent → 브로드캐스트
  Agenda: { 도메인: "식사 약속", 선택지: ["토요일", "일요일"] × ["18시", "19시", "20시"] × ["한식", "양식", "일식"] }

[Step 2] 각 디바이스 Local LLM → 선호 벡터 생성 (기기 내부에서만)
  Alice  원본: "이번 주 토요일 저녁 한식 먹고 싶어, 일요일도 괜찮긴 해"
         벡터: { 요일: {토: 0.9, 일: 0.6}, 시간대: {18시: 0.8, 19시: 0.6, 20시: 0.3}, 음식: {한식: 0.9, 양식: 0.3, 일식: 0.6} }

  Bob    원본: "일요일 저녁 늦게가 편해, 음식은 뭐든 OK"
         벡터: { 요일: {토: 0.3, 일: 0.9}, 시간대: {18시: 0.4, 19시: 0.6, 20시: 0.9}, 음식: {한식: 0.5, 양식: 0.7, 일식: 0.7} }

  Carol  원본: "토요일 18-19시가 제일 좋고 한식 선호"
         벡터: { 요일: {토: 0.8, 일: 0.6}, 시간대: {18시: 0.9, 19시: 0.7, 20시: 0.2}, 음식: {한식: 0.8, 양식: 0.5, 일식: 0.4} }

[Step 3] A2A Transport → 벡터 3개 Coordinator(Alice 디바이스)에 전송

[Step 4] Deterministic Solver → Borda Count 가중합 계산
  요일 합산: 토(0.9+0.3+0.8=2.0) vs 일(0.6+0.9+0.6=2.1) → 일요일 승
  시간대 합산: 18시(0.8+0.4+0.9=2.1) vs 19시(0.6+0.6+0.7=1.9) vs 20시(0.3+0.9+0.2=1.4) → 18시 승
  음식 합산: 한식(0.9+0.5+0.8=2.2) vs 일식(0.6+0.7+0.4=1.7) vs 양식(0.3+0.7+0.5=1.5) → 한식 승

[Step 5-6] 결과 브로드캐스트 → 전원에게 통보
  최종 합의: "일요일 18시, 한식"
  ※ 참여자 거부권 없음 — Bob이 원하던 20시와 다르지만 알고리즘 결과로 확정
```

**모듈 설명:**

| 모듈 | 위치 | 역할 | 입력 | 출력 |
|---|---|---|---|---|
| Personal Proxy Agent | 각 디바이스 | 협상 생명주기 관리, 사용자 대리 실행 | 사용자 요청, 수신 메시지 | Agenda 브로드캐스트, 결과 사용자 통보 |
| Local LLM | 각 디바이스 | 원본 선호도·일정 해석 → 선호 벡터 생성 | 사용자 캘린더·선호 원본 데이터 | 정규화된 선호 벡터 (도메인 스키마 준수) |
| Preference Vector Schema | 도메인별 정의 | 벡터 구조 명세 (시간대 가중치, 카테고리 점수 등) | 도메인 설계 결정 | 벡터 포맷 정의 |
| A2A Transport | 디바이스 간 | 디바이스 간 메시지 암호화 전송 및 인증 (NFR09) | 선호 벡터, Agenda | 암호화된 A2A 패킷 |
| Deterministic Solver | Initiator 디바이스 | Borda Count / 가중합으로 최적 후보 산출 — LLM 미사용 | N개 선호 벡터 | 최적 결과 1개 |
| Result Broadcaster | Initiator 디바이스 | 최종 결과를 전체 참여자에게 배포 | 최적 결과 | A2A 전송 결과 메시지 |

**QA 평가:**

| QA | 평가 | 근거 |
|---|:---:|---|
| 프라이버시 (NFR09·NFR12·QS-019) | ★★★ | 선호 벡터만 공유, 원본 PII 외부 노출 없음 |
| 토큰 효율 (NFR26) | ★★★ | Coordinator LLM 미사용. 디바이스당 1회 호출만 |
| 지연 (NFR02·QS-002) | ★★★ | 1라운드 통신으로 완료 |
| 일관성 (QA #4) | ★★☆ | 결정론적이라 동일 입력 → 동일 출력 재현 가능. 단, 최종 브로드캐스트에 원자적 전달 보장(2PC) 없음 |
| 협상 유연성 | ★☆☆ | 벡터로 표현 불가능한 맥락 정보("이번 주는 특별히...") 손실 |
| 장애 내성 (NFR07) | ★★☆ | Coordinator SPOF이나, 상태가 없어 복구 비교적 단순 |

**장점:**
- Coordinator가 LLM을 전혀 사용하지 않아 토큰 비용이 최소이고 결과가 항상 재현 가능하다
- 1라운드만으로 합의가 완료되어 지연이 가장 짧다
- 결정론적 특성으로 테스트·감사(audit)가 가장 용이하다 (NFR20·NFR21)

**단점:**
- 선호 벡터 스키마로 표현할 수 없는 맥락적 조건이 협상에서 누락된다
- 도메인마다 벡터 스키마를 별도로 설계해야 하므로 도메인 확장 시 사전 작업 필요
- 결과를 참여자가 거부할 수단이 없어 수락 의지가 낮을 수 있다

**적합한 시나리오**: 약속 시간·날짜처럼 선택지가 정형화된 협상.

**고도화 옵션 — Deterministic Solver를 NegMAS로 교체:**

Borda Count/가중합 대신 [NegMAS](https://negmas.readthedocs.io) 협상 프레임워크를 Coordinator Solver로 사용하는 변형이다. 아키텍처 골격(로컬 LLM → 점수 전달 → Coordinator 집계 → 브로드캐스트)은 동일하고, Solver 모듈만 교체된다.

| 항목 | 기본 Sol.1 | NegMAS 고도화 옵션 |
|---|---|---|
| Solver | Borda Count / 가중합 | SAO(Stacked Alternating Offers) 등 NegMAS 프로토콜 |
| 이슈 간 트레이드오프 | 불가 (이슈별 독립 집계) | 가능 ("시간 양보, 대신 장소 우선" 내부 처리) |
| 양보 전략 | 없음 | 시간 제약 기반 양보 곡선 설정 가능 |
| 표현력 | 스키마 범위로 제한 | 다중 이슈 동시 협상 지원 |
| 프라이버시·토큰 효율·라운드 수 | ← 동일하게 적용 → | ← 동일하게 적용 → |
| 참여자 거부권 | 없음 | 없음 (결과 강제 확정 동일) |

**도입 전 확인 필요 사항:**
- NegMAS는 Python 기반 연구용 프레임워크다 — 온디바이스 또는 경량 Coordinator 환경에서의 실행 가능성·성능을 별도로 검증해야 한다
- NegMAS가 Coordinator에서만 실행(점수 수집 후 내부 연산)되는지, 디바이스 간 오퍼 교환까지 하는지에 따라 구조가 달라진다 — 전자만 Sol.1 범주에 해당

### Sol.2 — Multi-round Commitment Protocol (FIPA Contract-Net 변형)

협상 개시자가 Agenda만 브로드캐스트한 뒤 각 참여자 LLM이 라운드별로 수락·역제안을 반복하며 합의에 수렴하는 방식이다. Coordinator 역시 LLM을 사용해 역제안들을 반영한 새 후보안을 생성하므로, 라운드마다 최신 맥락이 반영되어 협상 유연성이 가장 높다. 단, 토큰 비용과 지연이 라운드 수에 비례해 증가하기 때문에 반드시 최대 라운드 상한(max_rounds)과 교착 감지 로직을 함께 설계해야 한다.

> **Two-Phase Abstract Constraint Exchange 흡수 메모**: 이 프로토콜은 이 솔루션의 파라미터 변형이다 — `max_rounds=1`, `coordinator_uses_llm=false`로 제한하면 구조가 동일해진다. 독립 솔루션으로 유지할 설계 갈림길이 없어 Sol.2에 통합한다.

> 📐 설계도: [DP08-sol2.drawio](참고자료/DP08-sol2.drawio)

**구조:**
```
Round 1: Initiator가 협상 주제(Agenda)만 브로드캐스트
Round 2: 각 참여자가 "가능 여부 + 최소 조건"만 응답 (로컬 LLM)
Round 3: Initiator가 후보안 제시 (Coordinator LLM 사용)
Round 4: 각 참여자가 수락/역제안 결정 (로컬 LLM)
... (최대 라운드 수 상한까지 반복)
최종: Coordinator가 합의 확정 + 2PC 커밋
```

**워크플로우 상세:**

1. **Agenda 브로드캐스트 (Round 1)**: Initiator의 Personal Proxy Agent가 협상 주제(도메인, 조건 범위, 최대 라운드 수)만 전송 — 선호도 정보 미포함
2. **최소 조건 응답 (Round 1 응답)**: 각 참여자 Local LLM이 Agenda를 해석하여 "가능 여부 + 최소 수용 조건"만 응답. Session State Manager가 라운드 상태를 영속화
3. **후보안 생성 (Round 2)**: Coordinator LLM이 전체 응답을 취합하여 모든 최소 조건을 만족하는 후보안 생성 → 브로드캐스트
4. **수락/역제안 (Round 2 응답)**: 각 참여자 Local LLM이 후보안 평가 → 수락(accept) / 거절(reject) / 역제안(counter) 중 하나를 응답
5. **반복 (Round 3~N)**: 합의 미달 시 Coordinator LLM이 역제안들을 반영해 새 후보안 생성. 매 라운드 Session State Manager가 체크포인트 저장 (NFR06)
6. **합의 확정**: 전원 수락 시 Coordinator가 2PC Phase 1(Prepare) 전송 → 전원 ACK → Phase 2(Commit) → 원자적 상태 확정
7. **교착 처리**: max_rounds 초과 시 Round Manager가 교착 선언 → 부분 합의 또는 협상 중단 (NFR-14)

**예시 (3인 식사 약속 — Alice 협상 개시, 참여자: Bob·Carol, max_rounds=3):**

```
[Round 1] Alice → 브로드캐스트
  Agenda: "이번 주말 저녁 식사 모임 날짜·시간·장소 정하자. 선호랑 불가 조건만 알려줘."

[Round 1 응답] 각 Local LLM → 최소 조건만 전송
  Bob   응답: "일요일이 좋아. 18시 이전은 어렵고, 음식은 뭐든 OK"
  Carol 응답: "토요일 18시-19시 선호. 한식이나 양식 좋아"
  ※ 이 응답은 Coordinator(Alice 디바이스)로 전달됨
     원본 캘린더 원문("삼성 팀 회식 토요일 6시")·상세 위치는 기기 밖으로 나가지 않음
     LLM이 추상화한 가용 조건과 선호만 전달 — 원본 PII와 추상화 결과의 구분이 핵심

[Round 2] Coordinator LLM → 응답 취합 후 후보안 생성
  판단: Bob은 일요일, Carol은 토요일 선호 → 충돌. 양쪽 모두 언급 없는 일요일 19시 + 양식으로 중간점 탐색
  후보안: "일요일 19시, 양식 어때요?"

[Round 2 응답]
  Bob   응답: "accept"
  Carol 응답: "counter — 일요일은 좀 빠듯해. 토요일 19시 한식은 어때?"

[Round 3] Coordinator LLM → 역제안 반영
  판단: Carol의 역제안을 Bob이 수용할 수 있는지 확인 필요 → 토요일 19시 한식 제안
  후보안: "토요일 19시, 한식으로 하자"

[Round 3 응답]
  Bob   응답: "accept" (18시 이후면 OK라고 했으므로)
  Carol 응답: "accept"

[최종] 2PC Commit → 전원 ACK
  확정: "토요일 19시, 한식"
  세션 체크포인트 삭제, 각 디바이스 캘린더에 반영
```

**모듈 설명:**

| 모듈 | 위치 | 역할 | 입력 | 출력 |
|---|---|---|---|---|
| Personal Proxy Agent | 각 디바이스 | 라운드 생명주기 관리, 사용자 대리 실행 | 수신 메시지, 사용자 설정 | 응답 메시지, 사용자 통보 |
| Local LLM | 각 디바이스 | Agenda·후보안 해석, 수락/역제안 판단 | Agenda 또는 후보안 + 사용자 원본 선호 | accept / reject / counter 메시지 |
| Round Manager | 각 디바이스 | 현재 라운드 추적, 응답 타임아웃 감지 | 라운드 번호, 수신 메시지 타임스탬프 | 타임아웃 이벤트, 라운드 전환 신호 |
| Session State Manager | 각 디바이스 | 라운드별 상태 체크포인트 저장 (NFR06) | 라운드 응답 | 영속화된 세션 스냅샷 |
| Coordinator LLM | Initiator 디바이스 | 전체 응답 취합 → 다음 후보안 생성 | N개 참여자 응답 (accept/reject/counter) | 후보안 메시지 |
| 2PC Commit Manager | Initiator 디바이스 | 최종 합의 원자적 확정 (Phase 1: Prepare → Phase 2: Commit) | 전원 수락 신호 | Commit 또는 Abort 결과 |
| A2A Transport | 디바이스 간 | 라운드별 메시지 암호화 전송 및 인증 (NFR09) | 응답·후보안 메시지 | 암호화된 A2A 패킷 |

**QA 평가:**

| QA | 평가 | 근거 |
|---|:---:|---|
| 프라이버시 (NFR09·NFR12·QS-019) | ★★☆ | Round 1에서 자연어 선호·조건이 Coordinator로 전달됨. 원본 캘린더 원문은 기기 밖 미전송이나, LLM 추상화 결과에 일정 맥락이 노출될 수 있음 |
| 토큰 효율 (NFR26) | ★☆☆ | 라운드마다 참여자 전원 + Coordinator LLM 호출 |
| 지연 (NFR02·QS-002) | ★☆☆ | N라운드 × 참여자 수 × 네트워크 왕복 누적 |
| 일관성 (QA #4) | ★★☆ | 라운드 단위 상태 저장으로 재개 가능, 최종 2PC 커밋 |
| 협상 유연성 | ★★★ | LLM이 라운드마다 최신 맥락을 반영해 판단 |
| 장애 내성 (NFR07) | ★★☆ | 라운드 단위 체크포인트로 재개 가능 (NFR06·QS-016) |

**장점:**
- LLM이 매 라운드 최신 맥락을 반영하므로 협상 유연성이 가장 높다
- 라운드 단위 세션 저장으로 앱 종료 후 재개가 가능하다 (NFR06)
- 인간의 협상 방식과 가장 유사해 사용자가 결과를 신뢰하기 쉽다 (NFR13)

**단점:**
- 라운드마다 참여자 전원과 Coordinator가 LLM을 호출해 토큰 비용이 가장 크다
- 라운드 수가 늘수록 지연이 선형으로 증가한다
- 최대 라운드 상한 + 교착 감지 로직을 반드시 설계해야 한다 (NFR-14)

**적합한 시나리오**: 식사 메뉴·예산 협상처럼 맥락적 판단이 필요한 협상.

### Sol.3 — Async Shared State Board (비동기 수렴)

Coordinator 없이 참여자들이 공유 보드에 비동기로 자신의 상태 요약을 게시하고, 각 디바이스가 보드를 polling하며 합의 가능 여부를 독립적으로 판단하는 방식이다. 참여자가 동시에 온라인일 필요가 없어 장기·비동기 협상에 적합하고, Coordinator가 존재하지 않아 단일 장애점(SPOF)이 없다. 대신 보드에 요약 데이터가 저장되어 상대적으로 프라이버시 위험이 높고, 낙관적 잠금(optimistic locking)으로 일관성을 직접 보장해야 하는 구현 복잡도가 따른다.

> 📐 설계도: [DP08-sol3.drawio](참고자료/DP08-sol3.drawio)

**구조:**
```
[공유 협상 보드 — 암호화된 임시 저장소, P2P 또는 경량 클라우드]
  각 참여자가 자신의 가용 슬롯/선호 요약을 비동기로 게시
  각 디바이스 에이전트가 보드를 polling하며 로컬에서 합의 가능성 판단
  합의 가능 상태 감지 시 Claim → 낙관적 잠금으로 확정
```

**워크플로우 상세:**

1. **세션 생성**: Initiator의 Personal Proxy Agent가 공유 보드에 협상 세션 생성 (세션 ID, 참여자 목록, 도메인, 만료 시각)
2. **상태 게시 (비동기)**: 각 참여자 Local LLM이 사용자 가용 슬롯·선호 요약 생성 → Board Writer가 암호화하여 보드에 비동기 게시. 참여자마다 독립적으로 동작 — 동시 온라인 불필요
3. **보드 폴링**: Board Poller가 설정된 주기로 보드를 읽어 타 참여자 상태 수신. 변경 감지 시 Local LLM에 전달
4. **합의 가능성 판단**: Local LLM이 수신된 요약들을 해석하여 현재 상태에서 합의 가능한 후보 존재 여부를 판단
5. **Claim 시도**: 합의 가능 상태 감지 시 Optimistic Lock Manager가 현재 보드 버전 번호를 포함한 Claim 요청을 보드에 전송
6. **충돌 처리**: 다른 참여자가 동시에 Claim한 경우 버전 불일치 → 충돌 해결 정책(선착순 우선 또는 재협상)에 따라 처리 후 재시도
7. **합의 확정**: Claim 성공 시 최종 합의 결과를 보드에 게시 → 전체 참여자에게 공개
8. **결과 수신**: 각 참여자 Board Poller가 최종 결과를 polling으로 수신 → Personal Proxy Agent가 사용자에게 통보

**예시 (3인 식사 약속 — Alice 협상 개시, 참여자: Bob·Carol, 세션 만료: 3일):**

```
[Day 0 오전] Alice → 보드에 세션 생성
  Session: { id: "mtg-2026-abc", 참여자: [Alice, Bob, Carol], 도메인: "식사 약속", 만료: Day 3 자정 }

[Day 0 오후] Alice → Board Writer가 암호화 후 게시
  Alice 엔트리: { 가용: ["토 18-21시", "일 17-20시"], 선호: ["한식", "강남권"] }
  (원본 캘린더·위치 데이터는 기기 내부에만 존재)

[Day 1 오전] Bob → Board Writer가 게시 (Alice와 동시 온라인 불필요)
  Bob 엔트리: { 가용: ["일 20-23시"], 선호: ["양식 또는 무관", "지역 무관"] }

[Day 1 저녁] Carol → Board Writer가 게시
  Carol 엔트리: { 가용: ["토 18-20시", "일 18-21시"], 선호: ["한식", "강남권"] }

[Day 1 저녁, Carol 게시 직후] Consensus Detection Engine → 중첩 분석
  토요일: Alice(18-21시) ∩ Carol(18-20시) = 18-20시 가능. Bob 가용 없음 → 불가
  일요일: Alice(17-20시) ∩ Bob(20-23시) = 20시만. Carol(18-21시) ∩ 20시 = 가능
  → 일요일 20시 합의 후보 존재, 선호 음식: 한식(Alice+Carol) 우세

[Claim] Carol 디바이스 Optimistic Lock Manager → Claim 시도
  요청: { 합의: "일요일 20시, 한식", board_version: 3 }
  → 충돌 없음 → Claim 성공

[Day 1 저녁] 결과 보드에 게시
  Final: "일요일 20시, 한식, 강남권"

[Day 2 오전] Alice·Bob Board Poller → polling으로 최종 결과 수신 → 각 캘린더에 반영
  ※ Bob은 Day 1에 오프라인이었지만 polling으로 결과 확인 가능
```

**모듈 설명:**

| 모듈 | 위치 | 역할 | 입력 | 출력 |
|---|---|---|---|---|
| Personal Proxy Agent | 각 디바이스 | 보드 상호작용 생명주기 관리, 사용자 대리 | 사용자 요청, 보드 이벤트 | 세션 생성 요청, 게시 요청, 사용자 통보 |
| Local LLM | 각 디바이스 | 사용자 선호 요약 생성, 타 참여자 요약 해석, 합의 가능성 판단 | 사용자 원본 선호 + 보드 수신 요약들 | 암호화 전 요약 텍스트, 합의 판단 결과 |
| Board Writer | 각 디바이스 | 요약 암호화 후 보드에 비동기 게시 (NFR12) | 요약 텍스트 | 암호화된 보드 엔트리 |
| Board Poller | 각 디바이스 | 설정 주기로 보드 상태 polling, 변경 감지 | 보드 엔드포인트, 폴링 주기 | 최신 보드 스냅샷 (타 참여자 엔트리) |
| Optimistic Lock Manager | 각 디바이스 | 버전 번호 기반 Claim — 충돌 시 재시도 또는 충돌 해결 | 합의 후보 + 현재 보드 버전 | Claim 성공/실패, 충돌 해결 결과 |
| Consensus Detection Engine | 보드 또는 각 디바이스 | 모든 참여자 엔트리 중첩 여부 검사 → 합의 가능 상태 선언 | 보드 전체 엔트리 | 합의 가능 신호 또는 대기 유지 |
| Shared Board | P2P 또는 경량 클라우드 | 암호화된 참여자 상태 임시 저장, 버전 관리, 세션 만료 처리 (NFR12) | 보드 엔트리 (암호화) | 버전화된 엔트리 목록 |

**QA 평가:**

| QA | 평가 | 근거 |
|---|:---:|---|
| 프라이버시 (NFR09·NFR12·QS-019) | ★★☆ | 보드에 요약 데이터 저장 — 암호화 필수, 상대적 노출 위험 |
| 토큰 효율 (NFR26) | ★★☆ | polling 빈도에 따라 LLM 호출 제어 가능, 가변적 |
| 지연 (NFR02·QS-002) | ★★☆ | polling 주기 의존적. 실시간 협상에 부적합 |
| 일관성 (QA #4) | ★☆☆ | 낙관적 잠금 + 충돌 해결 로직 필요. 강한 일관성 보장 어려움 |
| 협상 유연성 | ★★☆ | 보드 스키마 내에서 자유로운 편이나 실시간 역제안은 어렵다 |
| 장애 내성 (NFR07) | ★★★ | Coordinator 없음(SPOF 제거). 참여자 이탈·재접속 자유 |

**장점:**
- Coordinator가 없어 단일 장애점이 존재하지 않는다
- 참여자가 동시에 온라인일 필요가 없어 장기·비동기 협상에 적합하다
- N명 추가·이탈이 자유로워 그룹 협상 확장성이 가장 높다

**단점:**
- 공유 보드에 요약 데이터가 저장되어 타 솔루션 대비 프라이버시 위험이 상대적으로 높다
- 낙관적 잠금과 충돌 해결 로직 구현이 복잡해 일관성 보장이 가장 어렵다
- polling 기반 구조라 실시간 협상에는 반응성이 떨어진다

**적합한 시나리오**: 여행 계획처럼 참가자가 동시 온라인이 아닌 장기 조율.

## 5. 권고 방향 — 계층적 조합

단일 솔루션 선택이 아니라 **협상 도메인 특성에 따라 계층화**:

| 협상 유형 | 권장 솔루션 | 이유 |
|---|---|---|
| 정형화된 선택 (시간·날짜) | Sol.1 (Deterministic) 우선 | LLM 호출 최소, 일관성 완벽, 결과 재현 가능 |
| 맥락적 판단 필요 (식당·활동) | Sol.2 (Multi-round), max_rounds 상한 설정 | 유연성 확보 + NFR26 통제를 위한 라운드 상한 필수 |
| 비동기·장기 조율 (여행 계획) | Sol.3 (Async Board) | 동시 온라인 불필요, SPOF 없음 |

**"어느 디바이스 LLM이 맡는가"에 대한 결론**:
- LLM 추론은 무조건 각자 로컬 (프라이버시·비용)
- 집계·조율만 Coordinator (협상 개시자 디바이스). Sol.1 Coordinator는 LLM 미사용(결정론적), Sol.2 Coordinator는 LLM 사용 — 각 디바이스의 로컬 LLM은 세 솔루션 모두 사용
- Coordinator 이탈 시 재선출 프로토콜 필요 (NFR07 대응) — Sol.3만 해당 없음

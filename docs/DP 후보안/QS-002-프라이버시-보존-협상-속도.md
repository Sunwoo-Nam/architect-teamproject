# QS-002 / QS-009 / QS-019 — 프라이버시 보존 협상 속도 후보 구조 설계

## 0. 설계 범위

본 문서는 **QA-001(A2A 협상 완료 시간 최소화, 최우선)** 을 달성하되,
다음 두 NFR을 **반드시 만족하는 제약 하에서** 협상 속도를 최적화하는 후보 구조를 설계한다.

- **NFR-005** 서버 LLM PII 전송 차단 (`R_pii_leak = 0%`, QS-009)
- **NFR-006** 협상 제안 내 불필요 개인/기기 정보 노출 차단 (`R_unnecessary_info = 0%`, QS-019)

> 설계 원칙: **NFR은 협상 속도보다 위에 둔다.** 속도를 위해 프라이버시 필터를 우회·생략하는 어떤 설계도 채택 후보가 될 수 없다. 즉 본 설계의 목표는 "프라이버시 게이트를 통과하면서도 빠른 협상"이며, "빠르지만 게이트를 건너뛰는 협상"이 아니다.

---

## 1. 문제 식별 (Problem Identification)

### 1.1 협상 시간의 구조적 분해

QS-002 측정 공식에 따르면 `T_negotiation`은 라운드의 합이며, 각 라운드 시간은 다음으로 구성된다.

```
T_negotiation ≈ Σ_round ( T_llm_inference + T_privacy_filter + T_network_rtt + T_persist )

T_llm_inference  : ProposalGenerator의 온디바이스 LLM 추론 시간 (라운드별 1~수 회)
T_privacy_filter : NFR-005/006을 위한 PII·불필요정보 분리/익명화 처리 시간
T_network_rtt    : A2AProtocolAdapter ↔ 상대 PPA 왕복 지연
T_persist        : NegotiationSessionDB 세션 상태 영속화 시간
```

### 1.2 핵심 긴장 관계 (Tension)

| # | 긴장 관계 | 설명 |
|---|-----------|------|
| T-1 | **프라이버시 필터 ↔ 라운드 지연** | NFR-005/006을 위해 매 제안/요청마다 PII 분리·익명화를 수행하면 `T_privacy_filter`가 모든 라운드에 누적되어 협상이 느려진다. |
| T-2 | **온디바이스 추론 ↔ 라운드 수** | 빠른 합의를 위해 더 좋은 제안을 생성하면 라운드 수(N)가 줄지만, 강한 추론은 `T_llm_inference`를 키운다. 약한 추론은 빠르지만 N이 늘어 총 시간이 증가한다. |
| T-3 | **속도 최적화 유혹 ↔ NFR 무결성** | "서버 LLM에 원문째 보내면 Task 분해가 빠르다", "DPA 컨텍스트를 통째로 제안에 실으면 LLM이 협상을 잘한다" 같은 속도 최적화는 NFR-005/006을 0%에서 깨뜨린다. → **금지선(red line)**. |

### 1.3 식별된 구조적 문제

- **P-1** 프라이버시 분리/익명화 로직이 협상 critical path에 직렬로 끼어 매 라운드 지연을 누적시킨다.
- **P-2** 라운드 수(N)가 협상 시간의 지배 변수인데, 이를 줄이려면 강한 추론이 필요하고 강한 추론은 온디바이스에서 느리다.
- **P-3** 프라이버시 게이트의 위치/형태에 따라 "0% 보장"과 "속도"가 동시에 달라진다. 게이트가 약하면 NFR 위반, 게이트가 무거우면 속도 저하.

## 2. 도출된 설계 포인트 (Design Points)

위 문제로부터, 프라이버시를 지키며 속도를 확보하기 위한 다섯 개의 설계 포인트를 도출한다.

| DP | 설계 포인트 | 대응 문제 | 관련 요구사항 |
|----|-------------|-----------|---------------|
| **DP-1** | **단일 강제 프라이버시 게이트(choke point)** — 기기 밖으로 나가는 모든 경로(서버 LLM행, 상대 PPA행)는 우회 불가능한 단일 필터 컴포넌트를 통과한다. 속도 때문에 우회하는 분기를 만들지 않는다. | P-3 | NFR-005, NFR-006 |
| **DP-2** | **토큰화/플레이스홀더 치환** — PII·기기식별자를 협상 시작 시 1회 토큰화하고, 이후 라운드는 토큰만 오간다. 원문 결합(de-tokenize)은 온디바이스 실행 단계에서만 수행. 매 라운드 재익명화를 제거하여 `T_privacy_filter`를 라운드당 비용에서 세션당 1회 비용으로 낮춘다. | P-1 | NFR-005, QA-001 |
| **DP-3** | **라운드 수(N) 최소화 우선** — 협상 시간의 지배 변수는 라운드 왕복 횟수다. 라운드당 추론을 약간 무겁게 하더라도 한 번에 합의 가능성이 높은 제안을 생성해 N을 줄이는 쪽이 총 `T_negotiation`에 유리하다. | P-2 | QA-001 |
| **DP-4** | **프라이버시-허용 협상 범위 사전 컴파일** — 협상 시작 시 AuthorizationScope + DPA 컨텍스트에서 "상대에게 노출 가능한 협상 변수 집합(가격범위·날짜범위 등 사실값만)"을 한 번에 추출해 둔다. 라운드 중에는 이 사전 컴파일된 안전 범위 안에서만 제안을 생성하므로, 매 제안마다 원문을 다시 필터링할 필요가 없다. | P-1, P-3 | NFR-006, QA-001 |
| **DP-5** | **필터 처리와 네트워크/추론의 파이프라이닝** — 불가피한 필터 비용은 critical path에서 빼서, 다음 라운드 준비(추론·인코딩)와 겹쳐 수행한다. 단, 전송 직전 최종 게이트 검증은 동기적으로 강제한다(우회 금지). | P-1 | QA-001 (NFR 유지) |

## 3. 후보 구조 (Candidate Architectures)

세 개의 후보는 **프라이버시 게이트를 어디에·어떤 형태로 두어 속도를 확보하는가**에서 갈린다.
모두 DP-1(단일 강제 게이트)을 공통 전제로 하며, NFR-005/006의 0% 보장을 깨지 않는다.

### CA-DP1: 게이트웨이 인라인 필터 (Synchronous Privacy Gate)

> 핵심 아이디어: 기존 도메인 모델의 단일 진입점을 그대로 활용해, **LLMGateway / A2AProtocolAdapter 내부에 동기 프라이버시 필터를 인라인**으로 둔다.

- 서버 LLM행: `IntentDetector` → **LLMGateway 내 PII 필터** → 서버 LLM
- 상대 PPA행: `ProposalGenerator` → **A2AProtocolAdapter 내 정보최소화 필터** → 상대 PPA
- 필터는 매 호출 시 동기 실행되며, 통과 못 하면 전송 자체가 차단된다.
- 적용 DP: DP-1(강제 게이트)

**장점**

- 구조 변경 최소. 기존 도메인 모델(LLMGateway, A2AProtocolAdapter가 이미 "단일 진입점"으로 명세됨)에 가장 충실.
- 게이트가 물리적으로 출구를 독점 → NFR-005/006 0% 보장이 구조적으로 가장 단순·명확. 검증·감사 로그 위치가 자명.
- PoC 단계 구현·디버깅이 쉽다(QA-007 추적성에도 유리).

**단점 및 트레이드오프**

- 매 라운드·매 호출마다 동기 필터 → `T_privacy_filter`가 critical path에 그대로 누적 (T-1 미해결). QA-001에 가장 불리.
- 필터 로직과 전송 어댑터의 책임이 한 컴포넌트에 섞여 응집도 저하 (QA-008 프로토콜 변경 비용에 약간 불리).

---

### CA-DP2: 세션 토큰화 + 안전범위 사전 컴파일 (Tokenized Negotiation Context)

> 핵심 아이디어: 협상 세션 시작 시 **PrivacyVault**가 PII·기기식별자를 1회 토큰화하고, **NegotiationScopeCompiler**가 "노출 가능한 협상 변수 집합"을 사전 컴파일한다. 이후 라운드는 토큰·안전범위만으로 진행 → 라운드당 필터 비용 제거.

- 세션 개시 시(1회): 원문 → `PrivacyVault`가 토큰 매핑 생성, `NegotiationScopeCompiler`가 AuthorizationScope+DPA 컨텍스트 → 안전 협상범위 산출
- 라운드 중(N회): `ProposalGenerator`는 토큰·안전범위만 다룬다. 원문 접근 불가.
- 온디바이스 실행 단계(합의 후): `PrivacyVault`가 토큰 → 원문 복원하여 실제 예약/기기 반영
- 적용 DP: DP-1, DP-2, DP-4

**장점**

- `T_privacy_filter`를 "라운드당 비용 → 세션당 1회 비용"으로 구조적으로 절감 (T-1 해결). 라운드가 많을수록(다라운드 협상·교착) 이득이 커진다 → QA-001에 가장 효과적.
- ProposalGenerator가 애초에 원문을 볼 수 없는 구조 → NFR-006 0%를 "필터로 거른다"가 아니라 "원문이 도달하지 못한다"로 보장(설계상 더 강함).
- 안전범위 사전 컴파일이 ProposalGenerator의 탐색 공간을 좁혀 라운드당 추론도 가벼워질 여지(부수적 속도 이득).

**단점 및 트레이드오프**

- 신규 컴포넌트(PrivacyVault, NegotiationScopeCompiler) 추가 → 구조 복잡도·구현 비용 증가.
- 토큰 매핑의 보관/만료/메모리 관리 필요(NFR-004 유휴 메모리, QS-023 세션 만료와 연계 검토 필요).
- 사전 컴파일된 안전범위가 협상 도중 바뀌어야 하는 경우(UC-022 직접 개입) 재컴파일 비용 발생.

---

### CA-DP3: 비동기 파이프라인 + 전송 직전 동기 게이트 (Pipelined Filter with Mandatory Egress Gate)

> 핵심 아이디어: 필터·익명화를 **다음 라운드 준비 작업과 비동기로 겹쳐(pipeline)** 수행해 critical path에서 빼되, **전송 직전(egress)에는 우회 불가능한 동기 최종 게이트**를 둬 NFR 0%를 사수한다.

- `ProposalGenerator`가 라운드 N 제안 생성과 동시에, 라운드 N+1 후보의 필터링을 백그라운드로 선행
- 네트워크 RTT 대기 시간 동안 다음 라운드 추론·익명화를 미리 수행 (DP-5)
- 단, 실제 전송 시점에는 `EgressPrivacyGate`가 동기 검증 → 미통과 시 차단(속도와 무관하게 강제)
- 적용 DP: DP-1, DP-5, (DP-3)

**장점**

- 불가피한 필터 비용을 네트워크/추론 대기와 겹쳐 체감 `T_negotiation` 단축 (T-1 완화). CA-DP1보다 빠름.
- 전송 직전 동기 게이트는 유지 → NFR-005/006 0% 보장 우회 없음(red line 사수).
- 기존 컴포넌트 구조를 크게 바꾸지 않고 실행 모델(동시성)만 개선.

**단점 및 트레이드오프**

- 동시성 도입 → 상태 관리 복잡도·경쟁 조건 위험 증가. PoC에서 디버깅 난도 상승(QA-007 추적성에 불리).
- 선행 필터링한 라운드가 실제로 안 쓰이면 온디바이스 연산·배터리 낭비 (NFR-003 배터리, QA-003과 상충 가능).
- 투기적 선행 계산은 협상 흐름이 비결정적(LLM)이라 적중률이 낮을 수 있어 이득이 제한적.

## 4. 후보 구조 비교 및 종속/상충 관계

| 항목 | CA-DP1 인라인 게이트 | CA-DP2 토큰화+사전컴파일 | CA-DP3 비동기 파이프라인 |
|------|----------------------|--------------------------|--------------------------|
| 협상 속도 (QA-001) | △ 라운드당 필터 누적 | ◎ 세션 1회로 절감 | ○ 체감 단축(겹침) |
| NFR-005/006 0% 보장 강도 | ○ 게이트로 차단 | ◎ 원문 미도달(설계상 차단) | ○ egress 게이트로 차단 |
| 구현 복잡도 | ◎ 가장 단순 | △ 신규 컴포넌트 2개 | △ 동시성 |
| 추적성 (QA-007) | ◎ | ○ | △ |
| 배터리 (NFR-003/QA-003) | ○ | ○ | △ 투기 계산 위험 |
| 다라운드/교착 협상에서 이득 | 작음 | 큼 | 중간 |

### 종속·상충 관계

- **공통 전제**: 세 후보 모두 DP-1(단일 강제 게이트)을 깨지 않는다. 게이트 우회 분기는 어떤 후보에서도 금지.
- **상충**: CA-DP1 ↔ CA-DP2 (필터를 "매 라운드 인라인"으로 둘 것인가 vs "세션 1회 토큰화"로 옮길 것인가 — 게이트의 형태가 양립 불가).
- **종속/보완**: CA-DP3은 CA-DP1 또는 CA-DP2 위에 얹는 실행모델 최적화로 결합 가능(독립 채택도 가능하나, CA-DP2 + CA-DP3 조합이 속도 측면 최대).

### 권고 (수석 아키텍트 의견)

- **1차 채택 권고: CA-DP2.** 협상 시간의 지배 변수는 라운드 수와 라운드당 필터 비용인데, CA-DP2는 후자를 구조적으로 제거하면서 NFR-006을 "필터링"이 아니라 "원문 미도달"로 더 강하게 보장한다. PoC의 핵심 가설(자율 협상의 신뢰성)에 가장 부합.
- **PoC 초기엔 CA-DP1로 시작**하여 0% 보장과 추적성을 먼저 확보한 뒤, 측정상 `T_privacy_filter`가 병목으로 확인되면 CA-DP2로 이행하는 단계적 전략을 제안한다.
- **CA-DP3은 다라운드/교착 빈도가 높다고 측정될 때** 보완 후보로 검토. 배터리(NFR-003) 영향 측정이 선결 조건.

> 최종 채택은 Phase 6(candidate-evaluator)에서 `qualities.md` 우선순위에 따라 정량 평가 후 결정한다.

### **DP 09. 협상 진행 중 인텐트(Intent) 변경 및 취소 시 동시성 제어** 

#### **1. 풀고자 하는 문제** 
A의 메타 에이전트가 B와 "토요일 저녁 식사"로 합의해가는 도중(In-flight Transaction), A가 인간 채널(메신저)에 "일요일 점심으로 하자"라고 입력했습니다. 모니터링 시스템(IDS)이 이를 맥락으로 파악하여 에이전트 프레임워크로 조건 변경을 요청했을 때, 기존 분산 트랜잭션을 꼬임 없이 방향 전환해야 합니다.

#### **2. 아키텍처적 난제** 
 * **경합 상태 (Race Condition):** 두 개 이상의 프로세스가 공유 상태를 동시에 변경하려 할 때 발생하는 논리적 충돌입니다. A의 IDS가 '변경 요청(일요일)'을 발생시키는 시점과, 상대방(B)이 A2A 프로토콜로 '토요일 확정' 응답을 보내오는 시점이 정확히 교차(Cross)할 경우 시스템의 분산 상태가 파괴될 수 있습니다.
   
 * **채널 비동기화 (Channel Asynchrony):** Human Channel(화면 텍스트)에서 파악한 묵시적 맥락만으로 시스템의 상태 전이(State Transition)를 독단적으로 확정해서는 안 되며, 반드시 Agent Channel(A2A 프로토콜)의 명시적 제어 메시지가 동반되어야만 상태 변경이 유효해진다는 제약입니다.  
   
#### **3. 해결 방안 1: 비관적 락 기반 강제 중단 및 재생성 (Pessimistic Abort & Re-initiate)** 
IDS로부터 새로운 의도가 인입되면, 즉시 기존 협상 세션에 배타적 락(Exclusive Lock)을 걸고 진행을 차단합니다. B에게 A2A 프로토콜로 CANCEL 제어 메시지를 브로드캐스팅하여 기존 협상을 무효화한 뒤, 메모리를 초기화하고 "일요일 점심"이라는 새로운 세션 ID로 처음부터 다시 협상을 시작(Cold Start)합니다.
#### **4. 해결 방안 2: 낙관적 프롬프트 주입 및 상태 업데이트 (Optimistic Context Mutation & State Update)** 
기존 세션을 유지한 채, A의 메타 에이전트의 LLM 컨텍스트 윈도우(프롬프트)에 [시스템 강제 주입: 일요일 점심으로 조건 변경]을 런타임에 동적으로 주입(Inject)합니다. 동시에 B에게 A2A 프로토콜로 UPDATE_PROPOSAL 제어 메시지를 발송하여, 기존 협상 세션 내에서 에이전트가 자율적이고 자연스럽게 대화 방향을 수정하도록 유도합니다.
#### **5. Quality Attribute Trade-off 종합 평가 (10대 QA 기준)**

| 품질 속성 (QA) | 해결 방안 1 (비관적 락/재시작) | 해결 방안 2 (낙관적 프롬프트 주입) | 아키텍트의 분석 (Trade-off) |
| :--- | :--- | :--- | :--- |
| **동시성 (교착/경합 방지)** | **[+] 완벽 통제** | **[-] 심각한 경합 위험** | 방안 1은 락을 걸어 트랜잭션을 직렬화하므로 충돌이 없습니다. 방안 2는 변경 프롬프트 주입과 B의 수락 메시지가 교차할 때 레이스 컨디션이 발생하여 시스템이 멈출 수 있습니다. |
| **기능적 정확성 (결과 보장)** | **[+] 극대화** | **[-] 훼손 (환각 리스크)** | 방안 1은 기계적인 If-Else 분기로 동작하므로 결과가 100% 예측 가능합니다. 방안 2는 LLM이 주입된 프롬프트를 환각(Hallucination)으로 무시할 리스크를 통제할 수 없습니다. |
| **무결성** | **[+] 보장** | **[-] 취약** | 방안 2는 시스템의 상태 전이(State Transition) 제어권을 확정적 로직이 아닌 확률론적인 LLM에게 넘겨주게 되어, 의도치 않은 권한 초과 행동을 유발할 수 있습니다. |
| **수용성 (토큰 버짓 최적화)** | **[-] 최악 (매몰 비용)** | **[+] 극대화** | 방안 1은 이전에 협상하며 소모한 수천 개의 토큰을 전부 폐기(Sunk Cost)하고 다시 시작해야 합니다. 방안 2는 과거 문맥을 재활용하므로 한정된 NPU 토큰 예산을 아낄 수 있습니다. |
| **시간 반응성 (지연 시간)** | **[-] 심각한 지연** | **[+] 우수** | 방안 1은 상대방과 Handshake부터 다시 해야 하므로 협상 완료 시간이 배로 늘어납니다. 방안 2는 흐름을 타면서 즉각 궤도를 수정합니다. |
| **상호운용성 (유연성)** | **[-] 경직됨** | **[+] 극대화** | 방안 1은 상대방에게 일방적인 CANCEL을 던져 인간의 대화 흐름을 단절시킵니다. 방안 2는 인간 비서처럼 융통성 있게(Adaptability) 협상 방향을 선회합니다. |

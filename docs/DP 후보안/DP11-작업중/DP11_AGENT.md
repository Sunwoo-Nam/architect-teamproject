# DP11 작업 에이전트 지침

## 1. 이 문서의 목적

이 문서는 `DP11. 다중 Intent의 동시 발생 및 실행 중 변경 처리 방식`을 구체화할 때 따라야 할 작업 지침임.

DP 구체화 작업의 최종 목표는 단순한 아이디어 나열이 아니라, 전체 프로젝트의 시나리오와 설계 구조 안에서 DP11이 풀어야 하는 문제를 해결할 수 있는 **서로 비등한 2개 솔루션**을 선정하고, 두 솔루션 사이의 선택이 왜 어려운지를 **QA 항목 간 trade-off**로 설명하는 것임.

따라서 DP11 작업자는 다음을 산출해야 함.

1. DP11이 정확히 어떤 아키텍처 문제를 푸는지 명확히 정의
2. 그 문제를 실제로 해결할 수 있는 솔루션 후보 2개 선정
3. 두 후보가 모두 합리적이어서 쉽게 하나를 고르기 어렵다는 점 설명
4. 선택의 어려움을 성능, 신뢰성, 복구성, 추적성, 자원 효율, 변경 용이성 등 QA trade-off로 구조화
5. 필요 시 최종 권고안을 제시하되, 권고의 근거를 QA 우선순위와 프로젝트 제약에 연결

## 4. 솔루션 후보 선정 원칙

최종 솔루션 2개는 다음 조건을 만족해야 함.

1. **둘 다 문제를 끝까지 해결해야 함**
   - 한 후보가 명백히 미완성, 위험, 또는 장난감 수준이면 안 됨.
   - 동시 실행, 관계 라벨 반영, 실패 회복을 모두 다룰 수 있어야 함.

2. **둘 사이의 선택이 어려워야 함**
   - 한쪽이 모든 QA에서 압승하면 DP가 아니라 단순 구현 결정이 됨.
   - 각 후보는 최소 2개 이상의 중요한 QA에서 상대보다 명확한 장점을 가져야 함.

3. **서로 다른 아키텍처 원리를 가져야 함**
   - 단순 파라미터 차이, 로그를 더 많이 남기는 정도, 캐시 TTL 차이처럼 구현 세부 조정에 그치면 안 됨.
   - 상태 소유권, 실행 제어권, 동시성 모델, 복구 모델 중 하나 이상이 본질적으로 달라야 함.

4. **프로젝트 제약 안에서 가능해야 함**
   - Android 온디바이스 중심, IDS/Orchestrator/Meta Agent/Sub-Agent 계층, A2A 프로토콜, 서버 LLM/온디바이스 LLM 배치 전제를 깨지 않아야 함.

5. **QA trade-off로 설명 가능해야 함**
   - "좋다/나쁘다"가 아니라 어떤 QA를 얻고 어떤 QA를 희생하는지 표로 설명할 수 있어야 함.
   - QA 명칭은 ISO/IEC 25010(2011)의 **상위 카테고리**(기능 적합성·성능 효율성·호환성·사용성·신뢰성·보안성·유지보수성·이식성) 중에서만 사용함.
     기밀성·무결성·가용성·결함 허용성·복구 용이성·기능적 정확성·시간 반응성·자원 활용성·수용성·상호운용성·변경 용이성·테스트 용이성 등 **하위(sub-characteristic) 명칭은 trade-off 표에 쓰지 않음.**
     필요 시 본문 서술에서만 보조적으로 인용함.
   - 본 프로젝트의 정량 QA(`docs/06-QAS.md`의 QA-001~008, NFR-001~007)는 위 상위 카테고리의 인스턴스로 봄.
     (예: QA-001 협상 완료 시간 → 성능 효율성, QA-007 추적 가능성 → 유지보수성, QA-002 동적 실행 성공률 → 신뢰성.)

## 5. 최종 DP 문서 권장 구조

DP11 최종 문서는 다음 구조를 권장함.

```markdown
# DP 11. 다중 Intent의 동시 발생 및 실행 중 변경·실패 처리 방식

## 1. 풀고자 하는 문제
## 2. 아키텍처적 난제
## 3. 해결 방안 1: ...
## 4. 해결 방안 2: ...
## 5. Quality Attribute Trade-off 종합 평가
## 6. 최종 판단 또는 권고
## 7. 남은 Open Questions
```

각 해결 방안은 최소한 다음을 포함할 것.

- 상태 모델
- 동시 실행 방식
- 병합/대체/차단/취소 처리 방식
- 실패 회복 방식
- 외부 side-effect 처리 방식
- Orchestrator, Meta Agent, Sub-Agent, Tool, 저장소의 책임 분담
- 장점과 약점

## 6. 작성 시 주의사항

- 한 후보를 의도적으로 허술하게 만들어 다른 후보를 돋보이게 하지 말 것.
- QA trade-off 표에서는 각 후보의 승패를 과장하지 말고, 어떤 조건에서 어느 후보가 유리한지 구체적으로 쓸 것.
- QA trade-off 표의 열 라벨·평가 축 명칭은 반드시 ISO/IEC 25010(2011) 상위 카테고리 8종(기능 적합성·성능 효율성·호환성·사용성·신뢰성·보안성·유지보수성·이식성) 중에서만 고를 것. ISO/IEC 25010:2023 개정판 항목(예: Safety, Interaction Capability, Flexibility)을 도입하고 싶다면 도입 사실을 본문에 명시할 것.
- 원문에 없는 가정이나 새로 추가한 메커니즘은 "추론" 또는 "후보 설계상 추가"임을 명시할 것.

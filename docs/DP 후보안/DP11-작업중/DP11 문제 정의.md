# DP11 문제 정의 (재작성 초안)

---

## 상황

Orchestrator는 Intent를 받아 실행하는 동안, 새로운 Intent가 계속 들어올 수 있습니다.

```
"토요일 저녁 식사 예약해줘"          ← Intent A 시작, 식당 검색 중
       ... 10초 후 ...
"아이도 갈 수 있는 곳으로"           ← Intent B 도착 (A에 조건 추가)
       ... 5초 후 ...
"내일 날씨도 알려줘"                ← Intent C 도착 (A와 무관)
       ... 30초 후 ...
"아 그냥 일요일 점심으로 바꿔"       ← Intent D 도착 (A의 시간 조건 대체)
```

이 시점에서 시스템은 이미 식당 후보를 검색해 두었거나, 일부 예약 hold·캘린더 임시 블록을 잡아 둔 상태일 수 있습니다. 또한 실행 중인 Task가 외부 사정으로 실패하는 경우도 함께 다뤄야 합니다.

---

## 전제: IDS와 Orchestrator의 책임 분담

IDS(Intent Detection System)가 Intent를 MAF Orchestrator로 전달할 때, 다음 정보를 함께 보냅니다.

- 각 Intent의 ID (IDS가 부여)
- Intent에 대한 (어느 정도의) 분석 결과
- 이전 Intent와의 관계 정보 — 관계의 종류(**병합 · 대체 · 차단 · 취소 · 무관**)까지 IDS가 분류해서 전달

따라서 "이 Intent가 기존 Intent와 어떤 관계인가"의 판단 자체는 **IDS의 책임**이며, 그 결과는 MAF Orchestrator에 입력으로 도착합니다.
MAF Orchestrator의 결정은 그 관계 정보와 현재 실행 상태를 바탕으로 **활성 Intent 집합 위에서 실제 실행을 어떻게 조직할 것인가**입니다.

---

## Orchestrator가 풀어야 할 질문

새 Intent가 들어왔을 때, 또는 실행 중인 Task에 변화(완료·실패·외부 효과 발생)가 생겼을 때, Orchestrator는 다음 세 질문을 함께 풀어야 합니다.

### 질문 1. 동시 실행 — 여러 Intent 중 무엇을 어떻게 병렬로 실행할 것인가

활성 Intent와 그 안의 Task 중 무엇을 어떤 방식으로 병렬 실행할지를 정합니다.

| 하위 항목 | Orchestrator가 정해야 할 것 |
|---|---|
| **병렬 후보의 범위** | 어디까지를 병렬 후보로 볼 것인가 — IDS가 무관으로 분류한 Intent 쌍, 관계가 있어도 자원이 겹치지 않는 Intent 쌍, 같은 Intent 안의 서로 다른 Task까지 포함할 것인가. |
| **충돌 판단의 기준** | "충돌 없음"을 무엇으로 정의할 것인가 — 같은 외부 Resource를 건드리는지, 같은 디바이스·화면을 점유하는지, 한쪽의 결과가 다른 쪽의 입력인지 등 어떤 기준을 채택할 것인가. |
| **병렬 실행의 방식** | 충돌 없는 후보들을 어떤 방식으로 병렬화할 것인가 — 완전 동시 실행, 시간 분할(인터리브), 부분 결과 공유 등 어떤 형태로 자원을 나눌 것인가. |

### 질문 2. 관계 라벨의 처리 — 병합·대체·차단·취소를 어떻게 적용할 것인가

IDS가 보낸 관계 라벨을, 진행 중인 Task와 이미 발생한 외부 효과를 함께 고려해 실제 실행에 반영합니다.

| 하위 항목 | Orchestrator가 정해야 할 것 |
|---|---|
| **병합** | 진행 중 Task에 새 조건을 어떻게 끼워 넣을 것인가 — 이미 수행된 부분 중 새 조건과 어긋나는 결과를 어디까지 폐기·재실행할 것인가. |
| **대체** | 기존 Intent의 진행을 어디까지 살리고 어디부터 새로 계획할 것인가 — 이미 발생한 외부 효과를 보상할 것인가, 유지한 채 활용할 것인가. |
| **차단** | 진행 중인 Task를 어느 지점에서 멈출 것인가 — 이미 발생한 외부 효과는 어떻게 다룰 것인가. |
| **취소** | 아직 시작 안 한 Task는 제거, 진행 중 Task는 중단 요청, 이미 끝난 외부 효과는 보상 실행 — 그 범위를 어디까지로 확정할 것인가. |

### 질문 3. 실패 회복 — 중간에 실패한 Intent를 어떻게 이어서 진행할 것인가

실행 중인 Task가 외부 시스템 오류, 네트워크 단절, 사용자 응답 지연 등으로 실패했을 때 어디부터 어떻게 다시 진행할지를 정합니다.

| 하위 항목 | Orchestrator가 정해야 할 것 |
|---|---|
| **재개 범위** | 실패한 Intent 전체를 폐기할 것인가, 일부 진행을 살리고 실패 지점부터 다시 이어 갈 것인가. |
| **진행 상태의 표현** | "어디까지 진행했는가"를 어떻게 표현하고 보관할 것인가 — 재개에 필요한 최소 상태는 무엇인가. |
| **외부 효과의 처리** | 이미 발생한 외부 효과(예약 hold, 캘린더 임시 블록 등)는 그대로 이어 갈 것인가, 되돌리고 다시 시작할 것인가. |
| **영향 전파** | 실패가 다른 Intent에 어떻게 영향을 주는가 — 병합되어 있던 다른 조건, 대기 중이던 의존 Task에 어떻게 전파할 것인가. |

## 이 문제가 단순하지 않은 이유

세 질문이 서로 얽혀 있습니다.

```
질문 2 (관계 적용) ───→ 활성 Intent 집합과 진행 상태가 바뀜
                            │
                            ▼
질문 1 (동시 실행) ───→ 바뀐 집합 위에서 다시 무엇을 어떻게 병렬로 돌릴지 결정
                            │
                            ▼
질문 3 (실패 회복) ───→ 실패가 발생하면 부분 진행 상태를 갱신
                            │
                            ▼
                     다시 질문 2로: 부분 진행 상태가 다음에 들어올 Intent의
                                    관계 라벨을 어떻게 반영할지를 좌우함
```

또한 이 판단은 **한 번으로 끝나지 않습니다.**
Intent A가 살아있는 동안 Intent B, C, D, E가 순차적으로 들어올 수 있고, Task가 중간에 실패할 수도 있으며, 그때마다 위 세 질문을 다시 풀어야 합니다.

---

## 정리: Decision 포인트

이 DP에서 결정해야 하는 것은:

> **IDS가 관계 정보(관계 종류 포함)를 함께 전달하는 전제 위에서, MAF Orchestrator가 (1) 활성 Intent 집합의 동시 실행 구성, (2) 병합·대체·차단·취소 같은 관계 라벨의 실행 반영, (3) 실패한 Intent의 부분 회복을 어떤 구조로 다룰 것인가?**

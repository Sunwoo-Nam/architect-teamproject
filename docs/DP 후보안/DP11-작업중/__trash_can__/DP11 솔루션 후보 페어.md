# DP11 솔루션 후보 페어 (작업 인수인계용)

> 이 문서의 목적
> - DP11 솔루션 후보 페어들을 한 곳에 정리해 두기 위함.
> - 다른 LLM/작업자가 이어서 작업할 수 있도록 **여기까지의 진행 배경**과 **다음 결정 포인트**까지 함께 기재함.
> - 모든 작업은 `docs/DP 후보안/DP11-작업중/DP11_AGENT.md`의 지침을 따름.

---

## 1. 진행 배경 (어디까지 합의되어 있는가)

1. **DP11이 풀려는 문제**는 다음 세 질문을 통합적으로 다루는 것으로 정의됨 (`DP11 문제 정의.md` 참고).
   - 질문 1. 동시 실행 — 여러 Intent를 무엇을 기준으로 어떻게 병렬로 실행할 것인가
   - 질문 2. 관계 라벨의 처리 — IDS가 전달한 병합·대체·차단·취소를 진행 중 Task와 외부 효과에 어떻게 적용할 것인가
   - 질문 3. 실패 회복 — 중간에 실패한 Intent를 어디부터 어떻게 다시 진행할 것인가

   세 질문은 서로 얽혀 있으며, 한 번이 아니라 반복적으로 다시 풀어야 한다는 점이 본 문제의 본질적 어려움임.

2. **전제 (재확인)**: IDS가 Intent 관계 종류(병합·대체·차단·취소·무관)까지 분류해서 MAF Orchestrator에 전달함. 관계 판단 자체는 IDS의 책임이고, Orchestrator는 활성 Intent 집합 위에서 실행을 조직하는 책임을 가짐.

3. **솔루션 선정 원칙** (`DP11_AGENT.md` §4)
   - 두 후보 모두 동시 실행·관계 라벨·실패 회복을 끝까지 해결해야 함.
   - 두 후보는 **상태 소유권 / 실행 제어권 / 동시성 모델 / 복구 모델** 중 적어도 한 축에서 본질적으로 달라야 함.
   - 선택이 한쪽 압승이 아니어야 하며, QA trade-off로 설명 가능해야 함.

4. **QA 표기 규칙** (`DP11_AGENT.md` §4-5, §6 — 2026-05-23 추가)
   - QA 명칭은 **ISO/IEC 25010(2011) 상위 카테고리** 8종만 사용함.
     - 기능 적합성 · 성능 효율성 · 호환성 · 사용성 · 신뢰성 · 보안성 · 유지보수성 · 이식성
   - 하위 항목(기밀성·무결성·가용성·결함 허용성·복구 용이성·기능적 정확성·시간 반응성·자원 활용성·수용성·상호운용성·변경 용이성·테스트 용이성 등)은 trade-off **표에는 쓰지 않고** 본문 서술에서만 보조 인용함.
   - `docs/06-QAS.md`의 정량 QA(QA-001~008), NFR(NFR-001~007)은 위 상위 카테고리의 인스턴스로 본다.
     - 예) QA-001 협상 완료 시간 → 성능 효율성, QA-002 동적 실행 성공률 → 신뢰성, QA-004/006/007/008 → 유지보수성, NFR-001/002/005/006 → 보안성, NFR-003/004 → 성능 효율성, NFR-007 → 유지보수성.

5. **현재 단계**: 본격 DP 본문 작성 전, **두 정식 후보의 페어를 어떤 축으로 잡을지** 결정하는 단계임. 본 문서가 그 결정을 돕기 위한 후보 카탈로그.

## 2. 페어 후보 (총 4개)

각 페어는 두 후보가 동시 실행·관계 라벨·실패 회복 세 질문을 모두 다룰 수 있도록 잡혀 있으며, 둘 사이는 본질적으로 다른 아키텍처 원리에 기반함.
모든 trade-off 표는 ISO/IEC 25010(2011) 상위 카테고리만 사용함.

> 표기 규칙: `+` = 유리, `−` = 불리, `±` = 중립/양면, 공란 = 두 후보 사이에 사실상 차이 없음 또는 본 페어와 무관.

---

### Pair 1. 단일 Orchestrator 권위형 ↔ Intent-Actor 분산형

- **A. 중앙집중 권위 (Centralized Authority)**
  Orchestrator가 모든 활성 Intent와 Task 상태를 단일 저장소에서 직접 소유. 병합·대체·차단·취소·재시도 결정을 모두 Orchestrator가 트랜잭션 단위로 수행. 모든 변화가 한 곳을 지나므로 전역 일관성·추적성이 강함.
- **B. Intent별 분산 Actor**
  Intent마다 IntentAgent(또는 Meta Agent 인스턴스)를 띄워 자기 상태를 소유. Orchestrator는 라우터·게이트키퍼만 함. IDS의 관계 라벨은 해당 IntentAgent로 메시지로 전달. 격리성·동시성은 강하지만 전역 일관성·관측성은 별도 메커니즘이 필요.

차이의 본질: **상태 소유권**과 **실행 제어권**의 위치.

| 상위 QA | A (중앙집중) | B (분산 Actor) |
|---|---|---|
| 신뢰성 | + | − |
| 성능 효율성 | − | + |
| 유지보수성 | + | − |
| 보안성 |  |  |
| 호환성 |  |  |
| 기능 적합성 |  |  |

핵심 갈림: **신뢰성 ↔ 성능 효율성** (유지보수성은 A 우세 보조).

---

### Pair 2. Event Sourcing 기반 회복 ↔ Saga + Checkpoint 기반 회복

- **A. Event Sourcing**
  모든 도메인 사건(Intent 도착, 라벨 적용, Task 시작/완료/실패, 외부 효과 발생)을 append-only 로그로 남김. 실패 시 로그 재생으로 상태 재구성. 어떤 시점의 상태든 복원·디버그 가능. 저장 공간·이벤트 스키마 진화 부담이 큼.
- **B. Checkpoint + Saga**
  Task 경계마다 최소 상태를 체크포인트로 저장하고, 외부 효과는 Saga 보상 트랜잭션으로 되돌림. 단순·경량. 과거 어느 시점이 어땠는지 재현·감사 능력은 떨어짐.

차이의 본질: **복구 모델** — "전부 다시 만든다" vs "마지막 안전점에서 이어 간다".

| 상위 QA | A (Event Sourcing) | B (Saga+Checkpoint) |
|---|---|---|
| 유지보수성 | + | − |
| 성능 효율성 | − | + |
| 신뢰성 | + |  |
| 호환성 | − |  |
| 보안성 |  |  |
| 기능 적합성 |  |  |

핵심 갈림: **유지보수성 ↔ 성능 효율성**.
(본 프로젝트 06-QAS 우선순위 QA-001 협상 완료 시간 = 성능 효율성, QA-003 배터리 = 성능 효율성, NFR-003/004 자원 = 성능 효율성과 직접 충돌함. 한편 QA-007 추적성, NFR-007 시뮬레이션 가능 여부 = 유지보수성에 직접 기여.)

---

### Pair 3. Eager Re-plan ↔ Lazy Checkpoint Re-plan

- **A. 즉시 재계획 (Eager / Preemptive)**
  IDS의 관계 라벨이 도착하면 즉시 진행 중 Task에 인터럽트를 걸고 plan을 재구성. 사용자가 체감하는 반응성은 가장 빠름. 인터럽트 시점이 외부 부수효과 중간일 수 있어 보상·정합성 보장 비용이 큼.
- **B. 체크포인트 기반 지연 적용 (Lazy / Cooperative)**
  관계 라벨을 큐에 적재해 두고, 다음 Task 경계 같은 안전한 체크포인트에서만 plan에 반영. 일관성·예측 가능성은 매우 높지만, "취소했는데도 잠깐 더 진행되는" 사용자 체감 지연이 발생.

차이의 본질: **변경 적용 타이밍** (선점 vs 협조). 동시성과 정합성 사이의 균형 위치.

| 상위 QA | A (Eager) | B (Lazy) |
|---|---|---|
| 성능 효율성 | + | − |
| 신뢰성 | − | + |
| 기능 적합성 | − | + |
| 유지보수성 |  |  |
| 보안성 |  |  |
| 호환성 |  |  |

핵심 갈림: **성능 효율성 ↔ 신뢰성** (기능 적합성도 동일 방향으로 B 우세).

---

### Pair 4. 명시적 Workflow / State Machine ↔ LLM Planner 재호출

- **A. 명시적 Workflow / State Machine**
  활성 Intent들의 plan을 DAG·State Machine으로 정형화. 라벨 적용·실패 복구·병렬 후보 판단을 모두 그래프 변형 규칙으로 정의. 규칙 기반이라 결정성·검증 가능성·재현성이 강함. 새 Intent 유형·도메인 추가 시 규칙 확장 비용이 큼.
- **B. LLM Planner Re-invocation**
  Intent 도착·라벨 변경·Task 실패가 발생할 때마다, 현재 활성 Intent 집합과 부분 진행 상태를 컨텍스트로 LLM(Meta Agent)에 다시 plan을 요청. 새로운 상황·도메인에 대한 적응력은 높지만, 매 결정이 비결정적이고 추적·검증이 어려움. 온디바이스 LLM 호출 비용·지연도 변수.

차이의 본질: **결정 주체**가 코드/규칙인가, LLM 추론인가 (정형 vs 적응형).

| 상위 QA | A (Workflow) | B (LLM Planner) |
|---|---|---|
| 신뢰성 | + | − |
| 기능 적합성 | + | − |
| 유지보수성 | ± (테스트성 +, 변경 수용성 −) | ± (변경 수용성 +, 테스트성 −) |
| 성능 효율성 | + | − |
| 보안성 |  |  |
| 호환성 |  |  |

핵심 갈림: **신뢰성·기능 적합성 + 유지보수성(테스트성) ↔ 유지보수성(변경 수용성)**.
같은 "유지보수성" 안에서 방향이 갈리는 페어이므로, 채택 시 본문에서 양면을 분리해 서술할 필요가 있음.

## 3. 페어별 핵심 갈림 종합

| 페어 | 가장 두드러진 trade-off 축 | 대표 정량 QA 충돌 (06-QAS) |
|---|---|---|
| 1 (중앙 vs 분산) | 신뢰성 ↔ 성능 효율성 | QA-002 ↔ QA-001 |
| 2 (Event Sourcing vs Saga) | 유지보수성 ↔ 성능 효율성 | QA-007·NFR-007 ↔ QA-001·QA-003·NFR-003/004 |
| 3 (Eager vs Lazy) | 성능 효율성 ↔ 신뢰성 | QA-001 ↔ QA-002 |
| 4 (Workflow vs LLM Planner) | 신뢰성·기능 적합성 ↔ 유지보수성(변경 수용) | QA-002 ↔ QA-004·QA-006 |

> Pair 1·3은 상위 카테고리 수준에서 같은 축(신뢰성 ↔ 성능 효율성)에 놓이므로, 본문에서 차별성을 풀어내려면 "상태 소유권 / 변경 적용 타이밍"이라는 하위 메커니즘 차이를 서술로 드러내야 함.
> Pair 2·4는 trade-off 축 자체가 다른 카테고리로 분리되므로, "다른 QA를 희생하는 두 후보"라는 DP의 전형 그림을 만들기 가장 쉬움.

## 4. 다음 결정 포인트 (이어 작업할 LLM/작업자에게)

이 단계에서 사용자(혹은 다음 작업자)가 결정해야 할 것은 **어떤 페어를 정식 후보로 채택할지**임. 현재까지 도출된 후보 채택 시나리오는 다음과 같음.

1. **Pair 2 (Event Sourcing ↔ Saga + Checkpoint) 채택**
   - 장점: 06-QAS 우선순위와 가장 깔끔하게 충돌함 (유지보수성 ↔ 성능 효율성).
   - 다음 단계: 각 후보의 상태 모델·동시 실행 방식·라벨 처리 방식·실패 회복·외부 효과 처리·책임 분담(Orchestrator/Meta Agent/Sub-Agent/Tool/저장소)을 상세화. QA trade-off 종합 표 작성 (상위 카테고리만 사용).

2. **Pair 1 (중앙집중 ↔ 분산 Actor) 채택**
   - 장점: 아키텍처 원리(상태 소유권·실행 제어권) 차이가 가장 큼.
   - 다음 단계: 위와 동일.

3. **Pair 4 (Workflow ↔ LLM Planner) 채택**
   - 장점: 결정 주체(규칙 vs LLM)의 차이가 DP11 외 다른 DP(예: DP07 협상 워크플로 복구)와도 연결되는 큰 결정.
   - 주의: 유지보수성이 양면에 등장하므로, 표 안에서 양면을 분리하기 위해 본문 서술 강화 필요.

4. **현 4개 외 새 페어 추가 발굴**
   - 예) 자원 점유 모델(자원 사전 예약 vs 사후 회수), 디바이스 vs 서버 LLM 분담 모델 등.

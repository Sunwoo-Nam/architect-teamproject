### **DP 07. 비동기 턴제 협상 중 네트워크 단절 시의 상태 불일치 해소 및 복구 방법**

#### **1. 풀고자 하는 문제**
온디바이스 환경 특성상 제한된 스레드 자원과 이로 인한 시스템 강제 종료를 방지하기 위해, 에이전트 간의 통신은 비동기(Async) 방식을 가정합니다.
이러한 비동기 환경을 전제로, A와 B의 에이전트가 Rule/Threshold 기반의 턴제 협상(Turn-based Negotiation)을 주고받고 있습니다. 이때 네트워크 단절로 인해 특정 턴의 메시지가 유실되었을 때, 인간의 개입 없이 양쪽 에이전트의 협상 상태(현재 몇 번째 턴인지, 협상의 전체 제한 시간인 데드라인은 얼마나 남았는지 등)를 일치시키고 협상을 이어가거나 종료해야 합니다.

#### **2. 아키텍처적 난제**
 * **협상 상태의 어긋남 (State Mismatch):** 비동기 통신 특성상, A가 제안을 발송하고 B가 이를 수락(Accept)하여 응답을 보냈으나 이 응답 패킷이 네트워크 구간에서 유실되는 비대칭적 단절(Asymmetric Drop) 상황이 발생할 수 있습니다. 이 경우 B는 수락 후 다음 단계를 준비하지만, A는 응답 대기 시간 초과로 협상을 실패로 간주하는 상태 불일치가 발생합니다.
 * **재시도 로직의 부작용:** 네트워크 오류를 극복하기 위해 A가 메시지를 단순히 재전송할 경우, B의 에이전트가 이를 이전 턴의 재전송이 아닌 완전히 새로운 턴의 중복 제안으로 오인하여 협상 턴 카운트가 꼬이거나 불필요한 연산을 수행하는 논리적 결함이 발생할 수 있습니다.

#### **3. 제안 방안**

**3.1 제안 방안 1: 백오프 기반 메시지 재전송 및 수신부 중복 필터링**
A는 제안 발송 후 타임아웃이 발생하면 동일한 Turn_ID를 유지한 채 메시지를 재발송합니다. 이때 상대방이 다중 협상을 처리하느라 응답이 늦어지는 상황을 고려하여, 정적 시간이 아닌 지수적 백오프를 적용해 재시도 간격을 점진적으로 늘립니다.
 * **핵심 메커니즘:** B의 단말 앞단에는 중복 제거 로직이 배치됩니다. B가 이미 처리 완료한 Turn_ID의 메시지가 다시 유입될 경우, 이를 협상 에이전트로 넘기지 않고 이전에 반환했던 똑같은 결과 패킷을 그대로 다시 A에게 전송합니다.
 * **장점:** 메시지 유실 시에도 협상의 원래 턴을 그대로 유지하며 끝까지 완수할 수 있습니다.
 * **단점:** B의 아키텍처에 중복 상태를 관리하고 과거 응답을 캐싱하는 추가적인 저장 계층이 강제됩니다.

**3.2 제안 방안 2: 타임아웃 시 최종 상태 교환 및 덮어쓰기**
메시지 단위의 재시도를 수행하지 않습니다. 대신, 특정 턴에서 타임아웃이 발생하면 A와 B는 현재 자신이 들고 있는 마지막 상태 정보(현재 턴 수, 결과)를 서로 교환(Sync)하는 상태 확인 프로토콜을 가동합니다.
 * **핵심 메커니즘:** 상태 불일치가 확인되면, 더 많이 진행된 턴(Highest Turn ID)을 동기화 값으로 삼습니다. 예를 들어 A가 3번째 턴에 머물러 있는데 B의 상태가 4번째 턴 수락(Accept)이라면, A는 유실된 메시지 복구를 포기하고 자신의 로컬 상태를 B와 동일하게 4번째 턴 수락으로 동기화(Overwrite)합니다.
 * **장점:** 여러 개의 메시지가 연속으로 유실되더라도, 단 한 번의 상태 교환 패킷으로 양측의 어긋난 상태를 수학적으로 일치시킬 수 있어 트래픽 효율이 좋습니다.
 * **단점:** 협상 프레임워크(Negmas)의 내부 상태(State)를 외부에서 강제로 주입하고 덮어씌우는 코드를 프레임워크 레벨에 구현해야 합니다.

**3.3 제안 방안 3: 즉시 파기 및 재시작**
Rule/Threshold 기반 협상 세션의 생성 비용이 극도로 가볍다는 도메인 특성을 활용하여, 네트워크 단절 및 타임아웃 발생 시 꼬인 상태를 복구하려는 모든 시도를 포기합니다.
 * **핵심 메커니즘:** 타임아웃 발생 시, A는 해당 세션(Session_ID)을 에러(Abort) 처리하고 폐기합니다. 직후 완전히 새로운 Session_ID를 발급하여 B에게 "초기 상태부터 다시 협상 시작"을 요청하는 메시지를 전송합니다. B 역시 타임아웃이 발생하거나 파기된 Session_ID에 대한 메시지를 받으면 해당 세션을 메모리에서 삭제합니다.
 * **장점:** 재시도 필터링, 상태 동기화 등의 복잡한 예외 처리 코드가 모두 사라지므로 코어 프레임워크의 개발 복잡도가 극단적으로 낮아지며, 논리적 버그 발생 확률이 0%에 수렴합니다.
 * **단점:** 90% 이상 진행된 협상이라도 네트워크가 흔들리면 처음부터 다시 시작해야 하므로, 불안정한 네트워크 환경에서는 전체 합의 도출 시간이 지연될 수 있습니다.

#### **4. 종합 평가 (Trade-off)**

| 비교 항목 | 3.1 제안 방안 1 (재시도 및 중복 방어) | 3.2 제안 방안 2 (최종 상태 동기화) | 3.3 제안 방안 3 (Fail-Fast & Restart) |
| :--- | :--- | :--- | :--- |
| **복구 철학** | 과정(메시지)을 보장하여 턴의 흐름 유지 | 결과를 교환해 도달한 최종 상태를 일치시킴 | 상태 복구를 포기하고 트랜잭션을 초기화 |
| **구현 난이도** | **보통** (네트워크 모듈에 중복 체크 추가) | **어려움** (프레임워크 상태 강제 수정) | **매우 쉬움** (단순 세션 파기 로직만 필요) |
| **네트워크 부하** | **높음** (지연 시 재시도 패킷 누적) | **낮음** (타임아웃 시 1회 교환) | **보통** (단절 시 처음부터 턴을 다시 진행) |
| **장애 대처 속도** | **빠름** (백오프 주기에 따라 즉각 재시도) | **보통** (상태 확인 및 덮어쓰기 로직 수행) | **가장 빠름** (오류 즉시 세션 재시작) |
| **시스템 무결성** | 중복 필터링 버그 시 상태 오염 리스크 | 상태 덮어쓰기 중 충돌 리스크 | **완벽한 상태 격리** (새로운 세션이므로 꼬임 없음) |
